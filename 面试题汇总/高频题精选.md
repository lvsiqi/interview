## 一、☕ Java 核心

### JVM

**Q1: JVM 内存区域有哪些？**
> 线程私有：程序计数器、虚拟机栈（栈帧=局部变量表+操作数栈+动态链接+返回地址）、本地方法栈。线程共享：堆（对象实例）、方法区/元空间（类信息、常量池）。
> ⚡追问：元空间和永久代的区别？（元空间在本地内存，不受JVM堆限制，不会再出现 PermGen OOM）

**Q2: 类加载双亲委派机制？**
> 加载类时优先委托父类加载器，父类无法加载才自己加载。顺序：BootstrapClassLoader → ExtClassLoader → AppClassLoader。**好处**：防止核心类被覆盖（如自定义 java.lang.String 无效）。
> ⚡追问：哪些场景破坏了双亲委派？（SPI/JNDI用线程上下文类加载器；Tomcat自定义WebAppClassLoader；OSGI模块化）

**Q3: G1 和 CMS 的区别？**
> CMS：老年代并发标记清除，产生内存碎片，Stop-The-World时间短但不可预期。G1：将堆划分为等大小 Region，优先回收垃圾最多的 Region，可设置停顿目标（`-XX:MaxGCPauseMillis`），几乎不产生碎片。JDK9+ 默认 G1。
> ⚡追问：ZGC 的核心优势？（停顿 < 10ms，使用染色指针+读屏障实现并发压缩）

**Q4: 什么情况会触发 Full GC？**
> ① 老年代空间不足；② 方法区空间不足；③ 显式调用 System.gc()；④ Minor GC 后晋升老年代的对象大于老年代剩余空间；⑤ 大对象直接进入老年代且空间不足。
> ⚡追问：如何避免频繁Full GC？（调大老年代/元空间；优化对象生命周期；避免大对象；升级G1/ZGC）

### 并发编程

**Q5: synchronized 底层原理？锁升级过程？**
> 对象头的 Mark Word 存储锁状态。升级路径：**无锁 → 偏向锁**（CAS写线程ID，无竞争时零成本）→ **轻量级锁**（CAS自旋，适合短暂竞争）→ **重量级锁**（OS Mutex，线程阻塞挂起）。锁只能升级不能降级（偏向锁撤销除外）。
> ⚡追问：JDK15 废弃偏向锁的原因？（现代多核场景竞争普遍，偏向锁撤销开销反而更大）

**Q6: volatile 能保证原子性吗？**
> 不能。volatile 保证**可见性**（写后立即刷入主存，读时从主存取）和**有序性**（禁止指令重排，通过内存屏障实现），但 `i++` 这类复合操作仍不是原子的，需要用 `AtomicInteger` 或 `synchronized`。
> ⚡追问：双重检查锁单例为什么需要volatile？（防止对象半初始化：new的步骤是分配内存→初始化→赋引用，重排后其他线程可能拿到未初始化的对象）

**Q7: AQS 的核心原理？**
> AbstractQueuedSynchronizer 内部维护一个 **volatile int state**（锁状态）和一个 **CLH 变体双向链表**（等待队列）。加锁=CAS修改state；失败则封装成Node入队，自旋检查前驱节点是否为head，是则再次CAS抢锁，否则 park 挂起。释放锁时 unpark 后继节点。ReentrantLock/Semaphore/CountDownLatch 都基于 AQS。
> ⚡追问：公平锁和非公平锁的区别？（公平锁每次先检查队列是否有等待者；非公平锁直接CAS抢，吞吐量更高但可能饿死）

**Q8: 线程池核心参数和工作流程？**
> 7个参数：corePoolSize（常驻核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（非核心线程空闲存活时间）、unit、workQueue（任务队列）、threadFactory、handler（拒绝策略）。
> 流程：任务来→核心线程未满先建核心线程→核心线程满则入队→队满则建非核心线程→达到max则触发拒绝策略。
> ⚡追问：线上如何设置线程数？（CPU密集型：核数+1；IO密集型：核数 × (1 + IO等待时间/CPU时间)，通常 2N 或更大）

**Q9: ConcurrentHashMap 如何保证线程安全？**
> JDK8：数组+链表+红黑树，桶粒度 synchronized（只锁单个桶头节点）+ CAS（空桶直接CAS插入）。扩容时多线程协同迁移（transfer），读操作无锁（volatile 数组 + Node.val）。JDK7 是 Segment 分段锁，粒度更粗。
> ⚡追问：为什么 size() 不精确？（用 baseCount + CounterCell 数组分散计数，最终汇总，高并发下是近似值）

### 集合框架

**Q10: HashMap 扩容机制？**
> 初始容量16，负载因子0.75，阈值=容量×负载因子。超过阈值触发 resize，新容量=旧容量×2。迁移元素时利用 `(hash & oldCap) == 0` 判断在原位置还是`原位置+oldCap`，避免重新计算哈希。链表长度≥8且数组长度≥64时转红黑树；红黑树节点≤6时退化回链表。
> ⚡追问：为什么容量必须是2的幂？（`hash & (n-1)` 等价于 `hash % n`，位运算更快，且扩容迁移逻辑更简洁）

## 二、🗄️ 数据库

### MySQL

**Q11: B+ 树和 B 树的区别？为什么用 B+ 树？**
> B树：非叶节点也存数据，叶节点无链表。B+树：非叶节点只存索引（key），所有数据在叶节点，叶节点用双向链表串联。**B+树优势**：① 非叶节点更小，单页存更多key，树更矮，IO次数更少；② 叶节点链表支持范围查询（ORDER BY / BETWEEN）；③ 查询路径稳定（必到叶节点），性能一致。
> ⚡追问：为什么 MySQL 用 B+ 树而不用哈希索引？（哈希不支持范围查询和排序；B+树支持前缀匹配）

**Q12: MVCC 原理？**
> 每行记录有两个隐藏字段：**trx_id**（最后修改的事务ID）和 **roll_pointer**（指向 undo log 的指针，构成版本链）。读操作创建 **ReadView**（快照），包含活跃事务列表。判断可见性：trx_id < min_trx → 可见；trx_id > max_trx → 不可见；在区间内 → 检查是否在活跃列表，不在则可见。RC 每次查询创建新 ReadView；RR 只在事务开始时创建一次。
> ⚡追问：RR 级别下 MVCC 能否完全避免幻读？（快照读靠MVCC可以，但当前读（SELECT FOR UPDATE）仍需 Gap Lock 配合）

**Q13: 索引失效的常见场景？**
> ① 左模糊查询（`LIKE '%abc'`）；② 对索引列做函数或运算（`WHERE YEAR(date) = 2024`）；③ 隐式类型转换（字符串列用数字比较）；④ OR 连接非全索引列；⑤ 联合索引不满足最左前缀；⑥ `!=` / `NOT IN`（通常走全表扫描，取决于选择率）。
> ⚡追问：覆盖索引是什么？（查询的所有字段都在索引中，无需回表，EXPLAIN的Extra列显示 Using index）

**Q14: 说说 redo log、undo log、binlog 的区别？**
> **redo log**：InnoDB 特有，物理日志（记录页的修改），保证崩溃恢复（持久性）；Write-Ahead Log，先写日志再写磁盘。**undo log**：逻辑日志（记录反向操作），保证事务回滚和 MVCC 多版本读。**binlog**：Server 层，逻辑日志（SQL/行变更），用于主从复制和数据恢复；不限于InnoDB。**两阶段提交**：redo log prepare → binlog写入 → redo log commit，保证两者一致性。
> ⚡追问：为什么需要两阶段提交？（redo log 和 binlog 是两个独立文件，若只写一个后崩溃会导致主从数据不一致）

### Redis

**Q15: Redis 为什么快？**
> ① **纯内存操作**，无磁盘IO（持久化是异步的）；② **单线程命令处理**，无锁竞争（6.0+ 网络IO多线程，命令执行仍单线程）；③ **IO多路复用**（epoll），单线程处理大量并发连接；④ **高效数据结构**（SDS、跳表、压缩列表、整数集合），内存紧凑操作快。
> ⚡追问：Redis 6.0 的多线程是什么？（网络读写IO多线程化，命令解析和执行仍是单线程，避免并发问题）

**Q16: Redis 持久化 RDB vs AOF？**
> **RDB**：fork子进程快照，适合备份，恢复快，但可能丢失数分钟数据。**AOF**：记录每条写命令，`appendfsync always/everysec/no`，数据更完整但文件更大、重启恢复慢，支持 BGREWRITEAOF 压缩。**混合持久化**（推荐）：AOF 重写时先写 RDB 快照，再追加增量 AOF，兼顾速度和完整性。
> ⚡追问：AOF 重写期间新的写命令怎么处理？（写入 AOF 缓冲区，重写完成后追加到新文件末尾，原子替换）

**Q17: 缓存穿透、击穿、雪崩的区别和解决方案？**
> **穿透**：查询不存在的key，每次打到DB。解决：布隆过滤器（拦截一定不存在的key）或缓存空值（TTL短）。**击穿**：热点key过期瞬间大量请求直接打DB。解决：互斥锁（只允许一个请求重建缓存）或逻辑过期（缓存不设TTL，后台异步刷新）。**雪崩**：大量key同时过期或Redis宕机。解决：TTL加随机抖动；多级缓存；Redis 高可用集群；熔断降级。
> ⚡追问：如何用 Redisson 实现互斥防击穿？（tryLock + double check：获取锁后再查缓存，若已有则直接返回，避免重复重建）

## 三、📨 消息队列

**Q18: Kafka 为什么吞吐量高？**
> ① **顺序写磁盘**（append-only，磁盘顺序写性能接近内存）；② **PageCache**（数据先写OS页缓存，批量flush，减少随机IO）；③ **零拷贝**（sendfile系统调用，数据从PageCache直接到网卡，不经用户态）；④ **批量压缩**（Producer端批量发送，支持gzip/snappy/lz4）；⑤ **分区并行**（多分区多消费者并行消费）。
> ⚡追问：Kafka 消息不丢失如何保证？（Producer：acks=all + 重试；Broker：min.insync.replicas≥2；Consumer：手动提交offset，业务处理完再提交）

**Q19: Kafka 如何保证消息顺序消费？**
> Kafka 保证**分区内有序**，跨分区不保证。实现顺序：① 将相同业务key（如订单ID）发到同一分区（`key hash % partitionNum`）；② 一个分区只分配一个消费者线程；③ Producer 设置 `max.in.flight.requests.per.connection=1`（防止重试乱序，或开启幂等生产者自动保证）。
> ⚡追问：消费者重平衡时如何保证不乱序？（rebalance 前 commit 已处理的 offset，rebalance 后从 committed offset 重新消费，配合幂等处理重复）

## 四、⚡ 分布式

**Q20: CAP 理论？分布式系统如何取舍？**
> 分布式系统无法同时满足**一致性（C）、可用性（A）、分区容忍性（P）**三者。网络分区（P）在分布式系统中必然存在，所以实际选择是 **CP 或 AP**。CP：ZooKeeper、etcd（分区时牺牲可用性保强一致）。AP：Nacos服务发现、Eureka（分区时返回旧数据保可用性）。绝大多数业务系统选 AP + 最终一致性。
> ⚡追问：BASE理论是什么？（Basically Available 基本可用、Soft state 软状态、Eventually consistent 最终一致性，是AP系统的工程实践指导）

**Q21: 分布式事务如何选型？**
> **2PC**：强一致，有阻塞和单点问题，适合内部数据库（XA协议）。**TCC**：业务侵入性强，需实现try/confirm/cancel，适合响应快、一致性要求高的场景（如支付）。**Saga**：长事务，每步本地提交，失败触发补偿，适合步骤多的业务流程。**消息最终一致性**：性能最好，适合允许短暂不一致（如下单→扣库存异步）。**Seata AT**：自动补偿，侵入性小，适合同机房低延迟场景。
> ⚡追问：TCC 的空回滚、幂等、悬挂如何解决？（维护事务状态表：try前插入记录防悬挂，cancel时检查是否有try记录防空回滚，每个操作前后幂等校验）

**Q22: Raft 选举流程？**
> 每个节点有三种角色：Leader/Follower/Candidate。Follower 在 election timeout（随机150~300ms）内未收到心跳则变为 Candidate，自增 term，投票给自己，广播 RequestVote。获得**过半选票**则成为Leader，开始发送心跳（AppendEntries）。选举规则：每个节点每个term只投一票；日志更新（term更大或同term下index更大）的候选人才能获票。
> ⚡追问：Raft 和 ZAB 的主要区别？（Raft 用 term+index 标识日志，支持动态成员变更；ZAB 用 epoch+ZXID，是 ZooKeeper 专用，两者思想相似）

## 五、🏗️ 系统设计

**Q23: 秒杀系统核心设计要点？**
> ① **漏斗模型逐层过滤**：浏览器/CDN→Nginx限流→服务层预扣→Redis原子扣减；② **Redis Lua 脚本**原子扣减库存，防超卖；③ **异步下单**：扣减成功后发MQ，消费者异步创建订单；④ **库存分桶**：热点库存拆分为多个key（如`stock:1~10`），减少单key并发压力；⑤ **接口限流+熔断**：Sentinel/令牌桶；⑥ **防重复提交**：Token机制或userId+itemId唯一索引。
> ⚡追问：Redis扣减成功但MQ发送失败怎么办？（本地事务表：扣减成功后写DB本地消息表，独立定时任务轮询补发，保证最终一致）

**Q24: 如何设计一个高可用的分布式锁？**
> **Redis 单机锁**：SET key value NX EX ttl + Lua 脚本删除（保证原子性），用 UUID 作 value 防误删。**Redisson** 封装了看门狗续期（1/3 TTL 续期）和可重入。**RedLock**：向 N 个独立 Redis 节点申请，过半成功且耗时 < TTL 才算获锁，但存在时钟漂移争议，Martin Kleppmann 等人认为不够安全，生产大多用单主 Redis+哨兵。**ZK 锁**：临时顺序节点+监听前驱，天然公平无死锁，适合公平性要求高的场景。
> ⚡追问：锁持有方宕机，没来得及释放锁怎么办？（Redis锁靠TTL自动过期；ZK锁靠临时节点随Session过期自动删除）

## 六、🌐 网络

**Q25: TCP 三次握手为什么不能是两次？**
> 两次握手无法确认**客户端的接收能力**，且可能造成"已失效连接请求报文"问题：客户端早期的请求因延迟到达服务端，服务端误认为是新连接并分配资源等待，但客户端早已放弃，造成服务端资源浪费。三次握手双方各确认一次对方的发送和接收能力。
> ⚡追问：四次挥手为什么需要 TIME_WAIT 等待 2MSL？（等待确保最后一个ACK被对方收到（如丢失则重传FIN），也让网络中残留的重复数据包自然消亡，防止被新连接误接收）

**Q26: epoll 为什么比 select/poll 高效？**
> **select/poll**：每次调用需把所有 fd 从用户态拷贝到内核态；返回后需遍历所有 fd 找就绪事件，O(n) 复杂度；select 有 fd 数量限制（1024）。**epoll**：fd 注册到内核红黑树（一次性），就绪 fd 用回调函数加入就绪链表，`epoll_wait` 只返回就绪的 fd，O(1)；适合连接数多但活跃连接少的场景（如 Nginx、Redis）。
> ⚡追问：ET（边缘触发）和 LT（水平触发）区别？（LT：只要fd可读就一直通知，可以分多次读；ET：只在状态变化时通知一次，必须一次性读完/写完，效率更高但编程复杂）

## 七、💻 算法高频

**Q27: 快速排序核心思路和复杂度？**
> 选 pivot，partition 分成左（<pivot）右（>pivot）两部分，递归处理。平均 O(n log n)，最坏（有序数组选固定pivot）O(n²)。优化：**随机 pivot** 规避最坏情况；**三路快排**（< = >三分）处理大量重复元素。空间复杂度：递归栈 O(log n)。
> ⚡追问：快排和归并排序哪个更快？（快排常数小，实践中更快，但不稳定；归并稳定，常用于外排序和链表排序）

**Q28: 动态规划解题框架？**
> ① 确定 **状态**（dp含义）；② 确定 **转移方程**（如何从小问题推大问题）；③ 确定 **初始值** 和 **遍历顺序**；④ 考虑**空间压缩**（滚动数组，背包问题逆序遍历）。常见：背包（`dp[j] = max(dp[j], dp[j-w]+v)`）、最长公共子序列（`dp[i][j]`）、编辑距离（`min(dp[i-1][j]+1, dp[i][j-1]+1, dp[i-1][j-1]+replace)`）。
> ⚡追问：什么情况用贪心而不用DP？（贪心局部最优能推全局最优时，如区间调度、霍夫曼编码；DP适合局部最优不等于全局最优的场景）

## 八、🔍 场景题

**Q29: 线上 CPU 突然飙到 100%，如何排查？**
> ① `top` 找到 CPU 高的 Java 进程 PID；② `top -Hp <PID>` 找到 CPU 高的线程 TID；③ `printf '%x' <TID>` 转十六进制；④ `jstack <PID> | grep -A 30 <十六进制TID>` 查看线程栈。常见原因：**死循环**（while true无 sleep）、**正则回溯**（复杂正则+特殊输入）、**频繁 Full GC**（内存泄漏导致 GC 线程占满 CPU）。
> ⚡追问：Arthas 如何快速定位热方法？（`thread -n 3` 找最忙线程；`profiler start` + `profiler stop --format html` 火焰图分析）

**Q30: 接口响应时间突然变长，如何排查？**
> 按分层逐步排查：① **网络层**：ping/mtr 检查网络延迟；② **应用层**：Arthas `trace com.xx.Service method` 查各方法耗时，定位瓶颈；③ **DB层**：慢查询日志 + EXPLAIN 分析执行计划，检查是否索引失效；④ **缓存层**：Redis `slowlog get` 检查大key/热key；⑤ **线程池**：查线程池队列积压（`jstack`/监控）；⑥ **GC**：`jstat -gcutil <PID> 1000` 看 Full GC 频率。
> ⚡追问：如何建立接口性能基线？（正常时采集 P99/P999 延迟 + QPS，设置告警阈值；集成 Prometheus + Grafana 面板）

## 九、🎤 HR 软技能

**Q31: 你最大的缺点是什么？**
> 策略：选一个**真实但可控**的缺点，说明已在改进。例：「我在做技术方案决策时，倾向于过度调研细节，有时会影响交付速度。我后来给自己设了 timeboxing：技术调研不超过 X 小时，超时就带着已有信息做决策，效果明显改善。」避免：说「我太努力/太完美主义」（无诚意）；说「我不擅长沟通」（核心软技能硬伤）。

**Q32: 你为什么要离职/换工作？**
> 策略：**正面动机** > 负面评价，向往新机会而非逃离旧环境。框架：「现在的工作让我在 XX 方面有了很好的积累，但我希望在 YY 方向更深入发展（如分布式系统/更大规模业务），贵公司在 ZZ 领域的技术深度和业务规模让我非常感兴趣。」避免：抱怨领导、团队氛围、薪资（即使是真实原因）。
