## 一、HashMap 源码 ⭐⭐⭐⭐

### 1.1 底层结构
- JDK7：数组 + 链表
- JDK8：数组 + 链表 + 红黑树

### 1.2 核心参数
| 参数 | 值 | 含义 |
|------|-----|------|
| 初始容量 | 16 | 必顿是2的幂 |
| 负载因子 | 0.75 | 扩容阈値=容量×0.75 |
| 树化阈値 | 8 | 链表长度≥8且数组长度≥64时转红黑树 |
| 退化阈値 | 6 | 红黑树退化回链表 |

### 1.3 put 流程
```
put(k,v) → 计算hash（hashCode ^ hashCode>>>16）
  → index = (n-1) & hash
  → 桶为空？直接放入
  → key相同？覆盖value
  → 红黑树？调用树插入
  → 链表尾插，长度≥8且数组长度≥64→转红黑树
  → size > threshold → resize()扩容
```

### 1.4 hash 计算为什么要 ^ (h>>>16)
- 将高 16 位信息混入低 16 位
- 计算桶位置时只用低位，混入高位信息可**减少哈希碰撞**

### 1.5 扩容机制
- 触发：size > capacity × 0.75
- 新容量 = 旧容量 × 2
- 节点新位置：`hash & oldCap == 0` → 位置不变；否则 → +oldCap
- 无需重新计算hash，只判断一个bit，效率高

### 1.6 JDK7 死循环问题
- **根本原因**：JDK7扩容用**头插法**，多线程并发扩容导致链表成环，get时CPU100%
- **JDK8修复**：改用**尾插法**，保持链表顺序，不会成环
- ⚠️ JDK8 HashMap 仍线程不安全！并发用 ConcurrentHashMap

### 1.7 容量必顿是2的幂的原因
1. `(n-1) & hash` 等价于 `hash % n`，位运算更快
2. 扩容迁移时只需看一个bit，无需重计算hash

### 1.8 面试标准答法
> JDK8 HashMap底层数组+链表+红黑树，默认容量16负载因子0.75。
> put先计算hash（高低16位异或减少碰撞），尾插链表，超阈値扩容一倍。
> 扩容时节点位置只有两种，通过hash & oldCap判断，无需重计算hash。
> JDK7头插法导致并发扩容成环；JDK8改尾插法，但仍非线程安全。

### 1.9 常见追问
| 追问 | 关键答点 |
|------|----------|
| 容量为什么2的幂？ | 位运算代替取模+扩容时只看onebit |
| 负载因子为什么0.75？ | 空间与时间平衡，泊松分布下碰撞概率可接受临界値 |
| 树化阈値为什么8？ | 泊松分布超8概率极低，<8时链表性能可接受 |
| HashMap key可以是null吗？ | 可以，存index 0，hashCode为0 |

## 二、ArrayList vs LinkedList ⭐⭐

### 2.1 底层结构
- **ArrayList**：Object 数组，连续内存，默认容量10，扩容 × 1.5
- **LinkedList**：双向链表，每个节点含 prev/next 指针，同时实现 Deque

### 2.2 核心对比
| 操作 | ArrayList | LinkedList |
|------|-----------|------------|
| 随机访问 get(i) | **O(1)** | O(N) |
| 头部插入/删除 | O(N)（移位） | **O(1)** |
| 尾部插入 | O(1)（均摊） | O(1) |
| 中间插入/删除 | O(N) | O(N)找位置+O(1)改指针 |
| 内存 | 连续，CPU缓存友好 | 不连续，有额外指针开销 |

> ⚠️ 绝大多数场景用 ArrayList，内存连续缓存命中率高，实际性能优于 LinkedList

### 2.3 ArrayList 注意点
- 遍历时删除用 `iterator.remove()` 或 `removeIf()`，直接 remove 会抛 ConcurrentModificationException（fast-fail机制）
- 初始化时预估容量 `new ArrayList<>(1000)`，避免频繁扩容
- 扩容1.5倍：减少内存浪费（比2倍小）+ 减少扩容次数（比1倍大）

### 2.4 面试标准答法
> ArrayList数组结构随机访问O(1)，扩容1.5倍；LinkedList双向链表随机访问O(N)。
> 实际绝大多数用ArrayList，内存连续CPU缓存友好，LinkedList额外指针开销且缓存不友好。
> for-each删除报错是因为fast-fail检测到modCount变化，要用iterator.remove()。

## 三、CopyOnWriteArrayList 原理 ⭐⭐⭐

### 3.1 核心思想：写时复制
- 底层数组用 **volatile** 修饰保证可见性
- **写操作**：加锁 → 复制新数组 → 在新数组修改 → 替换引用 → 释放锁
- **读操作**：直接读，**完全不加锁**

### 3.2 优缺点
| | 说明 |
|--|------|
| ✅ 读不加锁 | 读性能极高 |
| ✅ 迭代安全 | 迭代的是快照，不会ConcurrentModificationException |
| ❌ 写开销大 | 每次写都复制整个数组 |
| ❌ 弱一致性 | 读可能读到旧快照数据 |
| ❌ 不适合大数组 | 复制成本与数组大小成正比 |

### 3.3 适用场景
- **读多写少**：白名单、黑名单、配置列表、监听器列表

### 3.4 线程安全方案对比
| 方案 | 原理 | 适用场景 |
|------|------|---------|
| Vector | 全方法synchronized | 基本淘汰 |
| synchronizedList | 包装器加锁 | 简单场景 |
| **CopyOnWriteArrayList** | 写时复制，读不加锁 | **读多写少** |
| ConcurrentLinkedQueue | CAS无锁 | 高并发队列 |

### 3.5 面试标准答法
> CopyOnWriteArrayList写时复制：写操作加锁复制数组在副本上改，volatile替换引用；读不加锁直接读。
> 优点：读极快、迭代安全；缺点：写开销大、弱一致性。
> 适合读多写少场景，如配置列表、白名单。

### 3.6 常见追问
| 追问 | 关键答点 |
|------|---------|
| CopyOnWrite和读写锁的区别？ | COW读完全不加锁；读写锁读时加读锁，写时阻塞读 |
| 弱一致性影响大吗？ | 取决于业务，读配置/白名单场景可接受 |
