## 一、HashMap 源码 ⭐⭐⭐⭐

### 1.1 底层结构
- JDK7：数组 + 链表（头插法）
- JDK8：数组 + 链表（尾插法） + 红黑树

```
JDK8 HashMap 结构：
┌─────────────────────────────────────────┐
│  [0]  [1]  [2]  [3]  [4]  [5]  ...     │ ← Node[] table（数组）
└──┬─────┬────────────────────────────────┘
   │     │
   ▼     ▼
  Node  Node → Node → Node              ← 链表（≤8个节点）
         ▼
       TreeNode → TreeNode → TreeNode   ← 红黑树（>8且数组≥64）
```

### 1.2 核心参数
| 参数 | 值 | 含义 |
|------|-----|------|
| 初始容量 | 16 | 必须是2的幂 |
| 负载因子 | 0.75 | 扩容阈值=容量×0.75（16×0.75=12时扩容） |
| 树化阈值 | 8 | 链表长度≥8**且**数组长度≥64时转红黑树 |
| 退化阈值 | 6 | 红黑树节点≤6时退化回链表（留2个缓冲防频繁转换） |
| 最小树化容量 | 64 | 链表≥8但数组<64时，优先扩容而非树化 |

### 1.3 put 流程（核心 ⭐）
```
put(key, value)
  ↓
① hash计算：(h = key.hashCode()) ^ (h >>> 16)  → 高低16位异或减少碰撞
  ↓
② 定位桶：index = (n-1) & hash  → 位运算代替取模
  ↓
③ 桶为空？→ 直接 new Node 放入
  ↓ 不为空
④ key相同（equals为true）？→ 覆盖 value，返回旧值
  ↓ key不同
⑤ 是红黑树？→ 调用 putTreeVal 树插入
  ↓ 是链表
⑥ 尾插法遍历链表：
   → 找到相同key → 覆盖
   → 到尾部 → 尾插新节点
   → 插入后链表长度 ≥ 8 且数组长度 ≥ 64 → treeifyBin() 转红黑树
  ↓
⑦ ++size > threshold → resize() 扩容2倍
```

### 1.4 hash 计算为什么要 ^ (h>>>16)

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

```
hashCode: 1111 1010 0011 0101   1100 0001 0010 1010
h >>> 16: 0000 0000 0000 0000   1111 1010 0011 0101
XOR结果:   1111 1010 0011 0101   0011 1011 0001 1111

定位桶时只用低位：index = (n-1) & hash
如果不混合高位，高16位信息完全丢失 → 碰撞概率增大
异或后低位融合了高位特征 → 减少碰撞
```

### 1.5 扩容机制（resize）
```
触发条件：size > capacity × loadFactor（默认12）

扩容流程：
  新容量 = 旧容量 × 2（左移1位）
  新阈值 = 新容量 × loadFactor

节点迁移（无需重新计算hash！）：
  hash & oldCap == 0 → 位置不变（低位链表）
  hash & oldCap != 0 → 位置 + oldCap（高位链表）

原理：容量翻倍，只是多看一个bit
  旧容量16：index = hash & 0x0F（低4位）
  新容量32：index = hash & 0x1F（低5位）
  → 只需判断第5位（即hash & oldCap），0不动，1偏移oldCap
```

### 1.6 JDK7 死循环问题 ⭐

**根本原因：** JDK7扩容用**头插法**，多线程并发扩容导致链表成环

```
原始链表：A → B → C

线程1执行到B时被挂起
线程2完成扩容，头插法反转：C → B → A

线程1恢复：继续头插
  → A.next = B，B.next = A（链表成环！）
  → get操作遍历到环时 → CPU 100% 死循环
```

**JDK8修复：** 改用**尾插法**，保持链表顺序不反转，不会成环

> ⚠️ JDK8 HashMap 仍然线程不安全！并发put可能数据覆盖、size不准。并发场景必须用 ConcurrentHashMap

### 1.7 容量必须是2的幂的原因
```
原因一：(n-1) & hash 等价于 hash % n，位运算比取模快得多
  n=16: n-1=0x0F=01111 → hash & 01111 = hash % 16

原因二：扩容迁移时只需看一个bit，无需重计算hash
  扩容从16→32：只看第5位bit → 0就不动，1就+16

如果指定非2幂的容量？
  HashMap 会自动向上取整到最近的2的幂
  new HashMap<>(10) → 实际容量16
  new HashMap<>(17) → 实际容量32
```

### 1.8 面试标准答法

**Q: 讲讲HashMap的底层原理？**

> JDK8 HashMap底层数组+链表+红黑树，默认容量16负载因子0.75。
> put时先计算hash（hashCode高低16位异或减少碰撞），用`(n-1)&hash`定位桶，尾插链表，链表≥8且数组≥64转红黑树。
> 扩容时容量翻倍，节点通过`hash & oldCap`判断新位置（0不动，1偏移），无需重计算hash。
> JDK7头插法导致并发扩容链表成环死循环，JDK8改尾插法修复，但HashMap仍非线程安全。

### 1.9 常见追问

| 追问 | 关键答点 |
|------|----------|
| 容量为什么2的幂？ | 位运算代替取模+扩容时只看one bit |
| 负载因子为什么0.75？ | 空间与时间平衡，泊松分布下碰撞概率可接受临界值 |
| 树化阈值为什么8？ | 泊松分布超8概率极低（不到千万分之一），<8时链表性能可接受 |
| HashMap key可以是null吗？ | 可以，null的hashCode为0，存index 0 |
| 红黑树何时退化？ | ≤6时退化，留缓冲避免在8附近频繁转换 |
| HashMap和Hashtable区别？ | HashMap允许null key/value、非线程安全；Hashtable方法加synchronized、不允许null |

## 二、ArrayList vs LinkedList ⭐⭐

### 2.1 底层结构

**ArrayList：**
```
Object[] elementData  ← 连续内存数组
默认容量：10
扩容：oldCapacity + (oldCapacity >> 1) = 1.5倍
  10 → 15 → 22 → 33 → ...

内存布局：[a][b][c][d][e][_][_]  ← 连续内存，CPU缓存友好
```

**LinkedList：**
```
双向链表 + 实现 Deque 接口

Node<E> {
    E item;
    Node<E> prev;
    Node<E> next;
}

内存布局：[prev|a|next] ↔ [prev|b|next] ↔ [prev|c|next]
  ← 不连续内存，每个节点额外24字节（prev指针8 + next指针8 + 对象头8）
```

### 2.2 核心对比

| 操作 | ArrayList | LinkedList | 说明 |
|------|-----------|------------|------|
| 随机访问 get(i) | **O(1)** | O(N) | 数组下标直接算 vs 链表需遍历 |
| 头部插入/删除 | O(N)（整体移位） | **O(1)** | 数组需搬移元素 vs 链表改指针 |
| 尾部插入 | O(1)均摊 | O(1) | 二者相当，ArrayList偶尔扩容 |
| 中间插入/删除 | O(N) | O(N)找+O(1)改 | 仍然都是O(N)级别 |
| 内存占用 | 连续，CPU缓存友好 | 不连续，额外指针开销大 | ArrayList实际性能远优于LinkedList |
| 随机访问场景 | ✅ 绝对优势 | ❌ | ArrayList |
| 频繁头部插入 | ❌ | ✅ | 但可用ArrayDeque替代 |

> ⚠️ 实际开发中**绝大多数场景用 ArrayList**。LinkedList 内存不连续导致CPU缓存不友好，虽然理论中间插入是O(1)改指针，但找到位置本身是O(N)，实测比ArrayList还慢。需要双端操作用 ArrayDeque 更好。

### 2.3 ArrayList 注意点

**遍历时删除：**
```java
// ❌ 错误：for-each中直接remove → ConcurrentModificationException
for (String s : list) {
    if (s.equals("bad")) list.remove(s);  // fast-fail!
}

// ✅ 正确方式1：iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("bad")) it.remove();
}

// ✅ 正确方式2：removeIf（推荐）
list.removeIf(s -> s.equals("bad"));
```

**fast-fail机制：** 每次结构修改 modCount++，迭代器检测到 modCount 变化就抛异常，防止并发修改导致数据错误

**初始化建议：**
```java
// 预估容量避免频繁扩容
List<Order> orders = new ArrayList<>(1000);

// 扩容1.5倍的原因：
// 比2倍小 → 减少内存浪费
// 比1倍大 → 减少扩容次数
// 1.5倍是平衡点
```

### 2.4 面试标准答法

**Q: ArrayList和LinkedList有什么区别？**

> ArrayList底层Object数组，连续内存CPU缓存友好，随机访问O(1)，扩容1.5倍。LinkedList双向链表，随机访问O(N)，额外指针开销大。
> 实际绝大多数用ArrayList，内存连续性带来的缓存优势远大于理论上的链表插入优势。for-each删元素会fast-fail，要用iterator.remove()或removeIf()。

### 2.5 常见追问

| 追问 | 关键答点 |
|------|----------|
| 什么时候用LinkedList？ | 几乎不用。需要双端操作用ArrayDeque更好 |
| fast-fail和fast-safe区别？ | fast-fail检测到并发修改抛异常（ArrayList）；fail-safe迭代副本不抛异常（CopyOnWriteArrayList） |
| ArrayList添加null？ | 可以，允许null元素 |

## 三、CopyOnWriteArrayList 原理 ⭐⭐⭐

### 3.1 核心思想：写时复制（COW）

```java
// 底层数组用 volatile 修饰保证可见性
private transient volatile Object[] array;

// 读操作：完全不加锁，直接读volatile数组
public E get(int index) {
    return (E) getArray()[index];  // 无锁，直接读
}

// 写操作：加锁 → 复制新数组 → 在新数组修改 → 替换引用 → 释放锁
public boolean add(E e) {
    synchronized (lock) {
        Object[] es = getArray();
        int len = es.length;
        es = Arrays.copyOf(es, len + 1);  // 复制新数组
        es[len] = e;                       // 在新数组修改
        setArray(es);                       // volatile替换引用
        return true;
    }
}
```

### 3.2 读写分离原理

```
读操作：读当前volatile数组引用 → 直接访问（可能是旧数组快照）
写操作：加锁 → 复制整个数组 → 修改副本 → volatile替换引用

时间线：
  T1: array → [A, B, C]
  T2: 写线程加锁，复制 → [A, B, C, D]，写线程替换引用
  T3: array → [A, B, C, D]

  在T2时刻，读线程仍然读到旧数组[A, B, C] → 弱一致性
  在T3之后，读线程读到新数组[A, B, C, D]
```

### 3.3 优缺点

| | 说明 |
|--|------|
| ✅ 读不加锁 | 读性能极高，适合高并发读 |
| ✅ 迭代安全 | 迭代的是快照数组，不会ConcurrentModificationException |
| ❌ 写开销大 | 每次写都复制整个数组，O(N)空间+时间 |
| ❌ 弱一致性 | 读可能读到旧快照数据（最终一致，不是强一致） |
| ❌ 不适合大数组 | 复制成本与数组大小成正比 |
| ❌ 内存占用 | 写时同时存在两个数组（旧+新），GC压力 |

### 3.4 适用场景

- **读多写少**：白名单、黑名单、配置列表、监听器列表
- **数组不大**：通常几十到几百元素
- **能接受弱一致性**：读到旧数据可接受

### 3.5 线程安全List方案对比

| 方案 | 原理 | 读性能 | 写性能 | 适用场景 |
|------|------|--------|--------|---------|
| Vector | 全方法synchronized | 低 | 低 | **基本淘汰** |
| synchronizedList | 包装器加锁 | 低 | 低 | 简单场景，不推荐 |
| **CopyOnWriteArrayList** | 写时复制，读不加锁 | **极高** | 低 | **读多写少** |
| ConcurrentLinkedQueue | CAS无锁队列 | 高 | 高 | 高并发队列 |

### 3.6 面试标准答法

**Q: 讲讲CopyOnWriteArrayList的原理？**

> CopyOnWriteArrayList核心是写时复制：底层volatile数组保证可见性，写操作加synchronized锁后复制整个数组到副本，在副本上修改，再用volatile替换引用；读操作完全不加锁，直接读当前数组。
> 优点：读极快、迭代安全（迭代快照不会ConcurrentModificationException）。缺点：写开销大（每次复制整个数组）、弱一致性（可能读到旧快照）、不适合大数组。
> 适用读多写少场景，如配置列表、白名单。

### 3.7 常见追问

| 追问 | 关键答点 |
|------|---------|
| CopyOnWrite和读写锁的区别？ | COW读完全不加锁，写时其他线程读旧快照不阻塞；读写锁读时加读锁，写时阻塞所有读，但强一致 |
| 弱一致性影响大吗？ | 取决于业务，读配置/白名单场景可接受；强一致场景不适用 |
| CopyOnWriteArraySet？ | 基于CopyOnWriteArrayList实现，add时遍历检查去重，适合小集合 |
| 写多的场景用什么？ | ConcurrentHashMap（Map场景）、ConcurrentLinkedQueue（队列场景）|
