## 一、HashMap 源码 ⭐⭐⭐⭐

### 1.1 底层结构
- JDK7：数组 + 链表（头插法）
- JDK8：数组 + 链表（尾插法） + 红黑树

```
JDK8 HashMap 结构：
┌─────────────────────────────────────────┐
│  [0]  [1]  [2]  [3]  [4]  [5]  ...     │ ← Node[] table（数组）
└──┬─────┬────────────────────────────────┘
   │     │
   ▼     ▼
  Node  Node → Node → Node              ← 链表（≤8个节点）
         ▼
       TreeNode → TreeNode → TreeNode   ← 红黑树（>8且数组≥64）
```

### 1.2 核心参数
| 参数 | 值 | 含义 |
|------|-----|------|
| 初始容量 | 16 | 必须是2的幂 |
| 负载因子 | 0.75 | 扩容阈值=容量×0.75（16×0.75=12时扩容） |
| 树化阈值 | 8 | 链表长度≥8**且**数组长度≥64时转红黑树 |
| 退化阈值 | 6 | 红黑树节点≤6时退化回链表（留2个缓冲防频繁转换） |
| 最小树化容量 | 64 | 链表≥8但数组<64时，优先扩容而非树化 |

### 1.3 put 流程（核心 ⭐）
```
put(key, value)
  ↓
① hash计算：(h = key.hashCode()) ^ (h >>> 16)  → 高低16位异或减少碰撞
  ↓
② 定位桶：index = (n-1) & hash  → 位运算代替取模
  ↓
③ 桶为空？→ 直接 new Node 放入
  ↓ 不为空
④ key相同（equals为true）？→ 覆盖 value，返回旧值
  ↓ key不同
⑤ 是红黑树？→ 调用 putTreeVal 树插入
  ↓ 是链表
⑥ 尾插法遍历链表：
   → 找到相同key → 覆盖
   → 到尾部 → 尾插新节点
   → 插入后链表长度 ≥ 8 且数组长度 ≥ 64 → treeifyBin() 转红黑树
  ↓
⑦ ++size > threshold → resize() 扩容2倍
```

### 1.4 hash 计算为什么要 ^ (h>>>16)

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

```
hashCode: 1111 1010 0011 0101   1100 0001 0010 1010
h >>> 16: 0000 0000 0000 0000   1111 1010 0011 0101
XOR结果:   1111 1010 0011 0101   0011 1011 0001 1111

定位桶时只用低位：index = (n-1) & hash
如果不混合高位，高16位信息完全丢失 → 碰撞概率增大
异或后低位融合了高位特征 → 减少碰撞
```

### 1.5 扩容机制（resize）
```
触发条件：size > capacity × loadFactor（默认12）

扩容流程：
  新容量 = 旧容量 × 2（左移1位）
  新阈值 = 新容量 × loadFactor

节点迁移（无需重新计算hash！）：
  hash & oldCap == 0 → 位置不变（低位链表）
  hash & oldCap != 0 → 位置 + oldCap（高位链表）

原理：容量翻倍，只是多看一个bit
  旧容量16：index = hash & 0x0F（低4位）
  新容量32：index = hash & 0x1F（低5位）
  → 只需判断第5位（即hash & oldCap），0不动，1偏移oldCap
```

### 1.6 JDK7 死循环问题 ⭐

**根本原因：** JDK7扩容用**头插法**，多线程并发扩容导致链表成环

```
原始链表：A → B → C

线程1执行到B时被挂起
线程2完成扩容，头插法反转：C → B → A

线程1恢复：继续头插
  → A.next = B，B.next = A（链表成环！）
  → get操作遍历到环时 → CPU 100% 死循环
```

**JDK8修复：** 改用**尾插法**，保持链表顺序不反转，不会成环

> ⚠️ JDK8 HashMap 仍然线程不安全！并发put可能数据覆盖、size不准。并发场景必须用 ConcurrentHashMap

### 1.7 容量必须是2的幂的原因
```
原因一：(n-1) & hash 等价于 hash % n，位运算比取模快得多
  n=16: n-1=0x0F=01111 → hash & 01111 = hash % 16

原因二：扩容迁移时只需看一个bit，无需重计算hash
  扩容从16→32：只看第5位bit → 0就不动，1就+16

如果指定非2幂的容量？
  HashMap 会自动向上取整到最近的2的幂
  new HashMap<>(10) → 实际容量16
  new HashMap<>(17) → 实际容量32
```

### 1.8 面试标准答法

**Q: 讲讲HashMap的底层原理？**

> JDK8 HashMap底层数组+链表+红黑树，默认容量16负载因子0.75。
> put时先计算hash（hashCode高低16位异或减少碰撞），用`(n-1)&hash`定位桶，尾插链表，链表≥8且数组≥64转红黑树。
> 扩容时容量翻倍，节点通过`hash & oldCap`判断新位置（0不动，1偏移），无需重计算hash。
> JDK7头插法导致并发扩容链表成环死循环，JDK8改尾插法修复，但HashMap仍非线程安全。

### 1.9 常见追问

| 追问 | 关键答点 |
|------|----------|
| 容量为什么2的幂？ | 位运算代替取模+扩容时只看one bit |
| 负载因子为什么0.75？ | 空间与时间平衡，泊松分布下碰撞概率可接受临界值 |
| 树化阈值为什么8？ | 泊松分布超8概率极低（不到千万分之一），<8时链表性能可接受 |
| HashMap key可以是null吗？ | 可以，null的hashCode为0，存index 0 |
| 红黑树何时退化？ | ≤6时退化，留缓冲避免在8附近频繁转换 |
| HashMap和Hashtable区别？ | HashMap允许null key/value、非线程安全；Hashtable方法加synchronized、不允许null |

## 二、ArrayList vs LinkedList ⭐⭐

### 2.1 底层结构

**ArrayList：**
```
Object[] elementData  ← 连续内存数组
默认容量：10
扩容：oldCapacity + (oldCapacity >> 1) = 1.5倍
  10 → 15 → 22 → 33 → ...

内存布局：[a][b][c][d][e][_][_]  ← 连续内存，CPU缓存友好
```

**LinkedList：**
```
双向链表 + 实现 Deque 接口

Node<E> {
    E item;
    Node<E> prev;
    Node<E> next;
}

内存布局：[prev|a|next] ↔ [prev|b|next] ↔ [prev|c|next]
  ← 不连续内存，每个节点额外24字节（prev指针8 + next指针8 + 对象头8）
```

### 2.2 核心对比

| 操作 | ArrayList | LinkedList | 说明 |
|------|-----------|------------|------|
| 随机访问 get(i) | **O(1)** | O(N) | 数组下标直接算 vs 链表需遍历 |
| 头部插入/删除 | O(N)（整体移位） | **O(1)** | 数组需搬移元素 vs 链表改指针 |
| 尾部插入 | O(1)均摊 | O(1) | 二者相当，ArrayList偶尔扩容 |
| 中间插入/删除 | O(N) | O(N)找+O(1)改 | 仍然都是O(N)级别 |
| 内存占用 | 连续，CPU缓存友好 | 不连续，额外指针开销大 | ArrayList实际性能远优于LinkedList |
| 随机访问场景 | ✅ 绝对优势 | ❌ | ArrayList |
| 频繁头部插入 | ❌ | ✅ | 但可用ArrayDeque替代 |

> ⚠️ 实际开发中**绝大多数场景用 ArrayList**。LinkedList 内存不连续导致CPU缓存不友好，虽然理论中间插入是O(1)改指针，但找到位置本身是O(N)，实测比ArrayList还慢。需要双端操作用 ArrayDeque 更好。

### 2.3 ArrayList 注意点

**遍历时删除：**
```java
// ❌ 错误：for-each中直接remove → ConcurrentModificationException
for (String s : list) {
    if (s.equals("bad")) list.remove(s);  // fast-fail!
}

// ✅ 正确方式1：iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if (it.next().equals("bad")) it.remove();
}

// ✅ 正确方式2：removeIf（推荐）
list.removeIf(s -> s.equals("bad"));
```

**fast-fail机制：** 每次结构修改 modCount++，迭代器检测到 modCount 变化就抛异常，防止并发修改导致数据错误

**初始化建议：**
```java
// 预估容量避免频繁扩容
List<Order> orders = new ArrayList<>(1000);

// 扩容1.5倍的原因：
// 比2倍小 → 减少内存浪费
// 比1倍大 → 减少扩容次数
// 1.5倍是平衡点
```

### 2.4 面试标准答法

**Q: ArrayList和LinkedList有什么区别？**

> ArrayList底层Object数组，连续内存CPU缓存友好，随机访问O(1)，扩容1.5倍。LinkedList双向链表，随机访问O(N)，额外指针开销大。
> 实际绝大多数用ArrayList，内存连续性带来的缓存优势远大于理论上的链表插入优势。for-each删元素会fast-fail，要用iterator.remove()或removeIf()。

### 2.5 常见追问

| 追问 | 关键答点 |
|------|----------|
| 什么时候用LinkedList？ | 几乎不用。需要双端操作用ArrayDeque更好 |
| fast-fail和fast-safe区别？ | fast-fail检测到并发修改抛异常（ArrayList）；fail-safe迭代副本不抛异常（CopyOnWriteArrayList） |
| ArrayList添加null？ | 可以，允许null元素 |

## 三、CopyOnWriteArrayList 原理 ⭐⭐⭐

### 3.1 核心思想：写时复制（COW）

```java
// 底层数组用 volatile 修饰保证可见性
private transient volatile Object[] array;

// 读操作：完全不加锁，直接读volatile数组
public E get(int index) {
    return (E) getArray()[index];  // 无锁，直接读
}

// 写操作：加锁 → 复制新数组 → 在新数组修改 → 替换引用 → 释放锁
public boolean add(E e) {
    synchronized (lock) {
        Object[] es = getArray();
        int len = es.length;
        es = Arrays.copyOf(es, len + 1);  // 复制新数组
        es[len] = e;                       // 在新数组修改
        setArray(es);                       // volatile替换引用
        return true;
    }
}
```

### 3.2 读写分离原理

```
读操作：读当前volatile数组引用 → 直接访问（可能是旧数组快照）
写操作：加锁 → 复制整个数组 → 修改副本 → volatile替换引用

时间线：
  T1: array → [A, B, C]
  T2: 写线程加锁，复制 → [A, B, C, D]，写线程替换引用
  T3: array → [A, B, C, D]

  在T2时刻，读线程仍然读到旧数组[A, B, C] → 弱一致性
  在T3之后，读线程读到新数组[A, B, C, D]
```

### 3.3 优缺点

| | 说明 |
|--|------|
| ✅ 读不加锁 | 读性能极高，适合高并发读 |
| ✅ 迭代安全 | 迭代的是快照数组，不会ConcurrentModificationException |
| ❌ 写开销大 | 每次写都复制整个数组，O(N)空间+时间 |
| ❌ 弱一致性 | 读可能读到旧快照数据（最终一致，不是强一致） |
| ❌ 不适合大数组 | 复制成本与数组大小成正比 |
| ❌ 内存占用 | 写时同时存在两个数组（旧+新），GC压力 |

### 3.4 适用场景

- **读多写少**：白名单、黑名单、配置列表、监听器列表
- **数组不大**：通常几十到几百元素
- **能接受弱一致性**：读到旧数据可接受

### 3.5 线程安全List方案对比

| 方案 | 原理 | 读性能 | 写性能 | 适用场景 |
|------|------|--------|--------|---------|
| Vector | 全方法synchronized | 低 | 低 | **基本淘汰** |
| synchronizedList | 包装器加锁 | 低 | 低 | 简单场景，不推荐 |
| **CopyOnWriteArrayList** | 写时复制，读不加锁 | **极高** | 低 | **读多写少** |
| ConcurrentLinkedQueue | CAS无锁队列 | 高 | 高 | 高并发队列 |

### 3.6 面试标准答法

**Q: 讲讲CopyOnWriteArrayList的原理？**

> CopyOnWriteArrayList核心是写时复制：底层volatile数组保证可见性，写操作加synchronized锁后复制整个数组到副本，在副本上修改，再用volatile替换引用；读操作完全不加锁，直接读当前数组。
> 优点：读极快、迭代安全（迭代快照不会ConcurrentModificationException）。缺点：写开销大（每次复制整个数组）、弱一致性（可能读到旧快照）、不适合大数组。
> 适用读多写少场景，如配置列表、白名单。

### 3.7 常见追问

| 追问 | 关键答点 |
|------|---------|
| CopyOnWrite和读写锁的区别？ | COW读完全不加锁，写时其他线程读旧快照不阻塞；读写锁读时加读锁，写时阻塞所有读，但强一致 |
| 弱一致性影响大吗？ | 取决于业务，读配置/白名单场景可接受；强一致场景不适用 |
| CopyOnWriteArraySet？ | 基于CopyOnWriteArrayList实现，add时遍历检查去重，适合小集合 |
| 写多的场景用什么？ | ConcurrentHashMap（Map场景）、ConcurrentLinkedQueue（队列场景）|

## 四、LinkedHashMap 与 TreeMap ⭐⭐⭐

### 4.1 LinkedHashMap：插入/访问有序的 HashMap

LinkedHashMap 继承 HashMap，在每个 Entry 额外维护 **双向链表**，记录插入顺序或访问顺序。

```java
// accessOrder=false（默认）：按插入顺序迭代
Map<String, Integer> insertOrder = new LinkedHashMap<>();

// accessOrder=true：按访问顺序迭代（每次get/put会将节点移到链表尾）
Map<String, Integer> lru = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Integer> eldest) {
        return size() > 100;  // 超过100个元素时，自动删除最久未访问的头节点
    }
};
```

**LRU缓存实现：** `LinkedHashMap(accessOrder=true)` + 重写 `removeEldestEntry` 即可实现 O(1) LRU。

| 特性 | HashMap | LinkedHashMap | TreeMap |
|------|---------|---------------|---------|
| 底层结构 | 数组+链表+红黑树 | 数组+链表+红黑树+**双向链表** | 红黑树 |
| 顺序 | 无序 | 插入/访问有序 | **Key自然序/自定义序** |
| get/put | O(1) | O(1) | **O(log N)** |
| 线程安全 | 否 | 否 | 否 |
| 适用场景 | 一般Map | LRU/有序迭代 | 范围查询/排序 |

### 4.2 TreeMap：基于红黑树的有序 Map

```java
// 按Key自然排序（String字典序）
TreeMap<String, Integer> treeMap = new TreeMap<>();

// 自定义排序（按字符串长度）
TreeMap<String, Integer> customMap = new TreeMap<>(
    Comparator.comparingInt(String::length).thenComparing(Comparator.naturalOrder())
);

// TreeMap特有的区间查询方法（实现了NavigableMap接口）
treeMap.firstKey();               // 最小Key
treeMap.lastKey();                // 最大Key
treeMap.floorKey("c");            // 不大于"c"的最大Key（≤）
treeMap.ceilingKey("c");          // 不小于"c"的最小Key（≥）
treeMap.subMap("a", "d");         // [a, d) 范围的子Map
treeMap.headMap("c");             // 严格小于"c"的子Map
treeMap.tailMap("c");             // 大于等于"c"的子Map
```

### 4.3 面试答法

**Q: LinkedHashMap 如何实现 LRU？**

> LinkedHashMap 继承 HashMap，通过在 Entry 里额外维护 before/after 指针形成双向链表，记录访问顺序（`accessOrder=true`）。每次 `get/put` 都将对应节点移到链表尾部，链表头部就是最久未访问的节点。重写 `removeEldestEntry` 方法，当 `size() > 容量` 时返回 true，LinkedHashMap 自动删除链表头（最久未访问）。整体 O(1) 实现 LRU。

**Q: TreeMap 和 HashMap 如何选择？**

> 如果只需要快速存取，选 HashMap（O(1)）；如果需要按 key 有序遍历或范围查询（floorKey/ceilingKey/subMap），选 TreeMap（O(log N)）。TreeMap 的 Key 必须实现 Comparable 或提供 Comparator。

## 五、ArrayDeque 与 PriorityQueue ⭐⭐⭐

### 5.1 ArrayDeque：高效双端队列

ArrayDeque 底层是**循环数组**，**既可当 Stack 使用，也可当 Queue 使用**，是官方推荐替代 `Stack` 和 `LinkedList` 队列的首选。

```java
Deque<Integer> deque = new ArrayDeque<>();

// 当 Stack 使用（替代 Stack 类）
deque.push(1);      // 等价于 addFirst
deque.push(2);
int top = deque.pop();     // 弹出栈顶 2（LIFO）
int peek = deque.peek();   // 查看栈顶不弹出

// 当 Queue 使用（替代 LinkedList）
deque.offer(1);    // 尾部入队
deque.offer(2);
int front = deque.poll();  // 头部出队（FIFO）

// 双端操作
deque.addFirst(0);  // 头部插入
deque.addLast(3);   // 尾部插入
deque.peekFirst();  // 查看头部
deque.peekLast();   // 查看尾部
```

| | Stack | LinkedList | ArrayDeque |
|--|-------|-----------|------------|
| 底层 | 继承Vector | 双向链表 | 循环数组 |
| 线程安全 | 是（同步开销大） | 否 | 否 |
| 性能 | 低 | 有指针开销 | **最优** |
| 推荐 | ❌ 不推荐 | 不推荐作队列 | ✅ 优先选择 |

### 5.2 PriorityQueue：堆（优先队列）

PriorityQueue 底层是**小顶堆**（完全二叉树用数组表示），默认自然排序（最小元素优先出队）。

```java
// 默认小顶堆（最小值优先）
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
minHeap.offer(3);
minHeap.offer(1);
minHeap.offer(2);
System.out.println(minHeap.poll()); // 输出 1（最小值）

// 大顶堆（最大值优先）
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());

// 自定义对象排序（按任务优先级）
PriorityQueue<Task> taskQueue = new PriorityQueue<>(
    Comparator.comparingInt(Task::getPriority)
);
```

**时间复杂度：**
- `offer()/add()`：O(log N)（向上堆化）
- `poll()`：O(log N)（向下堆化）
- `peek()`：O(1)
- `contains()`：O(N)（遍历数组，**注意坑**）

**典型场景：** Top K 问题（K大/K小元素）、Dijkstra最短路径、任务调度

```java
// Top K 最大元素：维护 K 个元素的小顶堆
public int[] topK(int[] nums, int k) {
    PriorityQueue<Integer> heap = new PriorityQueue<>(k);
    for (int num : nums) {
        heap.offer(num);
        if (heap.size() > k) heap.poll(); // 淘汰最小值，保留K个最大
    }
    return heap.stream().mapToInt(i -> i).toArray();
}
```

### 5.3 面试答法

**Q: 为什么用 ArrayDeque 替代 Stack？**

> Stack 继承 Vector，所有方法都是 synchronized 的，但我们通常在单线程下使用栈，同步开销毫无意义。ArrayDeque 底层是循环数组，无锁，push/pop 均摊 O(1)，且没有对象头/next指针内存开销。在单线程场景下 ArrayDeque 性能远优于 Stack。

**Q: 如何用 PriorityQueue 解决 Top K 问题？**

> Top K 最大问题：维护大小为 K 的**小顶堆**，遍历数据，堆未满直接放入，堆满则与堆顶比较，新元素更大则替换堆顶并重新堆化。最终堆里的 K 个元素就是最大的 K 个，时间复杂度 O(N log K)，优于全量排序的 O(N log N)。

## 六、HashSet 与 TreeSet ⭐⭐

### 6.1 实现原理

Set 接口的实现类底层均**基于对应的 Map**，Value 用一个固定的 `PRESENT`（Object 占位符）填充。

```java
// HashSet 底层是 HashMap
private transient HashMap<E,Object> map;
private static final Object PRESENT = new Object();

public boolean add(E e) {
    return map.put(e, PRESENT) == null;  // 利用 Map.put 的返回值判断是否新增
}

// TreeSet 底层是 TreeMap
private transient NavigableMap<E,Object> m;
```

| | HashSet | LinkedHashSet | TreeSet |
|--|---------|---------------|---------|
| 底层 | HashMap | LinkedHashMap | TreeMap |
| 顺序 | 无序 | 插入有序 | Key自然/自定义有序 |
| null | 允许 1 个 | 允许 1 个 | ❌ 不允许（排序不能比较null） |
| 查找 | O(1) | O(1) | O(log N) |

### 6.2 去重原理（重要！）

HashSet/HashMap 的去重依赖 **`hashCode()` + `equals()`** 的约定：

1. **先用 `hashCode()` 定位桶**：hashCode 相同才会进一步比较 equals
2. **再用 `equals()` 判断相等**：equals 判断为 true 则认为是同一元素

**经典陷阱：** 自定义对象存入 HashSet 必须**同时重写 hashCode 和 equals**，否则逻辑相等的对象会被当作不同元素。

```java
// 反例：只重写 equals，不重写 hashCode
User u1 = new User("Alice", 18);
User u2 = new User("Alice", 18);
Set<User> set = new HashSet<>();
set.add(u1);
set.add(u2);
System.out.println(set.size()); // 2！因为 hashCode 不同，定位到不同的桶
```

## 七、Collections 工具类 & 不可变集合 ⭐⭐

### 7.1 Collections 工具类常用方法

```java
List<Integer> list = new ArrayList<>(Arrays.asList(3, 1, 4, 1, 5, 9));

Collections.sort(list);                          // 自然排序（底层 TimSort）
Collections.sort(list, Comparator.reverseOrder()); // 逆序
Collections.shuffle(list);                       // 随机打乱
Collections.reverse(list);                       // 翻转
Collections.swap(list, 0, 1);                    // 交换元素
Collections.fill(list, 0);                       // 全部填充为0

int max = Collections.max(list);                 // 最大值
int min = Collections.min(list);                 // 最小值
int freq = Collections.frequency(list, 1);       // 元素1的出现次数

// 线程安全包装（已有 ConcurrentXxx 方案时不推荐，此方案性能差）
List<Integer> syncList = Collections.synchronizedList(list);
Map<String, Integer> syncMap = Collections.synchronizedMap(new HashMap<>());

// 不可修改视图（视图≠深拷贝，原集合修改仍影响视图）
List<Integer> unmodifiable = Collections.unmodifiableList(list);
```

### 7.2 Java 9+ 不可变集合工厂方法

```java
// Java 9+：List.of / Set.of / Map.of 创建真正不可变集合
List<String> immutableList = List.of("a", "b", "c");
Set<String> immutableSet  = Set.of("x", "y", "z");
Map<String, Integer> immutableMap = Map.of("a", 1, "b", 2);

// 超过10个KV对时用 ofEntries
Map<String, Integer> bigMap = Map.ofEntries(
    Map.entry("a", 1),
    Map.entry("b", 2)
);

// Java 10+：复制为不可变集合
List<Integer> copy = List.copyOf(list);  // 深拷贝，独立于原集合
```

**`List.of` vs `unmodifiableList` 区别：**

| | `List.of` | `unmodifiableList` |
|--|-----------|-------------------|
| 本质 | 真正不可变 | 不可修改的**视图** |
| 修改原集合 | N/A（无原集合） | **视图随之变化** |
| null 元素 | ❌ 不允许 | 取决于原集合 |
| 空间 | 紧凑内部实现 | 包装器有开销 |

### 7.3 Arrays 工具类常用方法

```java
int[] arr = {3, 1, 4, 1, 5};
Arrays.sort(arr);                         // 基本类型：双轴快排；对象：TimSort
Arrays.binarySearch(arr, 4);              // 二分查找（需先sort），返回下标
Arrays.fill(arr, 0);                      // 全部填0
int[] copy = Arrays.copyOf(arr, 3);       // 复制前3个
int[] range = Arrays.copyOfRange(arr, 1, 4); // 复制[1,4)区间
String s = Arrays.toString(arr);          // 打印数组

int[][] matrix = {{1,2},{3,4}};
String s2 = Arrays.deepToString(matrix);  // 打印二维数组

// 流式操作（Java 8+）
int sum = Arrays.stream(arr).sum();
OptionalInt max2 = Arrays.stream(arr).max();
```
