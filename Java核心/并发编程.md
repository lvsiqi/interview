## 一、synchronized 底层原理 & 锁升级 ⭐⭐⭐

### 1.1 底层实现
- 同步代码块：依赖 **monitorenter / monitorexit** 字节码指令
- 同步方法：依赖 **ACC_SYNCHRONIZED** 标志位
- 每个对象关联一个 **Monitor对象**（C++的 ObjectMonitor 实现）

```
┌─────────────────────────────────────────────────────┐
│                 ObjectMonitor                        │
│                                                     │
│   _owner: 当前持有锁的线程                            │
│   _count: 重入次数                                   │
│   _recursions: 递归重入计数                           │
│                                                     │
│   _EntryList ──→ [T2] → [T3] → [T4]                │
│   (阻塞队列)     竞争锁失败的线程在此排队              │
│                                                     │
│   _WaitSet  ──→ [T5] → [T6]                        │
│   (等待队列)     调用 wait() 后在此等待               │
│                                                     │
│   _cxq: 最近到达的竞争线程栈（ContentionList）        │
│         新来的竞争线程先进 _cxq 再转到 _EntryList      │
└─────────────────────────────────────────────────────┘

Monitor 工作流程：
  T1 获取锁 → _owner = T1, _count = 1
  T1 重入   → _count++
  T1.wait() → _owner = null, T1 移入 _WaitSet
  T2.notify()→ 将 _WaitSet 中一个线程移到 _EntryList
  T1 释放锁 → _count--, 为0时唤醒 _EntryList 中的线程
```

### 1.2 对象头 Mark Word

| 锁状态 | 存储内容 |
|--------|---------|
| 无锁 | hashCode、GC年龄 |
| 偏向锁 | 线程ID、Epoch |
| 轻量级锁 | 栈帧锁记录指针 |
| 重量级锁 | Monitor对象指针 |
| GC标记 | 空 |

### 1.3 锁升级过程：无锁 → 偏向锁 → 轻量级锁 → 重量级锁

- **偏向锁**：单线程场景，CAS将线程ID写入Mark Word，同线程进入无需CAS，零开销
- **轻量级锁**：多线程**交替**访问，CAS尝试获取锁 + 自旋等待（默认10次），避免线程切换
- **重量级锁**：多线程**真正竞争**，线程阻塞进 EntryList，涉及用户态/内核态切换，开销大

```
无锁
 ↓ 第一个线程访问
偏向锁（CAS写线程ID）
 ↓ 有其他线程竞争
撤销偏向锁 → 轻量级锁（CAS + 自旋）
 ↓ 自旋多次失败
重量级锁（线程阻塞进EntryList）
```

### 1.4 可重入原理
- 同一线程重入时 `_count + 1`，释放时 `_count - 1`，为 0 时真正释放锁

### 1.5 JVM 锁优化（JDK6+）⭐⭐⭐

| 优化 | 说明 |
|------|------|
| **锁消除** | JIT 逃逸分析发现锁对象不逃逸（线程私有），直接消除同步。如方法内 `new StringBuffer().append()` 的 synchronized |
| **锁粗化** | 连续对**同一对象**加锁解锁，合并为一次大范围加锁。如循环内反复 synchronized(obj) → 移到循环外 |
| **自适应自旋** | 轻量级锁自旋次数由 JVM 根据前次自旋成功率动态调整，成功率高则多自旋，反之少自旋 |

```java
// 锁消除示例：sb 不逃逸，JIT 会消除 append 中的 synchronized
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();
    sb.append(a);   // synchronized 内部有锁
    sb.append(b);   // 但 sb 是方法局部变量，不逃逸
    return sb.toString();
}
// JIT 优化后等效于无锁操作

// 锁粗化示例：
for (int i = 0; i < 1000; i++) {
    synchronized(lock) { list.add(i); }  // 循环内反复加锁
}
// JVM 可能优化为：
synchronized(lock) {
    for (int i = 0; i < 1000; i++) { list.add(i); }
}
```

### 1.6 面试标准答法
> **第一层（是什么）**：synchronized 是Java的内置锁，通过 monitorenter/monitorexit 指令实现，每个对象关联一个Monitor，保证互斥访问。
>
> **第二层（怎么做）**：锁信息存在对象头的Mark Word中，JDK6后引入锁升级：无锁→偏向锁→轻量级锁→重量级锁，在不同竞争程度下选择最优策略。
>
> **第三层（为什么）**：偏向锁消除无竞争开销，轻量级锁用CAS避免线程切换，只有真正竞争激烈时才升级重量级锁，大幅提升性能。
>
> **加分点**：JDK15后偏向锁被废弃，因为维护成本高且撤销需要在安全点暂停所有线程。JDK6+ 还有锁消除（逃逸分析消除无竞争锁）、锁粗化（合并连续加锁）、自适应自旋三种 JVM 层优化。

### 1.7 常见追问

| 追问 | 关键答点 |
|------|---------|
| 锁能降级吗？ | 只能升级不能降级（GC时除外） |
| 自旋多少次升级重量级？ | 默认10次，JVM自适应调整 |
| 偏向锁为什么JDK15废弃？ | 维护成本高，撤销需安全点STW |
| synchronized 和 ReentrantLock 区别？ | 见AQS章节 |
| 什么是锁消除/锁粗化？ | 消除：逃逸分析发现锁对象线程私有，去掉锁；粗化：合并连续同对象加锁 |
| wait() 和 sleep() 区别？ | wait 释放锁进 WaitSet，需 notify 唤醒；sleep 不释放锁，到时间自动恢复 |
| notify 和 notifyAll 选哪个？ | 一般用 notifyAll 更安全，notify 可能唤醒错误的等待线程导致信号丢失 |

## 二、volatile 原理 & 内存屏障 ⭐⭐⭐

### 2.1 Java 内存模型（JMM）⭐⭐⭐

```
┌──────────────────────────────────────────────────────┐
│                  主内存（Main Memory）                 │
│            共享变量的"规范真相"存储位置                  │
│        ┌──────┐  ┌──────┐  ┌──────┐                 │
│        │ var1 │  │ var2 │  │ var3 │                 │
│        └──┬───┘  └──┬───┘  └──┬───┘                 │
└───────────┼─────────┼─────────┼──────────────────────┘
   read/load│  read/load│  read/load│
            ▼         ▼         ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 线程1工作 │ │ 线程2工作 │ │ 线程3工作 │
│   内存    │ │   内存    │ │   内存    │
│ var1 副本 │ │ var2 副本 │ │ var3 副本 │
│(CPU缓存)  │ │(CPU缓存)  │ │(CPU缓存)  │
└──────────┘ └──────────┘ └──────────┘

JMM 规定：
- 所有共享变量存在主内存
- 每个线程有自己的工作内存（CPU 缓存/寄存器的抽象）
- 线程对变量的读写必须在工作内存中进行，不能直接读写主内存
- 线程间变量传递必须通过主内存"中转"
```

**并发编程三大核心问题：**

| 问题 | 说明 | 解决手段 |
|------|------|----------|
| **可见性** | 线程 A 修改变量，线程 B 不可见 | volatile / synchronized / final |
| **原子性** | 复合操作（i++）非原子 | synchronized / Atomic 类 / Lock |
| **有序性** | 指令重排序导致乱序执行 | volatile / synchronized / happens-before |

### 2.2 volatile 三大特性
- ✅ **可见性**：保证
- ✅ **有序性**：保证（禁止指令重排序）
- ❌ **原子性**：不保证

### 2.3 可见性原理
- 根源：多核CPU各有缓存，线程修改变量只写到本地缓存，其他线程看不到
- volatile 写：强制将修改值**立即刷新到主内存**，并通过 **lock前缀指令** 使其他CPU缓存失效
- volatile 读：强制从**主内存重新读取**，不使用CPU缓存
- 底层依赖：**MESI 缓存一致性协议** + **lock前缀指令**

### 2.4 有序性原理（内存屏障）
- 根源：JVM和CPU会对指令重排序提升性能，多线程下可能出问题

**内存屏障类型：**

| 屏障类型 | 作用 |
|---------|------|
| StoreStore | 禁止写操作重排序 |
| StoreLoad | 最强屏障，禁止所有重排序 |
| LoadLoad | 禁止读操作重排序 |
| LoadStore | 禁止读后写重排序 |

```
volatile写：[StoreStore屏障] → 写操作 → [StoreLoad屏障]
volatile读：读操作 → [LoadLoad屏障] → [LoadStore屏障]
```

### 2.5 为什么不保证原子性
- `i++` 分三步：读i → i+1 → 写i，volatile 无法保证三步整体原子
- 需要原子性用 `AtomicInteger` 或 `synchronized`

### 2.6 经典应用：DCL 双重检查锁
```java
private static volatile Singleton instance;

public static Singleton getInstance() {
    if (instance == null) {
        synchronized (Singleton.class) {
            if (instance == null) {
                instance = new Singleton(); // new分三步，可能重排序！
            }
        }
    }
    return instance;
}
```
- `new Singleton()` 三步：① 分配内存 ② 初始化对象 ③ 引用指向内存
- 没有volatile：②③可能重排序 → 其他线程拿到未初始化的对象
- 加了volatile：StoreStore屏障禁止②③重排序，保证安全

### 2.7 面试标准答法
> JMM 规定线程对变量的读写在工作内存中进行，并发三大问题是可见性、原子性、有序性。
> volatile 保证可见性和有序性，不保证原子性。
> 可见性通过 lock前缀指令 + MESI协议，强制写刷主内存、读从主内存取。
> 有序性通过插入内存屏障禁止指令重排序。
> 典型场景是 DCL单例，volatile 防止 new 对象时指令重排序导致返回未初始化对象。

### 2.8 常见追问

| 追问 | 关键答点 |
|------|---------|
| volatile 能替代 synchronized 吗？ | 不能，不保证原子性 |
| DCL为什么要加volatile？ | 防止new对象指令重排序，返回未初始化对象 |
| long/double 为什么建议加volatile？ | 32位JVM对64位变量读写非原子，可能读到半截数据 |

## 三、AQS原理 & ReentrantLock ⭐⭐⭐⭐

### 3.1 AQS 是什么
- 全称 `AbstractQueuedSynchronizer`，并发包核心基础框架
- `ReentrantLock`、`Semaphore`、`CountDownLatch`、`CyclicBarrier` 等均基于 AQS 实现

### 3.2 核心数据结构
- **state**：`volatile int`，同步状态。0=未占用，1=占用，>1=重入
- **CLH变体队列**：双向链表，管理等待线程
  - 每个 Node 含：线程引用、waitStatus、prev、next
  - `waitStatus` 关键值：0初始、-1(SIGNAL)后继需唤醒、1(CANCELLED)已取消

### 3.3 非公平锁加锁流程
```
lock() → CAS(state: 0→1)
  成功 → 设置owner为当前线程
  失败 → acquire(1)
           → tryAcquire()（是否重入？state+1）
           → 失败 → addWaiter()加入CLH队尾
           → acquireQueued()自旋 → park()挂起
```

### 3.4 解锁流程
```
unlock() → state-1 → state==0?
  是 → 清空owner → unpark(队列第一个有效节点)
  否 → 仅减少重入计数
```

### 3.5 公平锁 vs 非公平锁
| | 公平锁 | 非公平锁（默认）|
|--|--------|------|
| 加锁 | 严格按队列顺序 | 先CAS抢，抢不到再排队 |
| 吞吐量 | 低 | 高 |
| 饥饿 | 不会 | 可能 |
#### 公平锁加锁流程（与非公平锁对比）

**非公平锁：**
```
lock()
  ↓
直接 CAS 抢锁（不管队列有没有等待线程）
  ↓ 失败
acquire() → tryAcquire()
  ↓ 失败
入队 → park
```

**公平锁：**
```
lock()
  ↓
直接进入 acquire() → tryAcquire()
  ↓
先判断：hasQueuedPredecessors()？（队列中是否有比我更早的等待线程）
  ↓ 有                ↓ 无
直接入队park     CAS 尝试获取锁
                   ↓ 失败
                 入队 → park
```

**关键方法 `hasQueuedPredecessors()`：**
```java
// 返回 true 表示队列中有更早的等待线程，当前线程必须排队
public final boolean hasQueuedPredecessors() {
    Node h = head, t = tail;
    Node s;
    return h != t &&  // 队列不为空
        ((s = h.next) == null || s.thread != Thread.currentThread()); // 队头不是自己
}
```
> **核心差异**：公平锁每次都先检查队列，哪怕锁是空闲的也要排队；非公平锁直接插队抢，抢不到再排队，减少线程切换所以性能更好。

### 3.6 AQS 两种模式

#### 独占模式 —— ReentrantLock
- 同一时刻只有一个线程持有锁，见上方加解锁流程

---

#### 共享模式 —— Semaphore（信号量）

> 允许**最多 N 个线程**同时访问资源

```java
Semaphore semaphore = new Semaphore(3); // state = 3，最多3个线程同时访问
semaphore.acquire(); // state - 1，为负则阻塞
semaphore.release(); // state + 1，唤醒等待线程
```

**实现原理：**
```
state = N（许可证数量）

acquire()：
  CAS(state - 1)
  state >= 0 → 获取成功，继续执行
  state <  0 → 加入CLH队列，park阻塞

release()：
  CAS(state + 1)
  唤醒CLH队列中的后继节点（共享唤醒：会连续唤醒多个节点）
```

**共享模式唤醒特点：**  
- 唤醒一个节点后，该节点还会继续唤醒它的后继节点（**传播唤醒**）
- 独占模式只唤醒队头一个，共享模式会**级联唤醒**

---

#### 共享模式 —— CountDownLatch（倒计时器）

> 让一个（或多个）线程**等待其他线程完成**后再继续

```java
CountDownLatch latch = new CountDownLatch(3); // state = 3

// 工作线程完成后调用
latch.countDown(); // state - 1

// 主线程等待所有工作线程完成
latch.await();     // state != 0 则阻塞，为 0 则放行
```

**实现原理：**
```
state = N（需要等待的线程数）

countDown()：
  CAS(state - 1)
  state == 0 → 唤醒所有 await() 阻塞的线程（共享模式级联唤醒）

await()：
  state != 0 → 加入CLH队列，park阻塞
  state == 0 → 直接通过
```

**CountDownLatch vs CyclicBarrier 对比：**

| | CountDownLatch | CyclicBarrier |
|--|---------------|---------------|
| 方向 | N个线程通知1个线程 | N个线程互相等待 |
| 重置 | ❌ 一次性 | ✅ 可重复使用 |
| 实现 | 基于AQS | 基于ReentrantLock+Condition |
| 场景 | 主线程等子线程结果 | 多线程到达同一屏障点 |

---

### 3.7 Condition 条件变量 ⭐⭐⭐

```
synchronized 只有一个等待队列（WaitSet），所有 wait() 的线程混在一起
ReentrantLock + Condition 可以创建多个等待队列，更精准控制

lock
├── Condition A  →  [等待取数据的消费者线程]
└── Condition B  →  [等待放数据的生产者线程]
     signal A   →  只唤醒消费者
     signal B   →  只唤醒生产者
```

```java
// 经典生产者-消费者模式
ReentrantLock lock = new ReentrantLock();
Condition notFull  = lock.newCondition();  // 队列不满条件
Condition notEmpty = lock.newCondition();  // 队列不空条件
Queue<Integer> queue = new LinkedList<>();
int capacity = 10;

// 生产者
public void put(int val) throws InterruptedException {
    lock.lock();
    try {
        while (queue.size() == capacity)   // 队列满了
            notFull.await();               // 等待"不满"条件
        queue.offer(val);
        notEmpty.signal();                 // 通知消费者"不空了"
    } finally { lock.unlock(); }
}

// 消费者
public int take() throws InterruptedException {
    lock.lock();
    try {
        while (queue.isEmpty())            // 队列空了
            notEmpty.await();              // 等待"不空"条件
        int val = queue.poll();
        notFull.signal();                  // 通知生产者"不满了"
        return val;
    } finally { lock.unlock(); }
}
```

**Condition 原理：**
```
每个 Condition 维护一个独立的等待队列（单向链表）
await()  → 释放锁 → 当前线程封装为 Node 加入 Condition 等待队列 → park
signal() → 将 Condition 队列头节点转移到 AQS 的 CLH 队列尾部 → 等待重新竞争锁
signalAll() → 将 Condition 队列所有节点转移到 CLH 队列
```

### 3.8 ReentrantReadWriteLock 读写锁 ⭐⭐⭐

```
场景：读多写少，synchronized/ReentrantLock 读也互斥 → 性能浪费
读写锁：读读共享、读写互斥、写写互斥

     读锁 (共享)     写锁 (独占)
读锁    ✅ 共享       ❌ 互斥
写锁    ❌ 互斥       ❌ 互斥
```

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
ReadLock readLock = rwLock.readLock();
WriteLock writeLock = rwLock.writeLock();

// 读操作（多线程可同时持有）
readLock.lock();
try { return cache.get(key); }
finally { readLock.unlock(); }

// 写操作（独占）
writeLock.lock();
try { cache.put(key, value); }
finally { writeLock.unlock(); }
```

**实现原理（⭐面试高频）：**
```
利用 AQS 的 state（32位 int）高低位分割：
  高 16 位：读锁持有次数（共享锁计数）
  低 16 位：写锁重入次数（独占锁计数）

state = 0x00020001
  → 高16位 = 2，表示有2个线程持有读锁
  → 低16位 = 1，表示写锁重入1次

锁降级：持有写锁 → 获取读锁 → 释放写锁（✅ 允许）
锁升级：持有读锁 → 获取写锁（❌ 不允许，会死锁）
```

**StampedLock（JDK8+，乐观读）：**
```java
StampedLock sl = new StampedLock();

// 乐观读（不加锁，性能最高）
long stamp = sl.tryOptimisticRead();
int x = this.x, y = this.y;         // 读取共享变量
if (!sl.validate(stamp)) {           // 检查期间是否有写操作
    stamp = sl.readLock();            // 乐观读失败，升级为悲观读锁
    try { x = this.x; y = this.y; }
    finally { sl.unlockRead(stamp); }
}
// StampedLock 不支持重入、不支持 Condition
```

### 3.9 synchronized vs ReentrantLock

| 对比项 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| 实现 | JVM内置 | Java代码+AQS |
| 公平锁 | ❌ | ✅ |
| 可中断 | ❌ | ✅ lockInterruptibly() |
| 超时获取 | ❌ | ✅ tryLock(timeout) |
| 条件变量 | 1个wait/notify | 多个Condition |
| 自动释放 | ✅ | ❌ 需手动unlock |

### 3.10 面试标准答法
> AQS 通过 volatile 的 state 表示同步状态，通过 CLH 变体双向队列管理等待线程。
> 加锁时先 CAS 修改 state，失败则封装成 Node 加入队尾并 park 挂起；
> 解锁时 state-1 为0后唤醒队列第一个节点重新竞争。
> ReentrantLock 比 synchronized 更灵活：支持公平锁、可中断、超时、多条件变量，但需手动释放。

### 3.11 常见追问

| 追问 | 关键答点 |
|------|----------|
| AQS为什么用双向链表？ | 取消节点时需找前驱，O(1)操作 |
| 为什么要虚节点Head？ | 简化队列操作，Head代表持锁线程 |
| park/unpark vs wait/notify？ | park/unpark以线程为单位，可先unpark再park，更灵活 |
| 公平锁为什么慢？ | 每次要检查队列，不能插队，线程切换更多 |
| Condition await 和 Object wait 区别？ | Condition 支持多个等待队列、可中断、超时；wait 只有一个 WaitSet |
| 读写锁 state 怎么拆分的？ | 高16位读锁计数，低16位写锁计数 |
| 什么是锁降级？为什么不支持升级？ | 持有写锁→获取读锁→释放写锁叫降级；升级会死锁（多个读锁都想升级写锁互相等待） |

## 四、ThreadLocal原理 & 内存泄漏 ⭐⭐⭐

### 4.1 ThreadLocal是什么
- 为每个线程提供**独立的变量副本**，线程间互不干扰
- 典型场景：Spring事务Connection传递、登录用户信息传递、SimpleDateFormat线程安全

### 4.2 底层数据结构
- 数据不存在 ThreadLocal 本身，存在**每个 Thread 对象内部的 ThreadLocalMap** 中
- `ThreadLocalMap` 是一个数组结构（开放地址法），Entry 的 **key 是弱引用**，**value 是强引用**

```
Thread 对象
  └── threadLocals: ThreadLocalMap
        └── Entry[] table（默认初始容量16）
              [0] = null
              [1] = Entry(WeakRef(threadLocal_A), "value_A")
              [2] = null
              [3] = Entry(WeakRef(threadLocal_B), 42)
              ···

Entry 继承 WeakReference<ThreadLocal<?>>
  key  = WeakReference(ThreadLocal 对象) → GC 可回收
  value = 强引用（Object）               → GC 不可回收
```

### 4.3 Hash 与冲突解决

```
ThreadLocalMap 用的是「开放地址法 - 线性探测」，不是 HashMap 的链表法

hash 值：每个 ThreadLocal 有个固定的 threadLocalHashCode
  private final int threadLocalHashCode = nextHashCode();
  private static final int HASH_INCREMENT = 0x61c88647; // 斐波那契散列
  → 配合 2 的幂次容量，散列极均匀

冲突处理：
  下标 = hash & (len - 1)
  如果位置被占 → 向后探测 (i+1) % len
  遇到 key=null 的过期 Entry → 顺手清理（expungeStaleEntry）
  遇到空 slot → 放入

为什么不用链表法？
  ThreadLocalMap 预期 entry 很少（几个到几十个），线性探测缓存友好、内存省
```

### 4.3 get/set 流程
- `set(v)`：获取当前线程 → 获取其 ThreadLocalMap → 以当前ThreadLocal为key存入value
- `get()`：获取当前线程 → 获取其 ThreadLocalMap → 以当前ThreadLocal为key取出value

### 4.4 内存泄漏原理 ⭐
```
tl = null（ThreadLocal对象失去强引用）
  ↓ GC回收 ThreadLocal 对象
Entry.key = null
  ↓ 但 Entry.value 仍是强引用
只要线程不结束（如线程池），value 永远无法被回收 → 内存泄漏！
```
- **线程池场景最危险**：线程复用不销毁，value 持续堆积

### 4.5 解决方案
```java
try {
    threadLocal.set(value);
    // 业务代码
} finally {
    threadLocal.remove(); // ✅ 必须在finally中清理
}
```
- key 弱引用是**保底设计**，get/set 操作会顺手清理 key=null 的 Entry，但不能完全依赖

### 4.6 InheritableThreadLocal
- 解决父子线程传值问题，创建子线程时自动拷贝父线程的值
- **缺陷**：线程池中线程复用，传递失效
- **生产方案**：使用阿里开源的 `TransmittableThreadLocal（TTL）`

### 4.7 面试标准答法
> ThreadLocal 数据存在每个 Thread 的 ThreadLocalMap 中，以 ThreadLocal 为弱引用 key，value 是强引用。
> ThreadLocal 被GC后 key=null，但 value 因强引用不能回收，线程池场景下线程不销毁导致内存泄漏。
> 解决：用完后 finally 块中调用 remove()。线程池父子传值用 TTL。

### 4.8 常见追问

| 追问 | 关键答点 |
|------|----------|
| key为什么弱引用？ | 保底让key能被GC；get/set会清理key=null的Entry |
| ThreadLocalMap和HashMap区别？ | 开放地址法 vs 链表法；Entry继承WeakReference |
| 线程池为何更容易泄漏？ | 线程不销毁，value一直存在 |
| Spring事务怎么用ThreadLocal？ | TransactionSynchronizationManager 存储当前线程Connection |

## 五、线程池核心原理 & 参数调优 ⭐⭐⭐⭐

### 5.1 七大核心参数
| 参数 | 含义 |
|------|------|
| `corePoolSize` | 核心线程数，默认一直存活（`allowCoreThreadTimeOut(true)` 可回收） |
| `maximumPoolSize` | 最大线程数 |
| `keepAliveTime` | 空闲线程存活时间（默认只约束非核心线程，开启 allowCoreThreadTimeOut 后也约束核心线程） |
| `unit` | 时间单位 |
| `workQueue` | 任务队列 |
| `threadFactory` | 线程工厂，可指定线程名 |
| `handler` | 拒绝策略 |

### 5.2 任务执行流程
```
提交任务
  │
  ├─ 线程数 < corePoolSize?
  │    └─ YES → 创建核心线程执行（即使有空闲线程也创建）
  │
  ├─ 任务入 workQueue?
  │    └─ 队列未满 → 入队等待
  │
  ├─ 线程数 < maximumPoolSize?
  │    └─ YES → 创建非核心线程执行
  │
  └─ 执行拒绝策略 handler.rejectedExecution()
```

```
⚠️ 核心理解：队列优先于非核心线程！

  常见误解：先创建到 max 线程，用不完才入队
  正确流程：core → 队列 → max → 拒绝
  
  这意味着如果用无界队列（LinkedBlockingQueue），
  maximumPoolSize 永远不会生效！因为队列永远不满。
```

### 5.3 线程池生命周期

```
               shutdown()
RUNNING ──────────────→ SHUTDOWN ──────→ TIDYING ──→ TERMINATED
   │                                       ↑
   │       shutdownNow()                   │
   └──────────────────→ STOP ──────────────┘
   
RUNNING:    接受新任务，处理队列任务
SHUTDOWN:   不接受新任务，继续处理队列中已有任务
STOP:       不接受新任务，不处理队列任务，中断执行中的任务
TIDYING:    所有任务终止，workerCount=0
TERMINATED: terminated() 钩子方法执行完成
```

```java
// shutdown vs shutdownNow
executor.shutdown();      // 温和关闭：不接受新任务，等待已提交任务完成
executor.shutdownNow();   // 强制关闭：中断所有线程，返回未执行的任务列表

// 优雅关闭的标准写法
executor.shutdown();
if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
    executor.shutdownNow();              // 超时强制关闭
    executor.awaitTermination(10, TimeUnit.SECONDS);
}
```

### 5.3 四种拒绝策略
| 策略 | 行为 | 推荐场景 |
|------|------|--------|
| AbortPolicy（默认） | 抛异常 | 快速失败 |
| **CallerRunsPolicy** | 提交线程自己执行 | **生产推荐，不丢任务+反压** |
| DiscardPolicy | 静默丢弃 | 允许丢任务 |
| DiscardOldestPolicy | 丢最老任务 | 需要最新任务 |

### 5.4 常见队列
| 队列 | 特点 | 风险 |
|------|------|------|
| ArrayBlockingQueue | 有界数组，FIFO | 安全，推荐生产使用 |
| LinkedBlockingQueue | 默认无界（可设上限） | **无界 OOM 风险** |
| SynchronousQueue | 不存储，直接移交 | 必须有空闲线程否则创建新线程 |
| PriorityBlockingQueue | 无界优先级队列 | 无界 OOM 风险 |
| DelayQueue | 延迟获取，元素需实现 Delayed | 定时任务场景 |

### 5.5 Worker 线程复用原理

```java
// ThreadPoolExecutor 核心内部类
final void runWorker(Worker w) {
    Runnable task = w.firstTask;
    w.firstTask = null;
    while (task != null || (task = getTask()) != null) {  // 循环从队列取任务
        w.lock();
        try {
            beforeExecute(thread, task);     // 钩子方法（可重写）
            task.run();                       // 执行任务
            afterExecute(task, null);         // 钩子方法
        } finally {
            task = null;
            w.completedTasks++;
            w.unlock();
        }
    }
    // getTask() 返回 null → 线程退出循环 → 线程销毁
}

// getTask() 核心逻辑
private Runnable getTask() {
    // 核心线程：workQueue.take() → 阻塞等待（永不超时）
    // 非核心线程：workQueue.poll(keepAliveTime) → 超时返回null → 线程销毁
}
```

> **线程复用本质**：Worker 线程在 while 循环中不断从 workQueue 取任务执行，取不到才退出。

### 5.6 Executors 的坑
- `newFixedThreadPool`/`newSingleThreadExecutor`：LinkedBlockingQueue 无界 → **OOM**
- `newCachedThreadPool`/`newScheduledThreadPool`：maximumPoolSize 无限 → **OOM**
- **阿里规范：禁止使用 Executors，必须手动 new ThreadPoolExecutor**

### 5.7 线程数设置
| 任务类型 | 推荐线程数 |
|---------|----------|
| CPU密集型 | CPU核数 + 1 |
| IO密集型 | CPU核数 × 2，或 CPU核数×(1 + 等待时间/计算时间) |
> 公式是参考值，最终要**压测**决定

### 5.8 线程池异常处理 ⭐⭐

```java
// execute 提交：异常直接抛出，线程销毁后创建新线程
executor.execute(() -> { throw new RuntimeException("boom"); });
// → 控制台打印异常堆栈，该 Worker 线程死亡

// submit 提交：异常封装在 Future 中，get() 时抛出
Future<?> f = executor.submit(() -> { throw new RuntimeException("boom"); });
f.get();  // 此时才抛出 ExecutionException

// 推荐方案一：任务内 try-catch
executor.execute(() -> {
    try { riskyMethod(); }
    catch (Exception e) { log.error("任务异常", e); }
});

// 推荐方案二：自定义 ThreadFactory 设置 UncaughtExceptionHandler
Thread.setDefaultUncaughtExceptionHandler((t, e) ->
    log.error("线程{}异常: {}", t.getName(), e.getMessage(), e));

// 推荐方案三：重写 afterExecute
ThreadPoolExecutor executor = new ThreadPoolExecutor(...) {
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        if (t != null) log.error("任务异常", t);
        if (r instanceof Future<?>) {
            try { ((Future<?>) r).get(); }
            catch (Exception e) { log.error("Future异常", e); }
        }
    }
};
```

```java
executor.getActiveCount();       // 活跃线程数（正在执行任务）
executor.getCompletedTaskCount();// 已完成任务数
executor.getTaskCount();         // 总任务数（已完成 + 执行中 + 队列中）
executor.getQueue().size();      // 队列中等待任务数

// 2. 告警：队列堆积 > 阈值、活跃线程 = maxPoolSize、拒绝次数增加
// 3. 线程池命名！自定义 ThreadFactory 给线程池取名，出问题方便定位
```

### 5.10 面试标准答法
> 线程池7个核心参数，执行流程：核心线程→队列→非核心线程→拒绝策略，队列优先于非核心线程。
> 禁用Executors（无界队列/无限线程数OOM风险），手动创建并命名线程。
> 拒绝策略推荐CallerRunsPolicy，不丢任务还能反压。
> CPU密集型核数+1，IO密集型核数×2，最终依赖压测。

| 追问 | 关键答点 |

| 为什么不建议无界队列？ | max 永远不生效，任务堆积导致 OOM |
| 动态调整线程池参数？ | setCorePoolSize()、setMaximumPoolSize() 可运行时动态修改 |

## 六、ConcurrentHashMap 原理 ⭐⭐⭐⭐

### 6.1 JDK7 vs JDK8 对比

| | JDK7 | JDK8 |
|--|------|------|
| 实现 | 分段锁 Segment | CAS + synchronized |
| 锁粒度 | Segment级（16个 | 桶（数组槽位）级 |
| 数据结构 | 数组+链表 | 数组+链表+红黑树 |
| 并发度 | 最多16 | 数组长度（默认16） |

### 6.2 JDK8 put 流程（⭐面试重点）
```
① 计算 hash → spread(key.hashCode())
   (h ^ (h >>> 16)) & HASH_BITS  → 高低位混合，减少冲突

② 定位桶 → tab[(n-1) & hash]

③ 判断桶状态：
   桶为空 → CAS 插入新 Node（无锁操作，最快路径）
   桶是 ForwardingNode（hash=-1）→ 帮助扩容，再重试
   桶不为空 → synchronized(桶头节点) {
       链表：遍历链表，key 相同则更新 value，否则尾插
       红黑树：TreeBin 内部加锁后走红黑树插入
   }

④ 链表长度 > 8?
   数组长度 < 64 → 先扩容（tryPresize）
   数组长度 ≥ 64 → 转红黑树（treeifyBin）

⑤ addCount() → 更新元素计数 → 检查是否需要扩容
```

### 6.3 get 全程无锁
- Node 的 `val` 和 `next` 均为 **volatile**，保证可见性，get 无需加锁

### 6.4 多线程协助扩容 ⭐⭐⭐

```
扩容触发：元素数 > capacity × loadFactor（0.75）

扩容过程（transfer方法）：
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│  ✅  │  ✅  │  ✅  │ 🔄  │ 🔄  │      │      │      │  旧数组
│ FWD  │ FWD  │ FWD  │ T2   │ T1   │ 待分配│ 待分配│ 待分配│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
                       ↓     ↓
┌─────────────────────────────────────────────────────────┐
│                       新数组（2倍大小）                    │
└─────────────────────────────────────────────────────────┘

分配机制（stride，步长）：
  每个线程负责一段连续的桶（最小步长 16）
  从数组尾部向头部领取任务（CAS 修改 transferIndex）
  T1 领取 [6,7]，T2 领取 [3,5]，以此类推

迁移完成标记：
  迁移完的桶放置 ForwardingNode（hash=-1，指向新数组）
  → 读操作：遇到 FWD 去新数组查
  → 写操作：遇到 FWD 先帮助迁移（helpTransfer），再重试 put

优势：多线程并行迁移不同的桶段，大数组扩容不阻塞
```

### 6.5 size() 实现 —— LongAdder 思路
```
ConcurrentHashMap 的 size 计数拆分为多个 Cell 分散 CAS 热点：

  baseCount（无竞争时直接 CAS 累加）
  + CounterCell[] 数组（竞争时分散到不同 Cell）

put/remove 后调用 addCount()：
  ① 先 CAS 修改 baseCount
  ② CAS 失败（有竞争）→ 随机选一个 CounterCell CAS 累加
  最终 size = baseCount + Σ CounterCell[i].value

原理同 LongAdder：空间换时间，分散热点
```

### 6.6 面试标准答法
> JDK8 的 ConcurrentHashMap 用 CAS + synchronized（锁桶头节点）替代 JDK7 的分段锁，锁粒度从 Segment 细化到每个桶。
> put 流程：hash 定位桶 → 空桶 CAS 无锁插入 → 非空桶 synchronized 加锁插入 → 链表 > 8 且数组 ≥ 64 转红黑树。
> 扩容多线程协助，每个线程按步长领取桶段并行迁移，已迁移的桶放 ForwardingNode。
> size 用 baseCount + CounterCell[] 分散 CAS 统计，get 全程无锁（volatile 保证可见性）。

### 6.7 不允许 null key/value 原因
- 多线程下无法区分“key不存在”还是“value为null”

### 6.8 常见追问
| 追问 | 关键答点 |
|------|----------|
| 链表长度>8才转红黑树？ | 泊松分布，超8概率极低；<8时链表操作更快 |
| 红黑树何时退化回链表？ | 节点数 < 6 时退化，留缓冲避免频繁转换 |
| JDK7头插法有什么问题？ | 并发扩容导致链表成环，get时死循环；JDK8改尾插法 |
| 能保证复合操作原子性？ | 不能，需要用 putIfAbsent()/compute()/merge() 等原子方法 |
| 为什么 get 不需要加锁？ | Node 的 val 和 next 是 volatile，保证可见性 |
| 扩容时 put 怎么处理？ | 遇到 ForwardingNode 先帮助迁移，再重试 |
| transferIndex 是什么？ | 控制扩容进度的原子变量，线程 CAS 领取桶段任务 |

## 七、happens-before 规则 ⭐⭐⭐

### 7.1 是什么
- JMM（Java内存模型）定义的**内存可见性保证规则**
- A happens-before B：A的操作结果对B可见，不是时间顺序而是**可见性保证**
- 是程序员与JMM的**契约**：遵守规则就保证可见性

### 7.2 八大规则

| 规则 | 内容 |
|------|------|
| **程序顺序规则** | 同一线程内，前面操作 hb 后面操作 |
| **volatile变量规则** | volatile写 hb 同一变量的volatile读 |
| **监视器锁规则** | 解锁 hb 后续加锁 |
| **线程启动规则** | start() hb 子线程任何操作 |
| **线程终止规则** | 子线程所有操作 hb join()返回 |
| **线程中断规则** | interrupt() hb 被中断线程检测到中断 |
| **对象终结规则** | 构造函数 hb finalize() |
| **传递性规则** | A hb B，B hb C ⇒ A hb C |

### 7.3 经典题：flag未加volatile时线程B能看到x=42吗？
```java
int x = 0;
boolean flag = false;
// 线程A：x=42; flag=true;
// 线程B：if(flag) use(x);
```
- **不一定！** flag未加volatile，x=42与flag=true可能重排序
- **修复**：flag加volatile → volatile写 hb volatile读 → 传递性保证x=42对B可见

### 7.4 面试标准答法
> happens-before 是 JMM 与程序员的可见性契约。8条规则中最关键的是volatile规则、监视器锁规则和传递性。
> volatile可见性的本质就是volatile变量规则+传递性共同保证的。
> 时间先后不能保证可见性，只有happens-before才能保证。

### 7.5 常见追问

| 追问 | 关键答点 |
|------|---------|
| hb和volatile的关系？ | volatile是实现hb规则的手段之一 |
| 时间先后能保证可见性吗？ | 不能，只有hb才能保证 |
| synchronized为什么能保证可见性？ | 监视器锁规则：解锁 hb 加锁 |
| 传递性有什么实际意义？ | 可通过volatile变量"桥接"传递可见性 |

## 八、线程基础与生命周期 ⭐⭐⭐

### 8.1 创建线程的方式

| 方式 | 特点 |
|------|------|
| `extends Thread` | 简单但无法继承其他类 |
| `implements Runnable` | 推荐，接口可多实现，分离任务和线程 |
| `implements Callable<V>` + `FutureTask` | 有返回值、可抛异常 |
| **线程池 `ExecutorService`** | ⭐ 生产唯一推荐，复用线程，统一管理 |

```java
// Callable + FutureTask
Callable<Integer> task = () -> { return 42; };
FutureTask<Integer> ft = new FutureTask<>(task);
new Thread(ft).start();
int result = ft.get();  // 阻塞获取结果
```

> **面试回答**：严格来说只有 `new Thread().start()` 才真正创建线程，Runnable/Callable 只是任务定义。生产环境用线程池。

### 8.2 线程生命周期（6 种状态）

```
                   ┌──────────────────────────────────┐
                   │                                  │
    NEW ──start()──→ RUNNABLE ──────────────→ TERMINATED
                     │      ↑                    
                     │      │                    
            ┌────────┘      └────────┐           
            ▼                        │           
     ┌──────────────┐        ┌──────────────┐    
     │   BLOCKED    │        │   WAITING     │    
     │ 等待获取锁    │        │ wait()/join() │    
     └──────┬───────┘        │ park()        │    
            │                └──────┬───────┘    
            │                       │            
            │                ┌──────────────┐    
            │                │TIMED_WAITING │    
            │                │sleep()/wait(n)│   
            │                │ join(n)/park(n)│   
            │                └──────┬───────┘    
            └───── 获取锁 ──────────┘            
```

| 状态 | 触发条件 | 说明 |
|------|---------|------|
| **NEW** | `new Thread()` | 创建但未启动 |
| **RUNNABLE** | `start()` | 就绪 + 运行（Java 不区分） |
| **BLOCKED** | 等待 synchronized 锁 | 只有 synchronized 才触发此状态 |
| **WAITING** | `wait()` / `join()` / `LockSupport.park()` | 无限等待，需其他线程唤醒 |
| **TIMED_WAITING** | `sleep(n)` / `wait(n)` / `join(n)` | 有超时的等待 |
| **TERMINATED** | run() 执行完/异常退出 | 终态 |

> ⚠️ **BLOCKED vs WAITING**：BLOCKED 是等 synchronized 的锁；WAITING 是主动调用 wait/park。ReentrantLock 中等待锁的线程是 **WAITING** 状态（park），**不是** BLOCKED。

### 8.3 线程核心方法对比

| 方法 | 释放锁？ | 所属类 | 唤醒方式 |
|------|---------|-------|---------|
| `sleep(ms)` | ❌ 不释放 | Thread | 时间到 / interrupt |
| `wait()` | ✅ 释放 | Object | notify/notifyAll |
| `join()` | - | Thread | 等待目标线程结束 |
| `yield()` | ❌ | Thread | 让出时间片（不可靠） |
| `park()` | ❌ | LockSupport | unpark(thread) |

### 8.4 线程中断机制 ⭐⭐

```java
// interrupt() 不是强制终止线程！只是设置中断标志位

// 情况1：线程正在运行
thread.interrupt();           // 只设置标志位，线程需主动检查
while (!Thread.interrupted()) { // 检查并清除中断标志
    // 工作代码
}

// 情况2：线程在 sleep/wait/join 阻塞中
thread.interrupt();           // 抛出 InterruptedException，中断标志被清除
// → 需要在 catch 中决定：重新设置中断标志 or 退出

// 正确处理中断
try {
    Thread.sleep(1000);
} catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // ⭐ 恢复中断标志，让上层感知
    // 或者直接退出循环
}
```

**三个方法区别：**
| 方法 | 作用 |
|------|------|
| `thread.interrupt()` | 设置中断标志位（实例方法） |
| `Thread.interrupted()` | 返回中断状态并**清除**标志（静态方法） |
| `thread.isInterrupted()` | 返回中断状态，**不清除**标志（实例方法） |

### 8.5 面试标准答法
> 线程 6 种状态：NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED。
> BLOCKED 只针对 synchronized 锁等待，ReentrantLock 等待的线程是 WAITING（用 park）。
> sleep 不释放锁，wait 释放锁。中断不是强制终止，只是设标志位，线程需主动检查或在阻塞时被 InterruptedException 唤醒。
> 生产创建线程唯一推荐方式是线程池。

### 8.6 常见追问
| 追问 | 关键答点 |
|------|----------|
| start() 和 run() 的区别？ | start 创建新线程执行 run；直接调 run 是当前线程执行普通方法 |
| 线程能 start 两次吗？ | 不能，第二次抛 IllegalThreadStateException |
| 守护线程是什么？ | setDaemon(true)，所有非守护线程结束后 JVM 退出，守护线程随之终止 |
| 为什么 wait 在 Object 而不是 Thread？ | wait/notify 基于 Monitor 锁，锁属于对象不属于线程 |
| 如何优雅停止线程？ | volatile boolean 标志位 + interrupt()，或 Future.cancel(true) |

## 九、并发工具类与原子操作 ⭐⭐⭐⭐

### 9.1 Atomic 原子类 ⭐⭐⭐

#### CAS 原理（Compare And Swap）

```
CAS(V, Expected, New)
  V = 内存中的值
  Expected = 期望值
  New = 新值
  
  if (V == Expected)
      V = New; return true;    // 原子操作，CPU 指令级保证
  else
      return false;            // 值已被其他线程修改

底层：sun.misc.Unsafe 类 → 调用 CPU 的 cmpxchg 指令（x86）
```

**CAS 三大问题：**

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| **ABA 问题** | 值从 A→B→A，CAS 认为没变 | `AtomicStampedReference`（版本号）|
| **自旋开销** | 竞争激烈时一直自旋浪费 CPU | LongAdder 分散热点 / 适当退让 |
| **只能保证单变量原子** | 多个变量无法同时 CAS | AtomicReference 包装对象 / 锁 |

#### 常用原子类

| 类别 | 类 | 说明 |
|------|-----|------|
| 基本类型 | `AtomicInteger` / `AtomicLong` / `AtomicBoolean` | 单变量原子操作 |
| 引用类型 | `AtomicReference<V>` | 原子更新引用 |
| ABA 解决 | `AtomicStampedReference<V>` | 带版本号的引用 |
| 数组 | `AtomicIntegerArray` | 数组元素原子操作 |
| 字段更新 | `AtomicIntegerFieldUpdater` | 原子更新对象的 volatile 字段 |
| **高性能计数** | `LongAdder` / `LongAccumulator` | ⭐ 分散热点，高并发计数首选 |

#### LongAdder vs AtomicLong ⭐⭐⭐

```
AtomicLong：所有线程 CAS 同一个 value → 竞争激烈时大量自旋
LongAdder：分散到多个 Cell，最后求和

              AtomicLong                   LongAdder
         ┌──────────────┐          ┌───────────────────┐
         │   value=100  │          │  base = 50        │
         │  所有线程CAS  │          │  Cell[0] = 20     │
         │  同一个变量   │          │  Cell[1] = 15     │
         │ （竞争激烈）  │          │  Cell[2] = 15     │
         └──────────────┘          │  sum() = 100      │
                                   └───────────────────┘

LongAdder 适合：高并发写（计数器、统计），不需要精确实时值
AtomicLong 适合：需要精确值（如全局序列号），并发不太高
```

### 9.2 CompletableFuture ⭐⭐⭐⭐

> JDK8 引入的异步编程利器，支持链式编排和组合

```java
// ===== 创建异步任务 =====
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> {
    return queryFromDB();     // 异步执行（ForkJoinPool.commonPool）
});

// ===== 链式转换 =====
cf.thenApply(result -> process(result))     // 同步转换，同线程
  .thenApplyAsync(r -> format(r))           // 异步转换，新线程
  .thenAccept(r -> System.out.println(r))   // 消费结果，无返回值
  .thenRun(() -> log("done"));              // 执行动作，不关心结果

// ===== 异常处理 =====
cf.exceptionally(ex -> "default")           // 异常时返回默认值
  .handle((result, ex) -> {                 // 统一处理结果和异常
      return ex != null ? "error" : result;
  });

// ===== 组合多个异步任务 =====
CompletableFuture<String> cf1 = CompletableFuture.supplyAsync(() -> queryUser());
CompletableFuture<String> cf2 = CompletableFuture.supplyAsync(() -> queryOrder());

// 两个都完成后合并
cf1.thenCombine(cf2, (user, order) -> user + order);

// 任意一个完成即可
CompletableFuture.anyOf(cf1, cf2).thenAccept(r -> use(r));

// 全部完成
CompletableFuture.allOf(cf1, cf2).thenRun(() -> all done);
```

**实战：并行调用多个接口汇总：**

```java
// 串行：200ms + 300ms + 150ms = 650ms
// 并行：max(200, 300, 150) = 300ms

CompletableFuture<User> userCf = CompletableFuture.supplyAsync(
    () -> userService.getUser(userId), executor);
CompletableFuture<List<Order>> orderCf = CompletableFuture.supplyAsync(
    () -> orderService.getOrders(userId), executor);
CompletableFuture<Account> accountCf = CompletableFuture.supplyAsync(
    () -> accountService.getAccount(userId), executor);

CompletableFuture.allOf(userCf, orderCf, accountCf).join();

// 汇总结果
UserVO vo = new UserVO();
vo.setUser(userCf.get());
vo.setOrders(orderCf.get());
vo.setAccount(accountCf.get());
```

> ⚠️ **生产注意**：不要用默认的 `ForkJoinPool.commonPool()`，应传入自定义线程池。

### 9.3 ForkJoinPool 分治框架

```
任务拆分与合并（分治思想）：

         大任务
        /      \
    子任务1    子任务2
    /    \    /    \
   T1   T2  T3   T4  ← 足够小，直接计算
    \   /    \   /
     合并     合并
        \    /
         结果

工作窃取（Work Stealing）：
  每个线程有自己的双端队列
  线程空闲时从其他线程队列尾部"偷"任务
  → 提高 CPU 利用率，减少线程空闲
```

```java
// 并行求和示例
class SumTask extends RecursiveTask<Long> {
    private final long[] arr;
    private final int lo, hi;
    static final int THRESHOLD = 1000;

    protected Long compute() {
        if (hi - lo <= THRESHOLD) {
            long sum = 0;
            for (int i = lo; i < hi; i++) sum += arr[i];
            return sum;
        }
        int mid = (lo + hi) / 2;
        SumTask left = new SumTask(arr, lo, mid);
        SumTask right = new SumTask(arr, mid, hi);
        left.fork();            // 异步执行左半部分
        long rightResult = right.compute(); // 当前线程执行右半部分
        long leftResult = left.join();      // 等待左半部分结果
        return leftResult + rightResult;
    }
}
```

> **并行流（parallelStream）底层就是 ForkJoinPool**。生产中避免在 parallelStream 中做 IO 操作。

### 9.4 并发集合概览

| 类 | 线程安全方式 | 适用场景 |
|----|------------|---------|
| `ConcurrentHashMap` | CAS + synchronized | 读多写少 Map |
| `CopyOnWriteArrayList` | 写时复制（COW） | 读极多写极少 List |
| `ConcurrentLinkedQueue` | CAS 无锁 | 高并发队列 |
| `BlockingQueue` | Lock + Condition | 生产者-消费者 |
| `ConcurrentSkipListMap` | 跳表实现有序 Map | 需要有序 + 线程安全 |

**CopyOnWriteArrayList 原理：**
```
写操作：
  ① ReentrantLock 加锁
  ② 复制一份新数组
  ③ 在新数组上修改
  ④ 将引用指向新数组（volatile 保证可见性）
  ⑤ 解锁

读操作：
  直接读旧数组，无锁，完全不阻塞

特点：读不加锁性能极高，但写要复制数组开销大
适用：读多写极少的场景（如事件监听器列表、配置列表）
```

### 9.5 面试标准答法
> CAS 是并发基石，通过 CPU 指令保证比较+交换的原子性，问题是 ABA（用 AtomicStampedReference 解决）和自旋开销（LongAdder 分散热点）。
> CompletableFuture 是 JDK8 异步编程核心，支持链式转换、异常处理、多任务组合（allOf/anyOf），生产中要用自定义线程池。
> ForkJoinPool 基于分治+工作窃取，parallelStream 底层使用。

### 9.6 常见追问
| 追问 | 关键答点 |
|------|----------|
| CAS 底层怎么保证原子性？ | CPU 的 cmpxchg 指令 + 缓存锁/总线锁 |
| ABA 问题怎么解决？ | AtomicStampedReference 加版本号（stamp） |
| LongAdder 为什么快？ | 分散 CAS 到多个 Cell，最后 sum，减少竞争 |
| CompletableFuture 默认线程池？ | ForkJoinPool.commonPool()，生产要换自定义线程池 |
| CopyOnWriteArrayList 适合什么场景？ | 读极多写极少，如监听器列表。写多不适合（频繁复制数组开销大） |
| parallelStream 的坑？ | 默认共用 commonPool，IO 操作会阻塞其他任务 |
