## 一、设计秒杀系统 ⭐⭐⭐⭐⭐

> 字节必考题，考察高并发/架构设计/一致性/稳定性综合能力

### 1.1 需求分析与挑战

```
业务特征：
  瞬时高并发（10万+/s）、持续时间短（10~30分钟）
  商品库存极少（100件）、请求量极大（100万用户抢）
  结果最终：库存不能超卖，不能少卖

核心挑战：
  ① 超卖问题：并发扣库存出现负数
  ② 系统压垮：DB/服务扛不住瞬时流量
  ③ 恶意刷单：机器人重复提交
  ④ 数据一致性：库存扣减 + 订单创建 + 支付 跨操作一致性
```

---

### 1.2 整体架构（漏斗模型）

```
流量入口
  ↓ 10,000,000 rps（前端拦截90%）
前端层：
  ① 按钮置灰（点击后禁用，防重复提交）
  ② 倒计时开始前页面静态化（CDN托管）
  ③ 随机Token：开秒杀前下发一次性 token，请求必须携带

  ↓ 1,000,000 rps（CDN过滤静态资源）
接入层（Nginx）：
  ① IP限流（单IP每秒最多5次）
  ② 黑名单过滤
  ③ 请求合法性校验（黑名单/签名）

  ↓ 100,000 rps（Nginx层拦截80%）
网关层（Gateway）：
  ① 用户级别限流（同一userId令牌桶，每秒1个）
  ② 全局限流（令牌桶，只放 2×库存量 的请求进后端）
  ③ 接口鉴权（JWT验证）

  ↓ 200 rps（只有库存量2倍的请求）
秒杀服务：
  ① 本地缓存预判（AtomicInteger本地库存标识）
  ② Redis原子扣减库存（Lua脚本）
  ③ 写入MQ（异步创建订单）

  ↓ 100 rps（真实成功扣减库存的请求）
订单服务（MQ消费）：
  ① 幂等校验
  ② 创建订单（DB写入）
  ③ 发起支付流程

  ↓
数据库
```

### 1.3 防超卖方案（最核心）⭐⭐⭐⭐⭐

**方案一：Redis Lua 原子扣减（推荐）**

```lua
-- KEYS[1]=库存key, ARGV[1]=扣减数量, ARGV[2]=userId（幂等）
local stock_key = KEYS[1]
local uid_key   = KEYS[1] .. ":uids"  -- 记录已购用户
local count     = tonumber(ARGV[1])
local uid       = ARGV[2]

-- 幂等：同一用户不能重复购买
if redis.call('SISMEMBER', uid_key, uid) == 1 then
    return -2  -- 已购买
end

-- 查询当前库存
local stock = tonumber(redis.call('GET', stock_key))
if stock == nil or stock < count then
    return -1  -- 库存不足
end

-- 原子扣减库存 + 记录用户
redis.call('DECRBY', stock_key, count)
redis.call('SADD', uid_key, uid)
redis.call('EXPIRE', uid_key, 86400)  -- uid集合1天过期

return 1  -- 成功
```

```java
// Java 调用
public SeckillResult doSeckill(Long userId, Long skuId, int count) {
    String stockKey = "seckill:stock:" + skuId;
    
    // 第一道屏障：本地标志位（AtomicBoolean），库存为0直接拒绝
    if (soldOutMap.get(skuId)) {
        return SeckillResult.soldOut();
    }
    
    // 第二道屏障：Redis Lua 原子扣减
    Long result = redisLua.execute(SECKILL_SCRIPT, stockKey, count, userId.toString());
    
    if (result == -1L) {
        soldOutMap.put(skuId, true);  // 更新本地标志，后续直接拒绝
        return SeckillResult.soldOut();
    }
    if (result == -2L) {
        return SeckillResult.alreadyBought();
    }
    
    // 第三道屏障：写MQ，异步创建订单
    mqProducer.send(new SeckillOrderMessage(userId, skuId, count));
    return SeckillResult.success("排队中，请稍候...");
}
```

UPDATE seckill_stock

### 1.4 库存预热

```
秒杀开始前（提前 5~10 分钟）：
  ① 将本次秒杀的库存信息从 DB 加载到 Redis
     SET seckill:stock:skuId 100 EX 3600

  ② 将商品信息（标题/价格/图片URL）缓存到 Redis/CDN
     → 秒杀期间不读 DB

  ③ 下发限量 Token 给排队用户（可选）：
     活动开始前10分钟允许用户排队
     每个用户获得一个随机 token（存 Redis，1小时有效）
     没有 token 的请求在 Gateway 层直接拦截

库存预热一致性：
  预热后 Redis 库存 = DB 库存
  秒杀结束后：对账脚本比较 Redis 最终库存 与 DB 实际订单数
  若不一致 → 触发补偿（退款/补单）
```

---

### 1.5 异步下单流程

```
Redis 扣减成功 → 发送 MQ（RocketMQ）→ 订单消费者处理

MQ 消息内容：{ userId, skuId, count, requestId（幂等key）}

订单消费者：
  ① 幂等校验：SELECT * FROM order WHERE request_id = ? 
               若已存在 → 直接 ACK，不重复处理

  ② 创建订单：
     INSERT INTO orders (user_id, sku_id, status=PENDING_PAY, ...)

  ③ 锁定库存（DB 层二次确认）：
     UPDATE seckill_stock SET locked = locked + count WHERE sku_id = ?

  ④ 发起支付流程（调用支付服务或发支付MQ）

  ⑤ 支付超时处理（30分钟未支付自动取消）：
     延迟消息（RocketMQ 延迟消息 30min）
     到期后检查订单状态，未支付则：
       → 订单状态 → CANCELLED
       → Redis 库存还原（INCRBY）
       → DB 库存还原
```

---

### 1.6 超卖兜底与对账

```
多层防超卖：
  Layer1：前端限流（用户体验）
  Layer2：Gateway 令牌桶（控制进后端流量 ≤ 2×库存）
  Layer3：本地 AtomicBoolean 快速拒绝（已售罄标志）
  Layer4：Redis Lua 原子扣减（核心防线）
  Layer5：DB 乐观锁（最后兜底）

对账补偿（定时任务，秒杀结束后执行）：
  比较 DB 中实际成功订单数 vs 秒杀库存数
  若订单数 > 库存数 → 超卖！→ 对多出的订单退款+取消
  若订单数 < 库存数 → 少卖（可能有取消的） → 检查是否需要补卖
```

---

### 1.7 限流设计详解

```
层次限流（漏斗，每层过滤大部分流量）：

① Nginx 层（IP 限流）：
   limit_req_zone $binary_remote_addr zone=seckill:10m rate=10r/s;
   单IP每秒10次，防刷

② Gateway 层（用户限流）：
   userId 维度：令牌桶，每用户每秒1次（Redis token bucket）
   全局维度：令牌桶，总QPS = 库存量 × 2（只放入真实可能成功的请求）

③ 应用层（本地快速拒绝）：
   AtomicBoolean soldOut：已售罄时不再打 Redis，直接返回

④ Redis 层：
   Lua 脚本原子操作，天然串行无并发问题

⑤ MQ 层（背压）：
   消费者按处理能力消费，DB 不会被瞬时压垮
```

---

### 1.8 热点商品问题

```
超级热点（某个 skuId 的 Redis key QPS 极高）：
  → 单个 Redis 节点成为瓶颈（Redis 单线程，极限 10w QPS）

解法：Redis 热点分片（本地缓存 + 集群分散）
  ① 本地缓存 AtomicInteger 本地库存（每节点分配一定份额）：
     10台应用节点，总库存100，每台预分配10个本地配额
     本地原子扣减，扣完再领下一批（减少 Redis 压力）
     
  ② Redis 库存分桶：
     seckill:stock:skuId:0 ~ seckill:stock:skuId:9（10个桶）
     请求随机打到一个桶，每桶存 stock/10 的库存
     → Redis 压力分散到多个 key（可落到不同节点）

  ③ 库存合并：
     某桶用完，原子 GETSET 从其他桶"借"库存（需谨慎处理一致性）
```

### 1.9 面试标准答法

**Q: 秒杀系统如何设计？重点讲防超卖和限流。**

> 秒杀系统的核心是**漏斗模型**：用多层拦截把 100 万请求缩减到真实库存量级再落库。  
> **防超卖**：核心用 Redis Lua 脚本原子扣减，Lua 保证"查库存 + 扣减 + 记录用户"的原子性；同时在应用层维护本地 `AtomicBoolean` sold-out 标志，一旦 Redis 返回库存不足就不再打 Redis；数据库层用乐观锁做兜底，秒杀结束后对账补偿。  
> **限流**：Nginx IP 限流 → Gateway 用户级令牌桶（每人每秒1次）+ 全局令牌桶（只放库存2倍流量进后端） → 应用层本地标志快速拒绝 → 成功进入的请求写 MQ 异步下单，DB 不直接面对洪峰。  
> **超时取消**：用 RocketMQ 延迟消息，30分钟后检查未支付订单自动取消并回滚库存。

### 1.10 常见追问

**Q: Redis 扣减成功但 MQ 发送失败怎么办？**
> 出现了不一致：Redis 库存减少但订单未创建。解法：① 发 MQ 失败时立即 `INCRBY` 还原 Redis 库存，然后返回失败给用户（简单但有原子性问题）；② 使用 RocketMQ 事务消息：先发半消息，发送成功后扣 Redis，本地事务结果 Commit/Rollback；③ 更稳的做法：先写本地 DB 的 seckill_request 表（状态=PENDING）+ Redis 扣减 在同一本地事务中无法做到（跨中间件），实践上用"本地消息表" + 定时扫描补发 MQ。

**Q: 同一用户能重复购买吗？如何防止？**
> Redis Lua 脚本中用 `SADD` 将 userId 加入已购集合，`SISMEMBER` 校验是否已购。但这只是 Redis 层的防重，DB 层也需要建唯一索引 `UNIQUE(user_id, sku_id, activity_id)`，防止 MQ 重试时重复写入订单。两层配合：Redis 快速拦截，DB 唯一索引兜底保正确。

**Q: 如果 Redis 宕机，秒杀能继续吗？**
> Redis 宕机后 Lua 扣减失败，应触发降级：① 熔断器（Sentinel）检测到 Redis 异常，自动降级到 DB 乐观锁扣减（性能差但可用）；② 降级的同时严格限流（Gateway 把进后端流量降到 DB 能承受的量，如 100 QPS）；③ Redis 恢复后自动切回，同时将降级期间的 DB 库存同步回 Redis。关键是降级期间流量必须严格管控，否则 DB 扛不住。

**Q: 秒杀商品超级热点，Redis 单 key QPS 达到瓶颈怎么办？**
> Redis 单线程对单 key 的极限约 10w QPS，超级热点会成为瓶颈。解法：① 应用层本地分配库存配额（每节点 AtomicInteger，本地先扣，扣完再到 Redis 批量领），大幅减少 Redis 请求；② Redis 库存分桶（`stock:skuId:0` ~ `stock:skuId:N`，请求随机分散到多个 key，落到不同 Redis 节点）；③ 结合前两点：本地先消费，本地库存不足再从 Redis 某个桶批量补充（每次从 Redis 领取 min-batch 个，减少 Redis 交互次数）。

## 二、设计短链接系统

### 1. 需求分析

**功能需求：**
- 输入长 URL，返回短链（如 `https://t.co/AbCdEf`）
- 访问短链，302 跳转到原始长 URL
- 支持短链有效期（永久 / 自定义过期时间）
- 支持访问量统计（PV / UV）

**非功能需求：**
- 高可用（SLA 99.99%）
- 跳转延迟 < 10ms（核心体验）
- 读写比例约 100:1（重读轻写）
- 日生成短链约 1000w 条，存量 100 亿条

### 2. 核心设计

#### 2.1 短码生成方案对比

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **自增 ID + 62 进制编码** | DB/Redis INCR → 转 62 进制（0-9A-Za-z） | 无冲突，有序，实现简单 | 规律性强，可被猜测 |
| **MurmurHash 截取** | 对长 URL 哈希，取低 32 位转 62 进制 | 无序，难预测 | 有极低概率冲突，需检测重试 |
| **预分配号段** | Segment 服务批量申领 ID 段（如每次领 10w 条），本地消费 | 无 DB 单点压力 | 重启丢失号段末尾 |

**推荐：自增 ID + 62 进制（生产主流）**

```java
// 62进制编码：ID → 短码
private static final String CHARS = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";

public static String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.insert(0, CHARS.charAt((int)(id % 62)));
        id /= 62;
    }
    // 补齐6位（62^6 ≈ 568亿，够用）
    while (sb.length() < 6) sb.insert(0, '0');
    return sb.toString();
}
```

#### 2.2 数据库设计

```sql
CREATE TABLE short_link (
    id          BIGINT       PRIMARY KEY AUTO_INCREMENT,  -- 即为短链 ID
    short_code  VARCHAR(10)  NOT NULL UNIQUE,             -- 6位短码
    long_url    VARCHAR(2048) NOT NULL,                   -- 原始长URL
    user_id     BIGINT       NOT NULL,                    -- 创建者
    expire_time DATETIME     DEFAULT NULL,                -- NULL=永不过期
    click_count BIGINT       DEFAULT 0,                   -- 点击数（异步写）
    create_time DATETIME     DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_expire_time (expire_time)                   -- 过期清理扫表
);
```

#### 2.3 整体架构

```
用户请求 https://t.co/AbCdEf
    │
    ▼
[Nginx] → [Gateway 限流]
    │
    ▼
[短链跳转服务]
    │ 1. 查 Caffeine 本地缓存（命中率 ~80%）
    │ 2. 未命中 → 查 Redis（string: short_code → long_url，TTL与链接过期对齐）
    │ 3. 未命中 → 查 MySQL → 回填 Redis + Caffeine
    │ 4. 返回 302 Location: long_url
    │
    ▼
[统计服务]（异步，MQ 消费，批量写 click_count）
```

#### 2.4 301 vs 302 选型

| | 301 永久重定向 | 302 临时重定向 |
|---|---|---|
| 浏览器缓存 | ✅ 缓存，后续直接跳，不走服务 | ❌ 每次走服务 |
| 统计能力 | ❌ 缓存后无法统计 PV/UV | ✅ 每次经过服务，可精确统计 |
| CDN 压力 | 低 | 高 |
| **选型结论** | 纯跳转场景（不需统计）| **有统计需求用 302** |

#### 2.5 缓存防穿透

短链被大量请求不存在的 code（恶意）：

```
1. Bloom Filter 预热全量 short_code（FP率0.1%，内存 = 100亿 * 约1.2字节 ≈ 12GB）
2. 请求先过 Bloom Filter，不存在直接返回 404，不打 DB
3. 对合法但已过期的短链：Redis 存特殊值（"EXPIRED"），避免反复穿透
```

#### 2.6 高可用设计

- **短链生成服务**：无状态，水平扩展；ID 生成依赖 Redis INCR（主从 + Sentinel）
- **跳转服务**：多级缓存，Redis 宕机降级直查 DB（限流保护 DB）
- **数据库**：MySQL 主从读写分离；冷热分离（3 个月未访问的链接迁移 HBase）
- **过期清理**：定时任务扫 expire_time < NOW() 的记录，先逻辑删除再物理删除

### 3. 面试标准答法

> 短链系统本质是 KV 存储 + 跳转服务。核心挑战有三：① 短码要唯一且短——用 DB 自增 ID 转 62 进制编码，6 位可表示 568 亿条；② 跳转要极低延迟——用 Caffeine 本地缓存 + Redis 两级缓存，命中率达 99%，延迟 < 5ms；③ 统计要准确——用 302 每次走服务，MQ 异步批量写统计，不影响主链路。另外用 Bloom Filter 防止不存在短码的穿透攻击。301 vs 302 的本质区别是"缓存 vs 统计"，有统计需求必须用 302。

### 4. 常见追问

**Q: 如果同一个长 URL 被多次创建，会生成多个短码吗？**
> 取决于业务：① 允许重复创建（简单，每次自增 ID 不同）；② 去重（先对 long_url 做哈希查 DB/Redis，有则复用，无则新建）。去重方案要注意并发下可能双写，加分布式锁或数据库唯一索引 + allowDuplicateKeyException 处理。

**Q: 100 亿条数据怎么存储？MySQL 单表扛不住怎么办？**
> 按 short_code 哈希分 1024 张表（分库分表）或直接上 TiDB/OceanBase。分表键选 short_code 而非 user_id，因为跳转是按 short_code 查询最频繁的路径。另外 3 个月未访问的冷数据迁到 HBase/OSS，MySQL 只保留热数据（约 10 亿条）。

**Q: 短链被刷，统计数据失真怎么处理？**
> ① 对同一 IP + 短码，1 小时内只统计 1 次（Redis `SETEX ip:code 3600 1`，存在则过滤）；② UV 统计用 Redis HyperLogLog（`PFADD` + `PFCOUNT`，误差 < 0.81%，内存 12KB/key）；③ 异常流量检测：QPS 突增超设定阈值触发告警，人工审核短链。

**Q: 短链过期后如何清理？**
> 两种方式叠加：① 访问时实时检测 expire_time，过期返回 410 Gone 并异步发 MQ 触发删除；② 离线定时任务每天凌晨扫 expire_time 在未来 24h 内的记录，批量逻辑删除（`is_deleted=1`），次日物理删除。两者配合，不遗漏也不立即删除（给访问中的请求缓冲时间）。

**Q: 如何设计短链的自定义域名功能？**
> 租户维度存储 `tenant_id → custom_domain` 映射，Nginx 根据域名（SNI）路由到对应租户的短链服务。短码全局唯一（包含 tenant 前缀，如 `t1_AbCdEf`，解析时先提取 tenant_id 再查该租户的短链映射表），或者每个租户独立的短码空间（隔离性更好，但存储多 DB 实例）。

## 三、设计消息推送系统

### 1. 需求分析

**功能需求：**
- 支持多渠道推送：APP Push（iOS APNs / Android FCM）、站内消息、短信、邮件
- 支持推送类型：即时推送、定时推送、批量推送（千万级用户群发）
- 支持消息优先级（高/中/低，紧急通知不能延迟）
- 支持已读回执、推送统计（发送量/到达率/点击率）

**非功能需求：**
- 日推送量 10 亿条
- 单条消息到达延迟 < 3s（即时推送）
- 高可用，渠道抖动不影响其他渠道

### 2. 整体架构

```
[业务系统]  ────────────────────────────────
    │ 推送请求（HTTP/MQ）                    │  触发推送的3种来源
    ▼                                        │  1. 实时事件（下单/评论）
[推送平台接入层 Push Gateway]                │  2. 定时任务（营销活动）
    │                                        │  3. 运营后台（手动运营推）
    ▼
[任务调度层]
  ├── 即时任务 → 直接入队
  ├── 定时任务 → 延迟队列（Redis ZSet）→ 到点入队
  └── 群发任务 → 批量分片（按userId范围），并发入队

    ▼
[消息队列 Kafka - 按渠道分 Topic]
  ├── topic: push_app     (APP推送)
  ├── topic: push_sms     (短信)
  ├── topic: push_email   (邮件)
  └── topic: push_inbox   (站内信)

    ▼
[渠道发送服务 Channel Workers - 独立部署]
  ├── APP Push Worker → APNs (iOS) / FCM (Android)
  ├── SMS Worker      → 阿里云短信 / 腾讯短信（多渠道容灾）
  ├── Email Worker    → SendGrid / SES
  └── Inbox Worker    → 写DB + 推长连接

    ▼
[结果回执层]
  └── 回调/轮询获取送达状态 → 写推送日志 DB
```

### 3. 核心模块设计

#### 3.1 APP 长连接管理（站内实时推送）

```
用户 APP ←──── 长连接 (WebSocket/MQTT) ─────→ [接入层 Connection Server]
                                                      │ 维护 Map<userId, channel>
                                                      │ 用户登录时注册路由
                                                      ▼
                                              [路由表 Redis Hash]
                                              key: online:userId
                                              value: connectionServerId + channelId
```

发送流程：
1. 推送服务查 Redis 路由表，得知用户在哪台 Connection Server
2. 通过内部 RPC 定向到该 Server 发送
3. Connection Server 通过 channel 推给 APP

#### 3.2 离线消息存储（用户不在线时）

```sql
CREATE TABLE inbox_message (
    id          BIGINT    PRIMARY KEY AUTO_INCREMENT,
    user_id     BIGINT    NOT NULL,
    msg_id      VARCHAR(32) NOT NULL UNIQUE,  -- 业务幂等ID
    title       VARCHAR(200),
    content     TEXT,
    extra       JSON,                          -- 扩展字段（跳转URL等）
    is_read     TINYINT   DEFAULT 0,
    create_time DATETIME  DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_unread (user_id, is_read, create_time)
);
```

用户上线后拉取：`SELECT * FROM inbox_message WHERE user_id=? AND is_read=0 ORDER BY create_time DESC LIMIT 50`

#### 3.3 群发任务分片

千万级群发不能单线执行（太慢），需并行分片：

```
群发任务（target: 全量用户 5000w）
    │
    ▼ [任务拆分器]
  user_id [0, 5000w) → 拆成 500 个分片，每片 10w 用户
    │
    ▼ [分布式任务调度，如 XXL-Job]
  500 个 Worker 并发拉取 user_id 列表 → 批量推 Kafka
    │
    ▼ 进度监控：Redis INCR 已发送计数，达到总量则任务完成
```

#### 3.4 渠道路由与降级

```java
// 渠道选择策略
public ChannelType selectChannel(PushRequest req, UserProfile user) {
    // 1. 按用户偏好（用户设置了关闭APP推送，走短信）
    if (!user.isAppPushEnabled()) return SMS;
    // 2. 按用户在线状态
    if (onlineService.isOnline(user.getId())) return APP_PUSH;
    // 3. 高优先级离线用户走短信
    if (req.getPriority() == HIGH) return SMS;
    // 4. 普通优先级离线用户写站内信
    return INBOX;
}
```

渠道降级：APNs 超时 → 自动切阿里云短信（Sentinel + 降级规则）

#### 3.5 推送去重与幂等

```
推送幂等Key = bizType + bizId + userId（如 order_pay_123456_789）
发送前: SETNX idem:key 1 EX 86400  → 返回0说明已发送，跳过
发送成功: 回写推送日志 + 标记幂等Key已处理
```

### 4. 推送统计设计

```
发送 → 到达（APNs回调）→ 点击（APP上报）

DB: push_stat 表聚合天维度统计
Redis: 实时计数（INCR），每分钟 Flush DB
目标指标: 送达率 = 到达数/发送数 > 95%
```

### 5. 面试标准答法

> 消息推送系统的核心是多渠道统一接入 + 可靠投递。架构分四层：接入层统一收口，调度层处理即时/定时/群发三种模式，渠道层各自独立部署（APNs/FCM/短信/邮件），回执层追踪送达状态。关键难点：① 千万级群发用 XXL-Job 分片并发，不能单线推；② APP 长连接用 userId→serverId 的路由表（Redis Hash）实现精准定向推送；③ 用户离线时消息写入 inbox 表，上线后拉取；④ 渠道间相互隔离部署，APNs 抖动不影响短信渠道；⑤ 业务幂等 Key 防重复推送。

### 6. 常见追问

**Q: 推送系统如何保证消息不丢失？**
> 三层保障：① Kafka 持久化消息，Consumer ACK 确认消费，失败自动重试（max.attempts=3）；② 发送方记录 push_log（status=SENDING），定时扫描 SENDING 超 5min 的消息重推；③ APNs/FCM 有自己的 ACK 机制，失败回调后可选重推或写离线消息兜底。

**Q: iOS APNs 和 Android FCM 的区别？**
> APNs（苹果推送）是苹果维护的长连接通道，APP 必须通过 APNs 推送，无法自建长连接（省电限制）；FCM（Firebase）是谷歌推送服务，国内 Android 因为 GMS 被墙通常不用，改用厂商通道（华为/小米/OPPO Push）。因此国内 Android 推送需要同时对接华为、小米、OPPO、vivo 等多个厂商 SDK，用户设备匹配对应厂商通道。

**Q: 如何处理用户换设备？**
> 设备 Token 与 userId 绑定存在 `user_device` 表。用户登录新设备时，新 Token 覆盖旧 Token。旧 Token 推送会收到 APNs/FCM 的无效 Token 回调，此时将旧 Token 从表中删除，防止无效推送浪费资源（约 1-2% 的 Token 每天失效）。

**Q: 如何防止用户被频繁打扰（消息轰炸）？**
> 用户维度的推送频控：`INCR push:freq:{userId}:{date}`，`EXPIRE` 设为当天结束。超过阈值（如一天 5 条营销消息）则降级到低优先级队列或直接丢弃非紧急消息。紧急消息（如支付验证码）不受频控限制，通过 `priority=HIGH` 标识绕过。

**Q: 群发 5000 万用户，如何知道任务进度？**
> Redis INCR 计数器实时累计已入队数量，任务表记录总量，前端轮询 `progress = INCR_count / total_count * 100%`。分片任务维度的状态机：PENDING → RUNNING → DONE，所有分片 DONE 则主任务完成。同时用 HyperLogLog 去重统计真实覆盖 UV（防止用户多设备重复计数）。

## 四、设计评论系统

### 1. 需求分析

**功能需求：**
- 支持一级评论 + 二级回复（楼层制，主流社区方案）
- 评论支持点赞，按点赞数/时间排序
- 支持删除（软删除，子回复保留占位符"该评论已删除"）
- 评论分页（前端下拉加载更多）
- 敏感词过滤

**数据规模：** 某内容平台，日发评论 1000 万，单条内容最多 10 万条评论

### 2. 存储模型设计

**扁平化存储（Flat Model）**—— 业界主流

```sql
CREATE TABLE comment (
    id          BIGINT      PRIMARY KEY AUTO_INCREMENT,
    content_id  BIGINT      NOT NULL,          -- 所属内容（文章/视频ID）
    content_type TINYINT    NOT NULL,          -- 内容类型（1=文章，2=视频）
    root_id     BIGINT      NOT NULL DEFAULT 0,-- 根评论ID，自身为根时=0
    parent_id   BIGINT      NOT NULL DEFAULT 0,-- 直接父评论ID（用于@xxx 的显示）
    user_id     BIGINT      NOT NULL,          -- 评论者
    reply_to_uid BIGINT     DEFAULT 0,         -- 被回复者uid（@展示用）
    content     VARCHAR(1000) NOT NULL,
    like_count  INT         DEFAULT 0,
    state       TINYINT     DEFAULT 1,         -- 1=正常 0=已删除
    create_time DATETIME    DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_content_root  (content_id, content_type, root_id, create_time),
    INDEX idx_root_like     (root_id, like_count DESC)  -- 热评排序
);
```

**为何用扁平化而不是递归嵌套：**
- 树形嵌套（存 path `/1/2/3`）查询方便但层级无限深，前端渲染复杂
- 扁平化只支持二级（root + reply），约束了深度，翻页逻辑简单，主流平台（B站/微博）都是此方案

### 3. 分页方案

**一级评论分页（游标分页，避免 OFFSET 深翻页性能问题）：**

```sql
-- 首次加载（按时间正序）
SELECT * FROM comment
WHERE content_id = ? AND root_id = 0 AND state = 1
ORDER BY create_time ASC
LIMIT 20;

-- 下一页（cursor = 上一页最后一条的create_time + id）
SELECT * FROM comment
WHERE content_id = ? AND root_id = 0 AND state = 1
  AND (create_time > ? OR (create_time = ? AND id > ?))
ORDER BY create_time ASC
LIMIT 20;
```

**热评（Top N）：** 单独维护 Redis Sorted Set，score = like_count

```
key: hot_comment:{contentId}
ZADD hot_comment:{contentId} {likeCount} {commentId}
ZREVRANGE hot_comment:{contentId} 0 9   -- 取前10条热评ID
再 MGET comment:{commentId} 批量取评论详情
```

### 4. 点赞设计

**方案：Redis 计数 + 异步落库**

```
用户点赞：
1. Redis SADD like:{commentId} {userId}   -- 防重复点赞
   INCR like_count:{commentId}             -- 点赞计数

2. 发 MQ（或直接异步任务）定期将 Redis 计数回写 DB
   UPDATE comment SET like_count = ? WHERE id = ?

3. 取消点赞：SREM like:{commentId} {userId}，DECR like_count
```

**注意**：`SADD` 中存储所有点赞用户 userId，当点赞数过大（如热评百万点赞）内存爆炸。优化：改用 HyperLogLog 统计 UV（不能判断某用户是否点过赞），或 Bloom Filter（有 FP，可能说"你没点过"但其实点了，少见错误可接受）+ DB 精确兜底。

### 5. 评论数缓存

```
key: comment_count:{contentId}
写评论时：INCR comment_count:{contentId}
删评论时：DECR comment_count:{contentId}
展示时：GET，未命中查DB回填

定期校准：离线任务 COUNT(*) DB → 覆写 Redis（防长期漂移）
```

### 6. 敏感词过滤

**Aho-Corasick 多模式匹配算法（AC 自动机）**：
- 将百万敏感词构建 AC 自动机（一次构建，内存 ~200MB）
- 单条评论过滤时间复杂度 O(n)，n 为评论长度
- 命中后替换为 `***` 或直接拒绝发布

### 7. 面试标准答法

> 评论系统的难点在存储模型选型和高并发下的计数准确性。存储上选扁平化二级结构（根评论 + 回复），root_id=0 表示一级评论，root_id=comment_id 表示二级回复，天然限制层级防止前端渲染复杂度爆炸。分页用游标（create_time + id 组合游标）代替 OFFSET，避免百万级深翻页时 MySQL 全表扫。点赞用 Redis SADD 防重 + INCR 计数，异步批量回写 DB；热评维护独立的 Redis Sorted Set，按 like_count 排序，常规分页和热评分两条路查询。

### 8. 常见追问

**Q: 评论被删除，子回复怎么处理？**
> 软删除：`UPDATE comment SET state=0 WHERE id=?`，不物理删除数据。前端查询时如果 state=0，内容显示"该评论已删除"，但二级回复仍正常展示（可能还有有价值的讨论）。如果根评论软删且其下所有子评论也都软删，则可以触发物理删除的 T+7 离线任务。

**Q: 如何防止刷评论（恶意灌水）？**
> 多层防护：① 频率限制（同一用户每分钟最多发 5 条，Redis `INCR rate:{uid}:{minute}` + `EXPIRE`）；② 内容去重（MD5 hash 近 5 分钟内相同内容拒绝）；③ 机器学习垃圾评论模型（异步打标，命中则人工审核或自动隐藏）；④ 新用户限流（注册 7 天内每天最多 10 条）。

**Q: 评论数说 10 万，但实际查出来不足怎么办？**
> 评论数与评论列表的一致性是最终一致的，允许短暂不一致。两种方案：① 接受轻微不一致（展示 "10万+" 而非精确数，产品常见做法）；② 定时校准任务（每小时 `COUNT(*)` DB 刷新 Redis 计数）。关键是不要在发评论的主链路上做 `COUNT(*)` 全表统计，那是灾难性的性能开销。

**Q: 内容详情页需要展示评论数，但内容表和评论表分库了，如何关联？**
> 评论数不能跨库 JOIN，应该在内容表（或内容缓存）中冗余 `comment_count` 字段。写评论时发 MQ 事件，内容服务消费后更新自己的 `comment_count`（异步最终一致），展示时直接从内容缓存读，无需跨库。

## 五、设计朋友圈 / Feed 流系统

### 1. 需求分析

**功能需求：**
- 用户发布动态（文字 + 图片/视频）
- 关注用户发布内容后，粉丝的 Feed 流能看到
- Feed 流按时间倒序展示（也可按算法排序）
- 支持点赞、评论（复用评论系统）

**数据特点：**
- 大 V 有 5000 万粉丝，普通用户有 200 粉丝
- 用户关注关系表规模：微博级约 500 亿行

### 2. 三种 Feed 流方案对比

#### 2.1 推模式（写扩散 / Fan-out on Write）

```
大V发布动态
    │
    ▼ 异步 MQ
  遍历 5000w 粉丝
  → 向每个粉丝的 Timeline 写入这条动态的 ID

存储：Redis Sorted Set
  key: timeline:{userId}
  member: postId
  score: timestamp（发布时间）
```

| 优点 | 缺点 |
|---|---|
| 读取极快（直接查自己 Timeline） | 大 V 写放大严重（5000w 次写操作）|
| 实时性好 | 存储爆炸（每条动态写入 N 份）|
| 适合粉丝数少的用户 | 大 V 发帖后台延迟高 |

#### 2.2 拉模式（读扩散 / Fan-out on Read）

```
用户打开 Feed 流
    │
    ▼ 实时计算
  查询所有关注的人：SELECT followee_id FROM follow WHERE follower_id = ?
  → 并发查每个 followee 的最新动态
  → 合并排序（N 路归并）
  → 取前 20 条返回
```

| 优点 | 缺点 |
|---|---|
| 写入极简单（只写自己的 post 表）| 读时延高（关注 2000 人则 2000 次查询）|
| 无写放大 | 用户刷新 Feed 慢 |
| 适合发帖少的场景 | 并发高时 DB 压力大 |

#### 2.3 推拉结合（字节/微博主流方案）

```
策略：
  - 普通用户（粉丝 < 1w）：推模式，发帖时写入粉丝 Timeline
  - 大 V（粉丝 > 1w）：不主动推，粉丝读 Feed 时实时拉取大 V 的最新帖子

读取 Feed 流时：
  1. 读取自己 Timeline（Redis ZSet，包含关注的普通用户推过来的帖子）
  2. 查询自己关注的大 V 列表（通常 < 100 个）
  3. 并发查这些大 V 的最新帖子（每个大 V 只查 < 20 条）
  4. 合并排序，返回 Top 20

写入 Feed 流时：
  发帖 → MQ → 异步推给粉丝数 < 1w 的粉丝 Timeline
              → 大 V 不推，粉丝自己拉
```

### 3. 核心数据结构

```sql
-- 帖子表（按userId分库，或TiDB）
CREATE TABLE post (
    id          BIGINT PRIMARY KEY,   -- Snowflake ID（含时间戳，天然有序）
    user_id     BIGINT NOT NULL,
    content     TEXT,
    media_urls  JSON,                 -- 图片/视频列表
    like_count  INT DEFAULT 0,
    comment_count INT DEFAULT 0,
    create_time DATETIME,
    INDEX idx_user_time (user_id, create_time DESC)
);

-- 关注关系表（按follower_id分库）
CREATE TABLE follow (
    follower_id  BIGINT NOT NULL,     -- 关注者
    followee_id  BIGINT NOT NULL,     -- 被关注者
    create_time  DATETIME,
    PRIMARY KEY (follower_id, followee_id),
    INDEX idx_followee (followee_id)  -- 查某人的粉丝列表
);
```

**Timeline（Redis Sorted Set）：**
```
key:    feed_timeline:{userId}
member: {postId}           # Snowflake ID 天然时间有序
score:  publish_timestamp  # 也可直接用 postId 排序

ZADD feed_timeline:{uid} {ts} {postId}    # 推入
ZREVRANGE feed_timeline:{uid} 0 19        # 取最新20条
ZREMRANGEBYRANK feed_timeline:{uid} 200 -1  # 只保留最新200条，防内存膨胀
```

### 4. Feed 流拉取完整流程

```java
public List<Post> getFeed(long userId, long cursor, int size) {
    // 1. 从 Redis Timeline 取 postId 列表（游标分页）
    Set<Tuple> items = redis.zrevrangeByScoreWithScores(
        "feed_timeline:" + userId,
        cursor == 0 ? Double.MAX_VALUE : cursor,
        Double.MIN_VALUE,
        0, size
    );

    List<Long> postIds = new ArrayList<>(items.stream()
        .map(t -> Long.parseLong(t.getElement()))
        .collect(Collectors.toList()));

    // 2. 拉取关注大 V 的最新帖子，合并到 postIds
    List<Long> bigVIds = followService.getBigVFollowees(userId);
    if (!bigVIds.isEmpty()) {
        List<Long> bigVPostIds = postService.batchGetLatestPostIds(bigVIds, size);
        postIds.addAll(bigVPostIds);
    }

    // 3. 合并排序，取前 size 条
    postIds.sort(Comparator.reverseOrder()); // Snowflake ID 有序
    postIds = postIds.subList(0, Math.min(size, postIds.size()));

    // 4. 批量查帖子详情（先Redis，再DB）
    return postService.batchGetByIds(postIds);
}
```

### 5. 面试标准答法

> Feed 流的核心选型是推拉模式。纯推模式写放大（大 V 5000w 粉丝意味着每条动态写 5000w 次），纯拉模式读延迟高（关注 2000 人需 2000 次查询）。生产环境普遍用推拉结合：普通用户推（发帖时写粉丝 Timeline），大 V 不推（读时实时拉）。阈值通常是粉丝数 1 万，大 V 的粉丝关注量通常不超过 200 人，所以实时拉 200 个大 V 的最新帖可接受。Timeline 用 Redis Sorted Set，score 为发布时间戳，只保留最近 200 条防内存爆炸，更早的翻页走 DB 查询（用户极少翻到那么早）。

### 6. 常见追问

**Q: 大 V 拉模式，用户关注了 100 个大 V，每次刷新都要查 100 次，性能问题如何解决？**
> ① 每个大 V 的最新帖子单独缓存（`latest_posts:{bigVId}` Redis List，存最新 50 条 postId），100次查询变为 100 次 Redis lrange，耗时 < 10ms（并发执行）；② 关注的大 V 列表也缓存（用户的大 V 关注列表变动不频繁）；③ 如果大 V 粉丝超过 1000 万，也可对其"活跃粉丝"做推模式，非活跃粉丝来访时再拉（二级推拉）。

**Q: 如何实现算法推荐而不是纯时间序 Feed？**
> 算法排序在离线/近线计算层完成：① 用户的互动历史（点赞/评论/停留）作为特征；② 离线训练排序模型；③ 近线层：用户打开 Feed 时，实时从 Timeline 取 200 条候选，调用排序服务打分，返回 Top 20（候选生成→粗排→精排）。Redis Timeline 依然是时间序的候选集，排序发生在取出后的服务层，不影响存储结构。

**Q: 用户刚注册没有关注，Feed 流是空的怎么处理？**
> 冷启动三步走：① 新用户引导页推荐关注（基于手机通讯录/兴趣标签）；② 关注数 < 5 时，Feed 流混入平台推荐内容（热榜 + 兴趣推荐填充）；③ 关注内容足够后，纯 Follow 模式的 Feed 自然填满。本质是内容推荐系统兜底，不在本题范围内。

**Q: 帖子被删除，如何从所有粉丝的 Timeline 中清除？**
> 不建议遍历所有粉丝删 Redis（成本极高，5000w 次操作）。更好的做法：① Redis Timeline 中只存 postId，展示时批量查帖子详情，查到帖子 state=deleted 则跳过（前端过滤）；② 读时过滤（在 batchGetByIds 中过滤掉已删除帖子，不返回给前端）。代价是 Timeline 有"脏数据"，但节省了写放大，是合理的工程权衡。

## 六、设计分布式延迟任务系统

### 1. 需求分析

**典型场景：**
- 订单 30 分钟未支付自动取消
- 消息在指定时间发送（定时消息）
- 优惠券 7 天后过期提醒
- 延迟重试（失败后 5s/30s/5min 指数退避）

**核心要求：**
- 支持任意时延（秒级甚至毫秒级精度）
- 海量任务（千万级并发延迟任务）
- 高可用（单节点故障任务不丢失）
- 支持任务取消

### 2. 方案对比

| 方案 | 精度 | 容量 | 可靠性 | 分布式 | 适用场景 |
|---|---|---|---|---|---|
| JDK DelayQueue | ms 级 | JVM 内存限制 | 差（重启丢失）| 单机 | 单机小规模 |
| 时间轮 HashedWheelTimer | ms 级 | JVM 内存限制 | 差（重启丢失）| 单机 | 网络超时等 |
| **Redis ZSet** | 秒级 | 亿级（持久化）| 好（AOF/RDB）| 是 | **业务延迟任务** |
| RocketMQ 延迟消息 | 固定级别 | 高 | 高 | 是 | 有限延迟级别 |
| 数据库轮询 | 分钟级 | 千万级 | 高 | 是 | 兜底方案 |
| **自建时间轮 + DB** | 秒级 | 亿级 | 高 | 是 | **完整生产方案** |

### 3. 核心方案：Redis ZSet 延迟队列

**原理：** ZSet score = 执行时间戳，轮询弹出 score ≤ NOW 的任务

```java
// 投递延迟任务
public void submitDelayTask(String taskId, String taskData, long delaySeconds) {
    long executeAt = System.currentTimeMillis() / 1000 + delaySeconds;
    // key: delay_queue, score: 执行时间戳, member: taskId
    redis.zadd("delay_queue", executeAt, taskId);
    // 任务详情单独存储（避免ZSet member过大）
    redis.setex("task:" + taskId, delaySeconds + 3600, taskData);
}

// 轮询消费（每100ms执行一次）
@Scheduled(fixedDelay = 100)
public void pollDelayQueue() {
    long now = System.currentTimeMillis() / 1000;
    // 原子获取并删除 score 在 (0, now] 的最多10个任务
    Set<String> taskIds = redis.zrangeByScore("delay_queue", 0, now, 0, 10);
    if (taskIds.isEmpty()) return;
    for (String taskId : taskIds) {
        // 用 Lua 脚本保证原子性：检查并删除（防多节点重复消费）
        Long removed = redis.zrem("delay_queue", taskId);
        if (removed == 1) { // 抢到这个任务
            String taskData = redis.get("task:" + taskId);
            if (taskData != null) {
                processTask(taskId, taskData);
                redis.del("task:" + taskId);
            }
        }
    }
}
```

**分布式多节点竞争消费：** `ZRANGEBYSCORE` + `ZREM` Lua 原子操作，只有 ZREM 成功的节点才真正处理任务。

#### Lua 原子脚本（防止多节点重复消费）

```lua
-- KEYS[1] = delay_queue, ARGV[1] = now, ARGV[2] = batchSize
local tasks = redis.call('ZRANGEBYSCORE', KEYS[1], 0, ARGV[1], 'LIMIT', 0, ARGV[2])
local result = {}
for _, taskId in ipairs(tasks) do
    local removed = redis.call('ZREM', KEYS[1], taskId)
    if removed == 1 then
        table.insert(result, taskId)
    end
end
return result
```

### 4. 生产级完整方案：分层时间轮 + DB 兜底

```
任务投递
    │
    ▼
[持久化层 - MySQL]          -- 所有任务先写DB（可靠性保障）
task表: id, execute_time, status(PENDING/RUNNING/DONE/FAIL), task_data
    │
    ▼（近期任务，2小时内）
[Redis ZSet 时间轮]          -- 热数据，快速触发
    │
    ▼（到点弹出）
[MQ（Kafka/RocketMQ）]       -- 解耦执行层，支持重试
    │
    ▼
[Task Executor 集群]         -- 消费MQ，执行任务逻辑
    │
    ▼
[结果回写 DB]                -- 更新task状态为DONE

[DB 兜底扫描（每分钟）]       -- 扫描 execute_time < NOW AND status=PENDING
                              -- 重新推入 Redis ZSet （防Redis宕机导致任务漏执行）
```

**任务表设计：**
```sql
CREATE TABLE delay_task (
    id            BIGINT          PRIMARY KEY AUTO_INCREMENT,
    task_id       VARCHAR(64)     NOT NULL UNIQUE,    -- 业务幂等ID
    task_type     VARCHAR(50)     NOT NULL,            -- 任务类型
    task_data     TEXT,                                -- 任务参数（JSON）
    execute_time  DATETIME        NOT NULL,            -- 期望执行时间
    status        TINYINT         DEFAULT 0,           -- 0=PENDING 1=RUNNING 2=DONE 3=FAIL
    retry_count   INT             DEFAULT 0,
    max_retry     INT             DEFAULT 3,
    create_time   DATETIME        DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_execute_status (execute_time, status)   -- 兜底扫描用
);
```

### 5. 时间轮算法（HashedWheelTimer）原理

```
时间轮：一个环形数组，每个槽代表一个时间单位（如1s）
拨针（tick）每秒前进一格，执行当前槽的所有任务

任务放入槽：slot = (now + delay) % wheelSize
多圈数（round）：round = (now + delay) / wheelSize

实际执行时检查：round == 0 则执行，否则 round--

优点：O(1)添加和触发，内存高效
缺点：单机，精度受tick间隔限制，重启任务丢失
```

Netty 的 `HashedWheelTimer` 是经典实现，广泛用于连接超时管理。

### 6. 面试标准答法

> 分布式延迟任务系统的核心是"可靠性"和"精度"的权衡。生产方案分三层：① DB 作为可靠存储，所有任务先持久化，防止任何中间件宕机丢任务；② Redis ZSet 作为调度层（score=执行时间戳），高效触发近期任务（2 小时内），轮询间隔 100ms 保证秒级精度；③ MQ 作为执行层，解耦触发和执行，天然支持重试。兜底策略：DB 扫描每分钟扫 execute_time 已到期但 status 仍为 PENDING 的任务，重推 Redis ZSet，形成闭环。多节点竞争消费用 `ZRANGEBYSCORE + ZREM` Lua 原子脚本，只有 ZREM 成功的节点真正处理任务。

### 7. 常见追问

**Q: Redis ZSet 方案，如果任务量达到 1 亿，内存会不会撑不住？**
> Redis ZSet 单条任务约 64 字节（8B score + taskId string），1 亿条约 6GB，Redis 单实例通常给 32-64GB 内存，理论上够。但推荐只在 ZSet 中放"执行时间在 2 小时内"的任务，更远的任务只存 DB，2小时内临近时再推入 ZSet（按 execute_time 定期扫 DB 补充）。这样 ZSet 常驻数量可控在千万级以内。

**Q: 如何支持任务取消？**
> ① Redis ZSet 层：`ZREM delay_queue {taskId}` 直接删除；② DB 层：`UPDATE delay_task SET status=CANCELLED WHERE task_id=?`；③ 执行器层：Consumer 消费 MQ 消息时，先查 DB 任务状态，如果是 CANCELLED 则跳过执行。三层都处理才能保证取消可靠（Redis 已出队但 MQ 未消费的情况靠 ③ 兜底）。

**Q: 任务执行失败，如何做指数退避重试？**
> 执行失败时：① 更新 DB `retry_count++`，计算下次执行时间：`next_execute = now + base_delay * 2^retry_count`（指数退避）；② 将任务重新推入 Redis ZSet；③ 达到 `max_retry` 后标记 FAIL 并发告警。每次执行前加分布式锁（或 DB optimistic lock）防止重试期间重复执行。

**Q: 跟 RocketMQ 延迟消息比有什么区别？**
> RocketMQ 延迟消息只支持固定 18 个延迟级别（1s/5s/10s/30s/1m/.../2h），无法精确到任意时刻。自建 Redis ZSet 方案支持任意时延（精确到秒）、任意取消、查询任务状态，功能远比 RocketMQ 延迟消息丰富。但 RocketMQ 延迟消息可靠性由 MQ 保证，运维成本低。实践中：简单场景用 RocketMQ，复杂业务（需要取消、状态查询、精确时间）自建调度系统。

## 七、设计唯一 ID 生成器

### 1. 核心要求

- **全局唯一**：分布式环境下不重复
- **趋势递增**：利于 B+ 树索引顺序插入，减少页分裂
- **高性能**：单服务 > 100w QPS
- **高可用**：单点故障不影响业务
- **长度紧凑**：尽量短（DB 存储、网络传输友好）

### 2. 核心方案对比

| 方案 | 原理 | 性能 | 单调递增 | 有无规律 | 依赖 |
|---|---|---|---|---|---|
| DB 自增 | AUTO_INCREMENT | 低（DB 写） | ✅ 全局单调 | 有（可被预测）| MySQL |
| UUID | 随机 128 位 | 极高（本地）| ❌ 随机 | 无 | 无 |
| **Snowflake** | 时间戳+机器号+序列号 | 极高（本地）| ✅ 趋势递增 | 有（含时间戳）| 弱（时钟）|
| 号段模式 | 批量申请 ID 段 | 高 | ✅ 趋势递增 | 有规律 | DB/Redis |
| Redis INCR | 原子自增 | 高 | ✅ 全局单调 | 有 | Redis |

### 3. Snowflake 详解（字节系最爱）

```
64位 Long:
┌───────────────────────────────────────────────────────────────────────────────┐
│ 0 │       41位时间戳        │  5位数据中心  │  5位机器号  │    12位序列号    │
│ 0 │ (ms-epoch，可用69年)    │  (0-31)      │  (0-31)    │    (0-4095)     │
└───────────────────────────────────────────────────────────────────────────────┘

理论QPS上限：4096 * 1000 = 409.6w/s（单机，1ms内4096个ID）
实际: 单机百万QPS无压力
```

**完整 Java 实现（含时钟回拨处理）：**

```java
public class Snowflake {
    private static final long EPOCH       = 1700000000000L; // 自定义起始时间(2023-11)
    private static final long WORKER_BITS = 10L;  // 机器ID位数（5位DC+5位Worker）
    private static final long SEQ_BITS    = 12L;
    private static final long MAX_WORKER  = ~(-1L << WORKER_BITS);  // 1023
    private static final long MAX_SEQ     = ~(-1L << SEQ_BITS);     // 4095
    private static final long WORKER_SHIFT = SEQ_BITS;              // 12
    private static final long TIMESTAMP_SHIFT = SEQ_BITS + WORKER_BITS; // 22

    private final long workerId;
    private long lastTimestamp = -1L;
    private long sequence = 0L;

    public Snowflake(long workerId) {
        if (workerId > MAX_WORKER || workerId < 0)
            throw new IllegalArgumentException("workerId=" + workerId + " out of range");
        this.workerId = workerId;
    }

    public synchronized long nextId() {
        long now = System.currentTimeMillis();
        if (now < lastTimestamp) {
            // 时钟回拨：等待追上，超过5ms抛异常
            long diff = lastTimestamp - now;
            if (diff <= 5) {
                try { Thread.sleep(diff); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
                now = System.currentTimeMillis();
            } else {
                throw new RuntimeException("Clock moved backwards by " + diff + "ms");
            }
        }
        if (now == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQ;
            if (sequence == 0) {
                // 序列号用尽，等到下一毫秒
                now = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        lastTimestamp = now;
        return ((now - EPOCH) << TIMESTAMP_SHIFT)
                | (workerId << WORKER_SHIFT)
                | sequence;
    }

    private long waitNextMillis(long lastTs) {
        long ts = System.currentTimeMillis();
        while (ts <= lastTs) ts = System.currentTimeMillis();
        return ts;
    }
}
```

### 4. 号段模式（美团 Leaf 方案）

适用于对递增性要求更强、不希望 ID 泄露时间信息的场景：

```
号段服务（Leaf-Segment）:
┌─────────────────────────────────────────────────────────────┐
│  DB: id_alloc 表                                            │
│  biz_tag | max_id | step | update_time                      │
│  order   | 50000  | 1000 | 2024-01-01 00:00:00             │
└─────────────────────────────────────────────────────────────┘
     │
     │ 应用启动/号段用完时，UPDATE max_id = max_id + step，取走这段
     ▼
┌────────────────────────┐     ┌────────────────────────┐
│  节点A：内存中[1,1001)  │     │  节点B：内存中[1001,2001)│
│  当前=500              │     │  当前=1500              │
└────────────────────────┘     └────────────────────────┘
```

**双 Buffer 优化**：当前号段用到 10% 时，异步预取下一个号段，不阻塞 ID 发放。

### 5. workerId 自动分配方案

Snowflake 中 workerId 需要在分布式环境下不重复分配：

| 方案 | 原理 |
|---|---|
| 手动配置 | 启动参数指定，运维繁琐，扩容不友好 |
| ZooKeeper 临时节点 | 启动时创建 `/workers/{id}` 临时节点，宕机自动释放，其他节点可复用 |
| Redis INCR | `INCR worker_id_counter` 自增，`SETNX worker:{id} {ip}` 占位，简单可靠 |
| **K8s Pod 序号** | StatefulSet 的 Pod 名包含序号（如 leaf-0, leaf-1），直接解析作为 workerId |

### 6. 系统设计视角

**API 设计：**
```http
GET /id/next?bizType=order        # 获取单个ID
GET /id/batch?bizType=order&n=100 # 批量获取，减少网络RTT
```

**高可用部署：**
```
Client
  │ 随机/轮询
  ├──→ ID-Service-Node-1 (workerId=1)
  ├──→ ID-Service-Node-2 (workerId=2)
  └──→ ID-Service-Node-3 (workerId=3)

每个节点完全独立，内存计算，无单点，任意节点挂掉不影响其他节点
```

**监控指标：**
- ID 生成 QPS（分节点、分 bizType）
- 时钟回拨发生次数（> 0 立即告警）
- 号段剩余量（号段模式，剩余 < 10% 告警）
- ID 唯一性校验（定期抽样 DB 去重验证）

### 7. 面试标准答法

> 唯一 ID 生成器本质是"唯一性 + 递增性 + 高性能"的平衡。Snowflake 是最经典的方案：41 位毫秒时间戳 + 10 位机器 ID + 12 位序列号，单机理论 QPS 400w/s，趋势递增，无网络依赖，适合绝大多数场景。核心问题有两个：① 时钟回拨——时间戳倒退会生成重复 ID，解法是等待（< 5ms 等待追上）或切换到备用位（用序列号高位做后备）；② workerId 分配——分布式环境下 ① ZooKeeper 临时节点 ② Redis INCR 占位 ③ K8s StatefulSet 序号，三选一。对递增性要求更强的场景（如对账、审计日志）用号段模式（美团 Leaf），牺牲一点性能换全局单调递增。

### 8. 常见追问

**Q: Snowflake 生成的 ID 在什么情况下会重复？**
> 有三种情况：① **时钟回拨**（NTP 同步导致时间倒退，同一毫秒 + 同一 workerId + 同一 sequence 会重复）；② **workerId 重复**（两台机器分配到同一 workerId，则相同毫秒内序列号相同时产生重复 ID）；③ **单毫秒内序列号溢出**（超过 4095 后必须等到下一毫秒，如果逻辑错误没等待就直接复用 sequence），正确实现不会触发此项。生产中最常见的是时钟回拨，要有明确的处理策略（等待/抛异常/降级）。

**Q: 为什么 UUID 不推荐作为数据库主键？**
> UUID 是随机 128 位，有两个问题：① **索引性能差**：MySQL InnoDB 主键是聚簇索引，UUID 随机导致每次插入都要找随机位置，引发频繁的页分裂和磁盘随机 IO，写入性能相比自增 ID 下降 3-5 倍；② **存储空间大**：UUID 字符串 36 字节 vs BIGINT 8 字节，主键出现在所有二级索引中，存储放大明显。如果必须用（分布式无 ID 服务场景），改用 UUID v7（含时间戳，趋势有序，解决了随机问题）。

**Q: 号段模式 DB 宕机怎么办？**
> 应对措施：① **双 Buffer**：号段用到 10% 时提前异步取下一段，DB 宕机时当前 Buffer 至少还有大量 ID 可用（max = step * 0.9 时间窗口）；② **DB 主从**：号段表 Master 宕机时切从库（秒级切换，仅损失极少量 ID 连续性）；③ **step 配置大**：step=10000 时，即使 DB 宕 10 分钟，Buffer 里的 ID 也够发放（前提是 QPS 不超过 step/10min）；④ 兜底：降级到本地时间戳 + 随机数生成临时 ID，并在 DB 恢复后做 ID 对账修复。

**Q: 如何保证 ID 不泄露业务信息（如订单量）？**
> Snowflake ID 含时间戳，理论上竞争对手可以对比相邻时间的 ID 差值估算业务量。解法：① **ID 混淆**：对外暴露的 ID 经过双射映射（如 XOR + 位交换），内部存 Snowflake 原始值，对外展示混淆值；② **号段模式 + 随机起点**：号段内的 ID 从随机数开始（而非 `maxId - step`），使相邻订单 ID 差值无规律；③ **业务隔离**：不同业务类型用不同的 ID 空间（bizType 前缀），无法横向对比。
