## 一、限流算法 ⭐⭐⭐⭐⭐

### 1.1 四种限流算法对比

| 算法 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **固定窗口** | 固定时间段内计数，超阈值拒绝 | 简单 | 临界点突刺（窗口边界2倍流量穿透）| 不推荐 |
| **滑动窗口** | 时间窗口随时间滑动，多个小格统计 | 解决突刺，平滑 | 实现稍复杂 | 接口限流 |
| **漏桶** | 请求入桶，固定速率出桶，桶满则拒绝 | 强制匀速输出，保护下游 | 无法应对突发，合法突发也被限 | 保护数据库等脆弱下游 |
| **令牌桶** | 固定速率往桶里放令牌，请求取令牌，取到则通过 | **允许一定突发**（桶内积攒的令牌）| 令牌积累过多可能瞬间大量放行 | **最常用**，接口/API限流 |

---

### 1.2 令牌桶算法详解 ⭐⭐⭐⭐⭐

```
核心参数：
  capacity：桶容量（最大允许突发量）
  rate：令牌放入速率（QPS上限）

流程：
  ① 定时（或按需）以固定速率向桶中放令牌，超出桶容量则丢弃
  ② 请求到来时尝试取令牌：
     有令牌 → 请求通过
     无令牌 → 拒绝（或等待）

突发处理：
  平时流量低 → 令牌积累（最多积满 capacity 个）
  突发流量来时 → 消耗积累的令牌 → 允许短时高并发
  突发超过 capacity → 超出部分被拒绝

**Redis Lua 令牌桶实现（分布式限流，字节常考手写）：**

```lua
-- KEYS[1]=令牌桶key
-- ARGV[1]=令牌桶容量capacity, ARGV[2]=每秒放入速率rate
-- ARGV[3]=当前时间戳(秒,支持小数), ARGV[4]=本次请求消耗令牌数

local key      = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate     = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local cost     = tonumber(ARGV[4])

-- 读取桶的当前状态（令牌数 + 上次填充时间）
local info = redis.call('HMGET', key, 'tokens', 'last_time')
local tokens    = tonumber(info[1]) or capacity   -- 首次默认满桶
local last_time = tonumber(info[2]) or now

-- 计算距上次填充经过的时间，补充令牌
local elapsed = math.max(0, now - last_time)
local new_tokens = math.min(capacity, tokens + elapsed * rate)

-- 判断令牌是否充足
if new_tokens < cost then
    -- 令牌不足：更新状态后拒绝
    redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
    redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)
    return 0  -- 拒绝
end

-- 令牌充足：扣减后放行
new_tokens = new_tokens - cost
redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)
return 1  -- 通过
```

```java
// Java 调用示例
public boolean tryAcquire(String key, int capacity, int rate, int cost) {
    String luaScript = "...";  // 上方 Lua 脚本
    double now = System.currentTimeMillis() / 1000.0;  // 秒级时间戳（支持小数）
    Long result = (Long) redisTemplate.execute(
        new DefaultRedisScript<>(luaScript, Long.class),
        Collections.singletonList(key),
        capacity, rate, now, cost
    );
    return result != null && result == 1L;
}
```

**核心设计要点：**
```
① 用 HMSET 存 tokens（当前令牌数）+ last_time（上次计算时间）
② 每次请求时"按需补充"令牌（惰性计算，不用定时任务）：
   新令牌 = min(capacity, 旧令牌 + 经过秒数 × 速率)
③ 整个 Lua 脚本原子执行，无并发安全问题
④ 时间用秒级浮点数（支持 0.1 秒精度），比毫秒整数更灵活
```

### 1.3 滑动窗口算法详解

```
固定窗口的临界点突刺问题：
  窗口 [0s,1s] 允许100个请求，计数清零时刻 1s 前后各100个请求
  → 0.9s~1.1s 200ms内 200个请求通过，实际QPS可达1000

滑动窗口解法：
  将 1s 窗口切成 10 个 100ms 的小格（bucket）
  每次统计最近 1s（10个格）的总请求数
  随时间推进，最老的格被丢弃，加入最新的格
  → 任意1s内的请求总数不超过阈值

Sentinel 的 LeapArray 实现：
  数组长度 = windowCount（循环数组）
  每个 bucket 存：通过数/拒绝数/异常数/慢调用数
  时间戳对应的 bucket = (timestamp / bucketLengthMs) % windowCount
```

### 1.4 分布式限流

```
单机限流（Guava/Sentinel单机模式）：
  QPS 阈值配置的是单节点限制
  10台机器 × 每台100 QPS = 总100 QPS（配置不统一）
  → 适合各节点独立限流（保护本机资源）

分布式限流（需要全局统一）：
  方案一：Redis + Lua 脚本（滑动窗口或令牌桶）
  方案二：Sentinel 集群限流（TokenServer 集中发放令牌）
  方案三：API 网关层统一限流（Gateway RedisRateLimiter）

Redis 滑动窗口限流 Lua 脚本：
```

```lua
-- KEYS[1]=限流key, ARGV[1]=当前时间戳ms, ARGV[2]=窗口大小ms, ARGV[3]=阈值
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local threshold = tonumber(ARGV[3])

-- 删除窗口外的记录
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
-- 当前窗口内的请求数
local count = redis.call('ZCARD', key)
if count < threshold then
    redis.call('ZADD', key, now, now)    -- 记录本次请求时间戳
    redis.call('EXPIRE', key, math.ceil(window/1000) + 1)
    return 1  -- 通过
end
return 0  -- 拒绝
```

## 二、熔断 & 降级 ⭐⭐⭐⭐

### 2.1 熔断 vs 降级 vs 限流的区别

```
限流（Rate Limiting）：
  控制流量入口，超出阈值直接拒绝
  目的：保护自身不被压垮

熔断（Circuit Breaker）：
  自动检测下游服务的健康状态，异常率高时断开连接
  目的：防止级联故障蔓延（雪崩）

降级（Fallback）：
  服务不可用时返回兜底数据（缓存/默认值/提示语）
  目的：保证核心功能可用，牺牲非核心功能

关系：熔断触发后通常执行降级逻辑（fallback）
```

### 2.2 雪崩效应与防护

```
雪崩场景：
  A → B → C → D（服务调用链）
  D 响应慢 → C 线程池耗尽等待 D → B 线程池耗尽等待 C → A 挂掉
  → 一个底层服务故障，导致整条链路雪崩

防护手段（多层防御）：
① 超时机制：每个调用设合理超时，不无限等待
② 熔断（Sentinel/Hystrix）：下游故障率超阈值快速失败
③ 限流：控制进入量，防止超载
④ 舱壁隔离：线程池隔离，不同服务用不同线程池，故障不传染
⑤ 降级兜底：返回缓存/默认值，保证核心业务可用
⑥ 限制重试：失败重试加指数退避 + Jitter（抖动），防止重试风暴
```

## 三、缓存架构设计 ⭐⭐⭐⭐⭐

### 3.1 多级缓存架构

```
请求链路（越靠近用户越快）：

Client（浏览器缓存）
  ↓ 未命中
CDN（边缘节点缓存，静态资源/接口缓存）
  ↓ 未命中
Nginx（本地 Lua/Shared Memory 缓存，单机内存）
  ↓ 未命中
应用层本地缓存（Caffeine/Guava Cache，JVM堆内）
  ↓ 未命中
分布式缓存（Redis，共享，支持集群）
  ↓ 未命中
数据库（MySQL）

典型延迟参考：
  本地缓存：< 1ms
  Redis：  1~3ms
  MySQL：  10~100ms
```

### 3.2 本地缓存 Caffeine ⭐⭐⭐⭐

```
Caffeine 是 Java 性能最好的本地缓存（Guava Cache 的升级版）

核心特性：
  W-TinyLFU 淘汰算法（LFU改进，比LRU命中率更高）
  异步刷新（refreshAfterWrite）
  自动过期（expireAfterWrite / expireAfterAccess）
  最大 size 限制（evict 淘汰）
```

```java
Cache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)              // 最多10000条
    .expireAfterWrite(5, TimeUnit.MINUTES)  // 写入5分钟后过期
    .refreshAfterWrite(1, TimeUnit.MINUTES) // 1分钟后异步刷新（不阻塞请求）
    .recordStats()                    // 开启统计
    .build(key -> userService.load(key));   // 加载函数（CacheLoader）

// 使用
User user = cache.get(userId);  // 命中返回，未命中自动加载
```

**本地缓存一致性问题：**
```
多节点本地缓存，某节点更新DB后，其他节点本地缓存仍是旧数据
解法：
  ① 短过期时间（容忍短暂不一致）
  ② MQ广播失效通知（canal订阅binlog → 发MQ → 各节点清本地缓存）
  ③ 数据一致性要求高的场景不用本地缓存，只用Redis
```

### 3.3 缓存更新策略

```
① Cache-Aside（旁路缓存）— 最常用 ⭐⭐⭐⭐⭐
   读：先读缓存，命中返回；未命中读DB，写入缓存
   写：先更新DB，再删除缓存（不是更新缓存）

   为什么删除而不是更新缓存？
   → 避免并发写时缓存和DB不一致（两个写操作顺序不确定）
   → 删除简单，下次读时重建（懒加载）

② Read/Write Through（穿透读写）
   应用只操作缓存，缓存层负责同步DB（缓存框架封装了DB操作）
   少见，多用于数据库缓存层

③ Write Behind（写回）
   只写缓存，异步批量刷DB（类似OS的Page Cache dirty机制）
   性能最好，但有数据丢失风险
```

**Cache-Aside 的先删缓存 vs 先更新DB：**
```
❌ 先删缓存再更新DB（延迟双删也有问题）：
   ① 删缓存
   ② 另一请求读缓存未命中，读了旧DB值写入缓存
   ③ 更新DB
   → 缓存里是旧值！

✅ 先更新DB再删缓存（标准方案）：
   ① 更新DB（成功）
   ② 删缓存（成功）→ 下次读自动重建

   极端情况：①成功②失败 → 缓存未删，读到旧值
   解法：删缓存失败 → 重试（MQ异步重试）或 引入 binlog + canal 监听删除
```

## 四、读写分离 & 分库分表 ⭐⭐⭐⭐

### 4.1 读写分离

```
场景：读多写少（典型：读:写 = 9:1）
方案：主库负责写，从库负责读，主从复制同步数据

实现方式：
① ShardingSphere-JDBC（代码层，无需中间件）
   配置主从数据源，注解/AOP 分发读写

② MyCat / ProxySQL（代理层，中间件）
   应用无感知，中间件解析SQL自动路由

③ 手动（@Primary/@Qualifier多数据源）
```

**主从复制延迟问题：**
```
现象：写主库后立即读从库，可能读到旧数据（复制延迟 100ms~1s）
解决方案（按强度排序）：

① 容忍延迟（80%场景）：
   展示类数据（列表、详情）允许短暂不一致，用户无感知

② 读写路由到主库（强一致场景）：
   下单后查询订单状态 → 此请求强制走主库
   实现：请求上下文标记（ThreadLocal），ShardingSphere 支持 hint 路由

③ 等待同步（不推荐，性能差）：
   读前等待主从延迟消除（轮询 slave position）

④ 半同步复制（MySQL 5.7+）：
   主库写完等至少1个从库确认收到 → 降低延迟，但写性能略降
```

### 4.2 分库分表回顾（与MySQL.md联动）

```
何时需要分库分表：
  单表 > 2000万行 或 单库 > 500GB → 考虑分片
  先优化（索引/缓存/读写分离），再分片（复杂度大增）

分片键选择原则：
  高基数（值多样化）、查询频率高、均匀分布、业务相关
  差例：性别（基数低）、时间（热点集中）
  好例：userId、orderId（雪花ID高位取模）

跨库问题及解法：
  跨库 JOIN → 业务层聚合 / ES 冗余字段
  跨库事务 → 分布式事务（TCC/Saga）
  全局唯一ID → 雪花算法
  全局排序分页 → 各库查后合并排序（scatter-gather）
```

## 五、异步化与削峰 ⭐⭐⭐⭐

```
高并发三板斧：缓存 + 异步 + 限流

异步化场景：
  ① 非核心操作异步：下单成功后发邮件/短信/积分 → MQ异步
  ② 写操作削峰：秒杀减库存 → 先写MQ，消费端匀速消费
  ③ 批量合并：日志批量写DB，减少IO次数

削峰设计：
  流量洪峰（秒杀/大促）→ MQ作为缓冲层
  上游无限速，MQ堆积
  下游按处理能力匀速消费
  → DB/第三方服务不被压垮

异步场景的最终一致性：
  MQ消费失败 → 死信队列 → 告警+人工处理
  消费幂等 → 业务ID去重（唯一索引/Redis set）
```

## 六、面试标准答法

**Q: 令牌桶和漏桶的区别？**

> 漏桶强制匀速输出：请求放入桶中排队，以固定速率流出，桶满则丢弃，无论上游突发多大，下游始终匀速。令牌桶控制平均速率但允许突发：以固定速率往桶里放令牌，请求来了取令牌，平时令牌积累，突发时一次性消耗积攒的令牌，允许短时超过均值的流量。**接口限流用令牌桶**（允许合理突发更友好）；**保护脆弱下游用漏桶**（数据库等不能承受突发）。

**Q: 高并发场景如何保证缓存和数据库一致性？**

> 采用 Cache-Aside 模式：写操作时**先更新 DB，再删除缓存**（不更新缓存，避免并发写导致新旧值覆盖问题）；读操作时缓存未命中再查 DB 并回写缓存。删缓存失败时通过 MQ 异步重试，或通过 Canal 监听 binlog 自动删除。对一致性要求极高的场景（如库存），用 Redis + Lua 做原子操作，或强制读主库。

## 七、常见追问

**Q: 固定窗口限流的临界点突刺如何用滑动窗口解决？**
> 固定窗口在 1s 结束时刻前后各允许 100 个请求，0.2s 内 200 个通过。滑动窗口将 1s 切成 10 个 100ms bucket，统计最近 10 个 bucket 的总量，任何时刻向前推 1s 的请求总数都不超过 100，从根本上消除临界点突刺。代价是需要额外存储每个 bucket 的计数，但资源开销很小。

**Q: 多级缓存架构如何保证各层一致性？**
> 各层缓存都设置较短的 TTL 作为兜底，TTL 越靠近数据源越长（DB永久，Redis分钟级，本地缓存秒级）。变更时：① 先更新 DB；② 删 Redis 缓存；③ 发 MQ 通知各节点清本地缓存（广播消费）；④ CDN 层通过缓存刷新 API 失效。最终一致性即可，极短TTL作为保障，不追求强一致。

**Q: 秒杀系统如何设计限流？**
> 分层限流：① **前端**：按钮点击后置灰，防止重复提交；② **CDN/Nginx**：IP限流，单IP频率限制；③ **Gateway**：令牌桶全局限流，只放 1% 流量进入核心服务；④ **应用层**：Sentinel 接口级限流 + 本地令牌桶二次过滤；⑤ **Redis**：原子 decr 库存预扣，库存为 0 直接拒绝；⑥ **MQ**：通过的请求异步写入MQ，消费端匀速下单。整体是漏斗模型，每层都在过滤流量。

**Q: 读写分离后如何解决主从延迟下的强一致读？**
> 三种方案：① **强制读主库**：对一致性敏感的操作（下单后查订单）通过 Hint 或注解强制路由到主库；② **等待位点**：记录写操作时的主库 binlog 位点，读从库时等待从库同步到该位点后再读（ShardingSphere 支持）；③ **直接不用从库**：将缓存作为读扩展手段代替读写分离，一致性由缓存更新策略保证。生产中方案①最常用，成本最低。
