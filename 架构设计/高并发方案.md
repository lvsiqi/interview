## 一、限流算法 ⭐⭐⭐⭐⭐

### 1.1 四种限流算法对比

| 算法 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **固定窗口** | 固定时间段内计数，超阈值拒绝 | 简单 | 临界点突刺（窗口边界2倍流量穿透）| 不推荐 |
| **滑动窗口** | 时间窗口随时间滑动，多个小格统计 | 解决突刺，平滑 | 实现稍复杂 | 接口限流 |
| **漏桶** | 请求入桶，固定速率出桶，桶满则拒绝 | 强制匀速输出，保护下游 | 无法应对突发，合法突发也被限 | 保护数据库等脆弱下游 |
| **令牌桶** | 固定速率往桶里放令牌，请求取令牌，取到则通过 | **允许一定突发**（桶内积攒的令牌）| 令牌积累过多可能瞬间大量放行 | **最常用**，接口/API限流 |

### 1.2 令牌桶算法详解 ⭐⭐⭐⭐⭐

```
核心参数：
  capacity：桶容量（最大允许突发量）
  rate：令牌放入速率（QPS上限）

流程：
  ① 定时（或按需）以固定速率向桶中放令牌，超出桶容量则丢弃
  ② 请求到来时尝试取令牌：
     有令牌 → 请求通过
     无令牌 → 拒绝（或等待）

突发处理：
  平时流量低 → 令牌积累（最多积满 capacity 个）
  突发流量来时 → 消耗积累的令牌 → 允许短时高并发
  突发超过 capacity → 超出部分被拒绝
  ```

**Redis Lua 令牌桶实现（分布式限流，字节常考手写）：**

```lua
-- KEYS[1]=令牌桶key
-- ARGV[1]=令牌桶容量capacity, ARGV[2]=每秒放入速率rate
-- ARGV[3]=当前时间戳(秒,支持小数), ARGV[4]=本次请求消耗令牌数

local key      = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate     = tonumber(ARGV[2])
local now      = tonumber(ARGV[3])
local cost     = tonumber(ARGV[4])

-- 读取桶的当前状态（令牌数 + 上次填充时间）
local info = redis.call('HMGET', key, 'tokens', 'last_time')
local tokens    = tonumber(info[1]) or capacity   -- 首次默认满桶
local last_time = tonumber(info[2]) or now

-- 计算距上次填充经过的时间，补充令牌
local elapsed = math.max(0, now - last_time)
local new_tokens = math.min(capacity, tokens + elapsed * rate)

-- 判断令牌是否充足
if new_tokens < cost then
    -- 令牌不足：更新状态后拒绝
    redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
    redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1) -- 设置过期时间，避免垃圾key
    return 0  -- 拒绝
end

-- 令牌充足：扣减后放行
new_tokens = new_tokens - cost
redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)
return 1  -- 通过
```

```java
// Java 调用示例
public boolean tryAcquire(String key, int capacity, int rate, int cost) {
    String luaScript = "...";  // 上方 Lua 脚本
    double now = System.currentTimeMillis() / 1000.0;  // 秒级时间戳（支持小数）
    Long result = (Long) redisTemplate.execute(
        new DefaultRedisScript<>(luaScript, Long.class),
        Collections.singletonList(key),
        capacity, rate, now, cost
    );
    return result != null && result == 1L;
}
```

**核心设计要点：**
```
① 用 HMSET 存 tokens（当前令牌数）+ last_time（上次计算时间）
② 每次请求时"按需补充"令牌（惰性计算，不用定时任务）：
   新令牌 = min(capacity, 旧令牌 + 经过秒数 × 速率)
③ 整个 Lua 脚本原子执行，无并发安全问题
④ 时间用秒级浮点数（支持 0.1 秒精度），比毫秒整数更灵活
```

### 1.3 滑动窗口算法详解

```
固定窗口的临界点突刺问题：
  窗口 [0s,1s] 允许100个请求，计数清零时刻 1s 前后各100个请求
  → 0.9s~1.1s 200ms内 200个请求通过，实际QPS可达1000

滑动窗口解法：
  将 1s 窗口切成 10 个 100ms 的小格（bucket）
  每次统计最近 1s（10个格）的总请求数
  随时间推进，最老的格被丢弃，加入最新的格
  → 任意1s内的请求总数不超过阈值

Sentinel 的 LeapArray 实现：
  数组长度 = windowCount（循环数组）
  每个 bucket 存：通过数/拒绝数/异常数/慢调用数
  时间戳对应的 bucket = (timestamp / bucketLengthMs) % windowCount
```
**Redis 滑动窗口限流 Lua 脚本：**
```lua
-- 基于 Redis 有序集合（Sorted Set）的滑动窗口限流器
-- 使用自增序列保证 member 唯一，解决同一毫秒内多个请求的计数问题

-- KEYS[1]   : 限流键名，例如 "rate_limit:api:/users:123"
-- ARGV[1]   : 当前时间戳（毫秒），建议使用 Redis 的 TIME 命令获取以保证一致性
-- ARGV[2]   : 滑动窗口大小（毫秒），例如 1000 表示 1 秒
-- ARGV[3]   : 窗口内允许的最大请求数（阈值）

local key = KEYS[1]
local now = tonumber(ARGV[1])          -- 当前毫秒时间戳
local window = tonumber(ARGV[2])        -- 窗口大小（毫秒）
local threshold = tonumber(ARGV[3])     -- 阈值

-- 1. 生成唯一 member
--    使用一个独立的计数器键（key .. ':seq'）来生成自增序列号
--    这样 member = "当前时间戳:序列号"，确保同一毫秒内多个请求的 member 也不同
local seq = redis.call('INCR', key .. ':seq')
local member = now .. ':' .. seq

-- 2. 清理窗口外的旧记录
--    有序集合的 score 存储的是请求时间戳（毫秒）
--    删除所有 score 小于 (now - window) 的元素，即超出当前滑动窗口的老请求
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 3. 获取当前窗口内的请求总数
local count = redis.call('ZCARD', key)

-- 4. 判断是否允许本次请求
if count < threshold then
    -- 允许：将本次请求的记录加入有序集合
    --   score = now, member = 唯一标识
    redis.call('ZADD', key, now, member)
    
    -- 5. 设置限流主键的过期时间
    --    窗口大小转换为秒，并额外 +1 秒作为缓冲，确保窗口结束后的数据能及时释放
    redis.call('EXPIRE', key, math.ceil(window/1000) + 1)
    
    -- 6. 设置计数器键的过期时间（与主键一致）
    --    避免计数器键无限增长，占用内存
    redis.call('EXPIRE', key .. ':seq', math.ceil(window/1000) + 1)
    
    return 1  -- 返回 1 表示请求通过
else
    return 0  -- 返回 0 表示请求被限流
end
```

### 1.4 分布式限流 ⭐⭐⭐⭐

#### 1.4.1 为什么需要分布式限流

```
单机限流的问题：

场景：订单接口总限流 1000 QPS，部署了 10 台机器
  单机限流配 100 QPS/台 → 看似 10×100=1000 QPS

但实际情况不是这样：
  ① 流量不均匀：负载均衡不完全均匀，某些机器可能分到 200 QPS，其他才 50
     → 总量远没到 1000，部分机器就开始拒绝了
  ② 扩缩容后失效：扩到 20 台 → 20×100=2000 QPS（超了总限制）
     缩到 5 台 → 5×100=500 QPS（资源浪费）
     → 每次扩缩容都要改配置
  ③ 无法做全局精确控制：整个集群到底通过了多少请求？单机限流无法统计

分布式限流核心目标：
  不管有多少台机器，集群总 QPS 不超过设定的全局阈值
```

#### 1.4.2 方案一：Redis + Lua 脚本（最通用）

```
原理：用 Redis 做全局共享计数器，所有节点共同读写同一个 key
  → 每次请求先到 Redis 判断是否超限
  → Lua 脚本保证"读计数+判断+写计数"原子执行

两种实现：

① 滑动窗口（ZSet）：
   上面 1.3 已有 Lua 脚本，直接复用即可
   所有节点共用同一个 Redis key，天然是分布式的

② 令牌桶（Hash）：
   上面 1.2 已有 Lua 脚本，同理
   所有节点都请求同一个 Redis 令牌桶 key

优点：
  实现简单，逻辑透明
  与语言/框架无关，任何服务都能接入
  支持多维度限流（按 API、按用户、按 IP 分别设 key）

缺点：
  每次请求都要访问 Redis，增加 1~3ms 延迟
  Redis 成为瓶颈点：所有限流判断集中到 Redis
  Redis 宕机 → 限流失效（需降级策略）
```

**多维度限流 key 设计：**

```
按接口限流：  rate_limit:api:/order/create      → 全局 1000 QPS
按用户限流：  rate_limit:user:12345              → 单用户 10 QPS
按IP限流：    rate_limit:ip:192.168.1.100        → 单IP 50 QPS
组合限流：    同时配多个 key，任一超限则拒绝
```

#### 1.4.3 方案二：Sentinel 集群限流（TokenServer 模式）

```
原理：
  指定一台机器作为 TokenServer（令牌中心），其余为 TokenClient
  所有请求先向 TokenServer 申请令牌，拿到令牌才放行

                    ┌─────────────┐
                    │ TokenServer │ ← 持有全局令牌池（1000 QPS）
                    │ （令牌中心） │ ← 各 Client 通过 Netty 长连接申请
                    └──────┬──────┘
               ┌───────────┼───────────┐
               ↓           ↓           ↓
          Client A    Client B    Client C
          (服务实例)   (服务实例)   (服务实例)
          申请令牌     申请令牌     申请令牌

工作流程：
  ① TokenServer 启动，加载集群限流规则（全局 1000 QPS）
  ② 各 Client 启动后连接 TokenServer（Netty 长连接，低延迟）
  ③ 请求到来 → Client 向 TokenServer 请求令牌
     → 拿到令牌 → 放行；拿不到 → 拒绝
  ④ TokenServer 宕机 → 自动降級为单机限流模式（不影响可用性）

优点：
  Sentinel 原生支持，配置简单（Dashboard 可配）
  不依赖 Redis，延迟比 Redis 方案低
  TokenServer 宕机有降级机制

缺点：
  TokenServer 是单点（虽然可故障转移，但切换有短暂空窗）
  需要额外部署 TokenServer 进程
  适合 Java 生态，非 Java 服务无法直接使用
```

**配置示例：**

```java
// TokenClient 端配置
ClusterClientConfig clientConfig = new ClusterClientConfig();
clientConfig.setServerHost("token-server-ip");
clientConfig.setServerPort(18730);
ClusterClientConfigManager.applyNewConfig(clientConfig);

// 集群限流规则（在 Nacos/Dashboard 配置更方便）
FlowRule rule = new FlowRule();
rule.setResource("queryOrder");
rule.setCount(1000);                           // 全局 1000 QPS
rule.setClusterMode(true);                     // 开启集群模式
rule.setClusterConfig(new ClusterFlowConfig()
    .setFlowId(1001L)                          // 规则ID
    .setThresholdType(ClusterRuleConstant.FLOW_THRESHOLD_GLOBAL) // 全局阈值
);
```

#### 1.4.4 方案三：网关层统一限流（Gateway + Redis）

```
原理：
  所有外部请求必经 API 网关 → 在网关层做全局限流
  Spring Cloud Gateway 内置 RequestRateLimiter，基于 Redis 令牌桶

                        ┌──────────────┐
        用户请求 ──────→ │   Gateway    │ ──→ Redis（令牌桶）
                        │ RateLimiter  │ ←── 有令牌 → 转发到后端服务
                        │              │ ←── 无令牌 → 返回 429
                        └──────────────┘

优点：
  统一入口，不需要每个服务都接入限流逻辑
  天然分布式（多个 Gateway 实例共用 Redis 令牌桶）
  支持按路由/用户/IP 等维度灵活限流

缺点：
  只能限外部流量，服务间互调的流量管不到
  依赖 Redis（同方案一）
```

**Gateway 限流配置：**

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 500   # 令牌填充速率：500/秒
                redis-rate-limiter.burstCapacity: 1000   # 桶容量：允许突发1000
                redis-rate-limiter.requestedTokens: 1    # 每请求消耗1令牌
                key-resolver: "#{@userKeyResolver}"      # 限流维度
```

```java
// 自定义限流维度：按用户ID限流
@Bean
public KeyResolver userKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getHeaders().getFirst("X-User-Id")
    );
}

// 按IP限流
@Bean
public KeyResolver ipKeyResolver() {
    return exchange -> Mono.just(
        exchange.getRequest().getRemoteAddress().getAddress().getHostAddress()
    );
}
```

#### 1.4.5 三种方案对比

| 对比 | Redis + Lua | Sentinel 集群限流 | Gateway 网关限流 |
|------|------------|-----------------|----------------|
| 精确度 | 高（全局原子计数）| 高（TokenServer 集中控制）| 高（Redis 令牌桶）|
| 延迟 | 1~3ms（Redis 网络）| <1ms（Netty 长连接）| 1~3ms（Redis 网络）|
| 依赖 | Redis | TokenServer 进程 | Redis + Gateway |
| 适用范围 | 任意语言/服务 | Java 生态 | 仅外部入口流量 |
| 运维复杂度 | 低 | 中（需管理 TokenServer）| 低 |
| 故障降级 | 需自行实现 | 自动降级为单机限流 | 需自行实现 |
| 典型场景 | 通用方案 | Sentinel 体系内首选 | API 入口统一管控 |

#### 1.4.6 生产实践：分层限流

```
实际生产中不是只选一种，而是分层组合：

① 网关层（Gateway + Redis 令牌桶）：
   全局入口限流，保护整个集群不被外部流量压垮
   按路由/IP/用户多维度限流

② 服务层（Sentinel 集群限流 或 Redis Lua）：
   各服务按接口精细化限流
   服务间互调也能限流（网关层管不到）

③ 单机层（Sentinel 单机模式）：
   兜底保护，即使 Redis/TokenServer 都挂了
   保护单台机器不被打满 CPU/内存

漏斗模型：每一层过滤一部分流量，层层递减
  外部 100万 QPS → 网关放 10万 → 服务层放 1万 → DB 承受 1千
```

## 二、熔断 & 降级 ⭐⭐⭐⭐

### 2.1 熔断 vs 降级 vs 限流的区别

```
限流（Rate Limiting）：
  控制流量入口，超出阈值直接拒绝
  目的：保护自身不被压垮

熔断（Circuit Breaker）：
  自动检测下游服务的健康状态，异常率高时断开连接
  目的：防止级联故障蔓延（雪崩）

降级（Fallback）：
  服务不可用时返回兜底数据（缓存/默认值/提示语）
  目的：保证核心功能可用，牺牲非核心功能

关系：熔断触发后通常执行降级逻辑（fallback）
```

### 2.2 雪崩效应与防护

```
雪崩场景：
  A → B → C → D（服务调用链）
  D 响应慢 → C 线程池耗尽等待 D → B 线程池耗尽等待 C → A 挂掉
  → 一个底层服务故障，导致整条链路雪崩

防护手段（多层防御）：
① 超时机制：每个调用设合理超时，不无限等待
② 熔断（Sentinel/Hystrix）：下游故障率超阈值快速失败
③ 限流：控制进入量，防止超载
④ 舱壁隔离：线程池隔离，不同服务用不同线程池，故障不传染
⑤ 降级兜底：返回缓存/默认值，保证核心业务可用
⑥ 限制重试：失败重试加指数退避 + Jitter（抖动），防止重试风暴
```

### 2.3 Sentinel 流控规则 Java API ⭐⭐⭐⭐⭐

**Sentinel 三大核心概念：**
```
资源（Resource）：保护的目标，可以是接口/方法/代码块（用 @SentinelResource 标注）
规则（Rule）：    针对资源配置的流控/降级策略
 Slot Chain：    责任链处理（统计 → 规则检查 → 执行）
```

**流控规则（FlowRule）：**

```java
// 方式一：代码硬编码（开发测试用）
private static void initFlowRules() {
    List<FlowRule> rules = new ArrayList<>();

    FlowRule rule = new FlowRule();
    rule.setResource("queryOrder");          // 资源名
    rule.setGrade(RuleConstant.FLOW_GRADE_QPS); // 限流维度：QPS（另一种是并发线程数）
    rule.setCount(100);                      // QPS阈值：100
    rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT); // 超出直接拒绝
    // 其他流控效果：
    // CONTROL_BEHAVIOR_WARM_UP        预热（冷启动，10s内从低阈值升到设定值）
    // CONTROL_BEHAVIOR_RATE_LIMITER   排队等待（漏桶，设置超时时间）
    rules.add(rule);

    FlowRuleManager.loadRules(rules);
}

// 方式二：注解方式（Spring项目推荐）
@SentinelResource(
    value = "queryOrder",
    blockHandler = "queryOrderBlocked",    // 触发限流/熔断走此方法
    fallback = "queryOrderFallback"        // 业务异常走此方法
)
public OrderDTO queryOrder(Long orderId) {
    return orderService.getById(orderId);
}

// 限流时的兜底方法（参数必须含 BlockException）
public OrderDTO queryOrderBlocked(Long orderId, BlockException ex) {
    log.warn("queryOrder被限流, orderId={}", orderId);
    return OrderDTO.empty(); // 返回空对象
}

// 业务异常降级方法
public OrderDTO queryOrderFallback(Long orderId, Throwable t) {
    log.error("queryOrder降级, orderId={}", orderId, t);
    return OrderDTO.empty();
}
```

**降级规则（DegradeRule）：**

```java
private static void initDegradeRules() {
    List<DegradeRule> rules = new ArrayList<>();

    DegradeRule rule = new DegradeRule();
    rule.setResource("queryOrder");

    // 策略一：慢调用比例熔断
    rule.setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType());
    rule.setCount(1000);          // 慢调用RT阈值：超过1000ms算慢
    rule.setSlowRatioThreshold(0.5); // 慢调用比例超过50%则熔断
    rule.setMinRequestAmount(10);    // 最小请求数（样本不足不熔断）
    rule.setStatIntervalMs(10000);   // 统计时间窗口10s
    rule.setTimeWindow(5);           // 熔断后5s进入半开状态（探测恢复）

    // 策略二：异常比例熔断（改 grade 即可）
    // rule.setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType());
    // rule.setCount(0.5);  // 异常比例超50%熔断

    // 策略三：异常数熔断
    // rule.setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType());
    // rule.setCount(10);   // 异常数超10个熔断

    rules.add(rule);
    DegradeRuleManager.loadRules(rules);
}
```

**Sentinel 熔断状态机：**
```
关闭（Closed）→ 正常统计，判断是否达到熔断条件
    ↓ 触发阈值
打开（Open）  → 所有请求快速失败（走blockHandler），不调用下游
    ↓ 经过 timeWindow 秒
半开（Half-Open）→ 放一个探测请求，
    成功 → 回到关闭状态
    失败 → 重新打开，再等 timeWindow 秒
```

**Nacos 动态规则推送（生产推荐）：**

### 2.4 重试策略设计 ⭐⭐⭐

```
重试是高并发的双刃剑：
  合理重试 → 提高成功率（瞬时网络抖动自恢复）
  盲目重试 → 重试风暴 → 打垮下游（比原始请求更大的流量）

重试三原则：
  ① 只重试可恢复异常（超时/网络抖动/503），不重试业务异常（参数错误/余额不足）
  ② 限制重试次数（通常 2~3 次）
  ③ 指数退避 + 随机抖动（Jitter），避免所有节点同一时刻重试
```

**指数退避 + Jitter 算法：**

```
第1次重试：等待 base × 2^0 + random(0, jitter) = 1s + [0~0.5s]
第2次重试：等待 base × 2^1 + random(0, jitter) = 2s + [0~0.5s]
第3次重试：等待 base × 2^2 + random(0, jitter) = 4s + [0~0.5s]
最大等待不超过 maxDelay（如 30s）

为什么加 Jitter（随机抖动）？
  100个客户端同时超时 → 都在 1s 后重试 → 重试风暴
  加随机抖动后：各客户端在 1s~1.5s 之间随机重试 → 分散压力
```

```java
// Spring Retry 注解方式（最常用）
@Retryable(
    retryFor = {TimeoutException.class, ConnectException.class}, // 只重试特定异常
    maxAttempts = 3,
    backoff = @Backoff(
        delay = 1000,        // 初始等待1s
        multiplier = 2,      // 每次×2
        maxDelay = 10000,    // 最大等待10s
        random = true        // 开启 Jitter
    )
)
public OrderDTO queryOrder(Long orderId) {
    return orderClient.getOrder(orderId);
}

@Recover  // 重试全部失败后的降级方法
public OrderDTO queryOrderRecover(Exception e, Long orderId) {
    log.error("重试3次后仍失败, orderId={}", orderId, e);
    return OrderDTO.empty();
}
```

**重试风暴防护：**

```
场景：下游服务 B 宕机5s → 上游 A 的 1000 个请求全部超时
  → 每个请求重试3次 → 3000 个重试请求涌向 B
  → B 刚恢复就被打垮 → 再次宕机 → 死循环

防护手段：
  ① 指数退避 + Jitter（分散重试时间）
  ② 熔断器（Sentinel）：失败率高时直接熔断，不再重试
  ③ 重试预算（Retry Budget）：全局限制重试比例 ≤ 总请求10%
  ④ 限制并发重试数：信号量控制同时在重试的请求数不超过阈值
```

## 三、缓存架构设计 ⭐⭐⭐⭐⭐

### 3.1 多级缓存架构

```
请求链路（越靠近用户越快）：

Client（浏览器缓存）
  ↓ 未命中
CDN（边缘节点缓存，静态资源/接口缓存）
  ↓ 未命中
Nginx（本地 Lua/Shared Memory 缓存，单机内存）
  ↓ 未命中
应用层本地缓存（Caffeine/Guava Cache，JVM堆内）
  ↓ 未命中
分布式缓存（Redis，共享，支持集群）
  ↓ 未命中
数据库（MySQL）

典型延迟参考：
  本地缓存：< 1ms
  Redis：  1~3ms
  MySQL：  10~100ms
```

### 3.2 本地缓存 Caffeine ⭐⭐⭐⭐

```
Caffeine 是 Java 性能最好的本地缓存（Guava Cache 的升级版）

核心特性：
  W-TinyLFU 淘汰算法（LFU改进，比LRU命中率更高）
  异步刷新（refreshAfterWrite）
  自动过期（expireAfterWrite / expireAfterAccess）
  最大 size 限制（evict 淘汰）
```

```java
Cache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)              // 最多10000条
    .expireAfterWrite(5, TimeUnit.MINUTES)  // 写入5分钟后过期
    .refreshAfterWrite(1, TimeUnit.MINUTES) // 1分钟后异步刷新（不阻塞请求）
    .recordStats()                    // 开启统计
    .build(key -> userService.load(key));   // 加载函数（CacheLoader）

// 使用
User user = cache.get(userId);  // 命中返回，未命中自动加载
```

**本地缓存一致性问题：**
```
多节点本地缓存，某节点更新DB后，其他节点本地缓存仍是旧数据
解法：
  ① 短过期时间（容忍短暂不一致）
  ② MQ广播失效通知（canal订阅binlog → 发MQ → 各节点清本地缓存）
  ③ 数据一致性要求高的场景不用本地缓存，只用Redis
```

### 3.3 缓存更新策略

```
① Cache-Aside（旁路缓存）— 最常用 ⭐⭐⭐⭐⭐
   读：先读缓存，命中返回；未命中读DB，写入缓存
   写：先更新DB，再删除缓存（不是更新缓存）

   为什么删除而不是更新缓存？
   → 避免并发写时缓存和DB不一致（两个写操作顺序不确定）
   → 删除简单，下次读时重建（懒加载）

② Read/Write Through（穿透读写）
   应用只操作缓存，缓存层负责同步DB（缓存框架封装了DB操作）
   少见，多用于数据库缓存层

③ Write Behind（写回）
   只写缓存，异步批量刷DB（类似OS的Page Cache dirty机制）
   性能最好，但有数据丢失风险
```

**Cache-Aside 的先删缓存 vs 先更新DB：**
```
❌ 先删缓存再更新DB（延迟双删也有问题）：
   ① 删缓存
   ② 另一请求读缓存未命中，读了旧DB值写入缓存
   ③ 更新DB
   → 缓存里是旧值！

✅ 先更新DB再删缓存（标准方案）：
   ① 更新DB（成功）
   ② 删缓存（成功）→ 下次读自动重建

   极端情况：①成功②失败 → 缓存未删，读到旧值
   解法：删缓存失败 → 重试（MQ异步重试）或 引入 binlog + canal 监听删除
```

## 四、读写分离 & 分库分表 ⭐⭐⭐⭐

### 4.1 读写分离

```
场景：读多写少（典型：读:写 = 9:1）
方案：主库负责写，从库负责读，主从复制同步数据

实现方式：
① ShardingSphere-JDBC（代码层，无需中间件）
   配置主从数据源，注解/AOP 分发读写

② MyCat / ProxySQL（代理层，中间件）
   应用无感知，中间件解析SQL自动路由

③ 手动（@Primary/@Qualifier多数据源）
```

**主从复制延迟问题：**
```
现象：写主库后立即读从库，可能读到旧数据（复制延迟 100ms~1s）
解决方案（按强度排序）：

① 容忍延迟（80%场景）：
   展示类数据（列表、详情）允许短暂不一致，用户无感知

② 读写路由到主库（强一致场景）：
   下单后查询订单状态 → 此请求强制走主库
   实现：请求上下文标记（ThreadLocal），ShardingSphere 支持 hint 路由

③ 等待同步（不推荐，性能差）：
   读前等待主从延迟消除（轮询 slave position）

④ 半同步复制（MySQL 5.7+）：
   主库写完等至少1个从库确认收到 → 降低延迟，但写性能略降
```

### 4.2 分库分表回顾（与MySQL.md联动）

```
何时需要分库分表：
  单表 > 2000万行 或 单库 > 500GB → 考虑分片
  先优化（索引/缓存/读写分离），再分片（复杂度大增）

分片键选择原则：
  高基数（值多样化）、查询频率高、均匀分布、业务相关
  差例：性别（基数低）、时间（热点集中）
  好例：userId、orderId（雪花ID高位取模）

跨库问题及解法：
  跨库 JOIN → 业务层聚合 / ES 冗余字段
  跨库事务 → 分布式事务（TCC/Saga）
  全局唯一ID → 雪花算法
  全局排序分页 → 各库查后合并排序（scatter-gather）
```

## 五、异步化与削峰 ⭐⭐⭐⭐

### 5.1 异步化场景

```
高并发三板斧：缓存 + 异步 + 限流

异步化场景：
  ① 非核心操作异步：下单成功后发邮件/短信/积分 → MQ异步
  ② 写操作削峰：秒杀减库存 → 先写MQ，消费端匀速消费
  ③ 批量合并：日志批量写DB，减少IO次数

削峰设计：
  流量洪峰（秒杀/大促）→ MQ作为缓冲层
  上游无限速，MQ堆积
  下游按处理能力匀速消费
  → DB/第三方服务不被压垮

异步场景的最终一致性：
  MQ消费失败 → 死信队列 → 告警+人工处理
  消费幂等 → 业务ID去重（唯一索引/Redis set）
```

### 5.2 秒杀完整异步流程 ⭐⭐⭐⭐⭐

```
【秒杀系统完整流程（漏斗模型）】

用户点击「抢购」
    ↓
① 前端防重：按钮置灰 + 防抖（500ms内只发一次请求）
    ↓
② CDN/Nginx 层：
   - 静态页面走CDN（商品详情页提前预热）
   - IP级限流（单IP 5次/秒，防刷）
    ↓
③ Gateway 层：
   - 令牌桶全局限流（如10万QPS只放1万进来）
   - 用户Token校验（未登录直接拦截）
    ↓
④ 应用层快速预检（内存级，纳秒级响应）：
   - 本地标记：该商品库存是否耗尽（AtomicBoolean）
   - 库存耗尽 → 直接返回「已售罄」（不查Redis）
    ↓
⑤ Redis 原子扣减库存：
   - Lua 脚本：先判断库存 > 0，再 DECR
   - 库存 ≤ 0 → 返回失败，同时设本地售罄标记
   - 扣减成功 → 记录用户已参与（Redis Set 去重，防重复下单）
    ↓
⑥ 发送 MQ 消息（异步下单）：
   - 消息内容：userId + skuId + 订单流水号（预生成）
   - 快速返回前端「排队中，请稍候」
    ↓
⑦ MQ 消费端（匀速消费，保护DB）：
   - 幂等校验（订单流水号唯一索引）
   - 真正创建订单 + 扣减DB库存（乐观锁/Lua）
   - DB扣减失败（并发冲突）→ 库存回滚到Redis + 通知用户
    ↓
⑧ 结果通知：
   - WebSocket / 短轮询：前端定时查询订单状态
   - 订单创建成功 → 推送「抢购成功，去支付」
   - 订单失败 → 推送「手慢了，下次再来」
```

**关键代码：Redis 原子扣减库存**

```lua
-- KEYS[1]=库存key, KEYS[2]=用户去重set, ARGV[1]=userId, ARGV[2]=skuId
local stock_key = KEYS[1]
local user_set  = KEYS[2]
local userId    = ARGV[1]

-- 检查用户是否已参与（防重复下单）
if redis.call('SISMEMBER', user_set, userId) == 1 then
    return -1  -- 已参与，拒绝
end

-- 检查并扣减库存
local stock = tonumber(redis.call('GET', stock_key) or '0')
if stock <= 0 then
    return 0  -- 库存不足
end

redis.call('DECR', stock_key)
redis.call('SADD', user_set, userId)
redis.call('EXPIRE', user_set, 86400)  -- 去重set保留1天
return 1  -- 扣减成功
```

```java
@Service
public class SeckillService {
    // 本地售罄标记，避免每次都查Redis
    private final Map<Long, AtomicBoolean> soldOutMap = new ConcurrentHashMap<>();

    public SeckillResult doSeckill(Long userId, Long skuId) {
        // ① 本地快速过滤
        AtomicBoolean soldOut = soldOutMap.computeIfAbsent(skuId, k -> new AtomicBoolean(false));
        if (soldOut.get()) {
            return SeckillResult.fail("已售罄");
        }

        // ② Redis 原子扣减
        Long result = (Long) redisTemplate.execute(
            seckillScript,
            Arrays.asList("stock:" + skuId, "user:seckill:" + skuId),
            String.valueOf(userId)
        );

        if (result == null || result <= 0) {
            if (result != null && result == 0) {
                soldOut.set(true);  // 标记本地售罄
            }
            return result == -1 ? SeckillResult.fail("请勿重复下单") : SeckillResult.fail("已售罄");
        }

        // ③ 发 MQ 异步下单
        String orderNo = IdGenerator.nextId();  // 预生成订单号
        seckillMqProducer.send(new SeckillMessage(userId, skuId, orderNo));
        return SeckillResult.queuing(orderNo);  // 返回排队中
    }
}
```

## 六、幂等性设计 ⭐⭐⭐⭐⭐

### 6.1 为什么需要幂等

```
高并发场景中，以下情况会产生重复请求：
  ① 网络超时重试：客户端超时后重试，服务端可能已执行成功
  ② MQ 重复消费：消费者消费成功但 ACK 未到达 Broker，消息重推
  ③ 用户重复提交：多次点击按钮，短时间内发多个请求
  ④ 负载均衡重试：LB检测到超时，重试到其他节点

幂等性：同一操作执行多次，结果与执行一次相同
  √ 查询（GET）天然幂等
  × 创建（POST）、扣款、库存扣减 需要设计幂等
```

### 6.2 幂等方案汇总

**方案一：唯一索引（最简单可靠）**

```sql
-- 订单表加唯一索引，重复插入直接抛DuplicateKeyException
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    order_no VARCHAR(64) NOT NULL UNIQUE,  -- 业务唯一键
    user_id BIGINT,
    amount DECIMAL(10,2),
    status TINYINT DEFAULT 0
);
```

```java
// 捕获重复键异常，视为成功
try {
    orderMapper.insert(order);
} catch (DuplicateKeyException e) {
    log.warn("重复下单，order_no={}", order.getOrderNo());
    // 查出已存在的订单返回
    return orderMapper.selectByOrderNo(order.getOrderNo());
}
```

**方案二：Token 防重（防前端重复提交）**

```
流程：
① 进入下单页时，前端请求后端生成唯一 Token，存入 Redis（TTL 5分钟）
② 前端提交表单时携带 Token
③ 后端校验：Redis 中存在该 Token → 删除 Token → 处理业务
                Redis 中不存在 → 拒绝（重复请求）

关键：check + delete 必须原子操作（Lua 脚本）
```

```java
// 生成Token接口
public String generateToken(Long userId) {
    String token = UUID.randomUUID().toString();
    redis.setex("idempotent:token:" + token, 300, userId.toString());
    return token;
}

// 提交时校验并删除Token（Lua原子操作）
public boolean checkAndDeleteToken(String token) {
    String luaScript =
        "if redis.call('exists', KEYS[1]) == 1 then " +
        "  return redis.call('del', KEYS[1]) " +
        "else " +
        "  return 0 " +
        "end";
    Long result = (Long) redisTemplate.execute(
        new DefaultRedisScript<>(luaScript, Long.class),
        Collections.singletonList("idempotent:token:" + token)
    );
    return result != null && result == 1L;
}
```

**方案三：状态机（乐观锁 + 状态流转）**

```sql
-- 只允许正向状态流转，利用 WHERE 条件做乐观锁
UPDATE orders 
SET status = 2          -- 目标状态：已支付
WHERE order_no = '123' 
  AND status = 1;        -- 当前状态：待支付
-- 影响行数 = 0 说明已被其他线程处理，幂等返回
```

**方案四：Redis SETNX（分布式去重）**

```java
// MQ消费幂等：消费前用业务ID抢占Redis key
@RocketMQMessageListener(topic = "seckill-order", consumerGroup = "order-consumer")
public class SeckillOrderConsumer implements RocketMQListener<SeckillMessage> {
    @Override
    public void onMessage(SeckillMessage msg) {
        String idempotentKey = "consumed:" + msg.getOrderNo();
        // SETNX：返回true说明首次消费，返回false说明已消费过
        Boolean isFirst = redis.opsForValue()
            .setIfAbsent(idempotentKey, "1", 24, TimeUnit.HOURS);
        if (Boolean.FALSE.equals(isFirst)) {
            log.warn("重复消费，orderNo={}", msg.getOrderNo());
            return;  // 直接返回，不处理
        }
        // 正常处理业务
        createOrder(msg);
    }
}
```

### 6.3 各方案对比

| 方案 | 实现复杂度 | 性能 | 适用场景 |
|------|---------|------|--------|
| 唯一索引 | 低 | 中（DB写） | **创建类操作首选**，天然兜底 |
| Token防重 | 中 | 高（Redis） | 前端表单重复提交 |
| 状态机 | 低 | 高 | 有明确状态流转的业务（订单/审批）|
| Redis SETNX | 低 | 高（Redis） | **MQ消费幂等首选** |
| 乐观锁版本号 | 中 | 高 | 并发更新同一记录 |

### 6.4 面试标准答法

**Q: 如何保证接口幂等？**

> 根据场景选择方案：**创建类操作**（下单/支付）用数据库唯一索引，重复插入捕获 DuplicateKeyException 返回已有记录；**前端防重复提交**用 Token 机制，页面打开时生成唯一 Token 存 Redis，提交时 Lua 原子校验并删除 Token；**MQ消费幂等**用 Redis SETNX，消费前以消息ID为 key 抢占，抢不到说明已消费；**更新类操作**用乐观锁状态机，WHERE 条件限制当前状态，影响行数为 0 则幂等返回。核心原则：**用业务唯一ID识别重复，操作必须原子**（DB唯一索引或 Redis Lua）。

## 七、缓存三大问题 ⭐⭐⭐⭐⭐

### 7.1 缓存穿透

**定义：** 查询一个**根本不存在**的数据，缓存和DB都没有，每次请求都打穿到DB。

```
正常流程：请求 → 缓存命中 → 返回
穿透流程：请求不存在的 id=-1 → 缓存未命中 → 查DB → DB也没有
         → 不写缓存（没东西可写）→ 下次请求重复穿透
         → 恶意构造大量不存在的key → DB被打垮
```

**解决方案：**

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **缓存空值** | DB没查到时，缓存写入 `null`（短TTL，如60s） | 简单，成本低 | 占用缓存空间；数据插入后有短暂不一致 |
| **布隆过滤器** | 启动时把所有合法ID写入布隆过滤器，请求进来先过滤 | 完全拦截非法请求，不浪费DB资源 | 有误判（合法key被误拦）；需维护过滤器 |
| **接口参数校验** | 前置参数校验，非法格式/范围直接拒绝 | 简单有效（如ID必须>0）| 仅能拦截明显非法参数 |

```java
// 方案一：缓存空值示例
public User getUser(Long id) {
    String key = "user:" + id;
    String cached = redis.get(key);
    if (cached != null) {
        return "NULL".equals(cached) ? null : JSON.parse(cached, User.class);
    }
    User user = userMapper.selectById(id);
    if (user == null) {
        redis.setex(key, 60, "NULL");  // 缓存空值，TTL 60秒
    } else {
        redis.setex(key, 3600, JSON.toJSONString(user));
    }
    return user;
}
```

```java
// 方案二：布隆过滤器（Redisson）
RBloomFilter<Long> bloomFilter = redisson.getBloomFilter("user:bloom");
bloomFilter.tryInit(1_000_000L, 0.01);  // 100万元素，1%误判率

// 启动时预热（或数据变更时同步）
userMapper.selectAllIds().forEach(bloomFilter::add);

public User getUser(Long id) {
    if (!bloomFilter.contains(id)) {
        return null;  // 布隆过滤器判定不存在，直接拒绝
    }
    // 后续走正常缓存流程
}
```

### 7.2 缓存击穿

**定义：** 一个**热点 key 过期**的瞬间，大量并发请求同时穿透缓存，全部打到 DB。

```
场景：商品详情页 id=1001 是热点，QPS=10000
T0：key 过期
T0~T0+10ms：10000个请求全部缓存未命中 → 全部查DB → DB崩溃
T0+10ms：第一个请求写回缓存，后续请求才命中缓存
```

**解决方案：**

**① 互斥锁（Mutex Lock）— 保证只有一个请求重建缓存**

```java
public User getUser(Long id) {
    String key = "user:" + id;
    User user = redis.get(key);
    if (user != null) return user;

    // 缓存未命中，用分布式锁保证只有一个线程重建
    String lockKey = "lock:user:" + id;
    if (redis.setnx(lockKey, "1", 10)) {  // 抢锁，10秒超时
        try {
            // 双重检查（可能已被其他线程重建）
            user = redis.get(key);
            if (user == null) {
                user = userMapper.selectById(id);
                redis.setex(key, 3600, JSON.toJSONString(user));
            }
        } finally {
            redis.del(lockKey);  // 释放锁
        }
    } else {
        // 未抢到锁，短暂等待后重试（或直接查DB兜底）
        Thread.sleep(50);
        return getUser(id);  // 递归重试
    }
    return user;
}
```

**② 逻辑过期（热点数据永不过期）— 异步更新，不阻塞请求**

```java
// 缓存存储：value + 逻辑过期时间（不设Redis TTL，永不被动过期）
public class CacheData {
    private Object data;
    private long expireTime;  // 逻辑过期时间戳
}

public User getUser(Long id) {
    CacheData cached = redis.get("user:" + id);
    if (cached == null) return null;  // 永不过期，基本不会null

    if (cached.getExpireTime() > System.currentTimeMillis()) {
        return (User) cached.getData();  // 未逻辑过期，直接返回
    }

    // 逻辑已过期 → 异步重建，当前请求先返回旧数据
    if (redis.setnx("lock:user:" + id, "1", 10)) {
        executor.submit(() -> {
            try {
                User newUser = userMapper.selectById(id);
                redis.set("user:" + id, new CacheData(newUser, 
                    System.currentTimeMillis() + 3600_000));
            } finally {
                redis.del("lock:user:" + id);
            }
        });
    }
    return (User) cached.getData();  // 直接返回旧数据，不阻塞
}
```

| 方案 | 一致性 | 可用性 | 适用场景 |
|------|---------|--------|---------|
| 互斥锁 | 强一致 | 有等待延迟 | 数据一致性要求高 |
| 逻辑过期 | 短暂旧数据 | **不阻塞，高可用** | 热点数据，允许短暂不一致 |
| 永不过期+主动更新 | 最终一致 | 最高 | 极热点（首页推荐等）|

### 7.3 缓存雪崩

**定义：** 大量 key **同时过期** 或 Redis **宕机**，请求全部打到 DB，导致 DB 崩溃。

**类型一：大量 key 同时过期**

```
原因：批量数据设置了相同的 TTL（如 3600s），同一批次全部在 3600s 后到期

解决方案：
① TTL 加随机偏移：
   redis.setex(key, 3600 + ThreadLocalRandom.current().nextInt(600), value)
   各 key 在 3600~4200s 之间随机过期，分散过期时间

② 热点数据永不过期（逻辑过期 + 主动更新）

③ 多级缓存兜底（本地缓存 Caffeine）：
   Redis 雪崩时，本地缓存仍可命中（5~10秒内保护 DB）
```

**类型二：Redis 宕机**

```
所有请求直接打到 DB → DB 被打垮

解决方案（多层防御）：
① Redis 高可用：哨兵模式（故障自动切换）/ Cluster 集群（数据分片）
② 服务降级：Redis 不可用时，返回默认值/静态页面，拒绝写请求
③ 熔断降级（Sentinel）：检测到 Redis 异常率高 → 熔断，快速失败
④ 本地缓存兜底：Caffeine 作为二级缓存，延长数据存活时间
⑤ 限流保护 DB：Redis 宕机时只放少量请求查 DB，其余返回服务繁忙
```

```java
// 完整的多级缓存 + 雪崩防护示例
@Service
public class UserService {
    private final Cache<Long, User> localCache = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(10, TimeUnit.SECONDS)  // 本地缓存10秒，兜底
        .build();

    public User getUser(Long id) {
        // L1：本地缓存
        User user = localCache.getIfPresent(id);
        if (user != null) return user;

        // L2：Redis（TTL加随机偏移，防雪崩）
        try {
            String cached = redis.get("user:" + id);
            if (cached != null) {
                user = JSON.parseObject(cached, User.class);
                localCache.put(id, user);
                return user;
            }
        } catch (Exception e) {
            log.warn("Redis异常，降级查DB", e);
            // Redis宕机时降级走DB
        }

        // L3：数据库
        user = userMapper.selectById(id);
        if (user != null) {
            int ttl = 3600 + ThreadLocalRandom.current().nextInt(600); // 随机TTL
            redis.setex("user:" + id, ttl, JSON.toJSONString(user));
            localCache.put(id, user);
        }
        return user;
    }
}
```

### 7.4 三大问题总结对比

| | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
|--|---------|---------|---------|
| **触发条件** | 查不存在的key | 热点key过期瞬间 | 大量key同时过期/宕机 |
| **影响范围** | 特定不存在key | 单个热点key | **大范围** |
| **核心解法** | 布隆过滤器/缓存空值 | 互斥锁/逻辑过期 | TTL随机化/高可用/多级缓存 |
| **面试关键词** | 布隆过滤器 | 双重检查锁、逻辑过期 | 随机TTL、哨兵/Cluster |

### 7.5 面试标准答法

**Q: 缓存穿透、击穿、雪崩的区别和解决方案？**

> **穿透**：请求不存在的key，缓存和DB都没有，解决方案是布隆过滤器（拦截非法key）或缓存空值（短TTL）。**击穿**：热点key过期瞬间大量并发打到DB，解决方案是互斥锁（只有一个线程重建缓存）或逻辑过期（key永不过期，异步重建，请求先返回旧数据）。**雪崩**：大量key同时过期或Redis宕机，解决方案是TTL加随机偏移防止同时过期，配合本地缓存Caffeine兜底，Redis做哨兵/集群保高可用，极端情况熔断降级保护DB。

## 八、连接池与线程池调优 ⭐⭐⭐⭐

### 8.1 线程池核心参数（ThreadPoolExecutor）

```
七大参数：
  corePoolSize：    核心线程数（常驻，不回收）
  maximumPoolSize： 最大线程数（高峰时扩展）
  keepAliveTime：   非核心线程空闲存活时间
  workQueue：       任务等待队列（核心线程满了先入队）
  threadFactory：   线程工厂（自定义线程命名，方便排查）
  handler：         拒绝策略（队列也满了怎么办）

任务提交流程：
  新任务到来
    → 当前线程数 < corePoolSize？→ 创建核心线程执行
    → 核心线程满？→ 放入 workQueue
    → 队列满？→ 当前线程数 < maximumPoolSize？→ 创建非核心线程
    → 都满了 → 触发拒绝策略
```

**四种拒绝策略：**

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| AbortPolicy（默认）| 抛 RejectedExecutionException | 关键业务，不能丢 |
| CallerRunsPolicy | 调用者线程执行任务 | 不丢弃，但会阻塞调用者 |
| DiscardPolicy | 静默丢弃 | 可丢弃的非核心任务（日志）|
| DiscardOldestPolicy | 丢弃队列最老的任务 | 实时性要求高，旧数据无意义 |

**不同业务场景的线程池配置：**

```
① CPU 密集型（计算/加密/压缩）：
   corePoolSize = CPU核数 + 1
   队列用小容量（SynchronousQueue 或小 ArrayBlockingQueue）
   → 减少线程切换开销

② IO 密集型（HTTP调用/DB查询/文件读写）：
   corePoolSize = CPU核数 × 2（或更高，IO等待时CPU空闲）
   队列用较大容量（LinkedBlockingQueue(1000)）
   → IO等待时让其他线程使用CPU

③ 生产经验公式（参考）：
   线程数 = CPU核数 × 目标CPU利用率 × (1 + 等待时间/计算时间)
   实际必须压测调整，公式仅起步参考
```

**线程池隔离（关键！）：**

```
核心链路（下单/支付）和非核心链路（日志/通知）必须用不同线程池
  → 非核心任务耗尽线程不会影响核心业务

常见坑：所有异步任务共用一个 @Async 默认线程池
  → 某个慢任务（如发邮件）把线程池占满
  → 其他关键异步任务也被阻塞
  → 配置多个线程池，@Async("orderPool") 指定使用
```

### 8.2 HikariCP 连接池调优

```
HikariCP 是 SpringBoot 默认数据库连接池（性能最好）

核心参数：
  maximumPoolSize：  最大连接数（默认10，生产建议 20~50）
  minimumIdle：      最小空闲连接（建议 = maximumPoolSize，保持预热）
  connectionTimeout：获取连接超时（默认30s，建议 3~5s，快速失败）
  idleTimeout：      空闲连接存活时间（默认10min）
  maxLifetime：      连接最大生命周期（默认30min，需 < MySQL wait_timeout）

连接数经验公式：
  最大连接数 ≈ (CPU核数 × 2) + 有效磁盘数
  4核CPU → maximumPoolSize ≈ 10，实际不超过 50

常见坑：
  maximumPoolSize 设太大（200）→ MySQL 连接数耗尽，其他服务连不上
  connectionTimeout 太长 → 线程长时间等待连接，请求堆积
  maxLifetime > MySQL wait_timeout → 连接被MySQL关闭但池里还在用 → 报错
```

```yaml
# 生产推荐配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 20            # = maximumPoolSize，避免动态创建连接开销
      connection-timeout: 3000    # 3秒获取不到连接就快速失败
      idle-timeout: 600000        # 10分钟
      max-lifetime: 1800000       # 30分钟（< MySQL wait_timeout）
      pool-name: HikariPool-Order
```

### 8.3 Redis 连接池（Lettuce）

```
SpringBoot 默认使用 Lettuce（基于 Netty 的异步 Redis 客户端）

Lettuce vs Jedis：
  Lettuce：共享连接（单连接多路复用），线程安全，适合高并发
  Jedis：  连接池模式（每线程独占连接），需要更多连接数

Lettuce 连接池（需要连接池的场景）：
  默认关闭，大多数场景无需开启（单连接多路复用已够用）
  事务操作（MULTI）/ Pipeline 需要独占连接，此时需开启池
```

```yaml
spring:
  data:
    redis:
      lettuce:
        pool:
          max-active: 32       # 最大连接数
          max-idle: 16         # 最大空闲连接
          min-idle: 8          # 最小空闲连接
          max-wait: 3000ms     # 获取连接最大等待时间
```

### 8.4 面试标准答法

**Q: 线程池参数如何设置？**

> 分场景：**CPU密集型**（计算/加密）核心线程数 = CPU核数+1，小队列；**IO密集型**（网络调用/DB查询）核心线程数 = CPU核数×2，较大队列。公式只是参考，**生产必须压测调整**。关键原则：核心链路和非核心链路用不同线程池做隔离，防止慢任务拖垮核心业务。拒绝策略按业务选择：核心用 AbortPolicy 抛异常自行处理，非核心用 CallerRunsPolicy 不丢弃但降速。

## 九、常见追问

**Q: 令牌桶和漏桶的区别？**
> 漏桶强制匀速输出：请求入桶排队，固定速率流出，桶满则丢弃。令牌桶控制平均速率但允许突发：固定速率放令牌，平时积累，突发时消耗积攒的令牌。**接口限流用令牌桶**（允许合理突发）；**保护脆弱下游用漏桶**（DB等不能承受突发）。

**Q: 固定窗口限流的临界点突刺如何解决？**
> 滑动窗口：将 1s 切成 10 个 100ms bucket，统计最近 10 个 bucket 的总量，任意时刻向前推 1s 的请求总数都不超过阈值，从根本上消除临界突刺。Sentinel 的 LeapArray 就是这种实现。

**Q: 高并发场景如何保证缓存和数据库一致性？**
> Cache-Aside 模式：写操作**先更新 DB，再删除缓存**（不更新缓存，避免并发写覆盖）。删缓存失败时 MQ 异步重试，或 Canal 监听 binlog 自动删除。极高一致性场景用 Redis + Lua 原子操作或强制读主库。

**Q: 多级缓存架构如何保证各层一致性？**
> 各层设较短 TTL 兜底（DB永久，Redis分钟级，本地缓存秒级）。变更时：先更新 DB → 删 Redis → MQ 广播通知各节点清本地缓存 → CDN 刷新。最终一致性即可，极短 TTL 作为保障。

**Q: 缓存与 DB 数据不一致最大能容忍多长时间？**
> **TTL 就是最大不一致窗口**。按业务分级：商品价格 10s、用户信息 5min、配置数据 10min。要求实时正确的数据（如库存）不用缓存，或用 Canal 监听 binlog 实时删缓存。

**Q: 秒杀系统如何设计限流？**
> 漏斗模型分层限流：前端按钮置灰 → CDN/Nginx IP限流 → Gateway 令牌桶全局限流（只放1%） → 应用层 Sentinel + 本地售罄标记 → Redis Lua 原子扣库存 → MQ 异步下单匀速消费。每层过滤流量，层层递减。

**Q: 读写分离后如何解决主从延迟的强一致读？**
> ① **强制读主库**（最常用）：下单后查订单等敏感操作通过 Hint 路由到主库；② **等待位点**：读从库前等待同步到写操作的 binlog 位点；③ **缓存代替从库**：用 Redis 做读扩展，不依赖从库。

**Q: Sentinel 和 Hystrix 的核心区别？**
> Hystrix 已停维护。核心区别：① **隔离方式**：Hystrix 线程池隔离 vs Sentinel 信号量隔离（轻量无线程切换）；② **功能**：Sentinel 内置限流/熔断/降级/热点参数/集群限流，更全面；③ **运维**：Sentinel Dashboard + Nacos 动态推送规则。

**Q: 幂等设计的核心原则？**
> 两个关键：① **唯一ID** 识别重复请求（业务流水号/消息ID/Token）；② **原子操作** 保证检查和处理不被并发破坏（DB唯一索引/Redis Lua）。创建类用唯一索引，MQ消费用 SETNX，表单防重用 Token。

**Q: 线程池和连接池设置不当会导致什么问题？**
> 线程池设太小 → 任务排队超时；设太大 → 上下文切换开销大。连接池 maximumPoolSize 过大 → MySQL 连接耗尽影响其他服务；connectionTimeout 过长 → 请求堆积。核心原则：**不同业务隔离**（核心/非核心用不同池），**参数通过压测确定**。

**Q: 重试策略设计不当会怎样？**
> 盲目重试引发**重试风暴**：下游宕机 → 全部超时请求各重试3次 → 流量翻倍涌入 → 刚恢复就被打垮。正确做法：指数退避+Jitter 分散时间；熔断器在失败率高时切断不再重试；**只重试可恢复异常**（超时/503），不重试业务异常（参数错误/余额不足）。
