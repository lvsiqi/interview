# 线上问题监控与排查

> 本章系统化讲解生产环境全维度监控、排查流程和问题分析方法论  
> 涵盖：**CPU / 内存 / 磁盘 / 网络 / 文件句柄 / JVM / 进程 / 日志 / 监控体系**  
> 面试考察重点：**排查思路的系统性 + 命令的熟练度 + 根因分析能力**

## 一、线上问题排查方法论 ⭐⭐⭐

### 1.1 排查总纲：MELT 四维模型

```
┌────────────────────────────────────────────────────────────────┐
│                线上问题排查四维模型（MELT）                       │
├───────────────┬────────────────────────────────────────────────┤
│  Metrics      │ 指标：CPU / 内存 / 磁盘 / 网络 / JVM / 连接数    │
│  Events       │ 事件：部署 / 配置变更 / 扩缩容 / 故障事件         │
│  Logs         │ 日志：应用日志 / 中间件日志 / 系统日志             │
│  Traces       │ 链路：请求链路追踪 / 慢请求分析                    │
└───────────────┴────────────────────────────────────────────────┘
```

### 1.2 排查黄金流程（5 步法）

```
收到告警 / 用户反馈
    │
    ▼
Step 1: 确认影响范围
  → 全局 or 局部？哪些服务/接口/用户受影响？
  → 确认是否需要先止血（降级/限流/回滚）
    │
    ▼
Step 2: 看监控大盘（Grafana/Prometheus）
  → 先看全局指标：CPU、内存、网络、磁盘、QPS、RT、错误率
  → 找到异常时间点和异常维度
    │
    ▼
Step 3: 关联变更事件
  → 最近是否有发布/配置变更/扩缩容/依赖方变更？
  → 80% 的线上问题由变更引起
    │
    ▼
Step 4: 深入排查（自顶向下）
  → OS 层 → JVM 层 → 应用层 → 依赖层
  → 使用对应维度的命令/工具
    │
    ▼
Step 5: 根因确认 & 复盘
  → 定位根因 → 修复 → 验证 → 写 Postmortem
  → 补充监控/告警/预案
```

### 1.3 止血优先原则

```
⚠️ 先恢复，再排查！

常见止血手段（按速度排序）：
┌──────────────────┬──────────────────────────────────────────┐
│ 手段              │ 适用场景                                  │
├──────────────────┼──────────────────────────────────────────┤
│ 流量切走/摘节点    │ 单机故障，其他节点正常                      │
│ 服务降级          │ 非核心功能异常拖垮全局                      │
│ 限流              │ 流量突增导致系统过载                        │
│ 回滚发布          │ 新版本引入 Bug                             │
│ 配置回滚          │ 配置变更导致异常                            │
│ 扩容              │ 容量不足导致的性能下降                      │
│ 重启              │ 内存泄漏/线程泄漏的临时手段（非根治）         │
└──────────────────┴──────────────────────────────────────────┘
```

### 1.4 面试答法

> **Q: 聊聊你的线上问题排查思路？**
>
> 我一般遵循"先止血再排查"的原则，按 5 步走：
> 1. **确认影响范围**，评估是否需要立即降级/限流/回滚
> 2. **看监控大盘**，快速定位异常维度（CPU/内存/网络/QPS/RT/错误率）
> 3. **关联变更事件**，确认最近是否有发布/配置/扩缩容变更
> 4. **自顶向下深入排查**：OS 层 → JVM 层 → 应用层 → 依赖层
> 5. **修复验证 + 复盘**，补充监控和预防措施
>
> 具体用到的工具：OS 层用 top/vmstat/iostat/netstat，JVM 层用 jstat/jstack/jmap/Arthas，链路层用 SkyWalking/Jaeger，日志层用 ELK/grep/awk。80% 的线上问题由变更引起，所以关联变更事件非常关键。

## 二、CPU 异常排查 ⭐⭐⭐

### 2.1 监控指标

```
┌──────────────────────┬────────────────────────────────────────┐
│ 指标                  │ 含义 & 告警阈值                          │
├──────────────────────┼────────────────────────────────────────┤
│ CPU Usage (%)         │ 总体使用率，持续 > 80% 告警               │
│ CPU User (%)          │ 用户态占比高 → 应用计算密集                │
│ CPU System (%)        │ 内核态占比高 → 系统调用 / IO / 上下文切换   │
│ CPU iowait (%)        │ IO 等待占比高 → 磁盘 IO 瓶颈              │
│ Load Average          │ 1/5/15 分钟负载，> CPU 核心数则过载         │
│ Context Switch        │ 上下文切换次数，频繁切换说明线程竞争激烈     │
└──────────────────────┴────────────────────────────────────────┘
```

### 2.2 排查命令速查

```bash
# ========== 快速查看 CPU 全貌 ==========

# 1. top - 实时查看进程 CPU 使用率
top -c                        # 按 P 排序（CPU），按 1 查看每核
top -H -p <PID>               # 查看某进程的线程级 CPU 使用

# 2. uptime - 快速看负载
uptime
# 输出示例: load average: 8.32, 6.15, 3.08
# 含义: 1分钟/5分钟/15分钟平均负载（递增说明问题正在恶化）

# 3. vmstat - CPU/内存/IO/上下文切换综合视图
vmstat 1 5                    # 每 1 秒输出，共 5 次
#  procs  -----cpu-----
#  r  b   us sy id wa st
# 12  0   85  8  5  2  0
# r: 运行队列长度（> CPU核数 → 过载）
# us: 用户态（应用代码）
# sy: 内核态（系统调用）
# wa: IO 等待
# id: 空闲

# 4. mpstat - 每个 CPU 核心的使用情况
mpstat -P ALL 1 3             # 每秒输出所有核心，共 3 次
# 某个核心 100% 其他空闲 → 可能单线程死循环

# 5. pidstat - 进程级别 CPU 统计
pidstat -u 1 5                # 每秒输出进程 CPU 使用
pidstat -t -p <PID> 1 5       # 线程级别

# ========== Java 进程 CPU 高排查（经典 4 步） ==========

# Step 1: 定位高 CPU 进程
top -c                        # 找到 PID

# Step 2: 定位高 CPU 线程
top -H -p <PID>               # 找到 TID

# Step 3: 线程 ID 转 16 进制
printf '%x\n' <TID>           # 例如 12345 → 0x3039

# Step 4: jstack 导出并搜索
jstack <PID> > /tmp/thread.log
grep -A 30 '0x3039' /tmp/thread.log
```

### 2.3 常见根因速查

| 现象 | 根因 | 排查方式 |
|------|------|----------|
| us（用户态）高 | 应用代码计算密集 / 死循环 / 正则回溯 / 频繁 GC | jstack 看热点线程栈 |
| sy（内核态）高 | 频繁系统调用 / 锁竞争 / 线程上下文切换 | vmstat 看 cs；strace -p PID |
| wa（IO等待）高 | 磁盘 IO 瓶颈 / 大量日志写入 / 同步写操作 | iostat 看磁盘；iotop 看进程 IO |
| 负载高但 CPU 不高 | 大量线程阻塞在 IO / 锁等待 | vmstat 看 r 和 b 列 |
| 单核 100% | 单线程死循环 / GC 线程 | mpstat 看各核；jstack 定位 |
| CPU 周期性飙高 | 定时任务 / GC / 批量任务 | 对比 crontab 和 GC 日志时间 |

### 2.4 高级工具：Arthas

```bash
# Arthas 是阿里开源的 Java 诊断工具，字节面试常问

# 启动
java -jar arthas-boot.jar     # 选择目标 Java 进程

# CPU 分析
thread                         # 查看所有线程信息
thread -n 3                    # 显示 CPU 最高的 3 个线程
thread --state BLOCKED         # 查看所有阻塞线程
thread -b                      # 找出当前阻塞其他线程的线程（死锁侦测）

# 火焰图（CPU Profiler）
profiler start                 # 开始采样
profiler stop --format html    # 停止并生成火焰图
# 火焰图宽度 = 采样次数，越宽 = 越热的方法
```

## 三、内存异常排查 ⭐⭐⭐

### 3.1 监控指标

```
┌────────────────────────────┬───────────────────────────────────────────┐
│ 指标                        │ 含义 & 告警阈值                             │
├────────────────────────────┼───────────────────────────────────────────┤
│ 系统内存                                                                  │
├────────────────────────────┼───────────────────────────────────────────┤
│ Total / Used / Free         │ 物理总量 / 已用 / 空闲                       │
│ Available                   │ 真正可用（含 buffer/cache 可回收部分）         │
│ Swap Used                   │ 交换区使用量，> 0 说明物理内存紧张             │
│ OOM Killer 触发次数          │ 系统内存耗尽后内核杀进程                      │
├────────────────────────────┼───────────────────────────────────────────┤
│ JVM 内存                                                                  │
├────────────────────────────┼───────────────────────────────────────────┤
│ Heap Used / Max             │ 堆使用量 / 最大值，Used/Max > 80% 告警       │
│ Young GC 频率 & 耗时         │ 频繁 YGC 说明对象创建过快                    │
│ Full GC 频率 & 耗时          │ FGC > 1次/分钟 → 内存泄漏或配置不合理         │
│ Metaspace Used              │ 元空间使用量（动态类加载场景需关注）            │
│ Direct Memory               │ 堆外内存（Netty 场景）                       │
│ Thread Count                │ 线程数过多 → 栈内存 OOM                      │
└────────────────────────────┴───────────────────────────────────────────┘
```

### 3.2 系统内存排查命令

```bash
# ========== 系统内存 ==========

# 1. free - 快速查看系统内存
free -h
#              total   used   free   shared   buff/cache   available
# Mem:          32G    20G    1.5G    256M       10G          11G
# Swap:          4G    500M    3.5G
#
# ⚠️ 重点看 available（真正可用），不是 free
# ⚠️ Swap 使用量 > 0 说明物理内存紧张

# 2. 查看进程内存排行
ps aux --sort=-%mem | head -20
# 或用 top，按 M 键按内存排序

# 3. smaps - 查看进程内存详细映射
cat /proc/<PID>/smaps | grep -i rss | awk '{sum+=$2} END {print sum/1024 " MB"}'

# 4. Swap 使用详情
for pid in $(ls /proc/ | grep '^[0-9]'); do
  swap=$(awk '/VmSwap/{print $2}' /proc/$pid/status 2>/dev/null)
  if [ "$swap" -gt 0 ] 2>/dev/null; then
    name=$(cat /proc/$pid/comm 2>/dev/null)
    echo "$pid $name $swap kB"
  fi
done | sort -k3 -rn | head -10

# 5. dmesg - 查看 OOM Killer 日志
dmesg -T | grep -i "oom\|killed"
# Out of memory: Killed process 12345 (java), UID 1000, total-vm:8389680kB
```

### 3.3 JVM 内存排查命令

```bash
# ========== JVM 内存 ==========

# 1. jstat - GC 实时监控（最常用）
jstat -gcutil <PID> 1000 10    # 每 1 秒输出，共 10 次
#  S0     S1     E      O      M     CCS    YGC   YGCT    FGC   FGCT     GCT
#  0.00  85.39  62.10  93.72  97.86  95.52  1250  12.8     15   8.62    21.4
#
# O（老年代）> 90% + FGC 持续增长 → 内存泄漏
# E（Eden）频繁 100% → 对象创建太快

# 2. jmap - 堆内存快照
# 查看堆统计
jmap -heap <PID>

# 查看对象占用排行（不需要 dump，线上快速定位）
jmap -histo:live <PID> | head -30
# num     #instances   #bytes  class name
#   1:     2561890    184345280  [B
#   2:     1025678     49232544  java.lang.String
# → 看哪个类实例异常多

# 生成 heap dump（⚠️ 会触发 Full GC，线上慎用）
jmap -dump:format=b,file=/tmp/heap.hprof <PID>
# 或者启动时加: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/

# 3. MAT / VisualVM 分析 heap dump
# → Leak Suspects Report → Dominator Tree → 找到占用最大的对象
# → GC Roots → 找到引用链 → 定位泄漏代码

# 4. Arthas 内存分析
# dashboard                    # 实时看堆/非堆/GC
# heapdump /tmp/heap.hprof    # 生成 heap dump
# memory                       # 查看内存区域使用情况
# vmtool --action getInstances --className java.util.HashMap --limit 10
```

### 3.4 常见根因速查

| OOM 类型 | 错误信息 | 常见原因 |
|----------|---------|---------|
| 堆内存溢出 | `java.lang.OutOfMemoryError: Java heap space` | 大对象、集合无限增长、内存泄漏 |
| 元空间溢出 | `OutOfMemoryError: Metaspace` | 动态类生成过多（CGLIB、Groovy） |
| 线程栈溢出 | `OutOfMemoryError: unable to create native thread` | 线程数过多，调小 -Xss 或限制线程数 |
| 直接内存溢出 | `OutOfMemoryError: Direct buffer memory` | NIO/Netty 堆外内存泄漏 |
| GC 超限 | `OutOfMemoryError: GC overhead limit exceeded` | GC 占用 >98% CPU 但回收 <2% 内存 |
| 栈溢出 | `StackOverflowError` | 递归太深 / 无限递归 |

### 3.5 内存泄漏排查 SOP

```
Step 1: 确认是否在泄漏
  jstat -gcutil <PID> 5000
  → 观察 O（老年代）是否在每次 Full GC 后持续增长
  → 若 FGC 后 O 不降 → 确认泄漏

Step 2: 快速定位可疑对象
  jmap -histo:live <PID> | head -30
  → 找到实例数/占用异常大的类

Step 3: 生成 Heap Dump 详细分析
  jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>
  → MAT 打开 → Leak Suspects → Dominator Tree

Step 4: 找 GC Root 引用链
  → MAT: 右键可疑对象 → Path to GC Roots → exclude weak/soft references
  → 找到持有引用的代码位置

Step 5: 修复并验证
  → 修复代码 → 灰度发布 → 观察 FGC 后 O 区是否正常回落
```

## 四、磁盘异常排查 ⭐⭐

### 4.1 监控指标

```
┌────────────────────────┬──────────────────────────────────────────────┐
│ 指标                    │ 含义 & 告警阈值                                │
├────────────────────────┼──────────────────────────────────────────────┤
│ Disk Usage (%)          │ 磁盘使用率，> 85% 告警，> 95% 紧急              │
│ Inode Usage (%)         │ Inode 使用率（小文件过多），> 85% 告警           │
│ IO Util (%)             │ 磁盘 IO 利用率，> 80% 说明瓶颈                  │
│ IO Await (ms)           │ IO 平均等待时间，> 20ms 需关注                   │
│ IOPS                    │ 每秒 IO 操作数（SSD 一般 5000~80000）            │
│ Throughput (MB/s)       │ 吞吐量（HDD ~150MB/s, SSD ~500MB/s+）          │
│ Read/Write Bytes        │ 读写字节数                                     │
└────────────────────────┴──────────────────────────────────────────────┘
```

### 4.2 排查命令速查

```bash
# ========== 磁盘空间 ==========

# 1. df - 查看磁盘使用率
df -h                         # 查看所有挂载点
df -ih                        # 查看 Inode 使用率（小文件太多场景）

# 2. du - 定位大目录/大文件
du -sh /*                     # 查看根目录下各目录大小
du -sh /var/log/*             # 查看日志目录
du -h --max-depth=2 / | sort -rh | head -20   # 全局 Top 20 大目录

# 3. 快速找大文件
find / -type f -size +500M -exec ls -lh {} \; 2>/dev/null
# 或按修改时间
find /var/log -type f -mtime -1 -size +100M   # 最近 1 天的大文件

# 4. 已删除但未释放的文件（进程仍持有 fd）
lsof +L1 | grep deleted
# 常见场景：日志文件被删除但进程未重启，空间不释放
# 解决：重启进程，或 > /proc/<PID>/fd/<FD> 清空

# ========== 磁盘 IO ==========

# 5. iostat - 磁盘 IO 统计
iostat -xdm 1 5               # 每秒输出，共 5 次
# Device  rrqm/s  wrqm/s  r/s   w/s   rMB/s  wMB/s  await  %util
# sda       0.0     50.2   1.5  200.3   0.1    25.6    5.2   65.3
#
# %util > 80% → 磁盘瓶颈
# await > 20ms → IO 等待过长
# w/s 很高 → 写操作频繁（日志？binlog？）

# 6. iotop - 进程级 IO 排行
iotop -oP                     # 只显示有 IO 的进程
# 找到 IO 最高的进程

# 7. 查看进程的 IO 统计
cat /proc/<PID>/io
# rchar: 读字节数
# wchar: 写字节数
# read_bytes: 实际磁盘读
# write_bytes: 实际磁盘写
```

### 4.3 常见问题 & 解决方案

| 问题 | 现象 | 排查 & 解决 |
|------|------|-------------|
| 磁盘空间满 | `No space left on device` | `df -h` + `du` 找大文件；清理日志/临时文件；配置日志轮转 |
| Inode 耗尽 | 空间有余但无法创建文件 | `df -ih` 检查 Inode；`find / -xdev -type f \| cut -d/ -f2-3 \| sort \| uniq -c \| sort -rn \| head` 找小文件多的目录 |
| 已删除文件不释放 | df 和 du 结果差异大 | `lsof +L1 \| grep deleted` 找到进程，重启或清空 fd |
| 写 IO 过高 | %util 高、wa 高 | iotop 找进程；优化日志级别/异步写入/binlog 配置 |
| 磁盘读慢 | await 高 | 检查是否 HDD 随机读过多；考虑换 SSD；加缓存减少读 |

## 五、网络异常排查 ⭐⭐⭐

### 5.1 监控指标

```
┌────────────────────────────┬──────────────────────────────────────────┐
│ 指标                        │ 含义 & 告警阈值                            │
├────────────────────────────┼──────────────────────────────────────────┤
│ 连接状态                                                                 │
├────────────────────────────┼──────────────────────────────────────────┤
│ ESTABLISHED 数              │ 当前活跃连接数                              │
│ TIME_WAIT 数                │ > 10000 需关注端口耗尽                      │
│ CLOSE_WAIT 数               │ > 100 说明应用未正确关闭连接（⚠️ 最危险）    │
│ SYN_RECV 队列               │ 半连接过多可能是 SYN Flood 攻击              │
├────────────────────────────┼──────────────────────────────────────────┤
│ 流量与延迟                                                               │
├────────────────────────────┼──────────────────────────────────────────┤
│ 带宽使用率 (Mbps)            │ > 80% 告警                                │
│ 包量 (pps)                  │ 包量过高（小包攻击）                        │
│ RTT (ms)                    │ 同机房 < 1ms，跨机房 < 5ms                 │
│ 丢包率 (%)                   │ > 0.1% 需告警                             │
│ 重传率 (%)                   │ > 1% 需告警                               │
├────────────────────────────┼──────────────────────────────────────────┤
│ Socket 缓冲区                                                            │
├────────────────────────────┼──────────────────────────────────────────┤
│ Recv-Q                      │ 接收缓冲区积压（应用处理不过来）              │
│ Send-Q                      │ 发送缓冲区积压（网络拥塞或对端慢）            │
└────────────────────────────┴──────────────────────────────────────────┘
```

### 5.2 排查命令速查

```bash
# ========== 连接状态 ==========

# 1. 查看各状态连接数统计
ss -s                          # 快速汇总
# 或详细分类
ss -ant | awk '{print $1}' | sort | uniq -c | sort -rn
# 输出示例:
#  2856 ESTAB
#  1247 TIME-WAIT
#   328 CLOSE-WAIT    ← ⚠️ 这个数字不应该大
#    56 LISTEN

# 2. CLOSE_WAIT 排查（最高频的网络故障面试题）
ss -antp | grep CLOSE-WAIT
# 找到 PID → jstack <PID> 看线程是否卡在 socket read/write
# 根因：应用代码没有正确关闭连接（缺少 finally close / try-with-resources）

# 3. TIME_WAIT 排查
ss -ant | grep TIME-WAIT | wc -l
# > 10000 时可能导致端口耗尽
# 优化：
#   sysctl net.ipv4.tcp_tw_reuse=1
#   使用连接池 / HTTP 长连接

# 4. 查看某端口连接详情
ss -antp | grep :8080          # 查 8080 端口所有连接
ss -antp | grep :8080 | grep ESTAB | wc -l  # 活跃连接数

# ========== 网络延迟 & 丢包 ==========

# 5. ping - 基本连通性和延迟
ping -c 10 <目标IP>
# 看 avg RTT 和丢包率

# 6. mtr - 全链路追踪（替代 traceroute + ping）
mtr -n -c 20 <目标IP>
# 显示每一跳的丢包率和延迟，快速定位哪段网络有问题

# 7. tcpdump - 抓包分析（终极武器）
tcpdump -i eth0 -nn host <IP> and port 8080 -c 100 -w /tmp/capture.pcap
# 分析 TCP 重传：
tcpdump -i eth0 -nn 'tcp[tcpflags] & (tcp-syn|tcp-fin) != 0' -c 50
# 生成的 pcap 文件可用 Wireshark 打开分析

# ========== 网络流量 ==========

# 8. sar - 网络流量统计
sar -n DEV 1 5                # 每秒输出，共 5 次
# IFACE  rxpck/s  txpck/s  rxkB/s  txkB/s  ...
# eth0    25000    20000    3000     2500

# 9. iftop - 实时流量排行（按连接维度）
iftop -n -i eth0              # 查看哪些连接占带宽最多

# 10. nload - 实时带宽视图
nload eth0                    # 直观的带宽图
```

### 5.4 常见网络问题对照表

| 问题 | 现象 | 排查 & 解决 |
|------|------|-------------|
| 连接泄漏 | CLOSE_WAIT 持续增长 | jstack 看卡在哪；检查代码是否正确关闭连接 |
| 端口耗尽 | TIME_WAIT 过多、`Cannot assign requested address` | 开启 tcp_tw_reuse；用连接池/长连接 |
| 连接超时 | `Connection timed out` | ping/mtr 检查连通性；telnet 测端口；检查防火墙 |
| 读超时 | `Read timed out` | 服务端处理慢；网络拥塞；检查 RT |
| DNS 解析慢 | 第一次请求慢，后续正常 | 检查 /etc/resolv.conf；加 DNS 缓存 |
| 带宽打满 | rxkB/s 或 txkB/s 接近网卡上限 | iftop 找大流量连接；是否有大量日志传输/文件下载 |
| TCP 重传过多 | 重传率 > 1% | tcpdump 分析；可能是网络设备问题/丢包 |

## 六、文件句柄（FD）异常排查 ⭐⭐

### 6.1 基础知识

```
一切皆文件：Linux 下文件/Socket/管道/设备都是文件描述符（File Descriptor）

文件句柄限制：
┌──────────────────────┬─────────────────────────────────────┐
│ 层级                  │ 查看 / 修改                           │
├──────────────────────┼─────────────────────────────────────┤
│ 系统级限制             │ cat /proc/sys/fs/file-max            │
│ （整个操作系统）       │ sysctl fs.file-max=1000000           │
├──────────────────────┼─────────────────────────────────────┤
│ 进程级限制             │ ulimit -n                            │
│ （单个进程）           │ 修改 /etc/security/limits.conf       │
│                      │ * soft nofile 65535                   │
│                      │ * hard nofile 65535                   │
├──────────────────────┼─────────────────────────────────────┤
│ 查看进程实际使用       │ ls /proc/<PID>/fd | wc -l            │
│ 或                    │ cat /proc/<PID>/limits               │
└──────────────────────┴─────────────────────────────────────┘

Java 进程启动时可查看：
  cat /proc/<PID>/limits | grep "open files"
  # Max open files  65535  65535  files
```

### 6.2 排查命令

```bash
# ========== 系统级 ==========

# 1. 系统当前打开的文件句柄数
cat /proc/sys/fs/file-nr
# 已分配  已使用(0)  最大值
# 18432     0        1048576

# 2. 查看系统级限制
cat /proc/sys/fs/file-max
# 1048576

# ========== 进程级 ==========

# 3. 查看某进程打开的 FD 数量
ls /proc/<PID>/fd | wc -l

# 4. 查看某进程 FD 详情（⭐ 最实用）
ls -la /proc/<PID>/fd
# lrwx------ 1 user user 0 ... 0 -> /dev/null
# lrwx------ 1 user user 0 ... 5 -> socket:[12345]    ← Socket
# lr-x------ 1 user user 0 ... 8 -> /var/log/app.log  ← 文件
# l-wx------ 1 user user 0 ... 12 -> pipe:[67890]     ← 管道

# 5. 统计 FD 类型分布
ls -la /proc/<PID>/fd | awk '{print $NF}' | sed 's/.*\///' | sort | uniq -c | sort -rn
# 快速看 socket / 文件 / pipe 各占多少

# 6. lsof 查看某进程打开的所有文件
lsof -p <PID> | wc -l          # 总数
lsof -p <PID> | grep socket    # 只看 socket
lsof -p <PID> | grep ".log"    # 只看日志文件

# 7. 排查 Socket FD 泄漏
lsof -p <PID> | grep "can't identify protocol"
# 这种说明 socket 处于异常状态

# 8. 查看所有进程的 FD 使用排行
for pid in $(ls /proc/ | grep '^[0-9]'); do
  count=$(ls /proc/$pid/fd 2>/dev/null | wc -l)
  name=$(cat /proc/$pid/comm 2>/dev/null)
  echo "$count $pid $name"
done | sort -rn | head -20

# ========== 告警相关 ==========

# 9. 查看进程 FD 上限和当前使用
cat /proc/<PID>/limits | grep "open files"
# 对比 ls /proc/<PID>/fd | wc -l
# 使用率 > 80% 需要告警
```

### 6.3 常见问题 & 解决方案

| 问题 | 错误信息 | 根因 & 解决 |
|------|---------|-------------|
| FD 耗尽 | `Too many open files` | 连接未关闭 / 文件未关闭 / ulimit 太小 |
| Socket 泄漏 | FD 中大量 socket 类型且持续增长 | HTTP 连接池配置不当 / 未用 try-with-resources |
| 文件泄漏 | FD 中大量文件句柄指向相同类型文件 | 循环打开文件未关闭 / 日志 appender 异常 |

```java
// ❌ 错误：未正确关闭，FD 泄漏
InputStream is = new FileInputStream("data.txt");
// 异常时不会关闭

// ✅ 正确：try-with-resources 自动关闭
try (InputStream is = new FileInputStream("data.txt")) {
    // 使用 is
}  // 自动调用 is.close()

// ❌ 错误：HTTP 连接未正确复用/关闭
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
// 用完未 disconnect

// ✅ 正确：使用连接池 + 确保归还
// 推荐 OkHttp / Apache HttpClient 连接池
```

```bash
# 1. 查看 Java 进程

ps -ef | grep java             # 传统方式

#   45 java.lang.Thread.State: RUNNABLE

pidstat -w -p <PID> 1 5        # 进程级上下文切换

# nvcswch/s: 非自愿上下文切换（时间片耗尽、锁竞争）

  sleep 5
done
# 线程数持续增长 → 线程泄漏
```

### 7.3 线程泄漏排查

```java
// 常见线程泄漏场景

// ❌ 1. 每次请求都新建线程池（没有复用）
public void handleRequest() {
    ExecutorService executor = Executors.newFixedThreadPool(10);
    executor.submit(() -> doSomething());
    // 没有 shutdown，线程池永远不会释放
}

// ❌ 2. Timer/ScheduledExecutor 未关闭
Timer timer = new Timer();
timer.schedule(task, 0, 1000);
// 服务停止时没有 timer.cancel()

// ✅ 正确做法：线程池全局复用，服务停止时关闭
@Bean
public ExecutorService taskExecutor() {
    return new ThreadPoolExecutor(5, 20, 60, TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(1000),
        new ThreadFactoryBuilder().setNameFormat("biz-worker-%d").build(),
        new ThreadPoolExecutor.CallerRunsPolicy());
}

@PreDestroy
public void shutdown() {
    taskExecutor.shutdown();
}
```

```
│ redis_exporter      │ Redis：内存/命中率/连接/命令统计     │
│ kafka_exporter      │ Kafka：Lag/ISR/吞吐               │
│ blackbox_exporter   │ 探针：HTTP/TCP/DNS/ICMP 可用性     │
└────────────────────┴──────────────────────────────────┘
```

### 10.3 关键告警规则参考

```yaml
# Prometheus 告警规则示例（面试时能说出来是加分项）

# CPU 使用率 > 80% 持续 5 分钟
- alert: HighCpuUsage
  expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
  for: 5m

# 内存使用率 > 85%
- alert: HighMemoryUsage
  expr: (1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100 > 85
  for: 5m

# 磁盘使用率 > 85%
- alert: DiskSpaceRunningOut
  expr: (1 - node_filesystem_avail_bytes / node_filesystem_size_bytes) * 100 > 85
  for: 5m

# FD 使用率 > 80%
- alert: TooManyOpenFiles
  expr: process_open_fds / process_max_fds * 100 > 80
  for: 5m

# Full GC 频繁（> 1次/分钟）
- alert: FrequentFullGC
  expr: rate(jvm_gc_collection_seconds_count{gc="G1 Old Generation"}[5m]) > 0.016
  for: 5m

# 接口 P99 延迟 > 1s
- alert: HighLatency
  expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 1
  for: 5m

# 接口错误率 > 1%
- alert: HighErrorRate
  expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m])
      / rate(http_server_requests_seconds_count[5m]) > 0.01
  for: 5m
```

### 10.4 面试答法

> **Q: 你们线上监控体系是怎么搭建的？**
>
> 我们采用四层监控架构：
> 1. **基础设施层**：node_exporter 采集 CPU/内存/磁盘/网络/FD 指标，Prometheus 存储，Grafana 看板
> 2. **中间件层**：mysqld_exporter 监控 MySQL 慢查询和连接数，redis_exporter 监控内存和命中率，kafka_exporter 监控消费 Lag
> 3. **应用层**：Spring Boot Actuator + Micrometer 暴露 JVM（GC/堆/线程）和接口指标（QPS/RT/P99/错误率），同时埋点业务指标
> 4. **链路追踪层**：SkyWalking 做全链路追踪，ELK 做日志集中化，支持按 traceId 串联全链路日志
>
> 告警方面，Alertmanager 配置分级告警（P0 电话 + P1 飞书 + P2 邮件），所有告警必须有对应的 Runbook（排查手册）。

## 十一、综合排查实战场景 ⭐⭐⭐

### 11.1 场景：服务响应变慢（RT 升高）

```
排查路径（自顶向下）：

1. 看监控大盘
   │── QPS 是否上升？（流量突增）
   │── 错误率是否上升？（某些请求异常）
   │── 哪些接口 RT 上升？（全局 or 局部）
   │
2. 看 OS 资源
   │── CPU：top → 是否过载？
   │── 内存：free → 是否用到 Swap？
   │── 磁盘：iostat → IO 等待高？
   │── 网络：ss → 连接数异常？
   │
3. 看 JVM
   │── jstat → GC 是否频繁？Full GC？
   │── jstack → 线程是否阻塞？死锁？
   │── 线程池队列是否积压？
   │
4. 看依赖
   │── MySQL → 慢查询？连接池满？
   │── Redis → 大 Key？热 Key？连接超时？
   │── RPC → 下游服务 RT？超时？
   │── MQ → 消费积压？
   │
5. 看链路追踪
   │── SkyWalking → 慢请求链路在哪个环节耗时最长？
   │── 日志 → 是否有异常堆栈？
   │
6. 结论
   └── 定位根因 → 修复 → 观察恢复 → 复盘
```

### 11.2 场景：服务不可用（接口 5xx）

```
止血 → 排查 → 修复

止血：
  → 是否可以回滚上次发布？
  → 是否可以摘掉故障节点？
  → 是否需要限流/降级？

排查：
  1. 看错误日志
     grep "ERROR" app.log | tail -50
     → 是什么异常？NullPointerException? TimeoutException? OOM?

  2. 看进程是否存活
     jps / ps -ef | grep java
     → 进程挂了？→ 看 OOM Killer (dmesg) / GC 日志 / 堆 dump

  3. 看连接
     ss -antp | grep :8080
     → 端口还在监听吗？连接数正常吗？

  4. 看线程
     jstack <PID> | grep -c "java.lang.Thread.State"
     → 线程数正常吗？是否都在 BLOCKED / WAITING？

  5. 看依赖
     → 是否某个依赖不可用导致请求堆积？
     → 数据库/Redis/MQ 是否宕机？
```

### 11.3 场景：内存持续增长（疑似泄漏）

```
验证泄漏 → 定位对象 → 找引用链 → 修复

Step 1: 确认是否泄漏
  jstat -gcutil <PID> 5000
  → 连续观察 10 分钟：每次 Full GC 后 Old 区使用率是否持续上升
  → 若每次 FGC 后 O 区 70% → 75% → 80% → 确认泄漏

Step 2: 快速看可疑对象
  jmap -histo:live <PID> | head -30
  → 哪个类的实例数/占用内存异常大

Step 3: Heap Dump 深度分析
  jmap -dump:live,format=b,file=/tmp/heap.hprof <PID>
  → MAT 打开 → Leak Suspects Report → Dominator Tree

Step 4: 找 GC Root 引用链
  MAT → 右键可疑对象 → Path to GC Roots → exclude weak/soft/phantom
  → 找到是哪个 static 变量/线程变量持有引用

Step 5: 修复验证
  → 常见修复：移除 static 集合无限放入、ThreadLocal 使用后 remove、
    关闭资源（连接/流）、EventListener 注销
  → 灰度发布 → 观察 Full GC 后老年代是否正常回落
```

### 11.4 场景：磁盘报警空间不足

```
Step 1: 确认哪个分区满了
  df -h
  → /dev/sda1  95%  ← 这个满了

Step 2: 定位大文件/大目录
  du -sh /* | sort -rh | head -10
  du -sh /var/log/* | sort -rh | head -10
  find / -type f -size +500M 2>/dev/null

Step 3: 检查已删除但未释放的文件
  lsof +L1 | grep deleted
  → 常见：日志已被 logrotate 删除，但 Java 进程仍持有 fd

Step 4: 清理
  → 清理大日志：> /var/log/xxx.log（清空文件内容但不删除）
  → 配置 logrotate 日志轮转
  → 清理临时文件/旧版本部署包
  → 如果是 lsof +L1 的问题：重启持有 fd 的进程
```

### 12.1 一表总览

## 📝 面试速查

```
Q: 线上 CPU 飙高怎么排查？
A: top找进程 → top -H找线程 → printf转16进制 → jstack看栈 → 分析根因
   （死循环/频繁GC/正则回溯/加解密）

Q: 内存泄漏怎么排查？
A: jstat看GC趋势（FGC后Old不降）→ jmap -histo看对象 → dump + MAT分析
   → Path to GC Roots找引用链 → 修复(static集合/ThreadLocal/未关闭资源)

Q: CLOSE_WAIT 过多什么原因？
A: 应用代码未正确关闭连接（缺少finally close）→ 用try-with-resources

Q: Too many open files 怎么解决？
A: 查ulimit -n → 查lsof -p PID看FD分布 → 找到泄漏的socket/file → 
   修复代码 + 调大limits.conf

Q: 磁盘空间满了怎么排查？
A: df -h找分区 → du -sh找大目录 → lsof +L1查已删除未释放 → 清理/轮转

Q: 你们监控体系怎么搭建的？
A: 四层：基础设施(node_exporter) + 中间件(mysqld/redis_exporter) 
   + 应用(Micrometer/Actuator) + 链路(SkyWalking) + 日志(ELK)
   全部接入Prometheus+Grafana看板，Alertmanager分级告警

Q: 服务 RT 升高怎么排查？
A: 大盘(QPS/错误率) → OS(CPU/内存/磁盘/网络) → JVM(GC/线程) 
   → 依赖(DB/Redis/RPC) → 链路追踪(SkyWalking慢链路)
```
