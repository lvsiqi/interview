## 一、连接池核心原理 ⭐⭐⭐

### 1.1 为什么需要连接池
```
无连接池：每次请求 → TCP三次握手 → MySQL鉴权 → 执行SQL → 关闭连接（~200ms/次）
有连接池：启动时创建连接 → 请求复用 → 归还连接（~1ms/次）
```

| 对比 | 无连接池 | 有连接池 |
|------|---------|---------|
| 连接创建开销 | 每次 3-5ms（TCP + 鉴权） | 启动时一次性创建 |
| 连接数控制 | 无限制，可能打满 DB | **最大连接数限制** |
| 资源回收 | 依赖 GC | 池管理、主动回收 |
| QPS 支撑 | 低（大量时间在建连） | 高（连接复用） |

### 1.2 连接池生命周期

```
① 池初始化（创建 minimumIdle 个连接写入池中）
② 获取连接（getConnection）
   → 池中有空闲连接 → 直接返回
   → 池中无空闲 → 当前连接数 < maximumPoolSize → 创建新连接
   → 当前连接数 = maximumPoolSize → 等待（超时抛异常）
③ 归还连接（connection.close() 实际是归还到池）
④ 连接检测（keepalive 心跳、空闲超时淘汰、最大生命周期回收）
```

### 1.3 核心参数模型

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `minimumIdle` | 最小空闲连接数 | = maximumPoolSize |
| `maximumPoolSize` | 最大连接数 | CPU核心数×2 + 磁盘数 |
| `connectionTimeout` | 获取连接最大等待时间 | 3000ms |
| `idleTimeout` | 空闲连接最大存活时间 | 600000ms（10min） |
| `maxLifetime` | 连接最大生命周期 | 1800000ms（30min，小于 DB wait_timeout） |
| `validationTimeout` | 连接活性校验超时 | 5000ms |

### 1.4 面试标准答法
> 连接池通过预创建、复用、回收机制避免频繁创建销毁TCP连接的开销。核心参数包括最大连接数（控制DB压力）、连接超时（避免无限等待）、最大生命周期（避免使用被DB端断开的连接）。连接用完归还到池中，不是真正关闭。

## 二、HikariCP 深入解析 ⭐⭐⭐

### 2.1 为什么 HikariCP 最快

HikariCP 是 **SpringBoot 2.x+ 默认连接池**，性能碾压其他方案：

| 优化点 | 实现 |
|--------|------|
| **FastList** | 替代 ArrayList，`get()` 不做范围检查；`remove()` 从尾部开始遍历 |
| **ConcurrentBag** | 无锁并发容器：ThreadLocal 优先获取本线程归还的连接（缓存亲和） |
| **字节码精简** | 编译后字节码极少，JIT 更容易内联优化 |
| **无锁设计** | CAS + `CopyOnWriteArrayList` 替代 synchronized |
| **代理类预编译** | 使用 Javassist 生成代理类，避免反射 |
| **精简调用栈** | `Connection.close()` → 直接归还，调用栈极短 |

### 2.2 ConcurrentBag 工作原理

```java
// 获取连接 - 三级查找策略
T borrow(long timeout) {
    // 1. 优先从 ThreadLocal 列表获取（无锁，最快）
    List<Object> list = threadList.get();
    for (逆序遍历 list) {
        if (CAS 标记为 IN_USE) return entry;
    }
    // 2. 从 sharedList CopyOnWriteArrayList 获取
    for (entry : sharedList) {
        if (CAS 标记为 IN_USE) return entry;
    }
    // 3. 都没有 → 通过 handoffQueue(SynchronousQueue) 等待归还
    return handoffQueue.poll(timeout);
}
```

### 2.3 连接状态机
```
NOT_IN_USE ──CAS──→ IN_USE ──close()──→ NOT_IN_USE
     ↓                                       ↓
  RESERVED（驱逐标记）               超过maxLifetime
     ↓                                       ↓
  REMOVED（彻底销毁）              REMOVED（彻底销毁）
```

### 2.4 HikariCP 配置模板
```yaml
spring:
  datasource:
    hikari:
      pool-name: HikariPool
      minimum-idle: 10           # 等于 maximumPoolSize 避免动态创建
      maximum-pool-size: 10      # CPU核心×2 + 磁盘数
      connection-timeout: 3000   # 3秒获取不到连接抛异常
      idle-timeout: 600000       # 10分钟空闲回收
      max-lifetime: 1800000      # 30分钟（小于 MySQL wait_timeout=28800）
      connection-test-query: SELECT 1
      leak-detection-threshold: 60000  # 60秒未归还视为泄漏
```

### 2.5 面试标准答法
> HikariCP 是 SpringBoot 默认连接池，最快的原因：① ConcurrentBag 无锁容器 + ThreadLocal 缓存亲和 ② FastList 去掉边界检查 ③ 字节码精简利于 JIT 内联 ④ Javassist 预编译代理类。获取连接三级查找：ThreadLocal → sharedList → handoffQueue。

## 三、Druid 特性与监控 ⭐⭐

### 3.1 Druid 核心特性

| 特性 | 说明 |
|------|------|
| **内置监控** | 提供 Web 监控面板（SQL执行统计、慢查询、连接池状态） |
| **SQL 防火墙** | `WallFilter` 拦截危险SQL（DROP、全表DELETE） |
| **Filter 链** | 类似 Servlet Filter，可插拔（日志/统计/防火墙） |
| **SQL 解析** | 内置 SQL Parser，支持 SQL 格式化和注入检测 |
| **多数据源** | 原生支持读写分离 |

### 3.2 监控配置
```yaml
spring:
  datasource:
    type: com.alibaba.druid.pool.DruidDataSource
    druid:
      initial-size: 5
      min-idle: 5
      max-active: 20
      max-wait: 3000
      # 监控
      stat-view-servlet:
        enabled: true
        url-pattern: /druid/*
        login-username: admin
        login-password: admin123
      filter:
        stat:
          enabled: true
          slow-sql-millis: 1000    # 超过1秒记录慢SQL
          log-slow-sql: true
        wall:
          enabled: true            # SQL防火墙
```

### 3.3 Filter 链机制
```
应用层
  → StatFilter（统计SQL执行时间、次数）
  → WallFilter（SQL防火墙，拦截危险操作）
  → Slf4jLogFilter（SQL日志输出）
  → JDBC Driver
```

### 3.4 HikariCP vs Druid

| 对比 | HikariCP | Druid |
|------|----------|-------|
| **性能** | ⭐⭐⭐⭐⭐（最快） | ⭐⭐⭐⭐ |
| **监控** | 无内置（需Prometheus/Micrometer） | ✅ 内置Web监控面板 |
| **SQL防火墙** | ❌ | ✅ WallFilter |
| **功能丰富度** | 极简（只做连接池） | 全面（监控/防火墙/Filter） |
| **社区** | Spring 官方默认 | 阿里开源，国内主流 |
| **适用场景** | 追求极致性能 | 需要监控/审计/安全 |

### 3.5 面试标准答法
> Druid 是阿里开源的连接池，核心优势是内置监控（SQL统计/慢查询/连接池状态）和 SQL 防火墙。通过 Filter 链机制扩展（Stat/Wall/Log）。性能略低于 HikariCP，但在需要监控审计的场景更适合。SpringBoot 2.x 默认 HikariCP，需要监控时可用 Druid。

## 四、参数调优指南 ⭐⭐⭐

### 4.1 最大连接数计算

**PostgreSQL 推荐公式**（同样适用 MySQL）：
```
maximumPoolSize = CPU核心数 × 2 + 有效磁盘数

例：4核CPU + 1块SSD → 4 × 2 + 1 = 9 ≈ 10
```

**经验法则**：
- **OLTP 短事务**：10-20 （绝大多数 Web 应用足够）
- **OLAP 长查询**：可适当增大到 30-50
- **微服务多实例**：总连接数 = 单实例 Pool × 实例数 ≤ MySQL `max_connections`

> ⚠️ **常见误区**：连接数越多越好 → 错！过多连接导致 DB 端上下文切换增加、内存不足，性能反而下降。

### 4.2 超时参数关系
```
connectionTimeout < validationTimeout < idleTimeout < maxLifetime < DB wait_timeout

推荐值：
  connectionTimeout: 3s
  validationTimeout: 5s
  idleTimeout: 10min
  maxLifetime: 30min
  MySQL wait_timeout: 8h（默认28800s）
```

### 4.3 minimumIdle 设置策略
| 策略 | 说明 |
|------|------|
| minimumIdle = maximumPoolSize | **推荐**：固定大小池，避免动态创建/销毁开销 |
| minimumIdle < maximumPoolSize | 低峰期减少空闲连接，但突发流量时需要等待创建 |

### 4.4 keepalive 机制
```yaml
# HikariCP 5.0+
keepalive-time: 30000  # 每30秒对空闲连接发探测包
```
- 防止连接被中间设备（防火墙/LB）4分钟空闲断开
- 防止 MySQL 端 `wait_timeout` 到期断开

### 4.5 面试标准答法
> 最大连接数推荐 CPU核×2+磁盘数（通常10-20足够），不是越多越好。超时参数遵循 connectionTimeout < maxLifetime < DB wait_timeout。minimumIdle 建议等于 maximumPoolSize 维持固定池大小。多实例部署时注意总连接数不超过 MySQL max_connections。

## 五、连接泄漏排查 ⭐⭐

### 5.1 什么是连接泄漏
> 从连接池获取连接后，**未正确归还**（close），导致池中可用连接越来越少，最终 `connectionTimeout` 抛出异常。

### 5.2 常见泄漏原因

| 原因 | 示例 |
|------|------|
| **未关闭连接** | try 中获取连接，异常后未 finally close |
| **事务未提交/回滚** | `@Transactional` 方法抛异常但被 catch 吞掉 |
| **ORM 无关连接** | `DataSource.getConnection()` 手动获取未手动归还 |
| **长事务** | 大事务持有连接时间过长，超过泄漏检测阈值 |

### 5.3 排查步骤

```bash
# 1. 查看连接池状态
# HikariCP 通过 JMX 或 Actuator
GET /actuator/metrics/hikaricp.connections.active
GET /actuator/metrics/hikaricp.connections.idle
GET /actuator/metrics/hikaricp.connections.pending

# 2. 开启泄漏检测
spring.datasource.hikari.leak-detection-threshold=60000

# 3. 日志中搜索泄漏警告
grep "Connection leak detection" app.log
# 会打印获取连接时的堆栈 → 定位泄漏代码

# 4. Druid 监控面板
# 查看"活跃连接数"是否持续增长（正常应波动）
# 查看"逻辑连接打开/关闭"数差值

# 5. MySQL 端检查
SHOW PROCESSLIST;              -- 查看当前连接
SHOW STATUS LIKE 'Threads%';   -- 查看连接数趋势
```

### 5.4 预防策略

```java
// 1. 使用 try-with-resources（推荐）
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // 执行SQL
} // 自动 close → 归还到池

// 2. Spring 事务管理（推荐）
@Transactional  // Spring 自动管理连接获取和归还
public void biz() { ... }

// 3. 避免手动 getConnection
// 使用 JdbcTemplate / MyBatis，框架自动管理连接
```

### 5.5 面试标准答法
> 连接泄漏是获取连接后未归还，排查方法：① HikariCP 开启 `leak-detection-threshold` 打印泄漏堆栈 ② Druid 监控面板查看活跃连接趋势 ③ MySQL `SHOW PROCESSLIST` 看连接状态。预防：用 try-with-resources 或 Spring 声明式事务，避免手动 getConnection。
