# 网络 & 操作系统

## 一、TCP 三次握手 & 四次挥手

### 1. 三次握手

```
客户端（Client）                     服务端（Server）
    │                                     │
    │  ① SYN（seq=x）                     │  LISTEN 状态
    │ ─────────────────────────────────→  │
    │                                     │  SYN_RCVD 状态
    │  ② SYN+ACK（seq=y, ack=x+1）        │
    │ ←─────────────────────────────────  │
    │  ESTABLISHED 状态                   │
    │  ③ ACK（seq=x+1, ack=y+1）          │
    │ ─────────────────────────────────→  │
    │                                     │  ESTABLISHED 状态
```

**为什么是三次而不是两次？**
- 两次握手只能保证客户端→服务端的单向信道可用
- 第三次 ACK 让服务端确认"客户端能收到服务端的数据"，完成双向信道验证
- 防止历史失效连接请求（网络延迟的旧 SYN）被服务端误当成新连接建立

**SYN Flood 攻击：** 攻击者发大量 SYN 不回应 ACK，服务端维护大量半开连接（SYN_RCVD），耗尽连接资源。
对策：`SYN Cookie`（服务端不存 SYN，将状态编码进 seq=y 返回，只有合法 ACK 才能重建状态）。

### 2. 四次挥手

```
主动关闭方（Client）                  被动关闭方（Server）
    │                                     │
    │  ① FIN（seq=u）                     │
    │ ─────────────────────────────────→  │
    │  FIN_WAIT_1                         │  CLOSE_WAIT
    │                                     │  （可继续发数据，半关闭状态）
    │  ② ACK（ack=u+1）                   │
    │ ←─────────────────────────────────  │
    │  FIN_WAIT_2                         │
    │                                     │  （Server 数据发完...）
    │  ③ FIN（seq=v）                     │
    │ ←─────────────────────────────────  │
    │                                     │  LAST_ACK
    │  ④ ACK（ack=v+1）                   │
    │ ─────────────────────────────────→  │
    │  TIME_WAIT（等待 2MSL）              │  CLOSED
    │  → CLOSED                           │
```

**为什么挥手需要四次而握手只需三次？**
- 握手时 SYN+ACK 可以合并（服务端同时确认+发起）
- 挥手时服务端收到 FIN 后，可能还有数据未发完（半关闭状态），ACK 和 FIN 必须分开发

**TIME_WAIT 为什么等待 2MSL（2 × 最大报文生存时间，约 60s）？**
1. 确保最后一个 ACK 能到达服务端（ACK 丢失时服务端会重发 FIN，需要时间重传）
2. 让网络中残留的旧连接报文全部消散，防止新连接收到旧连接的数据

**TIME_WAIT 过多的问题：**
- 大量短连接场景（如 HTTP/1.0 每次请求都新建连接）会堆积大量 TIME_WAIT，占用端口资源
- 解决：① 开启 `SO_REUSEADDR` 允许复用 TIME_WAIT 端口；② HTTP 长连接 Keep-Alive；③ `tcp_tw_reuse`（复用 TIME_WAIT 连接，仅客户端有效）

### 3. 常见面试追问

**Q: 为什么 TCP 建连需要三次，断连需要四次？**
> 建连时服务端可以把 SYN 和 ACK 合并成一个报文（同时表示"我收到了"和"我也想建连"）；断连时，服务端收到 FIN 只代表客户端不再发数据，但服务端可能仍有数据要发（半关闭状态），所以先单独 ACK 确认，等服务端数据发完再单独发 FIN。本质原因：TCP 是全双工，断连需要双方各自独立关闭自己的发送方向。

**Q: 如果第三次握手的 ACK 丢失了怎么办？**
> 服务端停留在 SYN_RCVD 状态，超时后会重发 SYN+ACK（默认重试 5 次，间隔指数退避）。客户端已经是 ESTABLISHED，收到重发的 SYN+ACK 后会再次发 ACK。若重试次数耗尽，服务端关闭该半开连接。

## 二、TCP 可靠性保证

### 1. 可靠性机制全景

```
┌─────────────────────────────────────────────────────────────┐
│                   TCP 可靠性机制                             │
├─────────────────┬───────────────────────────────────────────┤
│  序列号 & ACK    │ 每个字节都有序列号，接收方 ACK 确认已收到  │
│  超时重传        │ 未收到 ACK → RTO 超时 → 重发             │
│  快速重传        │ 收到 3 个重复 ACK → 立即重发（不等超时）  │
│  流量控制        │ 接收窗口（rwnd）限制发送速率，防止接收方   │
│                 │ 缓冲区溢出                                │
│  拥塞控制        │ 拥塞窗口（cwnd）根据网络状态动态调整      │
│  校验和          │ 检测数据传输中的错误（位翻转等）           │
│  连接管理        │ 三次握手/四次挥手保证双向通道可靠建立     │
└─────────────────┴───────────────────────────────────────────┘
```

### 2. 滑动窗口

```
发送方维护发送窗口（min(cwnd, rwnd)）：

已发已确认 │ 已发未确认 │ 可发未发 │ 不可发
─────────┼───────────┼─────────┼──────────
         ← 发送窗口 →
         
窗口滑动：收到 ACK 后，窗口向右移动
```

**零窗口问题：** 接收方 rwnd=0 时，发送方停止发送，开启 Persist Timer 定时发探测报文，等待窗口恢复。

### 3. 拥塞控制四阶段

```
cwnd
  │                  指数增长（慢启动）
  │               /
ssthresh──────────           线性增长（拥塞避免）
  │             /──────────────────
  │            /                    \  发生丢包
  │           /                      \（丢包 → ssthresh=cwnd/2，cwnd=1 重来）
  │          /
  │─────────
  1
  └──────────────────────────────────────── RTT

① 慢启动：cwnd < ssthresh，每个 RTT 翻倍（指数增长）
② 拥塞避免：cwnd ≥ ssthresh，每个 RTT 加 1（线性增长）
③ 快速重传：3 个重复 ACK → ssthresh = cwnd/2，cwnd = ssthresh（从中间重来，比从 1 开始快）
④ 快速恢复：配合快速重传，不从 1 开始，直接从新 ssthresh 开始拥塞避免
```

### 4. 面试标准答法

> TCP 可靠性靠五个机制协同保证：① **序列号+ACK**：对每个字节编号，接收方 ACK 确认，发送方知道哪些已收到；② **超时重传**：RTO 内未收到 ACK 则重传，RTO 基于 RTT 动态计算（Karn/Jacobson 算法）；③ **快速重传**：收到 3 个重复 ACK 意味着一个包丢了但后续包到了，不等超时立即重传；④ **流量控制**：接收方通过 TCP header 中的窗口字段（rwnd）告知发送方自己还能接收多少，防止缓冲区溢出；⑤ **拥塞控制**（慢启动+拥塞避免+快速恢复）动态探测网络容量上限，防止网络拥塞。

### 5. 常见追问

**Q: TCP 和 UDP 的区别，什么场景用 UDP？**
> TCP：面向连接、可靠、有序、流量/拥塞控制、较大头部开销（20B）。UDP：无连接、不可靠、无序、无拥塞控制、头部 8B。选 UDP 的场景：① 实时性优先于可靠性（音视频通话，丢帧可接受，重传反而增加延迟）；② 广播/多播（TCP 不支持）；③ DNS 查询（一问一答，短小，可自行实现重试）；④ 游戏信令（有些游戏自建可靠层在 UDP 上，KCP 协议）。QUIC（HTTP/3 底层）也是基于 UDP 自建可靠传输。

**Q: TCP 粘包 / 拆包是什么，如何解决？**
> TCP 是字节流协议，没有消息边界概念，发送的多条消息可能被合并（粘包）或拆分（拆包）。解决方案：① **固定长度**：每条消息固定字节数，满了就是一条；② **分隔符**：特殊字符（如 `\n`）分隔，接收方按分隔符切割；③ **长度前缀**：消息头 4 字节表示消息体长度，先读 4 字节，再读对应长度（最常用，Netty/PB/Dubbo 都是此方案）。

## 三、HTTP vs HTTPS

### 1. 核心区别

| | HTTP | HTTPS |
|---|---|---|
| 默认端口 | 80 | 443 |
| 传输层 | 明文 TCP | SSL/TLS 加密 |
| 安全性 | 可被中间人截获 | 加密 + 身份验证 |
| 速度 | 快 | 略慢（TLS 握手 + 加解密开销）|
| 证书 | 不需要 | 需要 CA 颁发的数字证书 |

### 2. TLS 握手过程（TLS 1.2）

```
Client                                Server
  │                                     │
  │  ① ClientHello                      │
  │  （TLS版本、随机数1、支持的加密套件）  │
  │ ─────────────────────────────────→  │
  │                                     │
  │  ② ServerHello + Certificate        │
  │  （选定加密套件、随机数2、服务端证书） │
  │ ←─────────────────────────────────  │
  │                                     │
  │  ③ 客户端验证证书（CA链验证）         │
  │    生成 Pre-Master-Secret（随机数3） │
  │    用服务端公钥加密，发送给服务端      │
  │  ClientKeyExchange                  │
  │ ─────────────────────────────────→  │
  │                                     │
  │  双方各自用随机数1+2+3 推导出         │
  │  对称加密 Session Key               │
  │                                     │
  │  ④ ChangeCipherSpec（准备切换加密）   │
  │  ⑤ Finished（握手摘要，验证一致性）  │
  │ ←─────────────→                     │
  │                                     │
  │  ═══ 后续用 Session Key 对称加密 ═══ │
```

**TLS 1.3 改进：** 1-RTT 握手（减少一次往返），0-RTT 会话复用（再次连接直接发数据），废弃 RSA 密钥交换（改用 ECDHE，保证前向安全）。

### 3. 证书验证（防中间人攻击）

```
CA（根证书颁发机构）
  └── 中间 CA（颁发的中间证书）
        └── 服务端证书（包含域名、公钥、CA签名）

验证流程：
1. 客户端收到服务端证书
2. 用中间 CA 公钥验证服务端证书签名 → 合法
3. 用根 CA 公钥验证中间 CA 证书签名 → 合法
4. 根证书预装在操作系统/浏览器中 → 信任锚点
5. 验证域名匹配（CN/SAN字段）
6. 验证证书未过期、未被吊销（OCSP/CRL）
```

### 4. HTTP/1.1 → HTTP/2 → HTTP/3

| | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---|---|---|---|
| **底层协议** | TCP | TCP | UDP（QUIC）|
| **多路复用** | 无（队头阻塞）| ✅ 二进制帧+流 | ✅ QUIC 流独立 |
| **头部压缩** | 无 | ✅ HPACK | ✅ QPACK |
| **服务端主动推送** | ❌ | ✅ Server Push | ✅ |
| **队头阻塞** | 应用层 | TCP 层仍有 | ✅ 消除（UDP）|
| **握手延迟** | 1-RTT TCP | 1-RTT TCP | 0-RTT |

### 5. 面试标准答法

> HTTP 和 HTTPS 的核心区别是 HTTPS 在 TCP 之上加了 TLS 层，实现了三件事：① **加密**（防窃听，对称加密传输数据，非对称加密交换密钥）；② **身份验证**（CA 证书机制，防中间人伪装服务端）；③ **完整性校验**（MAC，防数据篡改）。TLS 握手在 TCP 三次握手后进行，TLS 1.2 需要额外 2-RTT，TLS 1.3 优化到 1-RTT，HTTPS 延迟已经可以接受。

### 6. 常见追问

**Q: HTTPS 能防止所有中间人攻击吗？**
> 不能完全防止。以下情况仍有风险：① **证书信任问题**：客户端被植入伪造的根证书（企业/恶意软件），中间人可以用该根 CA 签发任意域名的证书；② **证书吊销未及时检查**：OCSP/CRL 有缓存，吊销不立即生效（OCSP Stapling 可缓解）；③ **SSLstrip 降级攻击**：将 HTTPS 降级为 HTTP（HSTS 头解决，`Strict-Transport-Security` 强制 HTTPS）。

**Q: HTTPS 的对称密钥是怎么协商出来的？**
> 以 TLS 1.2 RSA 为例：客户端生成随机 Pre-Master-Secret，用服务端公钥（证书中）加密发给服务端，服务端用私钥解密。双方各有：随机数1（Client Random）、随机数2（Server Random）、Pre-Master-Secret，通过 PRF（伪随机函数）推导出 Session Key（会话密钥），后续用 AES 等对称加密通信。TLS 1.3 改用 ECDHE：双方各生成临时公私钥对，交换公钥，独立推导出共享密钥（Diffie-Hellman），不传输任何密钥本身，即使私钥泄露也不影响历史会话（前向安全）。

## 四、零拷贝原理

### 1. 传统 IO 数据拷贝（4 次拷贝 + 4 次上下文切换）

```
应用程序读取文件并通过网络发送（read + write）：

磁盘 ──DMA拷贝──→ 内核 PageCache ──CPU拷贝──→ 用户缓冲区（read 返回）
                                                    │
           ──────────────────────────── CPU拷贝 ←──┘
           ↓
     内核 Socket 发送缓冲区 ──DMA拷贝──→ 网卡（发送）

切换：① 用户→内核（read）② 内核→用户（read 返回）③ 用户→内核（write）④ 内核→用户（write 返回）
```

### 2. sendfile 零拷贝（2 次拷贝 + 2 次上下文切换）

```java
// Linux sendfile 系统调用
// Java 中 FileChannel.transferTo() 底层使用 sendfile

FileChannel fileChannel = new FileInputStream("file.txt").getChannel();
SocketChannel socketChannel = serverSocket.accept();
// 数据不经过用户态，直接从 PageCache 传到 Socket 缓冲区
fileChannel.transferTo(0, fileChannel.size(), socketChannel);
```

```
磁盘 ──DMA拷贝──→ 内核 PageCache ──CPU拷贝──→ 内核 Socket 缓冲区 ──DMA拷贝──→ 网卡

切换：① 用户→内核（sendfile）② 内核→用户（sendfile 返回）
```

### 3. sendfile + DMA Scatter-Gather（真正的零 CPU 拷贝）

Linux 2.4+ 支持 DMA Scatter-Gather，连 CPU 拷贝也省掉：

```
磁盘 ──DMA拷贝──→ 内核 PageCache
                      │ 仅传文件描述符（fd + offset + length）给 Socket 缓冲区
                      ↓
              Socket 缓冲区（仅有 fd 信息，不复制数据）
                      │
                      ↓ DMA Scatter-Gather 协议
                    网卡直接从 PageCache 读

总拷贝：2 次 DMA（无 CPU 拷贝）
```

### 4. mmap + write（3 次拷贝）

```java
MappedByteBuffer buffer = fileChannel.map(FileChannel.MapMode.READ_ONLY, 0, size);
// buffer 是 PageCache 的用户态映射（不复制数据，只映射地址）
socketChannel.write(buffer);  // 用户态直接操作内核 PageCache
```

```
磁盘 ──DMA拷贝──→ 内核 PageCache
                  ↑ mmap 直接映射（无拷贝）
             用户态虚拟地址（buffer）
                  │ write 时
                  ↓ CPU拷贝
             Socket 发送缓冲区 ──DMA拷贝──→ 网卡
```

### 5. 各方案对比

| 方案 | CPU 拷贝 | DMA 拷贝 | 上下文切换 | 适用场景 |
|---|---|---|---|---|
| 传统 read+write | 2 | 2 | 4 | 通用 |
| mmap+write | 1 | 2 | 4 | 文件内容需修改 |
| sendfile | 1 | 2 | 2 | **文件直接发网络（Kafka、Nginx）**|
| sendfile+DMA SG | 0 | 2 | 2 | 大文件高吞吐（需网卡支持）|

**Kafka 使用零拷贝：** Consumer 拉取消息时，Broker 用 `FileChannel.transferTo()` 将日志文件直接发给 Socket，无需经过用户态，吞吐极高。

### 6. 面试标准答法

> 零拷贝是指减少数据在内存中的冗余复制，降低 CPU 干预。传统 IO 读文件发网络需要 4 次内存拷贝（磁盘→PageCache，PageCache→用户缓冲区，用户缓冲区→Socket 缓冲区，Socket 缓冲区→网卡）+ 4 次用户态/内核态切换。`sendfile` 系统调用跳过用户态缓冲区，数据从 PageCache 直接到 Socket 缓冲区（2 次 DMA + 1 次 CPU 拷贝），再配合 DMA Scatter-Gather 可以做到 0 次 CPU 拷贝。Java 的 `FileChannel.transferTo()` 底层就是 `sendfile`，Kafka/Nginx 都依赖它实现高吞吐数据传输。

## 五、IO 模型（BIO / NIO / AIO）

### 1. 五种 IO 模型（UNIX）

```
用户进程         内核
    │              │
    │─ recvfrom ──→│ 等待数据到达（内核等待网卡数据）
    │              │
    │   阻塞等待    │ 数据到达内核缓冲区
    │              │
    │              │ 将数据从内核复制到用户空间
    │ ←─ 返回 ─────│
    │              │
```

| 模型 | 等待数据 | 复制数据 | 特点 |
|---|---|---|---|
| **BIO（阻塞IO）** | 阻塞 | 阻塞 | 简单，一请求一线程，并发差 |
| **NIO（非阻塞IO）** | 不阻塞（轮询）| 阻塞 | 轮询浪费 CPU |
| **IO 多路复用** | select/epoll 阻塞 | 阻塞 | **一个线程监控多个 fd，Reactor 模式** |
| **信号驱动IO** | 不阻塞（信号）| 阻塞 | 少用 |
| **AIO（异步IO）** | 不阻塞 | 不阻塞 | 全异步，数据复制完才通知 |

### 2. BIO vs NIO 代码对比

**BIO（每个连接一个线程）：**
```java
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();  // 阻塞
    // 一个连接一个线程
    new Thread(() -> {
        byte[] buf = new byte[1024];
        socket.getInputStream().read(buf);  // 阻塞
        // 处理...
    }).start();
}
// 问题：10w 并发 = 10w 线程，内存爆炸，线程切换开销极大
```

**NIO（多路复用，Selector）：**
```java
Selector selector = Selector.open();
ServerSocketChannel server = ServerSocketChannel.open();
server.configureBlocking(false);
server.bind(new InetSocketAddress(8080));
server.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 阻塞等待，直到有事件就绪
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            SocketChannel client = server.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
        } else if (key.isReadable()) {
            SocketChannel client = (SocketChannel) key.channel();
            ByteBuffer buf = ByteBuffer.allocate(1024);
            client.read(buf);
            // 处理...
        }
    }
    keys.clear();
}
// 一个线程处理所有连接的 IO 事件，C10K 轻松应对
```

### 3. Reactor 模式（NIO 的编程模型）

#### 3.1 核心思想

Reactor 模式（也叫事件驱动模式）是一种处理并发 IO 的设计模式：

```
核心角色：
┌─────────────────────────────────────────────────────────────┐
│  Reactor（反应堆）                                          │
│    负责监听事件（accept/read/write），事件就绪后分发给 Handler │
│                                                             │
│  Acceptor                                                   │
│    处理新连接事件（OP_ACCEPT），将连接注册给后续 Reactor       │
│                                                             │
│  Handler                                                    │
│    处理具体的 IO 事件（读数据、编解码、业务逻辑、写回）         │
└─────────────────────────────────────────────────────────────┘

事件循环（Event Loop）：
  while (true) {
      select/epoll 阻塞等待事件
      for each 就绪事件:
          dispatch 到对应 Handler
  }
```

#### 3.2 单 Reactor 单线程（Redis 6.0 之前的命令处理）

```
                  ┌──────────────────────────────┐
                  │       单线程 Reactor           │
                  │                              │
Client ──connect──→  Acceptor                   │
Client ──read──────→  Handler1  → 业务处理 → write │
Client ──read──────→  Handler2  → 业务处理 → write │
                  └──────────────────────────────┘

一个线程负责：accept + read + 业务处理 + write
```

```java
// 简化版单 Reactor 单线程示意
public class SingleReactor implements Runnable {
    private final Selector selector;
    private final ServerSocketChannel serverChannel;

    public void run() {
        while (true) {
            selector.select();  // 阻塞等待任意事件就绪
            Iterator<SelectionKey> it = selector.selectedKeys().iterator();
            while (it.hasNext()) {
                SelectionKey key = it.next();
                it.remove();
                if (key.isAcceptable()) {
                    // 新连接：注册读事件
                    SocketChannel client = serverChannel.accept();
                    client.configureBlocking(false);
                    client.register(selector, SelectionKey.OP_READ);
                } else if (key.isReadable()) {
                    // 读数据 + 业务处理 + 写回 → 全在本线程
                    handle(key);
                }
            }
        }
    }
}
```

**优点：** 实现简单，无锁，无线程切换开销  
**缺点：** 单线程，业务处理阻塞会影响所有连接；无法利用多核  
**代表：** Redis 6.0 之前（命令处理单线程，IO 读写也是）

#### 3.3 单 Reactor 多线程

```
                  ┌─────────────────────────────────────────┐
                  │  Reactor 线程（只负责 IO 事件分发）        │
                  │                                         │
Client ──connect──→  Acceptor                              │
Client ──read──────→  Handler（只做 read + write，不做业务）  │
                  └─────────────┬───────────────────────────┘
                                │ 业务处理任务投递
                                ▼
                  ┌─────────────────────────────────────────┐
                  │  业务线程池（Thread Pool）                │
                  │  Worker1 / Worker2 / ... / WorkerN       │
                  │  负责：解码 + 业务逻辑 + 编码             │
                  └─────────────────────────────────────────┘
```

**优点：** IO 和业务解耦，业务处理不阻塞 Reactor；可利用多核处理业务  
**缺点：** Reactor 单线程仍是瓶颈（高并发接入时 accept + IO 都压在一个线程上）

#### 3.4 主从 Reactor 多线程（生产主流，Netty 默认）

```
客户端连接
    │
    ▼
┌───────────────────────────────────────────────────────────────┐
│  MainReactor（1 个线程，只处理 accept）                        │
│  监听 ServerSocketChannel 的 OP_ACCEPT 事件                    │
└──────────────────────┬────────────────────────────────────────┘
                       │ 新连接建立后，将 SocketChannel 注册给 SubReactor
                       ▼
┌──────────────────────────────────────────────────────┐
│  SubReactor 池（N 个线程，每个线程维护自己的 Selector） │
│                                                      │
│  SubReactor-0: 监听连接 C1、C4、C7... 的 read/write   │
│  SubReactor-1: 监听连接 C2、C5、C8... 的 read/write   │
│  SubReactor-2: 监听连接 C3、C6、C9... 的 read/write   │
└────────────────────────┬─────────────────────────────┘
                         │ IO 完成后，投递到业务线程池
                         ▼
┌──────────────────────────────────────────────────────┐
│  业务线程池（解码 + 业务逻辑 + 编码）                  │
└──────────────────────────────────────────────────────┘
```

**Netty 完整实现：**

```java
// MainReactor：bossGroup（默认1线程，只接受连接）
EventLoopGroup bossGroup = new NioEventLoopGroup(1);
// SubReactor：workerGroup（默认 CPU*2 线程，每个 EventLoop 一个 Selector）
EventLoopGroup workerGroup = new NioEventLoopGroup();

try {
    ServerBootstrap b = new ServerBootstrap();
    b.group(bossGroup, workerGroup)
     .channel(NioServerSocketChannel.class)
     .option(ChannelOption.SO_BACKLOG, 1024)           // 连接队列大小
     .childOption(ChannelOption.SO_KEEPALIVE, true)    // 开启心跳
     .childHandler(new ChannelInitializer<SocketChannel>() {
         @Override
         protected void initChannel(SocketChannel ch) {
             ChannelPipeline p = ch.pipeline();
             p.addLast(new LengthFieldBasedFrameDecoder(65536, 0, 4)); // 解决粘包
             p.addLast(new MyDecoder());      // 反序列化
             p.addLast(businessExecutor,      // 业务处理放独立线程池
                       new MyBusinessHandler());
             p.addLast(new MyEncoder());      // 序列化
         }
     });

    ChannelFuture f = b.bind(8080).sync();
    f.channel().closeFuture().sync();
} finally {
    bossGroup.shutdownGracefully();
    workerGroup.shutdownGracefully();
}
```

**Netty 的 Pipeline 模型：**

```
SocketChannel 收到数据
    │
    ▼ InboundHandler 链（从头到尾）
[LengthFieldDecoder] → [MyDecoder] → [MyBusinessHandler]
                                              │
                                              ▼ 业务处理完，写回
[MyEncoder] ← [OutboundHandler 链（从尾到头）]
    │
    ▼
SocketChannel 发送数据

每个 Handler 只做一件事（责任链模式），职责清晰，易扩展
```

#### 3.5 三种 Reactor 模式对比

| | 单 Reactor 单线程 | 单 Reactor 多线程 | 主从 Reactor 多线程 |
|---|---|---|---|
| **接受连接** | 主线程 | 主线程 | MainReactor 专用线程 |
| **IO 读写** | 主线程 | 主线程 | SubReactor 线程池 |
| **业务处理** | 主线程 | 业务线程池 | 业务线程池 |
| **多核利用** | ❌ | 部分（业务）| ✅ 全部 |
| **IO 瓶颈** | 有 | 有（单 Reactor）| 无（多 SubReactor）|
| **典型应用** | Redis 6.0 之前 | 部分旧版框架 | **Netty / Nginx** |

#### 3.6 Reactor vs Proactor

| | Reactor（同步 IO）| Proactor（异步 IO）|
|---|---|---|
| **触发时机** | IO **就绪**时通知（你来读）| IO **完成**时通知（读好了给你）|
| **谁做 IO** | Handler 自己调用 read/write | 内核完成 IO，结果放到缓冲区 |
| **Linux 支持** | epoll 完善支持 | AIO 支持不成熟 |
| **代表框架** | Netty、Nginx | Windows IOCP、Boost.Asio |
| **实际选型** | **Java/Linux 首选 Reactor** | Windows 高性能网络程序 |

### 4. 面试标准答法

> Java IO 模型经历了三个阶段。**BIO**（`java.io`）是同步阻塞，每个连接占用一个线程，C 级别并发时线程开销无法承受。**NIO**（`java.nio + Selector`）是 IO 多路复用，一个线程通过 Selector 监控多个 Channel 的就绪事件，本质上调用 OS 的 `epoll`（Linux）机制，实现 C10K 级别并发。**AIO**（`java.nio.AsynchronousChannel`）是真正的异步 IO，数据复制完成后才回调通知，但在 Linux 内核支持不成熟（底层仍是线程模拟），实际生产中 Netty 等框架更偏向 NIO + 主从 Reactor 多线程模式。

## 六、Epoll 原理

### 1. select / poll / epoll 对比

| | select | poll | epoll |
|---|---|---|---|
| **监控 fd 数量** | 1024（FD_SETSIZE）| 无限制 | 无限制 |
| **时间复杂度** | O(n) 每次遍历全量 | O(n) | O(1) 就绪事件 |
| **内核-用户空间通信** | 每次 select 都 copy fd_set | 每次 poll 都 copy pollfd | 仅注册时 copy，就绪事件通过回调通知 |
| **底层实现** | 位图 | 数组 | 红黑树 + 就绪链表 |
| **适合场景** | fd 少，活跃连接多 | fd 少 | **大量连接，少量活跃（C10K）** |

### 2. Epoll 三个核心接口

```c
// 1. 创建 epoll 实例（返回 epfd）
int epfd = epoll_create(1);

// 2. 注册/修改/删除感兴趣的 fd 和事件
struct epoll_event ev;
ev.events = EPOLLIN;        // 监听可读事件
ev.data.fd = sockfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);  // 注册

// 3. 等待就绪事件（只返回就绪的 fd，不是全量扫描）
struct epoll_event events[MAX_EVENTS];
int n = epoll_wait(epfd, events, MAX_EVENTS, timeout);
for (int i = 0; i < n; i++) {
    // events[i].data.fd 是就绪的 fd
    read(events[i].data.fd, buf, sizeof(buf));
}
```

### 3. Epoll 内核数据结构

```
epoll 实例（epfd）维护：
┌──────────────────────────────────────────────────────────────┐
│ 红黑树（rbr）                                                │
│   存储所有通过 epoll_ctl 注册的 fd（快速增删改，O(log n)）    │
│                                                              │
│ 就绪链表（rdllist）                                          │
│   存储已就绪的 fd（有数据可读/可写）                         │
│   epoll_wait 只需遍历此链表，返回就绪 fd                    │
│                                                              │
│ 回调机制：                                                   │
│   网卡数据到达 → 内核将对应 fd 从红黑树移入就绪链表           │
│   epoll_wait 检测到就绪链表非空 → 立即返回                  │
└──────────────────────────────────────────────────────────────┘
```

### 4. ET（边缘触发）vs LT（水平触发）

| | LT（Level Trigger，默认）| ET（Edge Trigger）|
|---|---|---|
| **触发时机** | 只要 fd 处于就绪状态就一直通知 | 仅在状态**变化时**通知一次 |
| **处理要求** | 不用一次读完，下次还会通知 | **必须一次读完**，否则错过事件 |
| **性能** | 略低（可能多次通知）| 略高（通知次数少）|
| **编程难度** | 简单 | 复杂（需循环读直到 EAGAIN）|
| **使用场景** | Nginx 默认 | Nginx 高性能模式 |

```c
// ET 模式下必须循环读
ev.events = EPOLLIN | EPOLLET;  // ET模式
// ...
while (true) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n == -1 && errno == EAGAIN) break;  // 读完了
    if (n == 0) { close(fd); break; }       // 连接关闭
    // 处理数据...
}
```

### 5. 面试标准答法

> epoll 相比 select/poll 有三个核心优势：① **不需要每次传递全量 fd 列表**：fd 注册一次存在内核红黑树中，select 每次调用都要把所有 fd 从用户态 copy 到内核；② **O(1) 获取就绪事件**：内核有就绪链表，有数据的 fd 通过回调直接加入链表，`epoll_wait` 只返回已就绪的 fd，而不是扫描全部 fd；③ **fd 数量无限制**：select 受 `FD_SETSIZE=1024` 限制。ET 模式（边缘触发）在状态变化时才通知，通知次数少，性能更好，但必须一次性读完数据，否则不再通知，编程更难。Nginx、Redis、Java NIO（Linux 下底层）都使用 epoll。

## 七、Linux 常用排查命令

### 1. CPU 问题排查

```bash
# 查看 CPU 使用率 top 进程
top -c                         # 实时监控（按 P 排序 CPU，按 M 排序内存）
top -H -p <pid>               # 查看指定进程的线程 CPU 使用率

# 找到 CPU 最高的线程
top -H -p <pid>               # 找到 CPU 占用高的线程 TID（十进制）
printf '%x\n' <TID>           # 转十六进制，如 12345 → 0x3039
jstack <pid> | grep '0x3039'  # 在堆栈中找对应线程，定位死循环/热点代码

# CPU 整体状态
vmstat 1 5                     # 每1秒打印5次（us:用户态、sy:系统态、id:空闲、wa:IO等待）
mpstat -P ALL 1               # 每核 CPU 使用率
```

**CPU 飙高排查步骤：**
```
1. top → 找 CPU 占用最高的 PID
2. top -H -p PID → 找占用最高的 TID
3. printf '%x' TID → 转16进制
4. jstack PID | grep 16进制 → 找到线程堆栈
5. 分析堆栈 → 死循环/GC 频繁/锁竞争
```
