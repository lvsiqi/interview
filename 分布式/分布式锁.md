# 分布式锁知识点

> 最后更新：2026年3月6日

---

## 一、为什么需要分布式锁

```
单机 JVM 锁（synchronized / ReentrantLock）只在同一进程内有效
分布式场景多个节点并发访问同一资源（库存、账户余额、幂等控制）
→ 需要跨进程、跨机器的互斥机制 → 分布式锁
```

**分布式锁三要素：**
```
互斥性：同一时刻只有一个客户端持有锁
安全性：锁只能由持有者释放（防止误删他人锁）
容错性：锁服务宕机后能自动释放（防止死锁）
```

## 二、Redis 分布式锁 ⭐⭐⭐⭐⭐

### 2.1 基础实现

```bash
# 加锁：SET key value NX PX 30000
# NX = Not Exists（不存在才写入，即加锁）
# PX 30000 = 30秒过期（防止宕机后死锁）
# value = 唯一标识（UUID，用于归属校验）
SET lock:order:123 "uuid-xxxx" NX PX 30000

# 释放锁：Lua 脚本保证"查+删"原子性
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**为什么释放锁必须用 Lua 脚本？**
```
错误做法（非原子）：
  if GET lock == myUUID:    # ① 查询
      DEL lock              # ② 删除
  问题：①和②之间锁可能过期被其他人拿走，DEL 删掉了别人的锁

Lua 脚本：Redis 单线程执行 Lua，GET+DEL 原子不可分割
```

---

### 2.2 Redisson 看门狗机制 ⭐⭐⭐⭐⭐

**问题：业务执行时间 > 锁超时时间 → 锁自动释放 → 并发安全问题**

```
Redisson 解决方案：看门狗（Watchdog）

加锁时若未指定 leaseTime（超时时间）：
  → 默认超时 30s（lockWatchdogTimeout）
  → 同时启动一个定时任务（看门狗线程），每 10s（30s/3）执行一次
  → 检查锁是否还被当前线程持有
    → 是 → 续期到 30s（重置过期时间）
    → 否 → 停止续期

客户端宕机后：
  看门狗线程随进程死亡，无法续期
  → 30s 后锁自动过期释放 → 防死锁
```

**Redisson 加锁底层 Lua 脚本（核心）：**
```lua
-- 加锁（可重入）
if (redis.call('exists', KEYS[1]) == 0) then        -- 锁不存在
    redis.call('hincrby', KEYS[1], ARGV[2], 1);     -- hash: field=线程ID, value=1
    redis.call('pexpire', KEYS[1], ARGV[1]);         -- 设置过期时间
    return nil;
end;
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then  -- 已是本线程持有（重入）
    redis.call('hincrby', KEYS[1], ARGV[2], 1);         -- 重入计数+1
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
return redis.call('pttl', KEYS[1]);  -- 加锁失败，返回剩余过期时间（用于等待）
```

**Redisson 可重入原理：**
```
用 Redis Hash 存储锁：
  key   = 锁名（如 lock:order:123）
  field = 客户端线程唯一标识（UUID + threadId）
  value = 重入次数

同一线程重复加锁 → hincrby +1（计数+1）
释放锁 → hincrby -1 → 计数为0时才真正删除
```

### 2.3 Redis 分布式锁的问题

```
① 主从切换丢锁（最大缺陷）：
   客户端在主节点加锁
   → 主节点还未同步到从节点时宕机
   → 从节点升为主节点（无锁记录）
   → 另一个客户端成功加锁
   → 两个客户端同时持锁！

② 锁过期时间难以估计：
   业务耗时不确定，时间太短死锁，太长等待时间长
   → Redisson 看门狗解决

③ GC STW 导致误判：
   客户端 GC 停顿期间锁自动过期
   → 其他客户端拿锁
   → GC恢复后原客户端以为自己还持锁 → 并发问题
   （RedLock 也无法彻底解决此问题）
```

### 2.4 RedLock（红锁）算法

**解决问题：单 Redis 节点主从切换丢锁**

```
原理：
  部署 N 个（推荐5个）完全独立的 Redis 节点（无主从关系）
  加锁流程：
  ① 记录加锁开始时间 startTime
  ② 依次向 N 个节点发送加锁请求（SET NX PX，设置很短的超时时间防单节点阻塞）
  ③ 成功加锁节点数 >= N/2+1（多数派）且 总耗时 < 锁过期时间 → 加锁成功
  ④ 否则 → 向所有节点发送解锁请求，加锁失败

  有效锁时间 = 设置的过期时间 - 获取锁的总耗时
```

**RedLock 的争议：**
```
Martin Kleppmann（《DDIA》作者）指出 RedLock 的缺陷：
① GC STW / 系统时钟漂移 → 锁过期后仍以为自己持锁 → 并发安全无法保证
② 强依赖各节点时钟精确同步

Antirez（Redis作者）反驳：
  大多数场景不需要绝对安全；上述问题在 ZooKeeper 下也存在

工程实践结论：
  对锁安全性要求极高（金融） → ZooKeeper 分布式锁
  高性能一般场景              → Redisson（单节点/主从）+ 看门狗
  RedLock 复杂度高收益有限    → 生产较少使用
```

## 三、ZooKeeper 分布式锁 ⭐⭐⭐⭐

### 3.1 实现原理

**利用 ZK 临时顺序节点：**

```
加锁流程：
① 在 /locks/order_123/ 下创建临时顺序节点
   → 如：/locks/order_123/lock-0000001、lock-0000002、lock-0000003

② 获取 /locks/order_123/ 下所有子节点，排序

③ 判断自己的节点是否是序号最小的：
   是 → 加锁成功，执行业务

   否 → 监听（Watch）前一个节点的删除事件（非最小，但只监听前一个，避免羊群效应）
        → 阻塞等待

④ 前一个节点删除（持锁者释放或宕机）→ 收到通知 → 重新检查自己是否最小 → 加锁
```

**释放锁：**
```
正常释放：删除自己的临时节点
客户端宕机：ZK 会话超时 → 临时节点自动删除 → 后续等待者收到通知
```

**为什么用临时节点而非永久节点？**
```
临时节点（EPHEMERAL）：客户端断开连接/宕机后自动删除
→ 天然防死锁，不需要手动设置过期时间
```

**为什么用顺序节点？**
```
解决公平锁：按创建顺序排队，先来先得，防止饥饿
解决羊群效应：只监听前一个节点，而非所有节点监听最小节点
  → 若所有节点都监听最小节点，最小节点释放时触发所有客户端唤醒，大量无效竞争
  → 只监听前一个：每次只唤醒一个等待者
```

---

### 3.2 Curator 框架实现

```java
// Apache Curator 封装了 ZK 分布式锁
InterProcessMutex lock = new InterProcessMutex(client, "/locks/order_123");

try {
    if (lock.acquire(30, TimeUnit.SECONDS)) {  // 尝试加锁，最多等30s
        // 业务逻辑
    }
} finally {
    lock.release();  // 释放锁（删除临时顺序节点）
}
```

## 四、Redis vs ZooKeeper 分布式锁对比 ⭐⭐⭐⭐⭐

| 对比 | Redis 分布式锁 | ZooKeeper 分布式锁 |
|------|--------------|------------------|
| **性能** | ⭐⭐⭐⭐⭐ 高（内存操作）| ⭐⭐⭐ 中（需要ZAB协议同步）|
| **可靠性** | ⭐⭐⭐ 主从切换有丢锁风险 | ⭐⭐⭐⭐⭐ 临时节点+ZAB强一致 |
| **死锁防护** | 过期时间（需Watchdog续期）| 临时节点，会话断开自动删除 |
| **公平锁** | ❌ 不支持（竞争抢占）| ✅ 顺序节点天然公平 |
| **可重入** | Redisson 支持（Hash计数）| Curator 支持 |
| **实现复杂度** | 低（Redisson开箱即用）| 中（需 Curator）|
| **适用场景** | 高并发、对强一致要求不极端 | 强一致、对正确性要求极高 |

**选型口诀：**
```
性能优先 → Redis（Redisson）
强一致优先 → ZooKeeper（Curator）
金融/库存核心 → ZooKeeper 或 数据库乐观锁（version字段）
```

## 五、数据库分布式锁（补充）

```
① 悲观锁（SELECT FOR UPDATE）：
   SELECT * FROM goods WHERE id=1 FOR UPDATE;
   → 行级排他锁，事务内互斥
   → 简单可靠，但性能差，不适合高并发

② 乐观锁（version 字段 / CAS）：
   UPDATE goods SET stock=stock-1, version=version+1
   WHERE id=1 AND version=#{oldVersion} AND stock>0
   → 无锁，通过版本号检测冲突，失败重试
   → 适合读多写少，冲突少的场景

乐观锁在秒杀场景的问题：
  高并发下大量 ABA 冲突 → 大量重试 → 性能退化
  解法：version 改为 stock 字段本身（WHERE stock >= #{count}），限制递减，减少 ABA
```

## 六、面试标准答法

**Q: Redis 分布式锁怎么实现？有什么问题？**

> 用 `SET key uuid NX PX 30000` 加锁（NX保证互斥，PX设置过期防死锁），释放时用 Lua 脚本原子执行 GET+DEL（校验 uuid 防误删）。核心问题：①主从切换时锁未同步可能丢锁导致双写；②业务时间超过锁超时时间会导致锁自动释放。Redisson 通过**看门狗**（每10s续期）解决超时问题，通过 Hash 结构 + 重入计数支持可重入。生产推荐 Redisson，高一致性场景用 ZooKeeper。

**Q: Redisson 看门狗的原理？**

> Redisson 加锁时若不指定 leaseTime，默认 30s 过期，同时启动一个定时任务（每10s执行一次），检查当前线程是否仍持锁，若持锁则重置过期时间到 30s。客户端宕机后看门狗随进程消亡，无法续期，30s 后锁自动过期，防止死锁。底层用 Redis Hash 存储锁（field 为 UUID+threadId，value 为重入次数），支持可重入锁。

## 七、常见追问

**Q: 为什么不用 SETNX + EXPIRE 两条命令，而用 SET NX PX？**
> `SETNX` + `EXPIRE` 是两条命令，非原子操作。若 `SETNX` 成功后进程宕机，`EXPIRE` 未执行，锁永不过期 → 死锁。`SET key value NX PX 30000` 是单条原子命令，Redis 2.6.12 后支持，一步完成加锁+设过期，没有中间状态。

**Q: 看门狗续期和业务执行是同一个线程吗？**
> 不是。看门狗是 Redisson 内部的 `TimeTask`（Netty 的 HashedWheelTimer 定时任务），独立运行于业务线程之外。它持有锁的引用，定期向 Redis 发送 Lua 续期脚本。业务线程释放锁后，看门狗自动取消定时任务。

**Q: ZooKeeper 分布式锁的羊群效应是什么？怎么避免？**
> 若所有等待的客户端都监听序号最小的节点，当最小节点释放时，N-1 个客户端同时收到通知，全部醒来竞争，但只有一个能成功，其余重新等待，产生大量无效通知。避免方式：每个客户端只监听**前一个序号**的节点，最小节点释放时只通知序号第二小的节点，形成链式唤醒，每次只唤醒一个。

**Q: 分布式锁配合本地锁有必要吗？**
> 有必要。分布式锁控制跨节点互斥，本地锁控制同节点内多线程互斥。在单节点高并发场景，先用本地锁过滤掉同进程并发，减少对 Redis/ZK 的请求次数，提升性能。典型用法：`synchronized(this)` 内层嵌套分布式锁，即"本地锁+分布式锁双重保护"。
