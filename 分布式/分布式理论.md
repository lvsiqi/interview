## 一、CAP 理论 & BASE 理论 ⭐⭐⭐⭐⭐

### 1.1 CAP 理论

**三个核心属性：**

```
C（Consistency，一致性）：
  任意节点在同一时刻读到的数据都是最新的（强一致）
  写操作完成后，所有节点立即可见最新值

A（Availability，可用性）：
  每个请求都能收到响应（不超时、不报错），但不保证是最新数据

P（Partition Tolerance，分区容错性）：
  网络分区（节点间通信中断）时，系统仍能继续提供服务
```

**核心结论：P 是分布式系统的前提，必须保证，因此只能在 C 和 A 之间二选一。**

```
CP 系统：保证一致性，牺牲可用性
  → 网络分区时拒绝请求（宁可不可用，也不返回脏数据）
  → 代表：ZooKeeper、HBase、etcd
  CP 系统的本质：
  用多台机器，模拟一台永不挂的强一致机器。
  写性能被单台 Leader 限制，
  加机器只能提高可用性和读能力，不能提高写吞吐量。
  这是为了强一致必须付出的代价。

AP 系统：保证可用性，牺牲强一致性（最终一致）
  → 网络分区时继续提供服务，但各节点数据可能暂时不一致
  → 代表：Eureka、Cassandra、DynamoDB、大多数 NoSQL

CA 系统：理论上存在，但在分布式场景中 P 不可避免，CA 只适用于单机
  → 传统关系型数据库（单节点 MySQL）
```

**常见组件归类：**

| 组件 | 类型 | 原因 |
|------|------|------|
| ZooKeeper | CP | Leader 选举期间拒绝写请求，保强一致 |
| Nacos（默认）| AP | 节点故障时仍返回旧数据，保可用 |
| Nacos（raft 模式）| CP | 开启 Raft 后强一致 |
| Redis 主从 | AP | 主从同步异步，主挂时可能丢数据 |
| MySQL 主从 | CP/AP 取决配置 | 半同步复制趋近 CP |
| Eureka | AP | 每个节点独立，不依赖 Leader |
| etcd | CP | Raft 协议，强一致 |

---

### 1.2 BASE 理论

> BASE 是 CAP 中 AP 的延伸，是对强一致性的妥协方案

```
BA（Basically Available，基本可用）：
  允许系统在故障时降级（响应时间延长、部分功能不可用），但核心功能可用
  例：双11大促时关闭评价功能，保商品购买可用

S（Soft State，软状态）：
  允许系统存在中间状态，数据不需要时刻一致
  例：订单状态"支付中"是一个软状态，最终会变为"支付成功"或"支付失败"

E（Eventually Consistent，最终一致性）：
  不保证任何时刻强一致，但数据经过一段时间同步后最终达到一致
  例：主从复制延迟 100ms，最终一致
```

**最终一致性的几种实现方式：**
```
① 读写 quorum（Cassandra，写超半数节点才成功，读超半数节点合并）
② 消息队列异步同步（最常用）
③ 定时对账/补偿任务
④ 向量时钟（分辨数据版本冲突）
```

### 1.3 面试标准答法

**Q: CAP 理论中为什么不能同时满足三者？**

> 假设有节点 A 和 B，发生网络分区（P 必须容忍）：若要保证 C（一致性），A 写入数据后必须等同步到 B 才返回成功，但 B 不可达，则只能返回失败或阻塞 → 牺牲了 A（可用性）；若要保证 A（可用性），A 立即返回成功，但 B 未同步 → 两节点数据不一致 → 牺牲了 C。分区不可避免，所以 C 和 A 只能取一。

## 二、一致性协议：Paxos & Raft ⭐⭐⭐⭐

### 2.1 为什么需要一致性协议

```
分布式系统多节点如何就某个值达成一致？
→ 需要一个算法保证：即使部分节点宕机、消息延迟，多数节点最终对同一个值达成共识
→ 这类算法统称为"共识算法"
```

---

### 2.2 Paxos 协议

**Basic Paxos（单值共识）：**

```
三种角色：
  Proposer（提议者）：发起提案
  Acceptor（接受者）：投票接受/拒绝提案
  Learner（学习者）：学习最终达成的值

两阶段流程：

Phase 1：Prepare（准备阶段）
  Proposer 发送 Prepare(n)，n 为提案编号（必须唯一递增）
  Acceptor 规则：
    若 n > 已承诺的最大提案号 → 回复 Promise(n, 已接受的最大值)，承诺不再接受编号<n的提案
    否则 → 直接拒绝

Phase 2：Accept（接受阶段）
  Proposer 收到多数派（>N/2）的 Promise 后：
    若有 Acceptor 已接受过值 → 必须使用响应中编号最大的那个值（不能自作主张）
    否则 → 使用自己的提案值
  发送 Accept(n, value)
  Acceptor：若 n >= 承诺的编号 → 接受，通知 Learner
  多数派接受 → 共识达成
```

**Paxos 的问题：**
```
① 难以理解和实现（活锁：两个 Proposer 交替递增提案号，导致永远无法达成）
② Basic Paxos 只能对单个值达成共识，实际系统需要日志序列 → Multi-Paxos（更复杂）
③ Leader 选举、日志复制等工程细节 Paxos 都未规定
```

---

### 2.3 Raft 协议（重点）⭐⭐⭐⭐⭐

> Raft 是 Paxos 的工程化替代品，**更易理解，生产中广泛使用**（etcd、TiKV、CockroachDB、Kafka KRaft）
> 核心思想：**先选出一个 Leader，一切以 Leader 为准，Leader 负责管理日志复制，保证所有节点状态一致。**

#### 2.3.1 三种角色与状态转换

```
Leader（领导者）：唯一，负责接收客户端请求、管理日志复制、发送心跳
Follower（跟随者）：被动接受 Leader 指令，响应投票请求，不主动发起任何请求
Candidate（候选人）：选举过程中的临时状态，由 Follower 超时后转变而来
```

**状态转换图：**
```
                    超时，发起选举
          ┌───────────────────────────┐
          │                           ▼
      Follower ◄──── 发现更高 Term ──── Candidate
          ▲                           │
          │                           │ 获得多数派投票
          │    发现更高 Term           ▼
          └─────────────────────── Leader
                                      │
                                      │ 持续发送心跳
                                      └──→ Follower 们保持跟随
```

- **Follower → Candidate**：选举超时未收到 Leader 心跳
- **Candidate → Leader**：获得多数派投票
- **Candidate → Follower**：发现更高 Term 的 Leader 或收到当前 Term 的 Leader 心跳
- **Leader → Follower**：收到更高 Term 的消息（说明自己已过期）

#### 2.3.2 Term（任期）

```
Raft 用 Term（逻辑时钟）划分时间，每个 Term 最多只有一个 Leader

Term 1          Term 2          Term 3          Term 4
┌──────────┐   ┌──────────┐   ┌──┐            ┌──────────┐
│选举 │正常 │   │选举│正常  │   │选举│(无Leader) │选举│正常  │
│    │工作  │   │   │工作   │   │失败│           │   │工作   │
└──────────┘   └──────────┘   └──┘            └──────────┘
```

**Term 的作用：**
```
① 识别过期信息：节点收到 term < 当前term 的消息 → 直接忽略（该消息来自旧 Leader）
② 自动退位：Leader/Candidate 收到 term > 当前term 的消息 → 立即变回 Follower
③ 防止脑裂：旧 Leader 网络恢复后，发现自己 term 过期 → 自动退位，不会出现两个有效 Leader
④ 每次选举 Term+1，相当于一个"朝代号"，用于分辨新旧
```

#### 2.3.3 Leader 选举（详解）

**触发条件：**
```
Follower 有一个选举超时计时器（Election Timeout）
在超时时间内未收到 Leader 的心跳（AppendEntries RPC）→ 认为 Leader 已宕机 → 发起选举
```

**为什么用随机超时（150~300ms）？（字节爱问）**
```
问题：如果所有 Follower 的超时时间相同
  → Leader 宕机后，所有 Follower 同时超时
  → 同时变成 Candidate，同时发起投票
  → 每个 Candidate 都先投自己一票 → 选票分裂（Split Vote）
  → 没人能获得多数派 → 本轮选举失败
  → 全部重新超时 → 再次同时发起 → 再次分裂 → 活锁！

解决：随机化超时时间（150~300ms 范围内随机取值）
  → 大概率只有一个节点最先超时
  → 它最先发起选举并拿到足够选票，快速当选
  → 其他节点超时前就收到了新 Leader 的心跳，不再发起选举
  → 避免了 Split Vote

为什么是 150~300ms？
  ① 下限（150ms）> 网络一次往返时间（通常 < 50ms）：避免网络抖动导致误判
  ② 上限（300ms）不宜太大：否则 Leader 宕机后选举太慢，服务不可用时间长
  ③ 范围要足够大：差值越大，同时超时的概率越低
  实际工程中可以根据网络环境调整（如跨机房部署可放大到 500~1000ms）
```

**完整选举流程：**
```
1. Follower 选举超时 → 转为 Candidate
2. Candidate 操作：
   a) 自增 currentTerm（Term+1）
   b) 投自己一票（voteFor = self）
   c) 重置选举超时计时器（防止自己超时重选干扰流程）
   d) 并行向所有其他节点发送 RequestVote RPC

3. RequestVote 请求包含：
   {term, candidateId, lastLogIndex, lastLogTerm}

4. 接收方处理规则：
   a) 如果请求的 term < 自己的 currentTerm → 直接拒绝
   b) 如果请求的 term > 自己的 currentTerm → 更新自己 term，转为 Follower
   c) 如果本 term 已投过票给别人 → 拒绝（每个 Term 只投一票，先到先得）
   d) 日志新鲜度检查（见下方）→ 不通过则拒绝
   e) 以上全部通过 → 投票，重置自己的选举超时计时器

5. 选举结果（三种情况）：
   a) 收到多数派投票 → 成为 Leader，立即发送心跳（空 AppendEntries）
   b) 收到其他节点的 AppendEntries（说明已有新 Leader）→ 自己转 Follower
   c) 超时仍未决出 → 重新随机超时，开始新一轮选举（Term 再+1）
```

**★ 日志新鲜度比较规则（Election Restriction，字节必问）：**
```
目的：保证新 Leader 一定包含所有已提交的日志，防止选出一个"落后"的 Leader

比较算法：
  先比较 lastLogTerm（候选人最后一条日志的任期号）
    候选人 lastLogTerm > 投票者 → 候选人更新 → 投票 ✅
    候选人 lastLogTerm < 投票者 → 候选人更旧 → 拒绝 ❌
    lastLogTerm 相同 → 再比较 lastLogIndex（日志长度/索引）
      候选人 lastLogIndex >= 投票者 → 投票 ✅
      候选人 lastLogIndex <  投票者 → 拒绝 ❌

为什么先比 Term 再比 Index？
  Term 更大 = 经历了更多轮 Leader，数据一定更新
  同 Term 下 Index 更大 = 收到了同一 Leader 更多的日志
  → 这样比较能保证：已被多数派复制（即已提交）的日志一定在新 Leader 里

举例：
  节点A：日志 [{term:1,idx:1}, {term:1,idx:2}, {term:2,idx:3}]  → lastLogTerm=2, lastLogIndex=3
  节点B：日志 [{term:1,idx:1}, {term:1,idx:2}, {term:1,idx:3}, {term:1,idx:4}] → lastLogTerm=1, lastLogIndex=4
  → A 的 lastLogTerm=2 > B 的 lastLogTerm=1 → A 更新（虽然 B 日志更长，但 A 有更高 Term 的日志）
```

#### 2.3.4 日志复制（详解）

**正常流程：**
```
1. Client 发送写请求 → Leader
2. Leader 将命令追加到本地日志（uncommitted 状态），分配 index 和 term
3. Leader 并行向所有 Follower 发送 AppendEntries RPC：
   {term, leaderId, prevLogIndex, prevLogTerm, entries[], leaderCommit}
4. Follower 收到后：
   a) 一致性检查：本地日志 prevLogIndex 位置的 term == prevLogTerm？
      → 是：追加 entries，回复成功
      → 否：回复失败（日志不一致，需要 Leader 回退重发）
5. Leader 收到多数派成功 → 提交该条目（commitIndex 前进）→ apply 到状态机 → 回复 Client
6. 后续心跳携带 leaderCommit → Follower 得知提交进度 → 也 apply 到自己的状态机
```

**日志不一致怎么处理（Leader 强制覆盖）？**
```
场景：Follower 崩溃恢复、网络分区恢复后，日志可能与 Leader 不一致
  → 可能缺少条目，也可能多出未提交的条目

解决：Leader 为每个 Follower 维护 nextIndex（下次发送的日志位置）
  初始值 = Leader 最后一条日志 index + 1
  
  AppendEntries 失败（一致性检查不通过）→ Leader 将该 Follower 的 nextIndex - 1，重试
  → 逐步回退，直到找到 Leader 和 Follower 日志一致的点
  → 从该点开始，Follower 用 Leader 的日志覆盖自己后续的所有日志

优化：Follower 在拒绝时返回冲突 term 和该 term 的第一条日志 index
  → Leader 可以一次跳过整个冲突 term 的日志，减少回退次数
```

**为什么只有 Leader 提交后才回复 Client？**
```
提交 = 写入多数派节点 → 即使 Leader 宕机，新 Leader 也一定有这条日志
→ 回复 Client "成功" 后，数据不会丢

如果 Leader 在多数派确认前宕机 → Client 超时 → 重试（需要幂等设计）
```

**心跳机制：**
```
Leader 定期（通常 50~150ms，远小于选举超时）发送空 AppendEntries 作为心跳
作用：
  ① 维持 Leader 地位，阻止 Follower 超时发起选举
  ② 携带 leaderCommit，同步提交进度
  ③ 检测 Follower 存活状态
```

#### 2.3.5 安全性保证（为什么不会丢数据/脑裂）

**已提交的日志永远不会丢失（核心安全性）：**
```
证明链路：
  ① 提交 = 多数派已复制该条目
  ② 选举 = 需要多数派投票
  ③ 两个多数派必然有交集（N/2+1 个节点的交集 >= 1）
  ④ 交集节点既有该条目，又会参与投票
  ⑤ 日志新鲜度检查保证：没有该条目的候选人拿不到交集节点的票
  ⑥ 所以新 Leader 一定包含所有已提交的条目 → 不会丢失
```

**为什么 Leader 不能提交旧 Term 的日志？**
```
问题场景（Raft 论文 Figure 8）：
  Leader 复制了旧 Term 的日志到多数派，但在提交前宕机
  新 Leader 可能不包含这条日志 → 如果旧 Leader 已提交，就会导致不一致

解决：Leader 只能通过提交当前 Term 的新日志来间接提交旧日志
  即：当前 Term 的日志被多数派确认时，之前所有日志也一并提交
  → 这样保证了不会出现"已提交但被覆盖"的情况
```

**脑裂（Network Partition）处理：**
```
场景：5节点集群 [A B C | D E]，网络分区，A是旧Leader在左侧

左侧（A B C）：A 仍是 Leader，可以正常读写（多数派 3/5）
右侧（D E）：超时后 D 发起选举，Term+1，但只有2票，无法当选 → 右侧无 Leader，拒绝写入

网络恢复后：
  D 的 Term 高于 A → A 发现更高 Term → 自动退位变 Follower
  集群重新选举（大概率 A/B/C 之一当选，因为日志更新）
  所有节点统一到新 Leader 的日志 → 一致性恢复

结论：分区期间只有包含多数派的一侧能正常工作，少数派一侧只读或不可用 → 不会脑裂
```

#### 2.3.6 日志压缩（Snapshot 快照）

```
问题：日志无限增长 → 磁盘耗尽 + 节点重启需要重放所有日志（太慢）

解决：定期对状态机拍快照（Snapshot），丢弃快照之前的所有日志

快照内容：
  ① 状态机当前状态
  ② 最后一条已包含日志的 index 和 term（lastIncludedIndex, lastIncludedTerm）

Follower 落后太多怎么办？
  Leader 发现 Follower 的 nextIndex 落在已被丢弃的日志范围内
  → 发送 InstallSnapshot RPC，直接传送整个快照
  → Follower 用快照恢复状态，再从快照点之后继续接收日志
```

#### 2.3.7 集群成员变更（Joint Consensus）

```
问题：直接切换集群配置（如从3节点扩到5节点）可能导致某一时刻出现两个多数派

解决（两阶段）：
  ① Leader 提交 C_old+new（联合配置）：此时决策需要旧配置和新配置各自的多数派同时同意
  ② C_old+new 提交后，Leader 提交 C_new（新配置）：正式切换

  →在整个过程中，任何时刻都不会出现两个独立的多数派
  → etcd 的实现简化为每次只增减一个节点（单步变更），避免了联合一致性的复杂度
```

#### 2.3.8 Raft 常见面试追问

**Q: 为什么要随机选举超时？固定超时行不行？**
> 不行。固定超时会导致所有 Follower 同时发起选举，选票分裂（Split Vote），每个人投自己一票拿不到多数派，陷入活锁。随机超时（150~300ms）让大概率只有一个节点最先超时并拿到多数票，快速完成选举。

**Q: Follower 宕机了怎么办？会影响集群吗？**
> 不影响，只要存活节点 >= N/2+1（多数派）。Leader 发送 AppendEntries 超时后会不断重试。Follower 恢复后，Leader 通过 nextIndex 回退机制把缺失的日志补齐。

**Q: Leader 宕机了未提交的日志会怎样？**
> 未被多数派复制的日志可能丢失（被新 Leader 覆盖）。Client 会因为超时没收到回复而重试，所以业务需要幂等设计。已被多数派复制但未提交的日志，新 Leader 通过提交当前 Term 的 no-op 日志间接提交它们。

**Q: Raft 的写性能瓶颈在哪？**
> 所有写请求必须经过 Leader → 单点写入，Leader 网络/磁盘是瓶颈。读请求可以做优化（ReadIndex/LeaseRead 读 Follower），但写无法分摊。这就是 CP 系统的代价：强一致 = Leader 序列化所有写操作。

**Q: Raft 能不能读 Follower？**
> 直接读 Follower 可能读到旧数据（Follower 的 commitIndex 落后于 Leader）。解决方案：
> ① **ReadIndex**：Follower 向 Leader 确认当前 commitIndex，等自己 apply 到该位置后再读 → 强一致但多一次网络往返
> ② **LeaseRead**：Leader 在心跳确认期间（Lease 有效期内）直接本地读，无需确认 → 性能最好，但依赖时钟（时钟漂移可能读到旧数据）

**Raft vs Paxos 对比：**

| 对比 | Paxos | Raft |
|------|-------|------|
| 可理解性 | 难 | 易（设计目标就是可理解性）|
| Leader | 无显式 Leader（Multi-Paxos有）| 强 Leader，集中日志管理 |
| 工程实现 | 复杂，细节未规定 | 明确规定了选举、日志复制、快照、成员变更 |
| 生产使用 | Zab（ZooKeeper变体）| etcd、TiKV、Kafka KRaft |
| 性能 | 低（两轮 RPC）| 正常时仅需一轮 RPC（Leader → 多数派确认）|
| 日志连续性 | 允许空洞（不要求连续）| 强制连续（无空洞），更易理解和实现 |

#### 2.3.9 面试标准答法

**Q: 完整描述 Raft 协议的工作原理？**

> Raft 把共识问题拆成三块：**Leader 选举、日志复制、安全性**。集群启动时所有节点是 Follower，各自启动随机选举超时计时器（150~300ms），最先超时的节点转为 Candidate，自增 Term 并向所有节点发送 `RequestVote`。投票遵循"每个 Term 只投一票"和"候选人日志不能比自己旧"两条规则。获得多数派投票后成为 Leader，立即发心跳占位。之后所有写请求经 Leader 写入本地日志，并行发送 `AppendEntries` 给 Follower，多数派确认后提交并回复 Client。安全性的核心是日志新鲜度检查——保证新 Leader 一定包含所有已提交日志，加上 Term 机制防止脑裂。日志无限增长通过 Snapshot 压缩解决。

### 2.4 ZAB 协议（ZooKeeper 的 Raft 变体）

```
ZAB（ZooKeeper Atomic Broadcast）≈ Raft 的早期工程实现（比 Raft 更早）

核心差异：
  Raft：日志索引（Index + Term）
  ZAB：ZXID（64位，高32位是epoch/任期，低32位是事务ID）选出来的leader不一定是日志最新的

选举完成后 ZAB 有额外的"数据同步阶段"，保证新 Leader 先同步旧数据再开始服务
```

### 3.3 雪花算法（Snowflake）⭐⭐⭐⭐⭐

**64 位结构：**

```
┌─────────────────────────────────────────────────────────────────┐
│  0  │        41位时间戳        │  5位机房  │  5位机器  │  12位序列号  │
└─────────────────────────────────────────────────────────────────┘
  1位    41位（毫秒时间戳差值）    5位         5位          12位
  符号位  约69年                  32个机房    32台机器     4096个/毫秒

总容量：32 × 32 × 4096 = 4,194,304 个ID/毫秒 = 约 42亿/秒
```

**各字段详解：**
```
① 1位符号位：始终为0，保证ID是正整数

② 41位时间戳（毫秒差值）：
   存储的是 当前时间 - 起始纪元（自定义，如2020-01-01）的差值
   支持 2^41 / (1000×3600×24×365) ≈ 69年

③ 10位工作机器ID（5位机房 + 5位机器）：
   支持 1024 台机器，需要部署时配置（从配置中心/Zookeeper获取）

④ 12位序列号：
   同一毫秒内的序列号，从0自增到4095（4096个）
   超出则等待到下一毫秒
```

**时钟回拨问题：**
```
问题：
  服务器 NTP 时间同步导致系统时间回退
  → 当前时间戳 < 上次生成ID的时间戳
  → 可能生成相同序号的ID → 重复！

解决方案：
① 检测回拨：记录上次生成ID的时间戳 lastTimestamp
   若 currentTimestamp < lastTimestamp → 时钟回拨了

② 容忍小范围回拨（< 5ms）：
   等待时钟追上来（自旋等待），回拨范围小则可接受

③ 大范围回拨：报错/告警，拒绝生成（宁可停服不出重复ID）

④ 美团 Leaf-Snowflake 方案：
   → 用 ZooKeeper 持久化节点记录最后一次使用的时间
   → 重启时检查，若ZK记录时间 > 当前时间 → 拒绝启动
   → 运行时维护 1ms 统计，若发生回拨则报警并切换到备用机器ID段
```

---

### 3.4 美团 Leaf 方案

**Leaf-Segment（号段模式）：**
```
DB 维护 biz_tag → max_id, step（步长）

获取ID流程：
  本地维护两个号段（双 buffer）：
    当前号段用完前，后台异步预取下一个号段
    → 避免号段用完时必须同步等待DB

  SQL：UPDATE leaf_alloc SET max_id = max_id + step WHERE biz_tag = ?
       SELECT max_id, step FROM leaf_alloc WHERE biz_tag = ?
  → CAS 更新，并发安全

优点：号段预取，正常情况几乎不需要访问DB；DB支持主从高可用
缺点：号段用完的瞬间（极短时间）访问DB；ID非绝对递增（重启丢号段末尾）
```

**Leaf-Snowflake（雪花模式）：**
```
解决雪花算法时钟回拨问题：
  启动时向 ZooKeeper 注册节点，ZK 分配 workerID（解决多节点ID冲突）
  持久化最后一次使用的时间戳到 ZK
  重启时校验时间，防止时钟回拨
```

### 3.5 雪花算法手写核心代码（字节高频）⭐⭐⭐⭐

```java
public class SnowflakeIdGenerator {
    // 起始纪元（2020-01-01 00:00:00）
    private static final long EPOCH = 1577836800000L;

    private static final long WORKER_ID_BITS  = 10L;  // 机器ID位数
    private static final long SEQUENCE_BITS   = 12L;  // 序列号位数

    private static final long MAX_WORKER_ID   = ~(-1L << WORKER_ID_BITS);  // 1023
    private static final long MAX_SEQUENCE    = ~(-1L << SEQUENCE_BITS);   // 4095

    // 移位偏移量
    private static final long WORKER_SHIFT    = SEQUENCE_BITS;             // 12
    private static final long TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS; // 22

    private final long workerId;
    private long lastTimestamp = -1L;
    private long sequence = 0L;

    public SnowflakeIdGenerator(long workerId) {
        if (workerId > MAX_WORKER_ID || workerId < 0)
            throw new IllegalArgumentException("workerId 超出范围");
        this.workerId = workerId;
    }

    public synchronized long nextId() {
        long now = System.currentTimeMillis();

        // 时钟回拨检测
        if (now < lastTimestamp) {
            long offset = lastTimestamp - now;
            if (offset <= 5) {
                // 小范围回拨（≤5ms）：等待追上
                try { Thread.sleep(offset << 1); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
                now = System.currentTimeMillis();
                if (now < lastTimestamp)
                    throw new RuntimeException("时钟回拨，拒绝生成ID");
            } else {
                throw new RuntimeException("时钟回拨超过5ms，拒绝生成ID");
            }
        }

        if (now == lastTimestamp) {
            // 同一毫秒内，序列号递增
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) {
                // 序列号溢出（同一毫秒超过4096个），等待下一毫秒
                now = waitNextMillis(lastTimestamp);
            }
        } else {
            // 新的毫秒，序列号归零
            sequence = 0L;
        }

        lastTimestamp = now;

        // 位运算组装64位ID
        return ((now - EPOCH) << TIMESTAMP_SHIFT)
             | (workerId << WORKER_SHIFT)
             | sequence;
    }

    private long waitNextMillis(long lastTs) {
        long ts;
        do { ts = System.currentTimeMillis(); } while (ts <= lastTs);
        return ts;
    }
}
```

**关键位运算解析：**
```
(now - EPOCH) << 22  → 时间戳放到高41位
workerId << 12       → 机器ID放到中间10位
sequence             → 序列号放到低12位
按位 OR（|）合并三部分 → 完整64位ID
```

### 3.6 面试标准答法

**Q: 雪花算法原理？时钟回拨问题怎么解决？**

> 雪花算法生成 64 位 long 型 ID：1位符号位 + 41位毫秒时间戳差值（支持约69年）+ 10位机器ID（5位机房+5位机器，支持1024台）+ 12位序列号（同一毫秒内最多4096个），理论吞吐约42亿/秒，纯本地生成，无中心依赖，趋势递增。时钟回拨时会生成重复ID，解决方案：①维护 `lastTimestamp`，检测到回拨时小范围（<5ms）自旋等待追赶；②大范围回拨直接报错拒绝；③美团 Leaf-Snowflake 通过 ZooKeeper 持久化时间戳，重启时做时间校验，运行时回拨则告警并切换备用 workerID 段。
