# 分布式事务知识点

> 最后更新：2026年3月6日

---

## 一、为什么需要分布式事务

```
单机事务：ACID 由数据库保证，一个连接管理全部操作
分布式场景：订单服务写订单库 + 库存服务写库存库 + 账户服务扣余额
  → 三个操作跨三个数据库，单机事务无法覆盖
  → 任意一个失败，其他已成功的操作如何回滚？→ 分布式事务问题
```

## 二、2PC（两阶段提交）⭐⭐⭐⭐

### 2.1 本质

```
2PC 本质是一个「分布式投票协议」：
  所有参与者就"是否提交"达成共识，由协调者收集投票结果做最终裁决
  → 一票否决制：任一参与者 Vote No → 全体回滚
  → 全票通过才提交

数据库层面的实现就是 XA 协议（X/Open DTP 模型）
```

### 2.2 XA 协议（2PC 的数据库标准实现）

```
XA 是由 X/Open 组织定义的分布式事务标准，MySQL 5.7+、PostgreSQL、Oracle 均支持

角色模型（DTP）：
  AP  (Application Program)       → 应用程序
  TM  (Transaction Manager)       → 事务管理器（协调者）
  RM  (Resource Manager)          → 资源管理器（参与者，即各数据库）

MySQL XA 关键命令：
  XA START 'xid'        → 开启 XA 事务分支
  XA END 'xid'          → 结束事务分支
  XA PREPARE 'xid'      → 进入 Prepare 状态（写 redo log，不提交）
  XA COMMIT 'xid'       → 提交
  XA ROLLBACK 'xid'     → 回滚
  XA RECOVER            → 查询处于 Prepare 状态的悬挂事务

⚠️ MySQL XA 的坑：
  - Prepare 后 MySQL 崩溃重启，悬挂事务需 XA RECOVER + 手动处理
  - 5.7 之前 XA Prepare 不写 binlog → 主从不一致（5.7+ 已修复）
  - InnoDB 的 XA 事务不支持 Savepoint
```

### 2.3 流程

```
角色：
  协调者（Coordinator）：事务发起方，负责协调决策
  参与者（Participant）：各个数据库节点

Phase 1：Prepare（准备阶段）
  协调者 → 向所有参与者发送 Prepare 请求
  参与者 → 执行本地事务（写 undo/redo log），但不提交
           → 投票：能提交则回复 Yes，否则回复 No

Phase 2：Commit / Rollback（提交/回滚阶段）
  所有参与者都回复 Yes → 协调者发送 Commit，所有参与者提交
  任意参与者回复 No 或超时 → 协调者发送 Rollback，所有参与者回滚
```

### 2.4 问题

```
① 同步阻塞：
   Prepare 后参与者持有锁等待协调者指令，期间资源一直被占用
   高并发下性能极差（锁持有时间 = 网络RTT + 最慢参与者的响应时间）

② 单点故障：
   协调者宕机 → 参与者永远等待（无法自行决定提交还是回滚）
   → 工程上可通过协调者HA（如 Seata TC 集群 + DB 持久化事务日志）缓解

③ 数据不一致（脑裂）：
   Commit 阶段协调者发了部分 Commit 后宕机
   → 收到的节点已提交，未收到的节点还在等待 → 数据不一致
   → 本质原因：Commit / Rollback 不是原子广播

④ 无超时回滚机制：
   参与者等待期间宕机重启，不知道该提交还是回滚
   → XA RECOVER 可查到悬挂事务，但需人工或脚本干预
```

### 2.5 3PC 的改进（了解）

```
3PC 在 2PC 基础上增加 CanCommit 阶段 + 超时机制：
  Phase 1：CanCommit → 只询问参与者是否有能力提交（不锁资源）
  Phase 2：PreCommit → 等价于 2PC 的 Prepare（锁资源）
  Phase 3：DoCommit  → 提交，参与者等待超时后自动提交（减少阻塞）

解决了：协调者宕机后参与者无限等待的问题
没解决：网络分区下的数据不一致（脑裂仍存在）
结论：3PC 工程上很少用，了解即可
```

## 三、TCC（Try-Confirm-Cancel）⭐⭐⭐⭐⭐

### 3.1 核心思想

```
将分布式操作拆分为三个阶段，业务自己实现补偿逻辑：

Try（预留）：
  检查业务可行性 + 预留资源（冻结而非真正扣减）
  例：库存从 100 冻结 10，变为 可用90 / 冻结10

Confirm（确认）：
  真正执行业务操作（消费预留的资源）
  例：冻结的 10 正式扣减，可用90 / 冻结0
  ⚠️ Confirm 必须保证幂等（可能重试）

Cancel（取消）：
  释放 Try 阶段预留的资源（补偿回滚）
  例：冻结的 10 释放，变回 可用100 / 冻结0
  ⚠️ Cancel 也必须保证幂等
```

### 3.2 TCC 的三个关键问题

```
① 空回滚（Cancel 先于 Try 执行）：
   Try 因网络超时未到达时，框架触发 Cancel
   此时 Cancel 不能真正回滚（因为 Try 根本没执行）
   解决：Try 阶段插入事务记录，Cancel 时检查记录是否存在
         若无记录 → 空回滚，直接返回成功

② 幂等（Confirm/Cancel 可能重试多次）：
   解决：每次操作前检查事务状态（已确认/已取消则直接返回）
         用数据库乐观锁或唯一约束防重

③ 悬挂（Try 比 Cancel 后到）：
   Cancel 已执行完成，被延迟的 Try 才到达
   → Try 执行会锁定资源，但 Cancel 不会再触发 → 资源永久冻结
   解决：Cancel 执行时记录事务ID，Try 阶段检查是否已有 Cancel 记录
         若有则拒绝执行 Try（直接返回失败）
```

### 3.3 具体示例：A 向 B 转账 100 元

```
Try 阶段：
  账户A：余额 1000 → 可用 900 / 冻结 100（检查余额够不够，够就冻结）
  账户B：无操作（或预创建待入账记录）

Confirm 阶段（Try 全部成功）：
  账户A：冻结 100 正式扣减 → 可用 900 / 冻结 0
  账户B：余额 +100

Cancel 阶段（Try 任一失败或超时）：
  账户A：冻结 100 释放 → 可用 1000 / 冻结 0
  账户B：删除待入账记录（如有）

关键：Try 阶段用"冻结字段"而非直接扣减
  → 这样即使 Try 成功后系统崩溃，钱只是冻结状态，不会丢也不会多
```

### 3.4 TCC 事务控制表设计

```
核心：用一张事务记录表统一解决空回滚、幂等、悬挂三个问题

tcc_transaction 表：
  | xid (全局事务ID) | branch_id | status (INIT/CONFIRMED/CANCELLED) | create_time |

  Try 阶段：  INSERT(xid, branch_id, INIT) → 插入成功才执行业务预留
  Confirm：   UPDATE status = CONFIRMED WHERE status = INIT → 更新成功才执行
  Cancel：    UPDATE status = CANCELLED WHERE status = INIT → 更新成功才执行

  空回滚：Cancel 时查到无记录 → INSERT(xid, branch_id, CANCELLED) 后直接返回
  悬挂：  Try 时发现已有 CANCELLED 记录 → 拒绝执行
  幂等：  Confirm/Cancel 时 status 不是 INIT → 说明已处理过，直接返回
```

### 3.5 TCC 适用场景

```
✅ 适合：对一致性要求高、短流程、资源能被预留的场景
   → 金融转账、库存扣减、优惠券核销

❌ 不适合：长流程、不能预留资源的场景（如调用第三方支付）
          业务侵入性强（每个服务都要实现 Try/Confirm/Cancel 三个接口）
```

## 四、Saga 模式 ⭐⭐⭐

### 4.1 核心思想

```
将长事务拆分为一系列本地事务 T1 → T2 → T3 → ... → Tn
每个本地事务对应一个补偿事务 C1, C2, C3, ... Cn

正常流程：T1 → T2 → T3 → ... → Tn（全部成功）
失败回滚：Ti 失败 → 执行 Ci-1 → ... → C2 → C1（反向补偿）

例：购票流程
  T1：创建订单   | C1：取消订单
  T2：扣减库存   | C2：还原库存
  T3：扣减余额   | C3：退款
  T4：出票       | C4：退票
```

### 4.2 两种协调方式

```
① 编排（Choreography）：
   各服务通过事件驱动，上一个服务完成后发布事件，下一个服务监听并执行
   优点：松耦合，无中心协调者
   缺点：流程分散，难以追踪全局状态，调试困难

② 编制（Orchestration）：
   由 Saga 编排器（中心化）按顺序调用各服务，负责补偿
   优点：流程清晰，便于监控（Seata Saga 模式）
   缺点：编排器是中心节点，有单点风险
```

### 4.3 Saga 的隔离性缺失与应对

```
Saga 每步都直接提交本地事务，其他事务可能读到中间状态（脏读）

例：T1扣库存已提交 → T2扣余额失败 → 执行C1回滚库存
    但在 C1 执行前，其他用户可能已读到了「库存已扣」的中间状态

应对策略（不求完美，够用即可）：
  ① 语义锁：在业务表加 status 字段（如 PROCESSING），补偿完成后才改为 DONE
     → 其他业务读到 PROCESSING 状态时知道是中间态，可选择等待或忽略
  
  ② 交换律：设计补偿操作满足交换律（执行顺序不影响最终结果）
     → 适用于加减类操作，如积分增减
  
  ③ 版本号：补偿时带 version 校验，防止覆盖其他事务的更新
  
  ④ 业务层面容忍：很多场景可接受短暂不一致（如订单显示"处理中"）
```

### 4.4 Saga vs TCC

| 对比 | TCC | Saga |
|------|-----|------|
| 隔离性 | 有（Try 预留资源）| 无（直接提交，可能读到中间状态）|
| 业务侵入 | 高（实现三接口）| 中（实现补偿接口）|
| 适用流程 | 短流程 | **长流程**（跨多服务的业务流程）|
| 一致性 | 最终一致 | 最终一致 |
| 典型场景 | 金融扣款 | 旅行预订（机票+酒店+用车）|

## 五、消息最终一致性 ⭐⭐⭐⭐⭐

### 5.1 核心思想

```
用 MQ 解耦事务，保证本地事务 + 消息发送的原子性，最终达到一致

核心问题：如何保证"本地事务提交"和"消息发送"同时成功？
  方案 A：先提交事务再发消息 → 事务成功但发消息失败 → 消息丢失
  方案 B：先发消息再提交事务 → 消息到了但事务失败 → 多余消息
  解法：本地消息表 / 事务消息（RocketMQ）
```

### 5.2 本地消息表（经典实现）

```
① 本地事务中同时写业务数据 + 消息记录（同一DB，同一事务）
   INSERT INTO orders ...
   INSERT INTO mq_outbox (id, topic, payload, status='PENDING') ...
   → COMMIT（两条写操作原子性由本地事务保证）

② 后台定时任务扫描 mq_outbox 表中 status='PENDING' 的记录
   → 投递到 MQ
   → 投递成功 → 更新 status='SENT'

③ 消费方消费消息 → 执行自己的业务逻辑（必须幂等）
   → 消费成功后 ACK

④ 定时任务对超时未发送的消息重试

优点：简单，不依赖特殊MQ
缺点：业务库多一张 outbox 表，定时轮询有延迟
```

### 5.3 CDC（Change Data Capture）方案

```
思路：不用定时轮询 outbox 表，而是监听数据库 binlog 自动捕获变更推送到 MQ

流程：
  ① 业务事务中写业务数据 + outbox 表（与本地消息表相同）
  ② CDC 组件（如 Debezium / Canal）监听 binlog
     → 捕获 outbox 表的 INSERT 事件
     → 自动投递到 MQ（Kafka / RocketMQ）
  ③ 消费方消费并处理（幂等）

优势（对比本地消息表）：
  - 无需定时轮询，近实时（延迟 ms 级）
  - 业务代码不需要写发消息逻辑，解耦更彻底
  - Debezium + Kafka Connect 开箱即用

劣势：
  - 引入 CDC 组件增加运维复杂度
  - binlog 格式需为 ROW 模式（MySQL 默认就是）
  - CDC 组件本身需要保证高可用

适用：中大型项目，已有 CDC 基础设施（如数据同步到 ES 也在用 Canal）
```

### 5.4 RocketMQ 事务消息（推荐）⭐⭐⭐⭐⭐

```
RocketMQ 原生支持事务消息，流程：

① 生产者发送 Half Message（半消息）到 Broker
   → Broker 存储但对消费者不可见

② Broker 返回发送成功

③ 生产者执行本地事务（扣款、下单等）

④ 根据本地事务结果：
   成功 → 发送 Commit → Broker 将消息对消费者可见
   失败 → 发送 Rollback → Broker 删除半消息

⑤ 若步骤④超时（生产者宕机）：
   Broker 定期回查（Transaction Check）：
   → 调用生产者的 checkLocalTransaction() 接口
   → 查询本地事务执行结果，再决定 Commit 还是 Rollback
   默认回查 15 次，超过则丢弃（可配置）

消费方：必须保证幂等（消息可能重复投递）
```

**流程图：**
```
Producer                     Broker                    Consumer
   |                            |                          |
   |---① Half Message---------->|                          |
   |<--② ACK(发送成功)-----------|                          |
   |                            |                          |
   |---执行本地事务               |                          |
   |                            |                          |
   |---④ Commit/Rollback------->|                          |
   |                            |---消息对Consumer可见----->|
   |                            |                          |
   |<--⑤ 超时回查(checkLocal)---|                          |
```

## 六、四种方案对比 ⭐⭐⭐⭐⭐

| 对比 | 2PC | TCC | Saga | 消息最终一致 |
|------|-----|-----|------|------------|
| **一致性** | 强一致（趋近）| 最终一致 | 最终一致 | 最终一致 |
| **隔离性** | 有（锁资源）| 有（预留）| 无 | 无 |
| **性能** | 差（同步阻塞）| 中 | 中 | 好（异步）|
| **可用性** | 低（协调者单点）| 高 | 高 | 高 |
| **业务侵入** | 无 | 高 | 中 | 低 |
| **实现复杂度** | 低 | 高 | 中 | 中 |
| **适用场景** | 跨库短事务 | 金融强一致短流程 | 长业务流程 | 异步解耦场景 |
| **框架支持** | Atomikos、SeataAT | Seata TCC | Seata Saga | RocketMQ事务消息 |

**选型建议：**
```
跨库（同一系统内多数据源）短事务 → Seata AT 模式（基于undolog，无侵入）
强一致金融场景（扣款/转账）      → TCC
长流程业务（订单全链路）          → Saga
服务间异步通知（下单通知库存）    → 消息最终一致性（RocketMQ事务消息）
```

## 七、Seata AT 模式（补充）⭐⭐⭐

### 7.1 核心机制

```
Seata AT 是 2PC 的优化版，自动生成补偿 SQL，对业务零侵入

角色：
  TC  (Transaction Coordinator)  → Seata Server，管理全局事务和分支事务
  TM  (Transaction Manager)      → 发起方（标注 @GlobalTransactional 的方法）
  RM  (Resource Manager)         → 参与方（各微服务的数据源代理）

一阶段：
  ① 拦截业务 SQL，查询 before image（修改前的数据快照）
  ② 执行业务 SQL
  ③ 查询 after image（修改后的数据快照）
  ④ 将 before/after image 写入 undo_log 表
  ⑤ 向 TC 注册分支事务，申请全局锁
  ⑥ 本地事务直接提交（释放DB锁！这是与标准2PC的最大区别）

二阶段（Commit）：
  → TC 通知各 RM 异步删除 undo_log，无需等待，性能好

二阶段（Rollback）：
  → TC 通知各 RM 根据 undo_log 的 before image 生成反向 SQL
  → 执行反向 SQL 前校验 after image 与当前数据是否一致（脏写检测）
  → 一致则回滚成功，不一致则需人工介入
```

### 7.2 全局锁与写隔离

```
核心问题：一阶段本地事务已提交（DB锁释放），如何防止其他事务脏写？

答案：Seata 引入 GlobalLock（全局锁），由 TC 统一管理

写隔离流程：
  事务A（Seata全局事务）：
    ① 执行 SQL，申请 GlobalLock → 获取成功
    ② 本地提交，DB锁释放
    ③ 此时 GlobalLock 仍被A持有

  事务B（也是 Seata 全局事务或标注 @GlobalLock）：
    ① 执行同一行 SQL，申请 GlobalLock → 被A持有，等待重试
    ② A 完成全局事务，释放 GlobalLock → B 获取成功，继续执行

  事务C（普通本地事务，绕过 Seata）：
    ① 直接写入成功（不走 GlobalLock）→ 脏写发生！
    ② A 回滚时发现 after image 不匹配 → 抛异常，需人工介入

⚠️ 核心结论：涉及分布式事务的表，所有写入都必须走 Seata 代理，不能有漏网之鱼
```

### 7.3 Seata AT vs XA 对比

| 对比 | Seata AT | XA |
|------|----------|-----|
| **锁粒度** | 一阶段提交后释放DB锁，持有全局锁 | Prepare后一直持有DB锁直到二阶段 |
| **性能** | 高（DB锁持有时间短） | 低（DB锁持有时间长） |
| **隔离性** | 依赖全局锁，有脏写风险 | 数据库级别保证，安全性更高 |
| **侵入性** | 无侵入（加注解即可） | 无侵入（数据库原生支持） |
| **数据库支持** | 需要 undo_log 表 | 需要数据库支持 XA 协议 |
| **适用场景** | 高并发场景，可接受极低概率脏写人工介入 | 强一致性场景，对性能要求不高 |

## 八、生产落地要点 ⭐⭐⭐

```
① 幂等是一切分布式事务的基石
   不管哪种方案（TCC/Saga/消息），下游操作都必须幂等
   常见手段：唯一约束、状态机校验、去重表、Redis SETNX

② 对账补偿是最后一道防线
   任何方案都不能100%保证一致性（网络分区、极端故障）
   → 定时对账任务（如 T+1 日终对账）校验上下游数据
   → 发现不一致 → 自动补偿或报警人工介入

③ 超时处理不能忽视
   TCC 的 Try 超时：框架自动触发 Cancel（要处理空回滚）
   Saga 的某步超时：编排器按超时策略（重试 / 补偿 / 挂起人工处理）
   消息投递超时：Broker 回查机制 + 对账兜底

④ 监控与告警
   - 全局事务成功率、平均耗时、超时率
   - 分支事务失败明细（哪个服务、哪个方法）
   - undo_log / outbox 表积压数量
   - Seata TC 节点健康状态

⑤ 避免分布式事务的最佳实践
   能用单库事务就不用分布式事务
   → 合理划分服务边界，减少跨服务写操作
   → 用事件驱动解耦（下单成功发事件 → 通知积分/物流/推荐）
   → 用最终一致性替代强一致性（大部分业务可以接受）
```

## 九、面试标准答法

**Q: 分布式事务有哪些方案？你们项目用的哪种？**

> 主流方案有四种：①**2PC** 强一致但性能差、协调者单点，生产少用；②**TCC** 业务侵入高，适合金融短流程强一致场景（扣款/转账），需要实现 Try/Confirm/Cancel 三接口并处理空回滚、幂等、悬挂三个关键问题；③**Saga** 适合跨多服务的长流程，无隔离性，靠补偿实现最终一致；④**消息最终一致性**（如 RocketMQ 事务消息）性能最好，通过半消息+本地事务+Broker回查保证原子性，适合异步解耦场景。项目中优先考虑 RocketMQ 事务消息（异步场景）或 Seata AT（同步场景无侵入）。

## 十、常见追问

**Q: TCC 的空回滚、悬挂、幂等如何解决？**
> **空回滚**：Try 未到达时 Cancel 先执行，记要在 Cancel 时查事务记录，无记录则直接返回成功（空补偿）。**悬挂**：Cancel 执行后 Try 才到达，Try 阶段查是否有 Cancel 记录，有则直接拒绝（不执行 Try）。**幂等**：Confirm/Cancel 执行前查事务状态，已处理则直接返回，或通过数据库唯一约束防重。三个问题需统一由一张事务记录表来支撑。

**Q: RocketMQ 事务消息回查次数超了怎么办？**
> 默认回查 15 次（间隔 60s 可配置），超过后 Broker 将消息丢弃（相当于 Rollback）。这意味着本地事务已执行成功但消息被丢弃 → 下游服务未被通知。需要通过**对账补偿任务**兜底：定期扫描业务表中状态异常的记录（已下单但下游未处理），手动补发消息或人工干预。

**Q: Seata AT 的脏写问题如何处理？**
> Seata AT 一阶段提交后释放本地锁，但全局事务锁（GlobalLock）仍持有。其他事务写同一行前若未申请 GlobalLock 直接写入（绕过 Seata），会导致 rollback 时 after image 对比不一致 → Seata 检测到脏写后不敢自动回滚，会抛出异常并告警，**需要人工介入**处理数据。所以 Seata AT 场景下，所有涉及相关表的写操作都应该走 Seata 事务，不能有"漏网之鱼"。

**Q: 最终一致性和强一致性如何选择？**
> 看业务容忍度：**金融核心操作**（扣款、转账）必须强一致（TCC），中间状态会导致资损，宁可牺牲性能和可用性；**非核心异步操作**（通知下游服务、积分发放、消息推送）可以接受短暂不一致，用消息最终一致性，性能和解耦收益更大。字节等大厂实践：核心链路 TCC，非核心链路消息驱动。

**Q: 本地消息表 vs RocketMQ 事务消息 vs CDC，怎么选？**
> **本地消息表**最简单，适合小团队、不想引入额外组件的场景，代价是多一张表和定时任务轮询延迟。**RocketMQ 事务消息**是最推荐的方案，Broker 原生支持半消息 + 回查，无需额外表和定时任务，但强依赖 RocketMQ。**CDC**（Canal/Debezium）实时性最好且业务代码最解耦，但需要维护 CDC 组件的高可用，适合已有 CDC 基础设施的中大型项目。

**Q: 分布式事务嵌套怎么办？（一个全局事务中调用另一个全局事务的服务）**
> Seata 支持事务传播：内层服务检测到已有全局事务 XID 时，会加入该事务（作为分支事务），而非新建全局事务。类似于 Spring `PROPAGATION_REQUIRED`。如果内层服务不希望加入外层事务，需要手动清除 XID 上下文或用 `@GlobalTransactional(propagation = NOT_SUPPORTED)` 隔离。

**Q: 如果不用任何框架，最简单的分布式事务怎么做？**
> **本地消息表 + 定时重试**。在本地事务中同时写业务数据和消息记录，后台任务扫描未投递的消息发送到MQ，下游消费时保证幂等。没有任何外部依赖，缺点是有轮询延迟且需要每个库加 outbox 表。这是最朴素但最可靠的方案。
