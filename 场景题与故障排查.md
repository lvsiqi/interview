# 场景题 & 故障排查

> 字节面试偏好：给现象 → 问思路 → 追问命令 → 问根因 → 问如何避免
> 核心考察：系统化排查思维 + 对底层原理的理解深度

---

## 一、线上 CPU 飙高如何排查

### 1. 排查全流程（黄金步骤）

```
现象：告警/监控显示 CPU 使用率持续 > 90%

Step 1: 确认是哪个进程占用 CPU 高
    top -c
    → 按 P 键（按 CPU 排序），找到 CPU 占用最高的进程 PID

Step 2: 找到该进程中 CPU 最高的线程
    top -H -p <PID>
    → 找到 CPU 占用最高的线程 TID（十进制）

Step 3: 将线程 ID 转成十六进制
    printf '%x\n' <TID>
    → 例如 TID=12345 → 0x3039

Step 4: 打印线程堆栈，定位热点代码
    jstack <PID> > /tmp/jstack.log
    grep -A 30 '0x3039' /tmp/jstack.log
    → 找到对应线程的栈帧

Step 5: 分析栈帧，判断根因
    → 死循环、频繁 GC、正则回溯、加解密计算...
```

### 2. 常见根因 & 对应栈帧特征

#### 根因一：死循环

```
栈帧特征：同一个方法反复出现，如：
  at com.example.OrderService.processOrder(OrderService.java:156)
  at com.example.OrderService.processOrder(OrderService.java:156)
  ...

常见场景：
  // 链表有环导致无限循环
  while (node.next != null) { node = node.next; }
  
  // HashMap 多线程 put 导致链表成环（JDK 7）
  // 避免：使用 ConcurrentHashMap
  
  // 自旋锁长时间 busy-wait
  while (!lock.tryLock()) { /* 空循环 */ }
```

#### 根因二：频繁 Full GC（GC 线程占满 CPU）

```
栈帧特征：
  at java.lang.ref.ReferenceQueue.remove(ReferenceQueue.java:144)
  at java.lang.ref.Finalizer$FinalizerThread.run(Finalizer.java:209)
  
或者 top 看到多个 java 进程 CPU 轮流高：GC 线程持续运行

排查：
  jstat -gcutil <PID> 1000    # 每1s看GC情况
  # 重点看 FGC（Full GC次数）和 FGCT（Full GC耗时）
  # 若 FGC 每分钟超过1次，说明堆内存严重不足

根因：内存泄漏 / 堆内存配置太小 / 大对象频繁创建
```

#### 根因三：正则表达式回溯（ReDoS）

```java
// 危险正则：含有嵌套量词，如 (a+)+ , (a|a)* 
Pattern p = Pattern.compile("(\\d+)+");
p.matcher("aaaaaaaaaaaaaaaaaaaaaaaaa!").matches(); // 指数级回溯！

栈帧特征：
  at java.util.regex.Pattern$GroupTail.match(Pattern.java:4717)
  at java.util.regex.Pattern$BranchConn.match(Pattern.java:4568)
  ...（大量重复 regex 相关栈帧）

解决：
  ① 重写正则，消除嵌套量词或歧义性分支
  ② 正则设置超时：Pattern.compile(regex, flags) + Matcher 加时间判断
  ③ 使用 org.jregex 或 RE2J（线性时间复杂度）
```

#### 根因四：序列化/加解密计算密集

```
栈帧特征：
  at com.fasterxml.jackson.core.JsonParser.nextToken(...)
  at sun.security.rsa.RSACore.modexp(...)

解决：
  ① JSON 序列化选高性能库（Jackson > Gson > Fastjson（反序列化漏洞慎用））
  ② 加密操作添加缓存，避免重复计算
  ③ 异步化非关键路径的 CPU 密集操作
```

### 3. JVM 内置工具快速定位

```bash
# 1. 生成线程快照
jstack -l <PID> > /tmp/thread_dump.log

# 2. 检测死锁（jstack 自动分析）
jstack <PID> | grep -A 20 "deadlock"

# 3. 线程状态统计（快速判断大量线程在做什么）
jstack <PID> | grep 'java.lang.Thread.State' | sort | uniq -c | sort -rn
# 输出示例：
#  800 WAITING (on object monitor)   → 大量线程在等锁
#   50 RUNNABLE                      → 正在运行中
#    5 BLOCKED                       → 被锁阻塞

# 4. Arthas（字节推荐，生产利器）
java -jar arthas-boot.jar <PID>
> thread -n 3          # 展示 CPU 占用前3的线程及堆栈
> thread -b            # 找出阻塞其他线程的线程（死锁分析）
> trace com.example.OrderService processOrder  # 方法链路耗时
> watch com.example.Service method '{params,returnObj}' -x 3  # 方法入参出参监控
```

### 4. 面试标准答法

> CPU 飙高的排查步骤是固定的四步：① `top -c` 找进程 PID；② `top -H -p PID` 找线程 TID；③ `printf '%x' TID` 转十六进制；④ `jstack PID | grep 十六进制` 找到线程堆栈。然后根据堆栈分析根因：死循环（同一代码行反复出现）、频繁 GC（GC 线程持续运行，配合 `jstat -gcutil` 确认）、正则回溯（大量 regex 相关栈帧）、序列化计算（Jackson/RSA 栈帧）。生产环境还会用 Arthas 的 `thread -n` 命令，相比手动 jstack 更高效。

### 5. 常见追问

**Q: CPU 飙高和内存 OOM 有什么关联？**
> Full GC 是两者的交汇点：当堆内存快耗尽时，JVM 会频繁执行 Full GC 试图回收对象，GC 线程会占用大量 CPU（有时超过 90%）。现象上看起来像 CPU 飙高，但根因是内存压力过大。`jstat -gcutil PID 1000` 如果 FGC 列一直在增长，说明是 GC 导致 CPU 高，需要从内存方向排查（堆快照 + MAT 分析内存泄漏）。

**Q: 如何在不重启的情况下动态修改线上代码逻辑排查问题？**
> 使用 Arthas 的热更新能力：`retransform` 命令可以在不重启 JVM 的情况下替换类字节码。或者用 `watch` 监控方法入参出参、`trace` 分析方法链路耗时、`ognl` 执行任意表达式。注意热更新有限制：不能增删字段/方法，只能修改方法体。

## 二、内存溢出如何排查

### 1. OOM 类型识别

```
常见 OOM 错误信息：

① java.lang.OutOfMemoryError: Java heap space
  → 堆内存不足，对象太多或有内存泄漏
  → 排查方向：堆快照 + MAT 分析

② java.lang.OutOfMemoryError: GC overhead limit exceeded
  → GC 占用超过 98% 时间，只回收了不到 2% 的内存
  → 本质也是堆内存泄漏/不足，比上面更严重

③ java.lang.OutOfMemoryError: Metaspace
  → 方法区（元空间）不足，类太多（动态代理/CGLib/JSP）
  → 排查：jmap -clstats 查看类数量；增大 -XX:MaxMetaspaceSize

④ java.lang.OutOfMemoryError: unable to create native thread
  → 线程数超过 OS 限制或物理内存不足
  → 排查：ps -eLf | grep java | wc -l 查线程数
  → 线程池设置不合理 / 线程泄漏（没有 shutdown）

⑤ java.lang.StackOverflowError
  → 非 OOM，递归太深导致栈溢出
  → 排查：jstack 看递归调用链
```

### 2. 堆内存 OOM 排查流程

```
Step 1: 先看 GC 日志 / 监控图
  # JVM 启动参数加上（提前配置好）：
  -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/var/log/gc.log
  -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof
  
  → OOM 时自动生成堆快照

Step 2: 没有提前配置，则手动生成堆快照
  jmap -dump:format=b,live,file=/tmp/heap.hprof <PID>
  # live: 只 dump 存活对象（先触发 Full GC，减小文件大小）
  # 注意：dump 操作会 STW，生产环境选业务低峰期

Step 3: 用 MAT（Eclipse Memory Analyzer）分析
  打开 .hprof 文件 → Leak Suspects Report
  关注：
  ① 占用最多内存的对象（Dominator Tree 视图）
  ② 内存泄漏嫌疑（Leak Suspects 自动分析）
  ③ 对象引用链（找到 GC Root → 为什么不能被回收）

Step 4: 结合代码定位根因
  常见根因见下节
```

### 3. 常见根因 & 代码示例

#### 根因一：静态集合无限增长（最常见）

```java
// 错误：static Map 作为缓存，只放不删
public class UserCache {
    private static final Map<Long, User> cache = new HashMap<>();
    
    public User get(long userId) {
        return cache.computeIfAbsent(userId, id -> loadFromDB(id));
        // 问题：随着用户增多，cache 无限膨胀，且 static → GC Root，永远不被回收
    }
}

// 修复：使用有界缓存（Caffeine / Guava Cache 设置 maximumSize + expireAfterWrite）
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(30))
    .build(userId -> loadFromDB(userId));
```

#### 根因二：ThreadLocal 内存泄漏

```java
// 错误：线程池场景下，ThreadLocal 用完不 remove
ExecutorService pool = Executors.newFixedThreadPool(10);
ThreadLocal<LargeObject> tl = new ThreadLocal<>();

pool.submit(() -> {
    tl.set(new LargeObject());  // 设置到线程的 ThreadLocalMap
    // ... 业务逻辑
    // ❌ 忘记 tl.remove()
    // 线程池的线程不销毁，ThreadLocalMap 里的 Value 永远不会被 GC
});

// 修复：
try {
    tl.set(new LargeObject());
    // 业务逻辑
} finally {
    tl.remove();  // ✅ 必须在 finally 里 remove
}
```

#### 根因三：监听器 / 回调未注销

```java
// 错误：注册了监听器但没有注销，持有外部对象引用
eventBus.register(this);  // this 里有大对象引用
// ... 对象生命周期结束，但 eventBus 还持有 this 的引用 → 无法 GC

// 修复：在对象销毁时注销
@PreDestroy
public void destroy() {
    eventBus.unregister(this);
}
```

#### 根因四：大文件/流未关闭

```java
// 错误：InputStream 未 close，占用大量内存和文件句柄
InputStream is = new FileInputStream("large_file.log");
byte[] content = is.readAllBytes();  // 整个文件读进内存！
// 忘记 is.close()

// 修复：try-with-resources + 流式读取（不一次性加载）
try (BufferedReader reader = new BufferedReader(new FileReader("large_file.log"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        // 逐行处理，内存占用固定
    }
}
```

### 4. MAT 分析要点

```
MAT 主要使用三个视图：

1. Dominator Tree（支配树）
   → 按内存占用倒序排列
   → 看最大的几个对象是什么，所属哪个包（业务代码还是框架）
   → 典型问题：java.util.HashMap$Entry[] 很大 → 静态 Map 泄漏

2. Leak Suspects（泄漏嫌疑报告）
   → MAT 自动分析，给出"Problem Suspect"
   → 通常直接指向问题类和引用链

3. OQL（对象查询语言）
   → 可以查询特定类的实例数量和内存占用
   → 如：SELECT * FROM java.util.HashMap WHERE size > 10000

关键指标：
  Shallow Heap：对象本身占用的内存（不含引用对象）
  Retained Heap：如果该对象被 GC，能释放的总内存（含所有引用对象）
  → Retained Heap 大的对象才是真正的"大对象"
```

### 5. 面试标准答法

> 排查内存 OOM 分三步：① 先确认 OOM 类型（heap space / Metaspace / unable to create thread），不同类型根因不同；② 通过 `jmap -dump:live,format=b,file=heap.hprof PID` 生成堆快照（提前配置 `-XX:+HeapDumpOnOutOfMemoryError` 自动触发）；③ 用 MAT 打开快照，重点看 Dominator Tree（最大对象）和 Leak Suspects Report（自动分析泄漏路径）。常见根因：static 集合无限增长、ThreadLocal 用完不 remove、监听器未注销、流未关闭。修复思路：有界缓存替换无界缓存、finally 块 remove ThreadLocal、对象销毁时注销监听器。

### 6. 常见追问

**Q: jmap -dump 对线上服务有影响吗？**
> 有。`jmap -dump` 会触发 Stop-The-World（全局暂停），对于大堆（如 16GB）STW 可能持续数十秒，生产流量会中断。3 个应对策略：① **提前配置**自动 dump（`-XX:+HeapDumpOnOutOfMemoryError`），只在真正 OOM 时触发一次；② **低峰期**手动执行，先切流（从负载均衡摘掉节点）再 dump；③ 使用 **JFR（Java Flight Recorder）**持续采集内存分配，无 STW 开销（JDK 11+）。

**Q: 元空间（Metaspace）OOM 常见场景是什么？**
> Metaspace 存的是类元数据（类名、方法、字节码等）。OOM 通常由以下场景引起：① **动态代理/CGLib 过度使用**（Spring AOP、Dubbo 服务注册每个接口都生成代理类）；② **JSP 热部署**（Tomcat/JBoss 每次重新部署会生成新的类加载器，旧的类无法卸载）；③ **Groovy 脚本动态编译**（每次编译产生新类）。排查：`jmap -clstats PID | sort -k3 -rn | head 20` 按类实例数排序，看哪个类加载器加载了大量类。

## 三、接口超时如何排查

### 1. 超时问题定位框架

```
接口超时 = 某一环节耗时过长

完整链路：
  用户 → LB → Gateway → 服务A → 服务B → 数据库/缓存/MQ

排查思路：二分法缩小范围
  ① 看监控/日志：是哪个服务返回慢？是偶发还是持续？
  ② 定位到具体服务后：是代码逻辑慢？外部依赖慢？资源等待？
  ③ 找到具体方法后：是 CPU 密集？IO 等待？锁竞争？
```

### 2. 分层排查

#### 第一层：网络层

```bash
# 1. 检查网络质量
ping <目标IP> -c 100           # 看丢包率和延迟
traceroute <目标IP>            # 看网络路径，找延迟高的跳

# 2. 检查 TCP 连接状态
ss -ant | grep -E 'ESTABLISHED|CLOSE_WAIT|TIME_WAIT' | wc -l
# CLOSE_WAIT 多 → 服务端 socket 没关闭，连接池耗尽
# TIME_WAIT 大量 → 频繁短连接，正常现象（可用 keep-alive 优化）

# 3. 连接池是否耗尽（线程在等待连接）
jstack <PID> | grep -A 10 "waiting to lock\|TIMED_WAITING\|HikariPool"
# 看是否大量线程在等待数据库/Redis 连接

# 4. 抓包分析具体请求的时延分布
tcpdump -i eth0 port 8080 -w /tmp/cap.pcap
# Wireshark 打开，过滤特定请求，看各阶段时间戳
```

#### 第二层：应用层（代码层）

```bash
# Arthas trace 追踪方法链路耗时（生产最有效）
> trace com.example.OrderController placeOrder '#cost > 100'
# 输出示例：
# ---[1523ms] com.example.OrderController.placeOrder()
#     +---[5ms] com.example.OrderService.validate()
#     +---[1510ms] com.example.OrderService.saveOrder()    ← 这里慢！
#         +---[1500ms] com.example.OrderMapper.insert()   ← SQL 慢！

# 查看实时方法调用统计
> monitor com.example.OrderService placeOrder -c 10
# 每10秒统计：调用次数、成功次数、失败次数、平均耗时、最大耗时
```

#### 第三层：数据库层

```sql
-- MySQL：查看当前正在执行的慢 SQL
SHOW PROCESSLIST;
-- 或
SELECT * FROM information_schema.INNODB_TRX;       -- 正在执行的事务
SELECT * FROM information_schema.INNODB_LOCKS;      -- 当前锁情况
SELECT * FROM information_schema.INNODB_LOCK_WAITS; -- 锁等待关系

-- 慢查询日志（提前开启）
SHOW VARIABLES LIKE 'slow_query_log%';
SHOW VARIABLES LIKE 'long_query_time';
-- 开启：SET GLOBAL slow_query_log = ON; SET GLOBAL long_query_time = 1;

-- 分析慢查询日志
-- mysqldumpslow -s t -t 10 /var/log/mysql/slow.log
-- 按总耗时排序，取前10条
```

#### 第四层：外部依赖（Redis/HTTP）

```bash
# Redis 慢查询日志
redis-cli SLOWLOG GET 10     # 最近10条慢命令
redis-cli SLOWLOG RESET      # 清空

# Redis 延迟监控
redis-cli --latency -h <host>   # 实时延迟监控
redis-cli --latency-history     # 历史延迟分布

# 检查 Redis 大 key（全量扫描，生产慎用）
redis-cli --bigkeys

# HTTP 下游服务超时排查
curl -o /dev/null -s -w "DNS: %{time_namelookup}s\n连接: %{time_connect}s\n首字节: %{time_starttransfer}s\n总时间: %{time_total}s\n" http://api/endpoint
```

### 3. 常见超时根因 & 解决方案

| 根因 | 现象 | 解决方案 |
|---|---|---|
| 慢 SQL（无索引/全表扫描）| DB 耗时高，`SHOW PROCESSLIST` 看到长事务 | EXPLAIN 分析，加索引，优化 SQL |
| 数据库连接池耗尽 | 线程大量等待连接，报 connection timeout | 增大连接池 `maximumPoolSize`，检查是否连接泄漏 |
| Redis 大 key 阻塞 | Redis 命令偶发耗时超 100ms | 拆分大 key，用 SCAN 替代 KEYS * |
| 下游服务超时 | 调用链路中某个下游一直慢 | 熔断降级（Sentinel），设置合理超时+重试 |
| 线程池队列积压 | 线程池任务队列满，新任务等待 | 调整线程池参数，或异步化非核心逻辑 |
| GC 暂停（STW）| 周期性超时，与 GC 日志时间吻合 | 优化内存使用，调整 GC 参数，或升级 ZGC |
| 热点锁竞争（synchronized）| jstack 大量 BLOCKED 线程 | 缩小锁粒度，ConcurrentHashMap，无锁化 |
| 网络丢包/延迟 | ping 丢包率高，traceroute 某跳延迟大 | 联系运维排查网络，检查带宽打满 |

### 4. 接口超时完整排查 SOP

```
排查步骤（从上到下，逐步缩小范围）：

1. 查告警时间 vs 代码部署时间
   → 是否恰好部署后开始超时（新代码引入）

2. 查监控：Grafana/Prometheus
   → CPU/内存/GC 是否异常？
   → DB 连接数/Redis 连接数是否打满？
   → JVM Heap 使用率是否飙高？

3. 查应用日志：grep ERROR/Exception
   → 是否有 Timeout / ConnectionRefused / SocketException

4. Arthas trace 定位具体慢方法
   → 找到耗时最长的一层

5. 根据慢的层次针对性排查：
   → 代码逻辑：算法复杂度？不必要的串行改并行？
   → SQL 慢：EXPLAIN 看执行计划，加索引/改写 SQL
   → Redis 慢：slowlog 看大 key / 热 key / 网络抖动
   → 下游超时：设置 fallback，增加熔断

6. 复现问题：用 Arthas 的 watch 命令抓慢请求的入参
   → 找到触发慢查询的具体数据特征（如特定 userId 的数据量很大）
```

### 5. 面试标准答法

> 接口超时排查核心是**分层缩小范围**：先看监控确认是哪层慢（网络/应用/DB/下游），再用 Arthas `trace` 命令追踪方法链路找到具体慢调用，最后针对根因解决。常见根因：① SQL 无索引（EXPLAIN 确认，补索引）；② 连接池耗尽（线程 jstack 看到大量等待连接，调大 maximumPoolSize）；③ Redis 大 key 阻塞（slowlog 确认，拆分 key）；④ GC STW 导致周期性超时（GC 日志时间与超时时间吻合，调整堆大小或换 ZGC）；⑤ 下游超时（设置熔断降级，避免雪崩）。

## 四、数据库慢查询优化

### 1. 慢查询排查入口

```sql
-- 1. 开启慢查询日志（5.7+ 动态开启，不需要重启）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;      -- 超过1秒记录
SET GLOBAL log_queries_not_using_indexes = ON;  -- 未走索引也记录

-- 2. 查看慢查询文件位置
SHOW VARIABLES LIKE 'slow_query_log_file';

-- 3. 分析工具（命令行）
-- mysqldumpslow -s t -t 20 /var/log/slow.log   # 按总耗时排序，前20条
-- 推荐：pt-query-digest（Percona Toolkit，更详细）
-- pt-query-digest /var/log/slow.log | head -100

-- 4. 查看当前执行中的慢请求
SELECT * FROM information_schema.PROCESSLIST
WHERE COMMAND != 'Sleep' AND TIME > 1
ORDER BY TIME DESC;
```

### 2. EXPLAIN 解读（核心工具）

```sql
EXPLAIN SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 1 AND o.created_at > '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 100;
```

| 字段 | 关注点 | 危险值 |
|---|---|---|
| **type** | 访问类型 | `ALL`（全表扫描）、`index`（索引全扫）→ 必须优化 |
| **key** | 实际用到的索引 | `NULL` → 没走索引 |
| **rows** | 预估扫描行数 | 值越大越差，百万级需优化 |
| **Extra** | 附加信息 | `Using filesort`（额外排序）、`Using temporary`（临时表）→ 优化 |
| **possible_keys** | 可能用的索引 | 有但 key=NULL → 统计信息不准，可 ANALYZE TABLE |

```
type 性能从好到差：
system > const > eq_ref > ref > range > index > ALL

const：主键/唯一索引 等值查询，最快
ref：非唯一索引等值查询
range：范围查询（>, <, BETWEEN, IN）
ALL：全表扫描，生产禁止出现（大表）
```

### 3. 索引优化策略

#### 策略一：联合索引最左前缀

```sql
-- 建立联合索引  (status, created_at, user_id)
CREATE INDEX idx_order_query ON orders(status, created_at, user_id);

-- ✅ 可走索引（满足最左前缀）
WHERE status = 1 AND created_at > '2024-01-01'

-- ✅ 覆盖索引（所需字段全在索引中，无需回表）
SELECT user_id FROM orders WHERE status = 1 AND created_at > '2024-01-01'

-- ❌ 不走索引（跳过了 status 列）
WHERE created_at > '2024-01-01' AND user_id = 100

-- ❌ 索引失效：对索引列做函数/运算
WHERE YEAR(created_at) = 2024     -- ❌ 函数导致失效
WHERE created_at + 1 > '2024-01'  -- ❌ 运算导致失效
WHERE status != 1                  -- != 在低选择率字段可能走全表
```

#### 策略二：覆盖索引避免回表

```sql
-- 场景：统计每个用户的订单数量
-- 原来：需要回表查 user_id
SELECT user_id, COUNT(*)
FROM orders
WHERE status = 1
GROUP BY user_id;

-- 添加联合索引 (status, user_id)，SQL 只需扫描索引，不需要回表
-- EXPLAIN 看 Extra 字段出现 "Using index" 即为覆盖索引

-- 原则：将 WHERE + GROUP BY + SELECT 中的字段都加入索引（减少回表）
```

#### 策略三：深分页优化

```sql
-- ❌ 深分页：偏移量大时需要扫描并丢弃大量数据（全表扫描）
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;

-- ✅ 方案1：游标分页（记录上次最大ID，适合顺序浏览）
SELECT * FROM orders WHERE id > :lastId ORDER BY id LIMIT 10;

-- ✅ 方案2：延迟关联（先走覆盖索引取ID，再回表查数据）
SELECT o.* FROM orders o
JOIN (
    SELECT id FROM orders ORDER BY id LIMIT 1000000, 10
) tmp ON o.id = tmp.id;
-- 内层只查 id（覆盖索引，无回表），外层根据10个ID取完整数据（10次回表）
```

#### 策略四：大 IN 优化

```sql
-- ❌ 大 IN 列表（IN 值很多时可能走全表）
SELECT * FROM orders WHERE user_id IN (1, 2, 3, ..., 10000);

-- ✅ 改成临时表 JOIN
CREATE TEMPORARY TABLE tmp_users (user_id BIGINT PRIMARY KEY);
INSERT INTO tmp_users VALUES (1), (2), ..., (10000);
SELECT o.* FROM orders o JOIN tmp_users t ON o.user_id = t.user_id;
```

### 4. SQL 改写优化

```sql
-- ❌ SELECT *（拉取所有字段，网络传输浪费，可能导致索引失效）
SELECT * FROM orders WHERE user_id = 100;
-- ✅ 按需查询字段
SELECT id, status, amount, created_at FROM orders WHERE user_id = 100;

-- ❌ 子查询（可能无法下推，产生临时表）
SELECT * FROM orders WHERE user_id IN (
    SELECT id FROM users WHERE city = '北京'
);
-- ✅ 改成 JOIN（优化器有更大优化空间）
SELECT o.* FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.city = '北京';

-- ❌ OR 条件（可能导致全表扫描）
SELECT * FROM orders WHERE status = 1 OR user_id = 100;
-- ✅ 改成 UNION ALL（两个索引分别走）
SELECT * FROM orders WHERE status = 1
UNION ALL
SELECT * FROM orders WHERE user_id = 100 AND status != 1;

-- ❌ 不必要的 ORDER BY RAND()（全表扫描+排序，极慢）
SELECT * FROM orders ORDER BY RAND() LIMIT 10;
-- ✅ 先算 COUNT，在应用层随机偏移
SELECT * FROM orders LIMIT :randomOffset, 1;
```

### 5. 表结构优化

```sql
-- 字段类型选择原则
-- ① 整型：TINYINT(1字节) < SMALLINT(2) < INT(4) < BIGINT(8)，选够用的最小类型
-- ② 字符串：CHAR（定长，查询快）vs VARCHAR（变长，省空间）
-- ③ 时间：DATETIME(8字节，有时区) vs TIMESTAMP(4字节，省空间但2038问题)
-- ④ 避免 TEXT/BLOB 存大字段（存到对象存储，DB 只存 URL）

-- 分区表（按时间范围分区，历史数据查询更快）
CREATE TABLE orders (
    id BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    ...
) PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION pmax  VALUES LESS THAN MAXVALUE
);
```

### 6. 面试标准答法

> 慢查询优化分四步：① 开慢查询日志 + `pt-query-digest` 找出最耗时的 SQL；② `EXPLAIN` 看执行计划，重点关注 `type`（有无全表扫描）、`key`（有无用索引）、`rows`（扫描行数）、`Extra`（有无 filesort/temporary）；③ 针对问题加索引（联合索引遵循最左前缀、覆盖索引避免回表）或改写 SQL（深分页用游标分页/延迟关联、大IN改JOIN、子查询改JOIN）；④ 表结构层面：合适的字段类型（避免大字段）、按时间分区。核心原则：减少扫描行数（索引）+ 减少数据传输量（按需查字段）+ 减少排序（索引覆盖 ORDER BY 字段）。

### 7. 常见追问

**Q: 索引并不是越多越好，为什么？**
> 索引的代价：① **写性能下降**：每次 INSERT/UPDATE/DELETE 都要维护所有相关索引（B+ 树的调整），索引越多写越慢；② **占用存储空间**：索引文件可能比数据文件还大；③ **优化器选择困难**：索引太多时优化器可能选错执行计划（statistics 不准时）。原则：只对高选择率（cardinality 高）、频繁查询的字段建索引，避免在小表、频繁更新的字段、长字段（TEXT）上建索引。

## 五、缓存与数据库数据一致性

### 1. 为什么会不一致

```
并发写场景（Cache-Aside 模式下的经典 Race Condition）：

线程A：写DB成功 ─────────────────────────────→ 删除缓存
线程B：           读缓存Miss → 读DB旧值 → 写入缓存（旧值写入！）
                                                    ↑
                                             此后缓存是旧值，DB是新值
                                             = 数据不一致！

问题根源：多个操作之间没有原子性
```

### 2. 主流一致性方案

#### 方案一：Cache-Aside + 删除缓存（推荐）

```
写操作步骤：
  1. 先更新数据库
  2. 再删除缓存（不是更新缓存）

读操作步骤：
  1. 读缓存，命中则返回
  2. 未命中则读数据库
  3. 将数据库结果写入缓存（设置过期时间）

为什么是删除缓存，而不是更新缓存？
  ① 更新缓存可能写入中间状态（多个字段的计算结果）
  ② 删除缓存更安全：下次读时从DB加载最新值
  ③ 缓存可能长期没被读（lazy loading更合理）
```

```java
// 标准 Cache-Aside 写实现
public void updateUser(User user) {
    // 1. 先更新 DB
    userMapper.update(user);
    // 2. 再删除缓存
    redisTemplate.delete("user:" + user.getId());
}

// 读实现
public User getUser(long userId) {
    String key = "user:" + userId;
    User user = (User) redisTemplate.opsForValue().get(key);
    if (user == null) {
        user = userMapper.selectById(userId);
        if (user != null) {
            redisTemplate.opsForValue().set(key, user, 30, TimeUnit.MINUTES);
        }
    }
    return user;
}
```

#### 方案二：先删缓存再更新数据库（不推荐）

```
问题：删缓存后到更新DB完成之间存在窗口期

线程A：删除缓存 ────────────────────────────→ 更新DB
线程B：          读缓存Miss → 读DB旧值 → 写入缓存（旧值！）

结果：缓存中是旧值，DB是新值，不一致（且持续到缓存过期）
```

#### 方案三：延迟双删（解决方案一的小概率问题）

```java
// 解决"先更新DB，再删缓存"的小概率 Race Condition
public void updateUserWithDelayedDoubleDel(User user) {
    // 1. 先删除缓存
    redisTemplate.delete("user:" + user.getId());
    // 2. 更新数据库
    userMapper.update(user);
    // 3. 延迟一段时间后再删一次缓存（等主从同步完成）
    // 延迟时间 > 主从同步延迟（通常 50~200ms）
    Thread.sleep(200);
    redisTemplate.delete("user:" + user.getId());
}

// 生产环境不能直接 sleep（阻塞当前线程），应该用异步延迟任务
CompletableFuture.runAsync(() -> {
    try { Thread.sleep(200); } catch (InterruptedException ignored) {}
    redisTemplate.delete("user:" + user.getId());
}, delayedExecutor);
```

#### 方案四：Canal + Binlog 异步同步（强一致场景）

```
基本原理：
  MySQL Binlog ──→ Canal 监听 ──→ 消费端删除/更新 Redis

优势：
  ① 与业务代码完全解耦，业务只需写 DB
  ② 基于 Binlog 顺序消费，天然保证顺序
  ③ Canal 模拟 MySQL Slave 协议，对 MySQL 无侵入

架构：
  Business App
       │ INSERT/UPDATE/DELETE
       ↓
    MySQL (Master)
       │ binlog
       ↓
    Canal Server (模拟 Slave)
       │ 解析 binlog
       ↓
    Kafka / RocketMQ
       │ 消费
       ↓
    Cache Updater Service → Redis DELETE/UPDATE

问题：有延迟（ms~s级），不适合强实时场景
```

```java
// Canal 客户端示例
CanalConnector connector = CanalConnectors.newSingleConnector(
    new InetSocketAddress("canal-server", 11111), "example", "", "");
connector.connect();
connector.subscribe("business_db\\.users");  // 订阅 users 表

while (true) {
    Message message = connector.getWithoutAck(100);
    for (CanalEntry.Entry entry : message.getEntries()) {
        if (entry.getEntryType() == ROWDATA) {
            RowChange rowChange = RowChange.parseFrom(entry.getStoreValue());
            String tableName = entry.getHeader().getTableName();
            for (RowData rowData : rowChange.getRowDatasList()) {
                long userId = getId(rowData);
                // 删除或更新 Redis 缓存
                redisTemplate.delete("user:" + userId);
            }
        }
    }
    connector.ack(message.getId());
}
```

### 3. 各方案对比

| 方案 | 一致性 | 复杂度 | 适用场景 |
|---|---|---|---|
| Cache-Aside（先更新DB，删缓存）| 最终一致（ms级）| 低 | **绝大多数业务场景** |
| 延迟双删 | 最终一致（比上面强）| 中 | 主从架构，读从库场景 |
| Canal + Binlog | 最终一致（ms~s）| 高 | 数据量大，需要完全解耦 |
| 分布式事务（2PC）| 强一致 | 极高 | 金融/支付等强一致场景 |

### 4. 特殊场景处理

#### 缓存删除失败怎么办

```java
// 问题：更新DB成功，但删除缓存失败（Redis宕机/超时）→ 数据长期不一致

// 解决方案1：重试 + MQ保证最终删除
public void updateUser(User user) {
    userMapper.update(user);  // 1. 更新DB
    try {
        redisTemplate.delete("user:" + user.getId()); // 2. 删缓存
    } catch (Exception e) {
        // 删缓存失败，发消息到MQ，异步重试
        mq.send(new CacheEvictMessage("user:" + user.getId()));
        log.error("删缓存失败，已发MQ重试", e);
    }
}

// 解决方案2：设置短过期时间（兜底）
// 即使删缓存失败，缓存过期后也会从DB加载最新值
redisTemplate.opsForValue().set(key, value, 5, TimeUnit.MINUTES); // 最多不一致5分钟
```

#### 主从延迟导致读到旧数据

```
场景：
  1. 写请求 → 更新主库 → 删除缓存
  2. 读请求 → 缓存 Miss → 读从库（主从延迟，从库是旧数据！）
  3. 旧数据写入缓存 → 持续不一致

解决方案：
  ① 延迟双删（等待主从同步完成后再删一次）
  ② 强制读主库：查询加 hint 或通过代理路由到主库（用于写后立即读场景）
  ③ 使用 Redis 版本号：写DB时更新版本号到Redis，读时比较版本号
```

#### 缓存击穿期间的并发写入

```java
// 问题：缓存 Miss 时，大量并发请求同时查 DB，同时写缓存，最后一个写的覆盖前面
// 利用 Redisson 分布式锁，只允许一个线程回填缓存

public User getUserWithLock(long userId) {
    String cacheKey = "user:" + userId;
    User user = (User) redis.get(cacheKey);
    if (user != null) return user;
    
    // 缓存 Miss，加锁重建缓存
    String lockKey = "lock:user:" + userId;
    RLock lock = redisson.getLock(lockKey);
    try {
        if (lock.tryLock(3, 5, TimeUnit.SECONDS)) {
            // 再次检查（Double Check，可能其他线程已经回填）
            user = (User) redis.get(cacheKey);
            if (user != null) return user;
            
            user = userMapper.selectById(userId);
            redis.setex(cacheKey, 300, user);
        }
    } finally {
        lock.unlock();
    }
    return user;
}
```

### 5. 面试标准答法

> 缓存与数据库一致性核心是处理**并发更新的竞争条件**。推荐的 Cache-Aside 模式写法是"先更新数据库，再删除缓存"（不是更新缓存，因为删除比更新更安全）。这个方案仍有小概率不一致：A 更新 DB 后、删缓存前，B 读缓存 Miss 把旧数据写入。对策是**延迟双删**（更新 DB 后延迟 200ms 再次删一次，覆盖 B 写入的旧缓存）。大规模场景用 **Canal 监听 Binlog** 异步更新缓存，与业务代码完全解耦，天然保证顺序。兜底措施是给缓存设置过期时间，即使删缓存失败也能最终一致。

### 6. 常见追问

**Q: 先删缓存再更新数据库，和先更新数据库再删缓存，哪个更好？**
> 先更新 DB 再删缓存（Cache-Aside 标准写法）更好。先删缓存的问题：删完到更新 DB 完成之间有窗口期，并发读会把 DB 的旧值写回缓存；先更新 DB 的问题：更新 DB 成功到删缓存之间也有短暂窗口期，但这个窗口期非常短（微秒级），且已经在 DB 写成功后，下次读一定能读到新值（只是极短暂的短暂旧值，而非长期旧值）。所以是两害相权取其轻——先更新 DB 再删缓存是更安全的方案。

**Q: 如何保证缓存和数据库操作的原子性？**
> 无法用传统 ACID 事务保证（Redis 和 MySQL 是两个独立系统），只能通过最终一致性方案。思路：① **补偿机制**：DB 操作成功后，删缓存失败则记录到"待重试队列"（本地消息表或 MQ），异步重试直到成功；② **Binlog 方案**：完全去掉业务层的缓存操作，由 Canal 监听 Binlog 来驱动缓存更新，天然保证最终一致；③ **Saga 模式**：允许不一致，通过补偿事务修正（适合长事务场景）。
