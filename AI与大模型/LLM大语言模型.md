# LLM 大语言模型

## 一、LLM 基础概念 ⭐⭐⭐

### 1.1 什么是大语言模型

```
大语言模型（Large Language Model）是基于 Transformer 架构、
在海量文本数据上预训练的深度神经网络，具备以下核心能力：

┌─────────────────────────────────────────────────────┐
│                   LLM 核心能力                       │
├──────────────┬──────────────────────────────────────┤
│  文本生成     │ 根据上下文生成连贯、有逻辑的文本     │
│  语言理解     │ 理解语义、意图、情感                 │
│  知识推理     │ 基于训练数据中的知识进行推理          │
│  上下文学习   │ In-Context Learning，少样本/零样本   │
│  指令跟随     │ 理解并执行自然语言指令               │
│  代码生成     │ 编写、解释、调试代码                 │
└──────────────┴──────────────────────────────────────┘
```

### 1.3 关键术语

| 术语 | 解释 |
|------|------|
| **Token** | 模型处理的最小文本单元（约 0.75 个英文单词或 0.5 个中文字符）|
| **上下文窗口** | 模型单次能处理的最大 Token 数量（如 GPT-4o 128K，Claude 200K）|
| **Temperature** | 控制输出随机性，0 最确定，1 最随机 |
| **Top-P** | 核采样，从累积概率前 P 的 Token 中采样 |
| **System Prompt** | 系统提示词，定义模型角色和行为规范 |
| **Few-shot** | 在 Prompt 中提供少量示例，引导模型输出格式 |
| **Hallucination** | 幻觉，模型生成看似合理但实际错误的内容 |
| **Alignment** | 对齐，让模型输出符合人类期望和价值观 |

## 二、Transformer 架构 ⭐⭐⭐

### 2.1 整体架构

```
原始 Transformer（Encoder-Decoder）：

输入序列 ──→ [Encoder] ──→ 编码表示 ──→ [Decoder] ──→ 输出序列

三种变体：
┌───────────────┬──────────────────┬─────────────────────┐
│ 架构           │ 代表模型          │ 适用任务             │
├───────────────┼──────────────────┼─────────────────────┤
│ Encoder-Only  │ BERT、RoBERTa     │ 文本分类、NER、语义理解 │
│ Decoder-Only  │ GPT、LLaMA、Qwen  │ 文本生成、对话（主流） │
│ Encoder-Decoder│ T5、BART          │ 翻译、摘要           │
└───────────────┴──────────────────┴─────────────────────┘

当前大模型几乎全部采用 Decoder-Only 架构（自回归生成）
```

### 2.2 自注意力机制（Self-Attention）

```
核心公式：Attention(Q, K, V) = softmax(QK^T / √d_k) · V

Q（Query）、K（Key）、V（Value）均由输入经线性变换得到

计算步骤：
1. 输入 X 分别乘以 W_Q、W_K、W_V 得到 Q、K、V     ← 线性映射
2. Q × K^T → 注意力分数矩阵（每个 Token 对其他 Token 的关注度）
3. 除以 √d_k → 缩放（防止点积过大导致 softmax 梯度消失）
4. softmax → 归一化为注意力权重
5. 权重 × V → 加权求和得到输出

直觉理解：
- Q = "我要查什么"（查询向量）
- K = "我这里有什么"（键向量）
- V = "我这里的具体内容"（值向量）
- QK^T = "查询与每个键的匹配度"
- softmax(QK^T)V = "按匹配度加权取值"
```

### 2.3 多头注意力（Multi-Head Attention）

```
Multi-Head 将 Q/K/V 拆成 h 个头，各自独立计算注意力后拼接：

MultiHead(Q,K,V) = Concat(head_1, ..., head_h) · W_O

好处：
- 每个头能关注不同维度的语义关系（语法、语义、位置等）
- h=32 或 64 是常见配置，d_k = d_model / h
```

### 2.4 位置编码（Positional Encoding）

```
Transformer 无循环结构，需要额外注入位置信息：

┌─────────────────┬────────────────────────────────────┐
│ 方案             │ 说明                               │
├─────────────────┼────────────────────────────────────┤
│ 正弦位置编码     │ 原始 Transformer，绝对位置          │
│ 可学习位置编码   │ BERT，可训练参数                    │
│ RoPE 旋转位置编码│ LLaMA/Qwen/GPT-NeoX，相对位置，    │
│                 │ 支持长度外推，当前主流               │
│ ALiBi           │ BLOOM，注意力偏置，线性外推          │
└─────────────────┴────────────────────────────────────┘
```

### 2.5 Transformer Block 内部结构

```
每个 Decoder Block（以 LLaMA 为例）：

Input
  │
  ├──→ RMSNorm（Pre-Norm，比 LayerNorm 更快）
  │       │
  │       ↓
  │     Masked Multi-Head Self-Attention（因果注意力，只看左侧）
  │       │ + GQA（Grouped-Query Attention，减少 KV 头数量以节省显存）
  │       │
  ├───── + ←── 残差连接
  │
  ├──→ RMSNorm
  │       │
  │       ↓
  │     FFN（SwiGLU 激活函数，比 ReLU 效果更好）
  │       │
  ├───── + ←── 残差连接
  │
  ↓
Output

典型配置（LLaMA 3 70B）：
- 80 层 Decoder Block
- 64 个注意力头
- 隐藏维度 8192
- GQA: 8 个 KV 头（减少 8 倍 KV cache 显存）
```

## 三、LLM 训练流程 ⭐⭐⭐

### 3.1 三阶段训练范式

```
┌─────────────────────────────────────────────────────────┐
│           LLM 训练三阶段 Pipeline                        │
├─────────────┬───────────────────────────────────────────┤
│ 第一阶段     │ 预训练（Pre-training）                    │
│             │ 海量无标注文本 → 自回归语言建模            │
│             │ 数据量: 数万亿 Token                      │
│             │ 目标: 学习语言知识和世界知识               │
│             │ 算力: 数千~上万 GPU，训练数周~数月         │
├─────────────┼───────────────────────────────────────────┤
│ 第二阶段     │ 监督微调（SFT, Supervised Fine-Tuning）   │
│             │ 高质量指令-回答对 → 学习指令跟随能力       │
│             │ 数据量: 数万~数十万条                     │
│             │ 目标: 学会按指令回答问题                   │
├─────────────┼───────────────────────────────────────────┤
│ 第三阶段     │ 人类反馈强化学习（RLHF）                  │
│             │ 人类标注偏好 → 训练奖励模型 → PPO 优化     │
│             │ 目标: 输出更符合人类偏好（有用、无害、诚实）│
└─────────────┴───────────────────────────────────────────┘

替代方案：DPO（Direct Preference Optimization）
- 跳过奖励模型，直接从偏好数据优化策略
- 更简单、训练更稳定，DeepSeek/LLaMA 3 等均采用
```

### 3.2 预训练目标

```
自回归语言模型（Causal LM）：

给定前面的 Token，预测下一个 Token：
P(x_t | x_1, x_2, ..., x_{t-1})

损失函数：交叉熵损失
L = -Σ log P(x_t | x_<t)

示例：
输入:  "今天 天气 真"
预测:  "好"（概率最高的下一个 Token）
```

## 四、LLM 推理优化 ⭐⭐⭐

### 4.1 推理过程

```
LLM 推理分两个阶段：

┌───────────────────────────────────────────────┐
│  Prefill 阶段（预填充）                        │
│  - 一次性处理整个输入 Prompt                   │
│  - 计算所有输入 Token 的 KV Cache              │
│  - 计算密集型（受算力约束）                    │
├───────────────────────────────────────────────┤
│  Decode 阶段（自回归生成）                     │
│  - 每次只生成一个 Token                        │
│  - 利用 KV Cache 避免重复计算                  │
│  - 内存密集型（受显存带宽约束）                │
│  - Token by Token，延迟累积                   │
└───────────────────────────────────────────────┘
```

### 4.2 KV Cache

```
KV Cache 原理：自回归生成时，已生成 Token 的 K/V 不变，缓存后复用

无 KV Cache：生成第 n 个 Token 需计算前 n-1 个 Token 的注意力 → O(n²)
有 KV Cache：只计算新 Token 与缓存 K/V 的注意力 → O(n)

KV Cache 显存 = 2 × layers × d_model × seq_len × batch × dtype_bytes
LLaMA 70B, seq=4096, batch=1: 约 10GB KV Cache

优化手段：
- GQA（Grouped-Query Attention）：KV 头数 < Q 头数，减少 KV Cache
- MQA（Multi-Query Attention）：所有 Q 头共享一组 KV
- PagedAttention（vLLM）：KV Cache 分页管理，避免显存碎片
```

### 4.3 主流推理框架

| 框架 | 核心特点 | 适用场景 |
|------|---------|---------|
| **vLLM** | PagedAttention + Continuous Batching，吞吐量最高 | 高并发在线服务 |
| **TensorRT-LLM** | NVIDIA 官方，深度融合 GPU 算子优化 | 追求极致性能 |
| **Ollama** | 一键本地部署，简单易用 | 本地开发调试 |
| **llama.cpp** | 纯 C++ 实现，支持 CPU 推理 | 边缘设备/无 GPU |
| **SGLang** | 结构化生成 + RadixAttention + 高级调度 | 复杂 Agent 场景 |

### 4.4 量化技术

```
量化 = 用更低位数表示模型参数，减少显存和加速推理

┌───────────┬────────────────────────────────────────────┐
│ 方法       │ 说明                                       │
├───────────┼────────────────────────────────────────────┤
│ FP16/BF16 │ 半精度，显存减半，几乎无精度损失（训练标配）│
│ INT8      │ 8bit 整数量化，显存减至 1/4               │
│ INT4/GPTQ │ 4bit 量化，70B 模型可用单张 A100 跑        │
│ AWQ       │ 激活感知权重量化，比 GPTQ 速度更快          │
│ GGUF      │ llama.cpp 格式，支持 CPU+GPU 混合推理       │
└───────────┴────────────────────────────────────────────┘

量化公式（均匀量化）：
q = round((x - min) / (max - min) × (2^bits - 1))
x' = q × (max - min) / (2^bits - 1) + min
```

### 5.1 提示词工程核心技巧

```
┌────────────────────────────────────────────────────┐

```

### 5.2 高级 Prompt 策略

| 策略 | 原理 | 示例 |
|------|------|------|
| **Zero-shot** | 无示例直接提问 | "翻译以下文本为英文：..." |
| **Few-shot** | 提供少量示例 | "示例1:...→... 示例2:...→... 现在：..." |
| **CoT（思维链）** | 引导逐步推理 | "让我们一步步分析这个问题..." |

## 六、MoE（混合专家模型）⭐⭐

### 6.1 MoE 架构原理

```
标准 FFN vs MoE FFN：

标准 FFN：
  Input → FFN → Output（所有参数都参与计算）

MoE FFN：
  Input → Router（门控网络）→ 选择 Top-K 个专家
         ├→ Expert 1 ──┐
         ├→ Expert 2 ──┤→ 加权求和 → Output
         └→ ...        ┘

DeepSeek-V3 配置：
- 总参数 671B（MoE）
- 每次激活约 37B（Top-8 from 256 experts）
- 推理开销约等于 37B 密集模型，但知识容量达 671B

优势：参数多但推理成本低→"大模型，小计算"
挑战：负载均衡、通信开销、训练稳定性
```

## 八、面试高频问题

### Q1: Transformer 中 Self-Attention 的时间和空间复杂度？
> 时间复杂度 O(n²d)：n 是序列长度，d 是维度。QK^T 的矩阵乘法是 O(n²d)，注意力权重乘 V 也是 O(n²d)。空间复杂度 O(n²)：需要存储 n×n 的注意力分数矩阵。这也是长上下文的主要瓶颈——序列长度翻倍，计算量和显存都翻 4 倍。优化方案包括 FlashAttention（IO 感知的精确注意力）、稀疏注意力、线性注意力等。

### Q2: 为什么现在大模型都用 Decoder-Only 架构？
> ① **训练效率**：自回归目标天然适合 next-token prediction，数据利用率高（每个位置都可计算 loss）；② **涌现能力**：规模增大后 Decoder-Only 模型展现出更强的 In-Context Learning 和推理能力；③ **通用性**：一个模型通过不同 Prompt 可做翻译、分类、问答、生成等各种任务（GPT 路线验证了这一点）；④ **工程简洁**：只有一个 Decoder Stack，实现简单、KV Cache 友好。

### Q3: LoRA 为什么有效？低秩假设是什么意思？
> LoRA 基于"低秩假设"：微调时模型权重的变化 ΔW 是低秩的，即可以分解为两个小矩阵 B×A（B:d×r, A:r×d, r<<d）。直觉上，微调一个领域任务并不需要改变模型的所有知识，只需调整一个低维子空间。LoRA 冻结原始权重 W，只训练 BA，参数量从 d² 降到 2dr。推理时 W+BA 可以合并，**不增加推理延迟**。实践中 r=8~64 通常就能达到接近全参微调的效果。

### Q4: 什么是幻觉（Hallucination）？如何缓解？
> 幻觉是模型生成看似合理但实际不正确的内容。原因：① 训练数据中的噪声和偏差；② 语言模型本质是基于统计的文本生成，不具备真正的事实核查能力；③ 对训练数据中未覆盖的知识进行"创造性填补"。缓解方案：① **RAG**（检索增强生成）：用外部知识库提供事实依据，最有效；② **SFT 训练**：用高质量数据训练模型说"我不确定"；③ **Self-Consistency**：多次采样比较一致性；④ **工具调用**：让模型调用搜索/计算器等外部工具获取准确信息。

### Q5: KV Cache 是什么？为什么是推理的显存瓶颈？
> 自回归生成时，每生成一个新 Token 都需要与之前所有 Token 做注意力计算。KV Cache 把已计算过的 Key 和 Value 缓存下来，避免重复计算（把 O(n²) 降到 O(n)）。但 KV Cache 的显存随 **序列长度 × 批次大小** 线性增长，长上下文和高并发场景下可能超过模型参数本身的显存占用。优化方案：GQA/MQA（减少 KV 头数）、PagedAttention（vLLM，按需分配+内存池化）、量化 KV Cache。
