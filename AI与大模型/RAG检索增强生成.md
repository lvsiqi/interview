# RAG 检索增强生成

## 一、RAG 核心概念 ⭐⭐⭐

### 1.1 什么是 RAG

```
RAG（Retrieval-Augmented Generation，检索增强生成）：
在 LLM 生成回答前，先从外部知识库中检索相关信息，
将检索结果作为上下文注入 Prompt，让模型基于真实数据回答。

┌──────────────────────────────────────────────────────┐
│                  RAG 核心流程                          │
│                                                      │
│  用户问题 ──→ 检索（Retrieval）──→ 增强（Augmentation）│
│                   │                     │             │
│              查询向量数据库           将检索结果注入   │
│              找到 Top-K 相关文档      Prompt 上下文    │
│                                         │             │
│                                    生成（Generation）  │
│                                    LLM 基于上下文回答  │
└──────────────────────────────────────────────────────┘
```

### 1.2 为什么需要 RAG

| 问题 | 纯 LLM | RAG 解决方案 |
|------|--------|------------|
| **知识时效性** | 训练数据有截止日期 | 知识库可实时更新 |
| **幻觉问题** | 可能编造不存在的信息 | 基于检索到的真实数据回答 |
| **领域知识** | 缺乏私有/专业领域知识 | 接入企业内部文档/数据库 |
| **可解释性** | 黑箱生成，无法溯源 | 可追溯到具体文档和段落 |
| **数据安全** | 数据需要发给模型训练 | 数据留在本地，仅检索时使用 |
| **成本** | 微调大模型成本高 | 无需微调，即插即用 |

### 1.3 RAG vs 微调 vs 长上下文

```
┌──────────────┬───────────────┬───────────────┬───────────────┐
│              │ RAG           │ 微调（SFT）    │ 长上下文       │
├──────────────┼───────────────┼───────────────┼───────────────┤
│ 知识更新     │ ✅ 实时更新     │ ❌ 需重新训练   │ ❌ 窗口有限    │
│ 成本         │ ✅ 低          │ ❌ 较高        │ ✅ 低          │
│ 幻觉控制     │ ✅ 有来源可查   │ ⚠️ 部分改善    │ ⚠️ 部分改善   │
│ 私有知识     │ ✅ 无需训练     │ ✅ 内化到模型   │ ⚠️ 窗口受限   │
│ 推理速度     │ ⚠️ 多一步检索  │ ✅ 同原模型    │ ❌ 上下文长慢  │
│ 适用场景     │ 知识密集型QA   │ 领域风格迁移   │ 文档分析       │
└──────────────┴───────────────┴───────────────┴───────────────┘

最佳实践：RAG + 微调 结合（先 RAG 检索，再用微调后的领域模型生成）
```

## 二、RAG 架构设计 ⭐⭐⭐

### 2.1 标准 RAG Pipeline

```
                    离线阶段（Indexing）
                    ┌─────────────────────────────────────┐
                    │                                     │
文档来源             │  文档加载 → 文本清洗 → 分块(Chunking)│
(PDF/Word/HTML/DB)──→│      │                              │
                    │      ↓                              │
                    │  Embedding 模型 → 向量化              │
                    │      │                              │
                    │      ↓                              │
                    │  存入向量数据库（Milvus/Pinecone...)  │
                    └─────────────────────────────────────┘

                    在线阶段（Querying）
                    ┌─────────────────────────────────────┐
                    │                                     │
用户问题 ────────────→│  Query Embedding → 向量检索 Top-K    │
                    │      │                              │
                    │      ↓                              │
                    │  Rerank 重排序 → 过滤 → 选取最相关   │
                    │      │                              │
                    │      ↓                              │
                    │  构建 Prompt（问题 + 检索结果）       │
                    │      │                              │
                    │      ↓                              │
                    │  LLM 生成回答 → 返回用户              │
                    └─────────────────────────────────────┘
```

### 2.2 文档分块策略（Chunking）⭐⭐

```
分块是 RAG 效果的关键环节，块太大→噪声多，块太小→语义不完整

┌─────────────────┬──────────────────────────────────────┐
│ 分块策略         │ 说明                                  │
├─────────────────┼──────────────────────────────────────┤
│ 固定大小分块     │ 按字符数/Token 数切割（如512 Token）   │
│                 │ 简单快速，但可能切断语义               │
├─────────────────┼──────────────────────────────────────┤
│ 重叠分块        │ 每个块之间有重叠部分（如 overlap=50）  │
│                 │ 避免边界信息丢失                       │
├─────────────────┼──────────────────────────────────────┤
│ 语义分块        │ 按段落/章节/语义边界切割               │
│                 │ 效果好但实现复杂                       │
├─────────────────┼──────────────────────────────────────┤
│ 递归分块        │ 按层级分隔符递归切割（\n\n→\n→.→空格） │
│                 │ LangChain 默认策略，兼顾效果和通用性   │
├─────────────────┼──────────────────────────────────────┤
│ 父子分块        │ 小块检索，返回时带上父级大块上下文      │
│                 │ 检索精确 + 上下文完整                  │
└─────────────────┴──────────────────────────────────────┘

推荐参数：
- chunk_size: 256~1024 Token（视文档类型而定）
- chunk_overlap: chunk_size 的 10%~20%
- 技术文档: 较大块（512~1024），保持代码完整
- FAQ 类: 较小块（128~256），一问一答
```

### 2.3 Embedding 模型

```
Embedding 模型将文本转换为高维向量，语义相近的文本向量距离近

主流 Embedding 模型：
┌──────────────────┬──────────┬────────┬────────────────┐
│ 模型              │ 维度      │ 语言    │ 特点            │
├──────────────────┼──────────┼────────┼────────────────┤
│ text-embedding-3 │ 256~3072 │ 多语言  │ OpenAI，效果好  │
│ bge-large-zh-v1.5│ 1024     │ 中文    │ 智源，中文最佳  │
│ bge-m3           │ 1024     │ 多语言  │ 多语言+多方式   │
│ GTE-Qwen2        │ 768~1024 │ 多语言  │ 阿里，性价比高  │
│ Jina-embeddings  │ 768      │ 多语言  │ 8192 Token 长文 │
│ E5-Mistral-7B    │ 4096     │ 多语言  │ LLM做Embedding │
└──────────────────┴──────────┴────────┴────────────────┘

相似度计算：
- 余弦相似度: cos(A,B) = A·B / (|A|·|B|)    ← 最常用
- 点积: A·B（向量归一化后等价于余弦相似度）
- 欧氏距离: √(Σ(a_i - b_i)²)
```

### 2.4 向量数据库

| 数据库 | 类型 | 核心特点 | 适用场景 |
|--------|------|---------|---------|
| **Milvus** | 开源独立数据库 | 分布式、高性能、GPU 加速、生态完善 | 生产级大规模向量检索 |
| **Qdrant** | 开源独立数据库 | Rust 实现、低内存、标量+向量混合过滤 | 需要复杂过滤的场景 |
| **Chroma** | 嵌入式 | 轻量、Python 优先、开箱即用 | 快速原型/小规模 |
| **Pinecone** | 全托管 SaaS | 免运维、自动扩容 | 不想管基础设施 |
| **Weaviate** | 开源独立数据库 | GraphQL API、多模态、混合搜索 | 多模态场景 |
| **Elasticsearch** | 已有搜索引擎 | kNN 向量检索 + 传统全文搜索 | 已有 ES 集群扩展 |
| **pgvector** | PG 扩展插件 | 基于 PostgreSQL，SQL 查询 | 已有 PG 不想加新组件 |

```
向量检索索引类型：
┌───────────┬──────────────────────────────────────────┐
│ 索引类型   │ 说明                                     │
├───────────┼──────────────────────────────────────────┤
│ Flat      │ 暴力搜索，100%精确，慢                    │
│ IVF       │ 倒排索引聚类，近似搜索，适合百万级         │
│ HNSW      │ 分层可导航小世界图，高召回+快速，内存占用大│
│ PQ        │ 乘积量化，压缩向量减少内存，牺牲精度      │
│ ScaNN     │ Google 出品，速度+精度均衡                │
└───────────┴──────────────────────────────────────────┘

生产推荐：HNSW（召回率高、延迟低，是 Milvus/Qdrant 默认索引）
```

## 三、高级 RAG 策略 ⭐⭐⭐

### 3.1 查询优化（Pre-Retrieval）

```
用户原始查询往往不够精确，需要优化后再检索：

┌─────────────────┬──────────────────────────────────────┐
│ 策略             │ 说明                                  │
├─────────────────┼──────────────────────────────────────┤
│ Query Rewriting │ LLM 改写查询，使其更适合检索           │
│                 │ "怎么部署" → "K8s Deployment 部署流程" │
├─────────────────┼──────────────────────────────────────┤
│ HyDE            │ 让 LLM 先生成假设性回答，再用回答去检索│
│ (Hypothetical   │ 假设回答的向量比问题更接近真实文档     │
│  Document       │                                      │
│  Embeddings)    │                                      │
├─────────────────┼──────────────────────────────────────┤
│ Sub-Question    │ 将复杂问题拆解为多个子问题分别检索     │
│ Decomposition   │ "A和B的区别" → 分别检索A和B的信息     │
├─────────────────┼──────────────────────────────────────┤
│ Step-Back       │ 先问更抽象的问题获取背景知识           │
│ Prompting       │ "Redis为什么快" → 先检索"内存数据库原理│
└─────────────────┴──────────────────────────────────────┘
```

### 3.2 混合检索（Hybrid Search）

```
向量检索（语义）+ 关键词检索（BM25）结合，互补优势：

用户查询
    ├──→ Sparse 检索（BM25 / TF-IDF）→ 关键词精确匹配
    │         适合专有名词、编号、代码
    │
    └──→ Dense 检索（向量相似度）→ 语义相似匹配
              适合同义词、自然语言表达
    
    └──→ RRF（Reciprocal Rank Fusion）→ 融合排序

RRF 公式：score(d) = Σ 1/(k + rank_i(d))
- k 通常取 60
- 将多路结果按倒数排名融合，简单有效

效果：混合检索 > 纯向量检索 > 纯关键词检索（多个基准测试验证）
```

### 3.3 重排序（Reranking）⭐⭐

```
检索返回 Top-K 后，用更精确（但更慢）的模型重排序：

向量检索 Top-20 ──→ Rerank 模型 ──→ 精排 Top-5 ──→ 送入 LLM

常用 Reranker：
┌───────────────────┬────────────────────────────────┐
│ 模型               │ 特点                           │
├───────────────────┼────────────────────────────────┤
│ bge-reranker-v2-m3│ BAAI 出品，多语言，效果好       │
│ Cohere Reranker   │ API 服务，简单易用              │
│ Cross-Encoder     │ 将 Query+Doc 拼接后分类，精确   │
│ RankLLM           │ 用 LLM 做排序，效果最好，最慢   │
└───────────────────┴────────────────────────────────┘

为什么需要 Rerank？
- Embedding 模型（Bi-Encoder）分别编码 Q 和 D，效率高但精度有限
- Reranker（Cross-Encoder）同时编码 Q+D，能捕捉更深层的交互关系
- Rerank 只处理 Top-K 个文档，计算量可接受
```

### 3.4 上下文增强（Post-Retrieval）

```
检索到文档后，在送入 LLM 前进行优化：

┌─────────────────┬──────────────────────────────────────┐
│ 策略             │ 说明                                  │
├─────────────────┼──────────────────────────────────────┤
│ Context         │ 将检索到的多个块进行压缩/摘要          │
│ Compression     │ 减少 Token 开销，去除无关信息          │
├─────────────────┼──────────────────────────────────────┤
│ Sentence Window │ 检索小块，但返回时扩展到包含其前后文   │
│                 │ 检索精确 + 上下文完整                  │
├─────────────────┼──────────────────────────────────────┤
│ Lost in Middle  │ 将最相关的内容放在 Prompt 头部和尾部   │
│                 │ LLM 对中间内容关注度低                 │
├─────────────────┼──────────────────────────────────────┤
│ 元数据过滤       │ 按时间、来源、类别等过滤文档           │
│                 │ 减少噪声、提高相关性                   │
└─────────────────┴──────────────────────────────────────┘
```

### 4.1 评估维度

```
RAG 效果评估需要分别评估检索质量和生成质量：

┌────────────────┬──────────────────────────────────────┐
│ 维度            │ 指标                                  │
├────────────────┼──────────────────────────────────────┤

└────────────────┴──────────────────────────────────────┘

评估工具：
- RAGAS：开源 RAG 评估框架（Faithfulness / Context Relevancy / Answer Relevancy）
- LlamaIndex 内置评估模块
- 人工评审（金标准）
```

## 五、RAG 技术栈与框架 ⭐⭐

### 5.1 主流 RAG 开发框架

| 框架 | 核心特点 | 适用场景 |
|------|---------|---------|
| **LangChain** | 组件丰富、生态最大、Chain/Agent 抽象 | 通用 RAG/Agent 开发 |
| **LlamaIndex** | 专注数据索引、丰富的数据连接器 | 文档问答、知识库 |
| **Haystack** | 端到端 Pipeline、组件化 | 企业级搜索 |
| **Dify** | 低代码可视化、开箱即用 | 快速构建 RAG 应用 |
| **RAGFlow** | 深度文档解析、可视化知识管理 | 复杂文档场景 |
| **FastGPT** | 工作流编排、知识库管理 | 快速搭建 AI 客服 |

### 5.2 Java 生态 RAG 方案

```java
// Spring AI + 向量数据库（pgvector）示例
@Service
public class RagService {
    @Autowired
    private ChatClient chatClient;
    
    @Autowired
    private VectorStore vectorStore;
    
    public String chat(String userQuery) {
        // 1. 向量检索相关文档
        List<Document> docs = vectorStore.similaritySearch(
            SearchRequest.query(userQuery)
                .withTopK(5)
                .withSimilarityThreshold(0.7)
        );
        
        // 2. 构建带上下文的 Prompt
        String context = docs.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n\n"));
        
        String prompt = """
            基于以下参考资料回答问题。如果资料中没有相关信息，请说明。
            
            参考资料：
            %s
            
            问题：%s
            """.formatted(context, userQuery);
        
        // 3. LLM 生成回答
        return chatClient.prompt(prompt).call().content();
    }
}
```

## 六、面试高频问题

### Q1: RAG 的效果不好怎么优化？
> 按流程逐步排查：① **检索环节**：检查分块策略（块大小是否合适）、Embedding 模型（换更好的如 bge-m3）、增加混合检索（BM25+向量）、增加 Rerank 重排序；② **生成环节**：优化 Prompt 模板（强调忠实于检索结果）、调整 Temperature（降低随机性）、尝试换更强的 LLM；③ **数据质量**：文档清洗（去除噪声、格式化）、丰富元数据、确保知识库覆盖完整；④ **评估驱动**：用 RAGAS 等框架量化评估，定位是检索问题还是生成问题。

### Q2: 向量数据库选型考虑什么因素？
> ① **数据规模**：百万以下用 Chroma/pgvector 够了，亿级以上需要 Milvus/Qdrant 分布式架构；② **现有技术栈**：已有 PG 可用 pgvector，已有 ES 可用 kNN 插件，尽量不增加新组件；③ **检索需求**：需要混合检索（向量+标量过滤）选 Qdrant/Milvus，纯向量选什么都行；④ **运维成本**：不想运维选 Pinecone SaaS，可运维选 Milvus/Qdrant；⑤ **性能要求**：高并发选 Milvus（GPU 加速），延迟敏感选 HNSW 索引。

### Q3: 分块策略怎么选择？
> 没有万能策略，需要根据文档类型调整：① **通用文档**：递归分块（LangChain RecursiveCharacterTextSplitter），chunk_size=512，overlap=50；② **技术文档/代码**：按函数/类/Markdown 标题切割，保持代码块完整；③ **FAQ/问答**：按 QA 对切割，一问一答为一块；④ **长文档**：父子分块策略，小块检索+大块返回上下文。关键原则：**块中应包含足够回答问题的完整信息，同时不引入太多噪声**。
