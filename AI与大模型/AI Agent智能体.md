# AI Agent 智能体开发

## 一、Agent 核心概念 ⭐⭐⭐

### 1.1 什么是 AI Agent

```
AI Agent（智能体）= LLM（大脑）+ 工具调用 + 规划推理 + 记忆

不同于简单的 LLM 对话，Agent 能够：
- 自主规划：将复杂任务分解为步骤
- 工具使用：调用外部 API/数据库/代码执行器
- 观察反思：根据执行结果调整策略
- 持久记忆：记住历史交互和中间结果

┌─────────────────────────────────────────────┐
│                 AI Agent 架构                 │
│                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │  规划    │  │  记忆    │  │  工具    │     │
│  │Planning │  │ Memory  │  │  Tools  │     │
│  └────┬────┘  └────┬────┘  └────┬────┘     │
│       │            │            │           │
│       └────────────┼────────────┘           │
│                    │                        │
│              ┌─────┴─────┐                  │
│              │  LLM 大脑  │                  │
│              │ (推理引擎) │                  │
│              └─────┬─────┘                  │
│                    │                        │
│              ┌─────┴─────┐                  │
│              │  行动执行   │                  │
│              │  Action    │                  │
│              └───────────┘                  │
└─────────────────────────────────────────────┘
```

### 1.2 Agent 与传统 Chatbot 的区别

| | 传统 Chatbot / LLM 对话 | AI Agent |
|---|---|---|
| **能力** | 单轮/多轮对话 | 自主规划+执行+反思 |
| **工具** | 无 | 可调用搜索/代码/API/数据库 |
| **复杂任务** | 一次性回答 | 多步骤分解执行 |
| **记忆** | 仅上下文窗口 | 短期+长期记忆 |
| **自主性** | 被动响应 | 主动规划和行动 |
| **输出** | 文本 | 文本+动作（发邮件/写代码/查数据）|

## 二、Agent 核心模块 ⭐⭐⭐

### 2.1 规划（Planning）

```
Agent 将复杂任务拆解为可执行步骤的能力：

┌─────────────────────────────────────────────────┐
│                  规划策略                         │
├──────────────┬──────────────────────────────────┤
│ 任务分解      │ 将大任务拆成小任务（Chain of       │
│ (Task         │ Thought、Tree of Thought）       │
│  Decomposition│                                  │
├──────────────┼──────────────────────────────────┤
│ ReAct        │ Reasoning + Acting 交替进行       │
│              │ Think → Act → Observe → Think...  │
│              │ 最经典的 Agent 推理范式             │
├──────────────┼──────────────────────────────────┤
│ Plan & Solve │ 先制定完整计划，再逐步执行         │
│              │ 适合需要全局规划的任务              │
├──────────────┼──────────────────────────────────┤
│ Reflexion    │ 执行后自我反思，修正错误           │
│              │ 类似人类 "复盘" 的能力             │
└──────────────┴──────────────────────────────────┘
```

**ReAct 范式（最核心）：**

```
用户: "帮我查一下北京今天的天气，然后据此推荐穿搭"

Agent 执行过程：

Thought 1: 用户想知道北京天气和穿搭推荐，我需要先查天气
Action 1:  call weather_api(city="北京")
Observation 1: {"temp": 12°C, "weather": "多云", "wind": "北风3级"}

Thought 2: 北京今天12度多云有风，偏凉。我来推荐穿搭
Action 2:  generate_response(基于天气数据推荐穿搭)
Observation 2: (最终回答)

Final Answer: 北京今天12°C，多云有北风。建议穿薄款外套+长裤...
```

### 2.2 工具调用（Tool Use / Function Calling）⭐⭐⭐

```
工具是 Agent 与外部世界交互的桥梁：

┌──────────────┬────────────────────────────────┐
│ 工具类型      │ 示例                           │
├──────────────┼────────────────────────────────┤
│ 搜索引擎      │ Google Search / Bing API       │
│ 代码执行器    │ Python Interpreter / Sandbox   │
│ API 调用      │ 天气API / 数据库查询 / 发邮件  │
│ 文件操作      │ 读写文件 / PDF解析             │
│ 知识库检索    │ 向量数据库 RAG 检索            │
│ 浏览器        │ 网页浏览 / 信息采集            │
│ 图像生成/分析 │ DALL-E / GPT-4V               │
└──────────────┴────────────────────────────────┘
```

**Function Calling 实现原理（OpenAI 为例）：**

```json
// 1. 定义工具（告诉 LLM 有哪些工具可用）
{
  "tools": [{
    "type": "function",
    "function": {
      "name": "search_knowledge_base",
      "description": "搜索内部知识库获取相关文档",
      "parameters": {
        "type": "object",
        "properties": {
          "query": {
            "type": "string",
            "description": "搜索关键词"
          },
          "top_k": {
            "type": "integer",
            "description": "返回结果数量",
            "default": 5
          }
        },
        "required": ["query"]
      }
    }
  }]
}

// 2. LLM 决定调用工具（返回 tool_calls）
{
  "role": "assistant",
  "tool_calls": [{
    "id": "call_abc123",
    "function": {
      "name": "search_knowledge_base",
      "arguments": "{\"query\": \"连接池参数调优\", \"top_k\": 3}"
    }
  }]
}

// 3. 应用层执行工具，返回结果
// 4. LLM 基于工具结果生成最终回答
```

│              │ - 用户画像/偏好记录                      │
├──────────────┼─────────────────────────────────────────┤
│ 情景记忆      │ 记住具体的过往经历和执行结果              │

├──────────────┼─────────────────────────────────────────┤

记忆管理关键问题：
- 何时存储：重要信息/用户偏好/执行结果
- 何时检索：与当前任务相关的历史经验
- 何时遗忘：过期/低价值/矛盾信息
- 摘要压缩：长对话超出窗口时生成摘要保留核心信息
```

### 3.2 LangGraph 工作流

```
LangGraph 用有向图定义 Agent 工作流：

from langgraph.graph import StateGraph, END

# 定义状态
class AgentState(TypedDict):
    messages: list[BaseMessage]
    next_action: str

# 定义节点（每个节点是一个处理函数）
def agent_node(state):
    """LLM 推理决策"""
    response = llm.invoke(state["messages"])
    return {"messages": [response], "next_action": parse_action(response)}

def tool_node(state):
    """执行工具"""
    result = execute_tool(state["next_action"])
    return {"messages": [ToolMessage(content=result)]}

# 构建图
graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)

# 定义边（流转逻辑）
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue,
    {"continue": "tools", "end": END})
graph.add_edge("tools", "agent")  # 工具执行后回到 agent

app = graph.compile()
```

```
图示：

        ┌─────────┐
   ──→  │  Agent  │ ──→ (判断) ──→ END
        │ (LLM)   │        │
        └────┬────┘        │ 需要工具
             ↑              ↓
             │       ┌───────────┐
             └────── │   Tools   │
                     │ (执行工具) │
                     └───────────┘
```

### 3.3 MCP（Model Context Protocol）⭐⭐

```
MCP 是 Anthropic 提出的开放协议，标准化 Agent 与外部工具/数据源的连接：

┌─────────────────────────────────────────────────────┐
│                   MCP 架构                           │
│                                                     │
│  AI 应用（Host）                                     │
│    ├── MCP Client ←──JSON-RPC──→ MCP Server A       │
│    │                              (文件系统工具)     │
│    ├── MCP Client ←──JSON-RPC──→ MCP Server B       │
│    │                              (数据库查询)       │
│    └── MCP Client ←──JSON-RPC──→ MCP Server C       │
│                                   (GitHub API)      │
└─────────────────────────────────────────────────────┘

MCP Server 提供三种能力：
- Tools（工具）：可执行的函数/API
- Resources（资源）：只读的数据/文件（类似 GET）
- Prompts（提示模板）：预定义的 Prompt 模板

MCP 的价值：
- 统一接口标准：一次开发，所有支持 MCP 的 Agent 都能用
- 类比 USB 接口：不同设备（工具）只需一种标准接口
- 生态复用：社区共享大量 MCP Server
```

## 五、Agent 开发实战要点 ⭐⭐⭐

### 5.1 工具设计原则

```
好的工具设计是 Agent 效果的关键：

✅ 正确做法：
- 函数名清晰描述功能：search_orders_by_user_id
- description 详细说明何时使用、输入输出含义
- 参数有类型约束和默认值
- 返回结构化数据（JSON）
- 做好错误处理，返回友好错误信息

❌ 常见错误：
- 工具太多（>20个）：LLM 选择困难，准确率下降
- 描述模糊：LLM 不知道何时该用
- 无错误处理：工具报错后 Agent 陷入死循环
- 权限过大：Agent 可以执行危险操作（删数据库）
```

### 5.2 安全与防护

```
Agent 安全是生产部署的关键问题：

┌──────────────┬──────────────────────────────────────┐
│ 风险          │ 防护措施                              │
├──────────────┼──────────────────────────────────────┤
│ Prompt 注入   │ 输入过滤 + 分离系统/用户 Prompt       │
│              │ 工具参数验证，禁止拼接为命令/SQL       │
├──────────────┼──────────────────────────────────────┤

## 六、面试高频问题

### Q1: Agent 和普通 RAG 有什么区别？
> RAG 是"检索→生成"的固定两步流程，Agent 是自主决策的智能体。Agent 更强大：① 可以**动态规划**多步任务，RAG 是固定流程；② Agent 可以使用**多种工具**（搜索、代码、API），RAG 只做检索；③ Agent 可以**反思和重试**（执行结果不好可以换策略），RAG 是一次性的；④ Agent 有**记忆能力**，可以跨会话保留信息。实际上，RAG 可以作为 Agent 的一个工具——Agent 判断需要查知识库时调用 RAG。

### Q2: 如何设计一个好的 Agent System Prompt？
> 好的 Agent System Prompt 包含：① **角色定义**：你是什么身份，具备什么能力；② **行为规范**：什么情况用什么工具，什么情况直接回答；③ **工具说明**：何时使用每个工具（最重要），输入输出格式；④ **限制约束**：不能做什么，遇到不确定怎么办（如"不确定时请询问用户"）；⑤ **输出格式**：最终回答的格式要求。关键是让 LLM 清楚知道**边界**——什么该做、什么不该做、什么需要人工确认。

### Q3: Agent 在生产环境中面临的主要挑战？
> ① **可靠性**：LLM 输出不确定，工具调用可能参数错误或失败，需要完善的错误处理和重试机制；② **延迟**：多步推理+工具调用，端到端延迟可能几十秒，需要流式输出和并行优化；③ **成本**：每步都要 LLM 推理，Token 消耗大，需要缓存和路由（简单任务不走 Agent）；④ **安全**：Prompt 注入风险、工具越权风险，需要沙箱+权限控制+审计；⑤ **评测**：Agent 行为难以用传统指标评估，需要端到端的任务完成率评测。

### Q4: 什么是 MCP？它解决了什么问题？
> MCP（Model Context Protocol）是 Anthropic 提出的开放协议，标准化 AI 应用与外部工具/数据源的连接方式。类比 USB 接口——以前每个 Agent 框架对接工具都要写不同的适配代码，MCP 统一了协议标准。MCP Server 提供 Tools（工具）、Resources（资源）、Prompts（模板）三种能力，MCP Client 通过 JSON-RPC 协议与 Server 通信。好处是：工具开发者只需实现一次 MCP Server，所有支持 MCP 的 AI 应用都能使用，大大降低工具集成成本。
