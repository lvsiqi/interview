## 一、JWT 认证机制 ⭐⭐⭐⭐

### 1.1 JWT 结构

```
JWT = Base64(Header) . Base64(Payload) . Signature

Header:  { "alg": "HS256", "typ": "JWT" }
Payload: { "sub": "userId", "exp": 1700000000, "role": "admin" }
Signature: HMACSHA256( base64(Header) + "." + base64(Payload), secret )
```

| 部分 | 内容 | 是否加密 |
|------|------|----------|
| Header | 算法类型 | 仅 Base64 编码，可解码 |
| Payload | 用户信息 + 过期时间 | 仅 Base64 编码，**不要放敏感信息** |
| Signature | HMAC/RSA 签名 | 签名防篡改，不可伪造 |

### 1.2 JWT 认证流程

```
① 用户登录 → 服务端验证账密
② 签发 JWT（含 userId、role、exp 等）→ 返回给客户端
③ 客户端存储（localStorage 或 Cookie）
④ 后续请求携带 JWT：Authorization: Bearer <token>
⑤ 服务端验签 → 解析 Payload → 获取用户信息
```

### 1.3 JWT vs Session 对比

| 维度 | JWT | Session |
|------|-----|---------|
| 存储位置 | 客户端（token） | 服务端（内存/Redis） |
| 扩展性 | 天然支持分布式 | 需 Session 共享（Redis） |
| 安全注销 | 难（token未过期即有效） | 易（删服务端Session即可） |
| 性能 | 无需查存储，验签即可 | 需查 Redis/内存 |
| token大小 | 较大（含信息） | 小（只有 sessionId） |

### 1.4 JWT 安全问题

| 漏洞 | 描述 | 防御 |
|------|------|------|
| `alg:none` 攻击 | 攻击者将 alg 改为 none 绕过签名验证 | 服务端强制指定算法，不接受 none |
| Payload 信息泄露 | Payload 仅 Base64 不加密 | 不存敏感信息（密码/银行卡等） |
| Token 被盗用 | XSS 窃取 localStorage 中的 token | 使用 HttpOnly Cookie 存储 |
| 未校验过期时间 | 忽略 exp 字段验证 | 务必校验 exp，拒绝过期 token |

### 1.5 Refresh Token 机制

```
Access Token：短期（15min~1h），用于携带请求
Refresh Token：长期（7d~30d），存服务端白名单，用于续期

续期流程：
① Access Token 过期 → 客户端用 Refresh Token 请求 /auth/refresh
② 服务端验证 Refresh Token → 重签新的 Access Token + 新的 Refresh Token
③ 旧 Refresh Token 立即失效（防重放）
```

### 1.6 HS256 vs RS256

| 算法 | 说明 | 使用场景 |
|------|------|----------|
| HS256 | 对称 HMAC，签发和验签用同一密钥 | 单体/内部服务 |
| RS256 | 非对称 RSA，私钥签发 + 公钥验签 | 多服务/对外开放（资源服务器只需公钥） |

### 1.7 面试标准答法
> JWT 由 Header、Payload、Signature 三部分 Base64 编码后用点拼接而成。Signature 用密钥对前两部分做 HMAC/RSA 签名防篡改。
> 优点：无状态、天然支持分布式；缺点：签发后无法主动失效，需配合 Redis 黑名单或短期 Token + Refresh Token 机制解决。
> 不能在 Payload 中存敏感信息，务必校验 exp，防止 alg:none 攻击。

### 1.8 常见追问
| 追问 | 关键答点 |
|------|----------|
| JWT 如何实现主动注销？ | Redis 维护 token 黑名单/jti黑名单；或改 token 版本号存 DB，登出后版本号+1 |
| Access Token 多短合适？ | 高安全场景 15min，普通场景 1~2h；配合 Refresh Token 自动续期 |
| RS256 为什么更安全？ | 资源服务器只需持有公钥，私钥只在认证中心，减少密钥暴露面 |

## 二、OAuth2.0 协议 ⭐⭐⭐

### 2.1 四个角色

| 角色 | 说明 |
|------|------|
| Resource Owner（资源所有者） | 用户 |
| Client（客户端） | 第三方应用 |
| Authorization Server（授权服务器） | 颁发 Token（如 GitHub OAuth 服务） |
| Resource Server（资源服务器） | 存放受保护资源的 API 服务 |

### 2.2 四种授权模式

| 模式 | 适用场景 | 安全性 |
|------|----------|--------|
| **授权码模式**（推荐）| 有服务端的 Web 应用 | 最高，code 换 token 在后端完成 |
| 隐式模式（Implicit） | 纯前端单页应用（已废弃） | 低，token 直接返回 URL |
| 密码模式 | 高度信任的内部应用 | 中，需暴露用户账密给 Client |
| 客户端凭证 | 服务间调用（M2M） | 高，无用户参与 |

### 2.3 授权码模式流程（最重要）

```
① 用户点击"第三方登录"→ Client 重定向到 Authorization Server
   GET /authorize?response_type=code&client_id=xxx&redirect_uri=xxx&scope=xxx&state=xxx

② 用户登录并授权 → Authorization Server 重定向回 redirect_uri，附带 code
   GET /callback?code=AUTH_CODE&state=xxx

③ Client 后端用 code 换 Access Token（携带 client_id + client_secret）
   POST /token  body: grant_type=authorization_code&code=AUTH_CODE&client_id=xxx&client_secret=xxx

④ Authorization Server 返回 Access Token（+ Refresh Token）

⑤ Client 用 Access Token 访问 Resource Server 获取资源
```

> **state 参数**：随机值，防 CSRF 攻击；回调时对比 state 是否一致

### 2.4 OpenID Connect (OIDC)

- OAuth2.0 只做**授权**（你能访问什么），不做**认证**（你是谁）
- OIDC = OAuth2.0 + ID Token（JWT），在授权的同时返回用户身份信息
- 常用于：第三方登录（微信、GitHub、Google 登录集成）

### 2.5 面试标准答法
> OAuth2.0 是授权框架，解决"第三方应用获取用户资源"的问题。推荐授权码模式：用户在授权服务器登录授权后返回 code，再由服务端后台用 code 换 token，避免 token 暴露在前端。
> OIDC 在 OAuth2 基础上增加 ID Token，实现用户身份认证。

### 2.6 常见追问
| 追问 | 关键答点 |
|------|----------|
| 为什么不直接返回 token 而是先返回 code？ | code 通过前端 URL 传输，若用 HTTPS 可被日志记录；code 只能用一次且短期有效，后端换 token 更安全 |
| state 参数的作用？ | 防 CSRF：攻击者构造授权链接无法知道随机 state，回调时校验不通过 |
| Token 如何吊销？ | 授权码模式支持 refresh_token revoke；资源服务器定期验证 token introspect 接口 |

## 三、Spring Security 原理 ⭐⭐⭐

### 3.1 核心架构：过滤器链

```
HTTP Request
    ↓
DelegatingFilterProxy（Servlet 容器 → Spring 容器桥梁）
    ↓
FilterChainProxy（SecurityFilterChain 管理器）
    ↓
SecurityFilterChain（过滤器链）
    ├── SecurityContextPersistenceFilter  → 从 Session 恢复 SecurityContext
    ├── UsernamePasswordAuthenticationFilter → 处理表单登录
    ├── BasicAuthenticationFilter          → HTTP Basic 认证
    ├── BearerTokenAuthenticationFilter    → JWT/OAuth Token 认证
    ├── ExceptionTranslationFilter         → 异常转 401/403
    └── FilterSecurityInterceptor          → 授权决策（最终拦截器）
```

### 3.2 认证流程

```
表单登录：
UsernamePasswordAuthenticationFilter
    → 构造 UsernamePasswordAuthenticationToken（未认证）
    → AuthenticationManager#authenticate()
    → 委托给 AuthenticationProvider（默认 DaoAuthenticationProvider）
    → 调用 UserDetailsService#loadUserByUsername()
    → 比对密码（PasswordEncoder.matches()）
    → 认证成功 → 存入 SecurityContextHolder

SecurityContextHolder：
    → 默认 ThreadLocal 模式存储，当前线程可随时获取认证信息
    → Authentication = { principal(UserDetails), credentials, authorities }
```

### 3.3 授权流程

```
FilterSecurityInterceptor
    → 获取当前请求需要的权限（ConfigAttribute）
    → 获取当前用户的 Authentication
    → AccessDecisionManager#decide()
        → AffirmativeBased（一票通过）
        → ConsensusBased（多数通过）
        → UnanimousBased（全票通过）
    → 投票器（AccessDecisionVoter）
        → RoleVoter：ROLE_ADMIN 等角色匹配
        → WebExpressionVoter：解析 hasRole()、hasPermission() 等 SpEL
```

### 3.4 JWT 集成方案

```java
// 自定义 JWT 过滤器，加在 UsernamePasswordAuthenticationFilter 之前
public class JwtAuthFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(...) {
        String token = extractToken(request);
        if (token != null && jwtUtil.validate(token)) {
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    jwtUtil.getUser(token), null, jwtUtil.getAuthorities(token));
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        filterChain.doFilter(request, response);
    }
}
```

### 3.5 @PreAuthorize & 方法级安全

```java
@EnableMethodSecurity  // 启用方法级安全
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long id) { ... }

@PreAuthorize("#userId == authentication.principal.id")  // SpEL 表达式
public UserInfo getUser(Long userId) { ... }

@PostAuthorize("returnObject.ownerId == authentication.principal.id")
public Order getOrder(Long orderId) { ... }
```

### 3.6 面试标准答法
> Spring Security 基于 Servlet Filter 链实现，核心是 SecurityFilterChain。认证流程：Filter → AuthenticationManager → AuthenticationProvider → UserDetailsService 加载用户 → 密码比对 → 存入 SecurityContextHolder（ThreadLocal）。
> 授权由 FilterSecurityInterceptor + AccessDecisionManager 决策，支持方法级 @PreAuthorize 注解。
> 与 JWT 集成：自定义 Filter 解析 Token 写入 SecurityContextHolder，完全无状态。

### 3.7 常见追问
| 追问 | 关键答点 |
|------|----------|
| SecurityContextHolder 线程安全吗？ | 默认 ThreadLocal 模式，每个请求线程独立；子线程需手动传递或改用 InheritableThreadLocal 模式 |
| 密码加密用什么？ | BCryptPasswordEncoder，自带 salt，不同密码每次加密结果不同 |
| 如何实现动态权限？ | 自定义 FilterInvocationSecurityMetadataSource 从 DB 读取 URL 权限配置 |
| RBAC 如何设计？ | 用户→角色→权限三表，权限粒度到接口/按钮；Spring Security authorities 中存 PERMISSION 字符串 |

## 四、SQL 注入 ⭐⭐⭐⭐⭐

### 4.1 原理

```sql
-- 正常 SQL
SELECT * FROM users WHERE username = 'alice' AND password = '123456'

-- 注入 payload：username = "' OR '1'='1"
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = ''
-- 结果：永远为 true，绕过登录验证
```

### 4.2 危害等级

| 类型 | 说明 |
|------|------|
| 数据泄露 | `UNION SELECT` 拼接查询其他表数据 |
| 数据破坏 | `'; DROP TABLE users;--` |
| 权限绕过 | 绕过登录认证 |
| RCE | MySQL `INTO OUTFILE` 写 WebShell / `xp_cmdshell`（SQL Server） |

### 4.3 防御方案

**① 预编译（PreparedStatement）—— 根本防御**
```java
// ❌ 字符串拼接，有注入风险
String sql = "SELECT * FROM users WHERE name = '" + name + "'";

// ✅ 预编译，参数绑定，SQL 结构固定
PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE name = ?");
ps.setString(1, name);
```

**② MyBatis `#{}` vs `${}`**
```xml
<!-- ✅ #{} 底层使用 PreparedStatement 参数绑定 -->
<select id="findUser">SELECT * FROM users WHERE name = #{name}</select>

<!-- ❌ ${} 直接字符串替换，有注入风险，仅用于动态表名/列名等结构部分 -->
<select id="findUser">SELECT * FROM users WHERE name = '${name}'</select>
```

**③ 输入校验**：白名单校验、限制特殊字符（`'`、`"`、`;`、`--`）

**④ 最小权限原则**：DB 账号只授予必要权限（SELECT/INSERT），禁止 DROP/CREATE

**⑤ WAF**：Web 应用防火墙过滤常见注入 Payload

### 4.4 面试标准答法
> SQL 注入是用户输入被拼接进 SQL 语句改变执行逻辑。根本防御是使用 PreparedStatement 预编译，SQL 结构在编译时固定，参数只作为数据不会被解析为 SQL 指令。MyBatis 中用 `#{}` 而非 `${}` 同理。

### 4.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| MyBatis 什么时候必须用 `${}`？ | 动态表名、列名、ORDER BY 字段（结构部分），此时需严格白名单校验 |
| 预编译能防所有注入吗？ | 能防参数注入；ORDER BY / 动态表名不能用预编译，需白名单校验 |
| 如何检测 SQL 注入漏洞？ | 工具：sqlmap；代码审计；输入单引号看报错 |

## 五、XSS 跨站脚本攻击 ⭐⭐⭐⭐

### 5.1 三种类型

| 类型 | 原理 | 示例 |
|------|------|------|
| **存储型**（持久化） | 恶意脚本存入 DB，所有访问者都受影响 | 评论框提交 `<script>steal(document.cookie)</script>` |
| **反射型**（非持久） | 脚本在 URL 参数中，诱导用户点击 | `?msg=<script>...</script>` |
| **DOM 型** | 前端 JS 直接操作 DOM 写入不可信内容 | `document.write(location.search)` |

### 5.2 危害

- 窃取 Cookie/Session → 账号劫持
- 键盘记录 → 账密泄露
- 页面篡改 → 钓鱼攻击
- 内网探测（配合 CSRF）

### 5.3 防御方案

**① 输出转义（最重要）**
```java
// Hutool / OWASP ESAPI 转义
String safe = HtmlUtil.escape(userInput);
// < → &lt;   > → &gt;   " → &quot;   ' → &#x27;   / → &#x2F;
```

**② CSP（Content Security Policy）**
```
// HTTP响应头，限制脚本只能从同域加载，禁止 inline script
Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-xxx'
```

**③ HttpOnly Cookie**
```
Set-Cookie: sessionId=xxx; HttpOnly; Secure; SameSite=Strict
// HttpOnly 禁止 JS 读取 Cookie
```

**④ 框架级防护**
- Vue/React 默认转义 `{{ }}` 插值，避免使用 `v-html`/`dangerouslySetInnerHTML`
- Thymeleaf `th:text` 自动转义，`th:utext` 不转义

### 5.4 面试标准答法
> XSS 是将恶意脚本注入网页并在其他用户浏览器中执行。存储型最危险（脚本持久化）。
> 防御：输出时转义 HTML 特殊字符；设置 HttpOnly Cookie 防窃取；配置 CSP 限制脚本来源；前端框架避免使用不安全的 DOM 操作 API。

### 5.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| 为什么存储型 XSS 危害最大？ | 脚本存入 DB，每次访问都触发，影响所有用户 |
| CSP nonce 是什么？ | 每次请求生成随机 nonce，内联脚本须携带，攻击者无法预知 nonce |
| 富文本编辑器如何防 XSS？ | 服务端用白名单过滤 HTML 标签（只保留 b/i/p 等），推荐 OWASP AntiSamy/jsoup 白名单清洗 |

## 六、CSRF 跨站请求伪造 ⭐⭐⭐

### 6.1 原理

```
① 用户已登录银行网站 bank.com，浏览器保留了 Cookie
② 用户访问恶意页面 evil.com
③ evil.com 页面中包含：
   <img src="https://bank.com/transfer?to=hacker&amount=10000">
④ 浏览器自动携带 bank.com 的 Cookie 发起请求
⑤ 银行服务器认为是合法请求，完成转账
```

### 6.2 防御方案

| 方案 | 原理 | 说明 |
|------|------|------|
| **CSRF Token** | 服务端生成随机 Token 嵌入表单，提交时验证 | Spring Security 默认启用；Ajax 请求需手动带 Header |
| **SameSite Cookie** | `SameSite=Strict/Lax`，跨站请求不携带 Cookie | 现代浏览器支持，最简单有效 |
| **Referer/Origin 校验** | 验证请求来源域名 | 弱防御，Referer 可被伪造或丢失 |
| **Double Submit Cookie** | Cookie 中存 Token，请求参数也带 Token，验证两者一致 | 无状态方案 |

### 6.3 CSRF vs XSS

| | CSRF | XSS |
|---|------|-----|
| 本质 | 借用用户身份（Cookie）发请求 | 在用户浏览器执行恶意脚本 |
| 是否执行脚本 | 否 | 是 |
| 防御重点 | Token/SameSite Cookie | 输出转义/CSP |

### 6.4 面试标准答法
> CSRF 利用用户已登录状态，诱导用户访问恶意页面，由浏览器自动携带 Cookie 发起伪造请求。
> 根本原因：Cookie 自动携带且服务端只验证 Cookie。防御：① SameSite=Strict Cookie（现代浏览器推荐）；② CSRF Token（服务端生成随机 Token 绑定 Session，提交时验证）。

### 6.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| SameSite=Lax vs Strict 区别？ | Strict：任何跨站请求不带 Cookie；Lax：顶层导航（点链接）允许，XHR/Form/img 不允许 |
| JWT 能防 CSRF 吗？ | 能。JWT 存 localStorage 不会被浏览器自动携带，无 Cookie 风险；但 XSS 更危险，需 HttpOnly Cookie 存放 |
| GET 请求能被 CSRF 攻击吗？ | 能（img/a 标签），所以修改型操作必须用 POST，且加 CSRF Token |

## 七、SSRF 服务端请求伪造 ⭐⭐⭐

### 7.1 原理

```
服务端接收用户输入的 URL，直接发起请求（如图片抓取、Webhook）
攻击者输入内网地址：http://192.168.1.1/admin 或 http://169.254.169.254/latest/meta-data/
服务端发起请求并将响应返回 → 泄露内网信息/云服务元数据
```

### 7.2 危害

- 探测内网拓扑（端口扫描）
- 访问云实例元数据（AWS/阿里云 `169.254.169.254` 可获取临时密钥）
- 攻击内网服务（Redis、Elasticsearch 无认证服务）
- 利用协议（file://、dict://、gopher://）读取本地文件

### 7.3 防御方案

```java
// ① URL 白名单校验
if (!allowedDomains.contains(new URL(userUrl).getHost())) {
    throw new SecurityException("非法 URL");
}

// ② 禁止私有 IP（防内网穿透）
InetAddress address = InetAddress.getByName(host);
if (address.isSiteLocalAddress() || address.isLoopbackAddress()
        || address.isLinkLocalAddress()) {
    throw new SecurityException("禁止访问内网地址");
}

// ③ 限制协议：只允许 http/https，禁止 file/dict/gopher
// ④ 响应内容不直接返回给用户
// ⑤ 网络层限制：服务器出口 ACL 禁止访问内网段
```

### 7.4 面试标准答法
> SSRF 是攻击者通过控制服务端发起的请求，访问原本不可达的内网资源。防御核心：URL 白名单 + 禁止访问私有 IP + 限制协议。云环境需特别防范元数据地址 `169.254.169.254`。

### 7.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| URL 白名单中 hostname 解析时如何防 DNS 重绑定？ | 解析 IP 后再次校验不是私有地址；或使用固定 IP 白名单而非域名 |
| 代码审计中哪些操作要重点关注？ | `HttpClient`/`RestTemplate`/`OkHttp` 发起请求时，入参是否来自用户输入 |

## 八、接口安全设计 ⭐⭐⭐

### 8.1 防重放攻击

```
方案：timestamp + nonce + signature

① 客户端生成随机 nonce，记录当前时间戳 timestamp
② 将请求参数 + timestamp + nonce + appSecret 按字典序拼接后 MD5/HMAC-SHA256
③ 将 timestamp、nonce、sign 放入请求头

服务端验证：
① 验证 timestamp 与服务端时间差 ≤ 5 分钟（防重放）
② Redis 查询 nonce 是否已使用（防重放）→ 存入 Redis，TTL = 5 分钟
③ 重新计算 sign 比对（防篡改）
```

### 8.2 敏感数据处理

| 场景 | 方案 |
|------|------|
| 传输加密 | HTTPS（TLS 1.2+）全程加密 |
| 密码存储 | BCrypt/Argon2 不可逆哈希（禁止 MD5/SHA1 明文存储） |
| 手机号/身份证 | 脱敏展示（138****1234）；存储 AES-256 加密 |
| 日志 | 自动过滤/脱敏 password、token、idCard 等敏感字段 |
| API 密钥 | 环境变量/配置中心，禁止硬编码在代码中 |

### 8.3 访问控制

```
① 接口鉴权：每个接口明确 @PreAuthorize 所需角色/权限
② 水平越权：业务层校验数据归属（userId 与当前登录用户一致）
   // ❌ 只凭 orderId 查询，未校验归属
   Order order = orderMapper.findById(orderId);
   // ✅ 加入 userId 过滤
   Order order = orderMapper.findByIdAndUserId(orderId, currentUserId);
③ 垂直越权：普通用户不能调用管理员接口（RBAC 权限体系）
```

### 8.4 面试标准答法
> 接口安全核心三点：① 认证（JWT/OAuth2 确认身份）；② 授权（RBAC 控制访问资源）；③ 防篡改/防重放（timestamp + nonce + HMAC 签名）。
> 敏感数据传输必须 HTTPS；存储密码用 BCrypt；防越权须在数据层做 userId 过滤。

### 8.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| 如何防止接口被暴力破解？ | 登录接口限流（IP+账号双维度）+ 验证码 + 账号锁定策略 |
| HTTPS 能防中间人攻击吗？ | 能（证书验证）；但客户端需严格校验证书（禁止 SSLContext.getInstance("SSL") 跳过验证） |
| 什么是 IDOR？ | 不安全的直接对象引用（水平越权的一种），如 URL 中直接暴露 orderId 且未鉴权 |

## 十、HTTPS & TLS 原理 ⭐⭐⭐

### 10.1 TLS 1.2 握手流程

```
Client                                    Server
  |                                          |
  |---------- ClientHello ---------------→  |  (支持的TLS版本、加密套件、随机数C)
  |                                          |
  |←--------- ServerHello ---------------   |  (选定加密套件、随机数S、证书)
  |←--------- Certificate ---------------   |
  |←--------- ServerHelloDone -----------   |
  |                                          |
  |  [验证证书链 → 取出服务端公钥]             |
  |                                          |
  |---------- ClientKeyExchange ----------→ |  (用公钥加密预主密钥 Pre-Master Secret)
  |---------- ChangeCipherSpec -----------→ |
  |---------- Finished -------------------→ |  (用三个随机数生成 Master Secret 加密验证)
  |                                          |
  |←--------- ChangeCipherSpec -----------  |
  |←--------- Finished -------------------  |
  |                                          |
  |======= 后续使用对称密钥加密通信 ==========|

对称密钥 = PRF(Pre-Master Secret, C随机数, S随机数)
```

### 10.2 TLS 1.3 改进

| 对比 | TLS 1.2 | TLS 1.3 |
|------|---------|---------|
| 握手次数 | 2-RTT | **1-RTT**（首次），0-RTT（会话复用） |
| 密钥交换 | RSA（私钥可解密历史流量） | **仅 ECDHE**（前向保密） |
| 废弃算法 | 支持 RC4/MD5 等弱算法 | 只保留安全算法套件 |
| 加密时机 | Finished 才开始加密 | 更早开始加密 |

> **前向保密（Forward Secrecy）**：每次会话临时生成密钥，即使长期私钥泄露，历史流量仍安全

### 10.3 证书链验证

```
根 CA 证书（内置操作系统/浏览器）
    ↓ 签名
中间 CA 证书
    ↓ 签名
服务端证书（包含域名、公钥、有效期）

验证：
① 用中间CA证书的公钥验证服务端证书签名
② 用根CA证书的公钥验证中间CA证书签名
③ 验证域名匹配（CN / SAN 字段）
④ 验证有效期
⑤ 检查 CRL/OCSP 是否吊销
```

### 10.4 面试标准答法
> HTTPS = HTTP + TLS，TLS 握手阶段用非对称加密（RSA/ECDHE）协商会话密钥，数据传输阶段用对称加密（AES-GCM）保证性能。
> TLS 1.3 将握手从 2-RTT 优化到 1-RTT，强制 ECDHE 保证前向保密，废弃了不安全算法。

### 10.5 常见追问
| 追问 | 关键答点 |
|------|----------|
| 对称加密和非对称加密为什么结合使用？ | 非对称加密慢但用来安全交换密钥；对称加密快用来加密数据 |
| HTTPS 一定安全吗？ | 不一定；中间人攻击（安装根证书）/ 证书签发机构被攻破 / 代码层 HTTPS 配置不当都可能不安全 |
| HTTP/2 和 TLS 的关系？ | HTTP/2 强制 TLS；HPACK 头部压缩；多路复用解决 HTTP/1.1 队头阻塞 |
